## Goal for this iteration

PLAN priority #1 iter B-gold — the bootstrap-cascade gold case is
the sole remaining piece of priority #1. Iter 372 / 374 landed the
runtime: cumulative chunk assembly + `targetPage:
maxViewingPage(p)` + `.pdf-page-placeholder` mount when
`lastPage===false` + IntersectionObserver demand-fetch chain. What
the existing first case in `verifyLivePdfMultiPage.spec.ts`
asserts (`pageWrapperCount ≥ 2`) is satisfied by *placeholder
alone* — i.e. the test passes without ever validating that the
demand-fetch chain successfully replaces the placeholder with a
real second page. Closing this gap closes M21.2 (per `369b_answer`)
and finishes priority #1.

## Approach

Add a third test in `verifyLivePdfMultiPage.spec.ts` exercising
the full bootstrap-cascade:

1. Fresh project, navigate, wait `.cm-content`.
2. Wait initial hello-world pdf-segment + canvas paint.
3. Ctrl+A → `keyboard.type(STATIC_TWO_PAGE)` (same atomic shape as
   case #1; reuses the proven mechanism, no fresh shape).
4. Wait for `.pdf-page-placeholder` element to appear in the DOM
   (signals: post-replace compile shipped page 1 with
   `lastPage=false` AND PdfViewer rendered page 1 + mounted the
   placeholder).
5. `placeholderEl.scrollIntoView()` → IntersectionObserver fires →
   `maxViewingPage` bumps to 2 → `view` frame → sidecar
   `kickForView(2)` → recompile with `targetPage=2`.
6. Await pdf-segment frame with `lastPage=true` and
   `shipoutPage ≥ 2`.
7. Assert placeholder gone and `.pdf-page:not(.pdf-page-placeholder)`
   count is ≥ 2 (two *real* rendered pages).

The wire-frame match for (6) requires the spec to parse the new
18-byte header. I'll extend `wireFrames.ts` to expose `lastPage`
(tri-state, mirroring `decodeFrame` on byte 17) alongside the
existing `shipoutPage` extraction, so future specs can match on
this signal without re-parsing the buffer. Pure additive change;
existing entries keep their shape.

Polling deadlines are bounded with explicit messages classifying
the failure mode (placeholder never mounted, scroll didn't fire
IO, lastPage=true never arrived, placeholder never removed).
Outer test timeout 240 s (initial cold compile up to 90 s, type +
re-compile 30-60 s, scroll → kickForView compile up to 60 s,
generous tail).

## Notes for future iterations

- **Spec already covers M21.2 — no separate 3-page spec needed.**
  The bootstrap-cascade case exercises max-visible → sidecar
  target on each placeholder→real transition; the same
  `kickForView(2)` path that would fire on a 3-page document's
  cascade fires here, so the additional shape is not required
  for coverage.
- **If the new case red-flakes intermittently**, the failure
  message will name which hop timed out. Likely candidates:
  (a) placeholder mount takes > 90 s on extreme cold-start →
  bump the inner `waitFor` budget (not the outer testInfo
  timeout), (b) `kickForView` raced the post-typing pending
  compile and was skipped → check `compileStatusEvents` for
  `running` arriving after the scroll, and (c) PdfViewer's
  `render` path silently rejects → `[PdfViewer] render
  failed:` console line caught by `authedPage` fixture.
- **`lastPage` byte parsing is duplicated** between
  `packages/protocol/src/index.ts` `decodeFrame` and this
  spec's `parseLastPage`. Acceptable — the protocol package
  isn't in playwright's transform path (same reason as the
  `TAG_PDF_SEGMENT` constant duplication in `wireFrames.ts`).
  If a fourth spec needs `lastPage`, lift to a fixture helper
  alongside `shipoutPage` extraction in `wireFrames.ts`.
- **Open after priority #1 closes:** plan-review (`N%10==1`,
  i.e. iter 381) becomes a meaningful next milestone for
  removing the priority-#1 narrative scaffolding from PLAN now
  that the work is done. The narrative is still useful for
  the next 2-3 iterations as context for the cascade gold
  case's first runs.
