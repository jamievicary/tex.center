## Approach + actions

1. **Root cause re-check.** INCIDENT-147 pointed at
   `apps/sidecar/src/index.ts:19` (`HOST` default
   `"127.0.0.1"`), but the Dockerfile sets `HOST=0.0.0.0`,
   which is what was actually in effect on the live Machines
   (the "listening at http://172.x.x.x" startup logs in
   INCIDENT-147 are Fastify enumerating IPv4 interfaces under
   a `0.0.0.0` bind, not loopback). `0.0.0.0` is IPv4 wildcard
   only; Fly 6PN is IPv6. Both sites needed the fix — either
   alone left the regression latent.

2. **Sidecar bind host.** `apps/sidecar/src/index.ts`: extracted
   `resolveBindHost(env)` + `DEFAULT_BIND_HOST = "::"`,
   replaced the inline default. `apps/sidecar/Dockerfile`:
   `HOST=0.0.0.0` → `HOST=::`. Node binds dual-stack on Linux
   for `"::"`, so IPv4 health probes still work.

3. **Regression tests.**
   - `apps/sidecar/test/bindHost.test.mjs` — asserts
     `DEFAULT_BIND_HOST === "::"`, `resolveBindHost({}) === "::"`,
     empty-string treated as unset, overrides honoured, and
     reads the Dockerfile to ban a regression to `0.0.0.0`.
   - `tests_normal/cases/test_sidecar_dockerfile.py::test_runtime_listens_on_all_interfaces`
     flipped from `HOST=0\.0\.0\.0` to `HOST=::`.
   - `tests_normal/cases/test_node_suites.py::test_sidecar_bind_host`
     runs the new mjs test under tsx.

4. **wsProxy 502 fix.** `apps/web/src/lib/server/wsProxy.ts`
   `dialAndPipe`: new `upstreamConnected` flag set in the
   `connect` handler. `cleanup` now writes
   `HTTP/1.1 502 Bad Gateway` to the client socket before
   destroying it, but only when the reason is
   `upstream-error` and the upstream never finished its
   handshake. Post-connect errors mid-pipe still destroy
   silently (the WS frame stream is already in flight).
   `apps/web/test/wsProxy.test.mjs` Case 4 strengthened to
   read the response, assert `HTTP/1.1 502`, wait for `close`
   (or already-destroyed), and assert an `upstream-error`
   event.
