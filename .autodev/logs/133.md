## Notes for future iterations

- M8.pw.3.1 (next): add a SvelteKit route that wraps
  `finalizeGoogleSession`. Design call: gate on
  `TEST_OAUTH_BYPASS_KEY` env (route returns 404 when unset); body
  is `{idToken}`; header `X-Test-Bypass: HMAC(body, key)`. Server
  still runs the real `verifyGoogleIdToken` (so JWKS regressions
  are caught) — the bypass only skips token-exchange, not the
  ID-token cryptographic check. Unit-test by extracting a pure
  `resolveTestCallback({ verifyIdToken, finalize, hmacVerify, ... })`
  orchestrator next to `resolveGoogleCallback`.
- Security envelope: a successful attack needs both the bypass key
  (a Fly secret) and a valid Google ID token for an allowlisted
  email — same hurdle as obtaining the real OAuth client_secret +
  a real user login. Acceptable for the single-user MVP per the
  M8.pw.3.0 design call.
- `priorSetCookies` is currently only used by the real callback to
  clear `tc_oauth_state`. The test route won't have a state cookie
  to clear, so it'll pass `[]` (the default). Keeping the
  parameter optional avoids forcing every caller to think about it.
