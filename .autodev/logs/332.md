## Goal for this iteration

PLAN priority #1 follow-up M20.3(a)2: gate the cold-boot
`loadCheckpoint` GET (and the idle-stop `persistCheckpoint` PUT)
on a new `Compiler.supportsCheckpoint` flag. All three production
compilers (`Fixture`, `SupertexOnce`, `SupertexDaemon`) return
`null` from `snapshot()` today — upstream supertex doesn't expose
a serialise wire yet (M7.4.2) — so on every cold boot the sidecar
makes a Tigris GET that is guaranteed to return null, costing
~0.27 s wallclock per project boot (`restoreMs:273` in the
iter-330 capture). On idle-stop the symmetric `snapshot()`-then-
`persistCheckpoint` is skipped today only because `snapshot()`
returns null and `persistCheckpoint` short-circuits on null, but
the snapshot call itself still runs.

`supportsCheckpoint = false` on all three current impls; flip to
`true` is what M7.4.2 (or the per-compiler equivalent) will do
when serialise lands. Keeps the wiring tested via
`serverCheckpointWiring`, just gates the network round-trips.

## Approach

1. `Compiler.supportsCheckpoint: readonly boolean` on the
   interface in `compiler/types.ts` with the rationale above.
2. `FixtureCompiler` / `SupertexOnceCompiler` /
   `SupertexDaemonCompiler` set `supportsCheckpoint = false`.
3. `server.ts ensureRestored`: short-circuit before calling
   `loadCheckpoint` when the project's compiler doesn't support
   checkpoints. `restoreMs` stays 0 (same as today's empty-blob
   path).
4. `server.ts persistAllCheckpoints`: skip per-project body
   (snapshot + persistCheckpoint) when `supportsCheckpoint = false`.
5. Update test stubs to set the flag — `false` everywhere except
   `serverCheckpointWiring.test.mjs`'s `RecordingCompiler` which
   exercises the live path (`true`). Add a 4th case to that test
   pinning that a `supportsCheckpoint:false` compiler with a
   pre-seeded checkpoint key skips both restore and snapshot
   (and the blob is never read on cold boot).
6. Gold `sidecarColdStartCoalescer` wrapper delegates the flag to
   its inner compiler.

  `readonly supportsCheckpoint = false` on each.

  `Promise.resolve()` before calling `loadCheckpoint` when the
  per-project compiler reports `!supportsCheckpoint`. `restoreMs`
  remains 0 (same as today's empty-blob path), so the cold-start
  log line shape is unchanged; only the network round-trip is
  removed.
- `apps/sidecar/src/server.ts persistAllCheckpoints()` —
  `continue` before the snapshot+PUT for any project whose
  compiler reports `!supportsCheckpoint`. Symmetric with the
  cold-boot skip.
- `apps/sidecar/test/serverCheckpointWiring.test.mjs` —
  `RecordingCompiler` opt-in: constructor takes
  `{ supportsCheckpoint = true }` so cases 1–3 keep exercising
  the live wiring with no other change. New

  `get`. New case 4 boots a server with a `RecordingCompiler`
  carrying `supportsCheckpoint:false`, pre-seeds a checkpoint at
  `projects/p4/checkpoint.bin`, runs a compile, then idle-stops:
  asserts (a) `restore` was never called, (b) `blobStore.get`
  was never invoked with the checkpoint key, (c) `snapshot` was
  never called on idle-stop, (d) the pre-seeded blob is intact
  on disk afterwards.

  `SlowFirstCompileCompiler` wrapper exposes
