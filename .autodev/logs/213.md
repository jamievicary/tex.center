## Goal for this iteration

Use the iter-212 diagnostic capture to land the GT-5 fix. The live
gold run on iter 212 produced this message inside
`last_gold_output.txt`:

```
framesSent: pre-edit=1, post-type=24, now=26
  (delta during type=23; a healthy GT-C run sends ≥1 frame per keystroke)
cursor at edit: startOffset=165, focused line="Hello, world!!Coalescer probe abc…"
pre-edit doc length=218, tail(120)="…window.\end{document}"
overlapErrors=0, controlFrames=8
  ctrl[3]={"type":"compile-status","state":"error","detail":"supertex-daemon: stdin not writable"}
  ctrl[5]={"type":"compile-status","state":"error","detail":"supertex-daemon: stdin not writable"}
  ctrl[7]={"type":"compile-status","state":"error","detail":"supertex-daemon: stdin not writable"}
```

Reading against log 212's interpretation guide: framesSent delta
is healthy (23 deltas for ~24 chars), ruling out the page → WS
edge (probe #2). The cursor lands on a body line, not
`\end{document}`, ruling out cross-spec state pollution (probe #1).
And the controlFrames contain `compile-status state:error`, which
the guide calls "probe #3 silent compile-error short-circuit" — but
the detail string is not a LaTeX error, it is the sidecar's own
`supertex-daemon: stdin not writable` raised by
`SupertexDaemonCompiler.writeStdin`. The daemon child has exited
between GT-4 and GT-5 (or mid-GT-5), and from that point onward
every recompile attempt finds `child.stdin.destroyed` and surfaces
the same error — there is no recovery path until the sidecar
process restarts.

So the GT-5 failure is NOT any of the three probes the plan
predicted, it is a missing-recovery bug in our own code: the
`SupertexDaemonCompiler` caches `readyPromise` permanently and
never re-spawns the child if the OS or the daemon kills it.

This iteration adds the recovery: detect dead-child state at the
top of `compile()` and reset for re-spawn.

## Approach

In `apps/sidecar/src/compiler/supertexDaemon.ts`:

1. Add `isChildDead()` — returns true if `spawnError`,
   `childExited`, or the stdin pipe is gone / destroyed.
2. Add `resetForRespawn()` — nulls out `child`, `childExited`,
   `spawnError`, the stdout/stderr buffers, the event queue and
   waiter, `stdoutEnded`, and crucially `readyPromise` so the
   next `ensureReady()` call falls through to a fresh
   `spawnAndWaitReady()`.
3. At the top of `compile()` (after the `busy` / `closing`
   guards, before `ensureReady()`), if `isChildDead()` is true,
   call `resetForRespawn()`.

Loses upstream incremental state on respawn — the next compile
runs from scratch. That is strictly better than wedging every
subsequent edit until the sidecar process is killed; an idle Fly
machine restart was previously the only path back.

In `apps/sidecar/test/supertexDaemonCompiler.test.mjs` add case 14:
spawn happy daemon, run one successful compile, `SIGKILL` the
child directly via the (TypeScript-private but runtime-accessible)
`child` handle, wait for the `exit` event, then run a second
compile and assert it succeeds and that a fresh child object is in
place.
