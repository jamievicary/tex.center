## Goal for this iteration

M8.pw.1.1.c — the local-target DB co-location design call +
helper. The PLAN listed three options ((a) PGlite-over-TCP via
`@electric-sql/pglite-socket`, (b) ephemeral system Postgres,
(c) shared in-process PGlite via a swappable `getDb()`).
Option (c)'s appeal in the PLAN ("no extra binaries/servers")
mis-modelled the topology: the SvelteKit dev server runs as a
*child process* of Playwright's `webServer` config, so module-
graph sharing can't reach it — cross-process state demands a
real transport. Option (b) needs a system Postgres binary on
every dev/CI box. So (a) is the only working choice.

## Approach

1. Bump `@electric-sql/pglite` 0.2.17 → 0.3.16 (lowest
   pglite-socket-compatible series) and add
   `@electric-sql/pglite-socket@^0.0.22`. Bumped in root
   `package.json` and `packages/db/package.json` devDeps;
   existing in-process PGlite tests (`test_pglite_migrations`,
   `test_pglite_users_sessions`, `test_pglite_projects`,
   `test_mint_session`) still pass against the bumped engine —
   verified `tests_normal/run_tests.sh` and
   `tests_gold/run_tests.sh` before writing any new code.
2. Land `tests_gold/lib/src/localDb.ts` —
   `startLocalDb({signingKey?, seedEmail?, seedGoogleSub?})` →
   `{url, port, db, signingKey, userId, close()}`. Internally
   boots PGlite, applies migrations from
   `packages/db/src/migrations/`, seeds one user (default
   allowlisted `jamievicary@gmail.com`), wraps in
   `PGLiteSocketServer` on `127.0.0.1:0`, returns a
   `postgres-js` DbHandle so the test driver writes via the
   same client family the dev server uses.
3. Gold-test: open a *second* `postgres-js` client against the
   same URL (modelling the dev server's `getDb()`) and verify a
   write through the driver-side handle is visible through the
   dev-server-side handle. This is the load-bearing claim for
   the whole approach.
4. Update PLAN: mark design + helper done under pw.1.1.c; queue
   the Playwright wiring (`globalSetup`/`globalTeardown` +
   fixture branch on project name) as the next ordinary
   iteration.

## Problems encountered

1. **`maxConnections` default is 1, not 100.** The second
   `postgres-js` client got `ECONNRESET` the instant it tried
   its first query — the first connection's startup handshake
   succeeded, but a parallel connection was rejected. Source
   inspection (`node_modules/@electric-sql/pglite-socket/src/index.ts:563`)
   shows the constructor sets
   `maxConnections = options.maxConnections ?? 1` despite the
   JSDoc saying 100. The helper sets `maxConnections: 16`
   explicitly. Comment in `localDb.ts` records the gotcha so
   the next reader doesn't relearn it from a stack trace.
2. **PGlite emits `DEBUG`-level NOTICEs by default** which
   postgres-js's default `onnotice` handler forwards to
   `console.log`. Test output became unreadable. Helper passes
   `onnotice: () => {}` to `createDb` and the gold-test
   silences its own second client the same way.
