Scope kept tight: existing inline `✎` / `×` / create-form affordances
stay; the new context menu wraps the same imperative flows
(`promptRename`, `window.confirm` + `onDeleteFile`, a new
`promptCreate` for the menu's "New file…"). No new wire path; no
sidecar contract change; pure UI.

## Approach

Pure-policy split, matching the existing M11.2a shape
(`fileTreeKeyboard.ts` + `decideFileRowAction` → unit test). New
module `apps/web/src/lib/fileTreeContextMenu.ts` exports:

- `menuItemsForFile(path, mainDocName)` → `[Rename…, Delete]` with
  `enabled=false` on both for `MAIN_DOC_NAME` (same guard as the
  `✎` / `×` buttons).
- `menuItemsForRoot()` → `[New file…]` enabled.
- `decideMenuKeyAction(ev)` → `{kind: "prev"|"next"|"activate"|
  "dismiss"} | null`. Ctrl/Alt/Meta suppress so OS shortcuts pass
  through; Shift is allowed (harmless on arrow nav).
- `moveMenuFocus(items, current, delta)` — wrapping focus advance
  that skips disabled entries; returns `current` unchanged if no
  enabled entries exist.
- `initialMenuFocus(items)` — first enabled entry, falls back to
  0 if none enabled.

`FileTree.svelte` holds the menu open state in a single
`$state<OpenMenu | null>` (x, y, items[], focused, optional path
for file menus). `oncontextmenu` on file rows opens the file menu
at clientX/clientY; `oncontextmenu` on the `.ft-host` wrapper opens
the root menu (gated on `onCreateFile` so unused trees stay
right-click-inert). `<svelte:window onpointerdown>` dismisses on
outside clicks; the menu's own `pointerdown` handler
`stopPropagation`s to prevent self-dismissal. `use:autoFocus`
moves DOM focus to the menu wrapper on open so arrow keys work
without a pre-click. Activation routes to `promptCreate` /
`promptRename` / `confirmDelete` — all three are the same
imperative flows the existing buttons use, so the wire behaviour
is identical.

CSS: `position: fixed` with `z-index: 1000`; the rendered menu
sits above any sibling content. Width-min 9rem; box-shadow for
elevation; `.focused:not(:disabled)` background highlights the
arrow-key-focused row.

## Notes for future iterations

- **Position-clamp.** The menu currently renders at exact
  `clientX/clientY`. If the click is too close to the viewport
  edges it clips off-screen. The Playwright spec works around
  this by clicking at fixed in-bounds coords (x=20, y=80); a
  follow-up could clamp inside the host's `getBoundingClientRect`
  before render. Out of scope for this iteration since the user-
  facing impact is cosmetic and the affordance is otherwise
  functional.
- **Keyboard activation under pointer-initiated open.** The
  current spec uses pointer-click on the focused menu item;
  Enter-via-keyboard activation was deferred after a Playwright
  dialog/focus interaction made the test fragile (the auto-focused
  menu wrapper appears to lose focus between the right-click and
  the keypress on some Chromium builds). The pure-module unit
  tests still cover the key→action mapping. A follow-up spec
  could explicitly `menu.focus()` from `page.evaluate(…)` before
  pressing Enter if a regression motivates it.
- **No new wire path.** All three menu actions reuse the existing
  `onCreateFile` / `onRenameFile` / `onDeleteFile` props the
  inline buttons already drive, which the editor page wires
  through `WsClient.send*` → sidecar. The new affordance is
  purely additive.
- **Maintenance cadence.** N=354, N%10==4, ordinary iteration.
  Next refactor N=360, next plan-review N=361. No deferral.
