## Goal for this iteration

Deferred refactor cron (N=193; cron rolled from iter 190 → 192 →
193 per logs). Single highest-leverage cleanup: unify the two
`createProjectPersistence` return branches in
`apps/sidecar/src/persistence.ts`. The `!blobStore` (in-memory) and
the blob-backed branch each carry a near-duplicate implementation
of `addFile` / `deleteFile` / `renameFile` — same validation order,
same Y.Doc mutation, same in-memory bookkeeping — differing only in
whether a blob op runs. ~50 lines of duplication; bugs would need
fixing twice, and the `memFiles` set is structurally redundant with
`knownFiles`.

## Approach

Single return object, `knownFiles` used by both modes, blob ops
gated on `canPersist && blobStore`. In-memory mode resolves
`canPersist` to `false` (hydration is a synchronous seed in an
async IIFE). `persistenceSeed.test.mjs` already awaits
`awaitHydrated()` before reading the doc, so the seed-deferral is
test-safe.

## Actions

- `apps/sidecar/src/persistence.ts`: collapsed the
  `if (!blobStore) { return { ... }; }` early-return into the
  main return object. `knownFiles`, `persistedByName`, and
  `canPersist` are now hoisted out of the blob-backed branch and
  shared by both modes. The async hydration IIFE handles in-memory
  seeding (synchronously inside the IIFE; `awaitHydrated()` is the
  call-site barrier all consumers already use) and falls through
  to blob hydration when `blobStore` is defined. Blob-op gates
  in `addFile`/`deleteFile`/`renameFile` widened to
  `canPersist && blobStore` so the TS narrower can prove `blobStore`
  non-null inside the block; `maybePersist` similarly. ~50 lines
  removed; the duplicate `memFiles` set is gone.
