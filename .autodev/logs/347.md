## Goal for this iteration

PLAN.md priority 2: Bug B investigation. User-reported "compile
runs but emits zero pdf-segments on cold-resume edit" (manual
repro in `.autodev/discussion/344_question.md`). The iter-346
commitment was to capture a prod daemon transcript and pin the
root cause.

## Approach

DEBUG_COMPILE_LOG is default-true in `apps/sidecar/src/server.ts`
(line 221: only suppressed when env explicitly "0" or "false"),
so `compile-source` + `daemon-stdin` + `daemon-round-done` records
are already streaming to `flyctl logs`. Pull a recent slice and
classify each compile cycle by `segments`, `maxShipout`,
`compileMs`, and the cold/warm context (machine reboot markers,
`reqId`).

## Iter-344 user transcript reconciled

Neither captured cycle reproduces the user's transcript exactly
(`0.0s — compile-status idle, no pdf-segment` on initial AND
`2.7s — compile-status idle, no pdf-segment` on edit). The 0.0s
initial matches the warm-reconnect cycle's <50 ms compile, but
the user reports the Machine was stopped on connect (so it should
have been a cold-resume taking ~5 s). The 2.7 s edit with zero
segments is more suspicious — an edit that produces a Yjs op
should result in `[edit detected at line N]` on stderr and at
least one shipout. Without a fresh user repro on the current
build, the root cause stays speculative.

## Resolution

**Land diagnostic-only instrumentation** in
`apps/sidecar/src/server.ts` that makes the next prod repro
self-explanatory:

1. New `replay-segments` log line at WS-connect, after the
   `lastSegments` for-loop and before `coalescer.kick`:
   `{ projectId, lastSegmentsLen, replaySent, replayBytes,
   socketOpen }`. Tracks whether the replay shipped any segment
   AND whether the socket survived the loop.

2. Add `lastSegmentsLen` to every `compile ok` log line so we
   can grep the in-memory cache state across successive compile
   cycles (a `segments=0, lastSegmentsLen=0` row is the
   smoking gun for "first compile silently no-op'd, leaving
   the cache empty").

These are pure observation changes — no behaviour modified, no
new failure mode introduced. Existing
`apps/sidecar/test/serverSegmentReplay.test.mjs` continues to
pin the replay behaviour; tests with `logger: false` ignore the
new fields.

## Notes for future iterations

- The `replay-segments` log line uses `break` rather than `return`
  on a closed socket so the log fires even when the loop didn't
  run to completion. The previous `return` would have suppressed
  both the log AND `coalescer.kick`; that suppression was probably
  unintentional. With the break-and-log shape, a closed socket
  exits the for-loop, logs the partial counts, and still falls
  through to `coalescer.kick` — which is the same kick a
  successful loop would have made. If a future iteration spots a
  problem with kicking the coalescer after the socket closed,
  guard the kick on `socket.readyState === socket.OPEN` and add
  a test.
- Once a user-driven Bug B repro shows up in prod logs,
  `flyctl logs | grep -E "replay-segments|compile ok|round-done"`
  on the affected project ID gives the full cycle in one greppable
  view. Decision tree at the bottom of `PLAN.md` priority #2.
- The warmup-fails-then-respawn cold-resume pattern (project
  `2efd808d` above) is a known inefficiency: the daemon eats ~5 s
  of cold-start cost on the *retry*, not in parallel with hydrate.
  If/when M20.3 closeout points at this as the dominant remaining
  latency, the fix is to defer `warmup()` until after
  `awaitHydrated()` resolves — or to pre-write a placeholder
  `main.tex` so warmup's daemon spawn can succeed before hydrate.
  Not load-bearing for Bug B.
