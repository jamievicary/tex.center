## Goal for this iteration

Pick up the iter-227 hand-off: GT-5 RED on the last gold pass
(third recurrence). Iter 227's notes scheduled this iteration
to diagnose via flyctl logs and decide whether the failure is
the iter-213 dead-child class (sidecar `SupertexDaemonCompiler`

cold when GT-5 runs). Then localise the bug.

## Findings (load-bearing)

The iter-227 GT-5 inline trace already contained enough signal
to rule out both hypotheses:

```
pdfSegmentFrames: pre-edit=1, post-wait=1
framesSent: pre-edit=1, post-type=24, now=28  (delta=23 — healthy)
overlapErrors=0, controlFrames=8
  ctrl[2]=running  ctrl[3]=idle
  ctrl[4]=running  ctrl[5]=idle
  ctrl[6]=running  ctrl[7]=idle
```

Daemon ran THREE full `compile-status running→idle` cycles
(initial seed + two post-edit), **no error frames, no `stdin not
writable` violation**. So iter-213's dead-child re-spawn path
isn't the cause — the daemon is alive and responsive. And
`framesSent` shows ~1 frame per keystroke during typing,
matching a healthy GT-3 baseline — not a globalSetup-coldness
issue at the Yjs/WS edge either.

The symptom maps cleanly onto the existing no-op path at
`apps/sidecar/src/compiler/supertexDaemon.ts:141`:

```ts
if (events.maxShipout < 0) {
  return { ok: true, segments: [] };
}
```

`events.maxShipout < 0` ⇔ the supertex daemon's `recompile`
round completed (round-done emitted) with no `[N.out]` shipout
events and no `[error …]` line — the upstream "no usable
rollback target → silently no-op" round shape iter 188
documented. The sidecar then dutifully:

- sets `compile-status: idle` (no error)
- ships zero pdf-segments (segments array is empty)

…which exactly produces the observed wire pattern.

The trace's pre-edit doc tail is
`…padding bytes to extend the typing window.\end{document}`,
cursor on that body line, doc length 218 — the shared
`liveProject` has been polluted by GT-D / GT-7 cumulative typing
before GT-5 runs. The combination of "polluted warm doc" + "edit
lands at a body offset past every extant supertex checkpoint
for this round" reproducibly drives the daemon into the no-op
rollback path.

The iter-755–758 upstream fix referenced in PLAN's
M9.editor-ux.regress.gt7 closed exactly the related class for
GT-8 (cold-start `\newpage` spam) — but evidently does NOT
cover the warm-doc body-edit shape. GT-8 went GREEN in iter 227,
GT-5 stayed RED.

I attempted to pull live `flyctl logs` for the failed run's
project (`5e640a76-696b-46e9-844e-043171c1e357`) to confirm the
no-op path was hit, but the per-project Machine had
idle-stopped by the time I queried (only 3 machines in
`started` state across the sidecar app; none owned the failed
project's compile traffic — logs from the 3 currently-running
ones contained zero references to that project id). This is
the right place to drop the diagnostic into app-logs themselves
so the next failure is locatable from log volume alone — the
work below.

## Approach

Add explicit app-log signal for the no-op path so the next live
GT-5 RED is diagnosable from `flyctl logs -a tex-center-sidecar`
in one grep, no per-machine spelunking required.

Concretely:

- `CompileSuccess.noopReason?: string` — optional field on the
  successful-compile shape (kept optional so `FixtureCompiler`,
  `SupertexOnceCompiler`, and the existing test stubs that
  return empty segments don't need changes).
- `SupertexDaemonCompiler.compile()` sets `noopReason` when
  taking the `events.maxShipout < 0` return path.
- `server.ts` `runCompile`: when `result.segments.length === 0`
  and `result.noopReason` is set, log at `warn` level
  (`"compile no-op (no pdf-segment shipped)"` with
  `projectId / elapsedMs / sourceLen / targetPage / noopReason`)
  instead of the usual info-level `"compile ok"`.

No wire-protocol change — the client still sees
`compile-status: running → idle` and no pdf-segment frame, same
as before. This is purely a diagnostic seam.

  test that drives a stub compiler returning the no-op shape
  and asserts (a) `compile-status: idle` still emitted, (b) no

  (d) the warn line surfaces in the pino log stream with the

  rather than a stub logger object (Fastify rejects custom
  logger objects with `FST_ERR_LOG_INVALID_LOGGER_CONFIG`
  unless they are pino-shaped — using a stream sidesteps that).

## Notes for future iterations

- **Next live gold run should grep `flyctl logs -a
  tex-center-sidecar | grep "compile no-op"` over the GT-5
  window.** If hits appear with `projectId` matching
  globalSetup's reported project id, the diagnosis is confirmed
  upstream-supertex and the fix moves to `vendor/supertex`. If
  no hits appear, the failure is something else entirely (the
  daemon path is hitting a different code branch — possibly a
  silent shipout but the assembled bytes equal the prior PDF
  and PDF.js renders to identical pixels, which would be the
  iter-188 byte-identical regression class).
- **The upstream supertex fix** would need to ensure
  `recompile,end` always produces ≥1 shipout on success, even
  when the edit lands past every extant checkpoint. The
  iter-755–758 fix worked for the GT-8 cold-start case; the
  GT-5 warm-doc body-edit case is a separate input shape and
  needs its own upstream slice.
- **An alternative local mitigation** (worth considering only
  if upstream is slow): the sidecar could re-issue the
  `recompile` with `targetPage = 0 / "end"` when it sees the
  no-op shape, OR force-rebuild the daemon's checkpoint chain
  by sending a synthetic full-document write. Both feel like
  papering over the upstream bug; not pursued this iteration.
- **GT-5 itself doesn't need a code-side change.** Its inline
  diagnostic captured the right signal; the failure message is
  already actionable. Don't add more probes — pull the
  sidecar-side warn lines from the next live RED run instead.
- Sidecar redeploy required to make the new warn-log visible
  live. Will happen on the next iteration that touches sidecar
  surface area or explicitly redeploys — not blocking this
  iteration's commit.
