## Approach

Yjs' Y.Doc-level update format already carries every Y.Text on
the doc, so no protocol change is needed. The shape:

- **Server hydration** (`apps/sidecar/src/persistence.ts`):
  `createProjectPersistence` switches from taking a `text: Y.Text`
  to taking a `doc: Y.Doc`. Hydration lists project files,
  fetches them in parallel, and bulk-inserts each into
  `doc.getText(<relative-path>)` inside a single `doc.transact`
  so observers see one coherent update. `main.tex` is still
  special: only its content is tracked for `maybePersist` (the
  diff-and-PUT path is unchanged).

  The empty-blob and missing-blob branches still behave as before
  (`persistedSource = ""` vs `null`), because we extract the
  main entry post-load.

- **Client** (`apps/web/src/lib/wsClient.ts`): new
  `getText(name)` accessor — thin wrapper around
  `this.doc.getText(name)`. Yjs auto-creates the type on first
  access, so an unknown name is safe (the server's later
  hydration update will populate it).

- **Editor binding** (`apps/web/src/routes/editor/+page.svelte`):
  `$effect` on `selected` rebinds `text = client.getText(selected)`.
  The existing `{#key text}` block remounts the CodeMirror view
  so `yCollab` re-attaches to the new Y.Text cleanly.

- **Foot-gun guard.** Today only `main.tex` is persisted back, so
  letting the user type into `refs.bib` and watching their edits
  silently vanish on reconnect would be bad UX. `Editor.svelte`
  grows a `readOnly?: boolean` prop (adds
  `EditorState.readOnly.of(true)` when set); the page passes
  `selected !== MAIN_DOC_NAME`. When multi-file persistence
  lands, drop the guard.

## Notes for future iterations

- Multi-file persistence is the natural next slice: extend
  `maybePersist` to take a name+source (or observe per-file Y.Text
  changes) and drop the `editorReadOnly` guard. Probably wants a
  debounced per-file persist trigger separate from the compile
  trigger, since non-main edits don't fire a recompile.
- The file-list emission in `server.ts` still calls
  `listProjectFiles` independently of persistence's hydration —
  one round-trip duplicated. Could be deduplicated by having
  `ProjectPersistence` expose `files()`, but the cost is small
  and the responsibilities are distinct (file-list reflects the
  blob store; persistence's job is to mutate the Y.Doc). Leave
  for now.
