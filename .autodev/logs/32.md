## Goal for this iteration

Lay the foundation for M5 (auth) with the smallest coherent slice
that doesn't yet require an apps/web server tier: a pure-logic
`packages/auth` leaf module with the two primitives every later
auth iteration will reuse — the email allowlist policy and
HMAC-signed session tokens. PLAN's "current focus" listed M3.5
(not actionable in-repo), multi-file (multi-iteration, needs
frontend work), and M5 scaffolding. M5.0 here is testable today
with no I/O, lets the OAuth-callback iteration focus on PKCE +
JWKS verify, and gives the cookie-mint/verify code one obvious
home shared by web and any future control-plane Fastify app.

## Approach

1. New workspace package `@tex-center/auth`:
   - `src/allowlist.ts` — `ALLOWED_EMAILS` (single entry,
     `jamievicary@gmail.com` per GOAL.md), `isAllowedEmail`
     (case-insensitive, trimmed, robust against null/undefined/
     non-string inputs that could leak past TS).
   - `src/session.ts` — `signSessionToken({sid, exp}, key)` /
     `verifySessionToken(token, key, nowSeconds)`. Token shape
     `<payloadB64u>.<sigB64u>`, HMAC-SHA256, constant-time
     signature compare, caller-supplied clock (deterministic
     tests, prod stub-ability). Discriminated `VerifyResult` with
     reasons `malformed | bad-signature | expired | bad-payload`.
   - `src/index.ts` — barrel.
   - `package.json` / `tsconfig.json` mirror `@tex-center/blobs`.
2. Test suite `packages/auth/test/auth.test.mjs` — round-trip,
   wrong key, expired-at-exp and expired-after, tampered payload,
   tampered signature, malformed shapes (no dot, multi-dot, empty
   half), invalid base64url chars, valid signature on non-JSON
   payload (the `bad-payload` branch), and sign-time validation
   (empty key, empty sid, non-integer exp).
3. Wire into `tests_normal/cases/test_node_suites.py` as
   `test_auth`. `pnpm install` to refresh the lockfile so
   `--frozen-lockfile` accepts the new workspace package.

## Notes for future iterations

- M5.1 (the OAuth callback iteration) needs an apps/web *server*
  tier — today `apps/web` is `adapter-static`. The tradeoff is
  swap to `adapter-node` (one runtime, simpler topology) versus
  split a small Fastify control-plane app alongside `apps/sidecar`
  (matches the GOAL architecture more literally). Worth picking
  deliberately at that iteration; either way, this module is the
  cookie-layer dependency.
- The signing key is intentionally caller-supplied. M5.1 should
  load it from env (`SESSION_SIGNING_KEY`, base64-encoded 32+
  bytes) and refuse to boot if absent. No fallback / no auto-
  generated key — silent rotation would invalidate every live
  session.
- The session row (uuid `sid`) is the source of truth: the cookie
  is just a signed pointer with a server-side expiry cap. The
  `sessions` table in `packages/db` already matches this shape.
