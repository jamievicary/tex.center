## Problems encountered

### Fly trial expired mid-deploy — 5-minute machine cap

Five minutes after the sidecar app machine started, Fly emitted:

```
[runner] Trial machine stopping. To run for longer than 5m0s,
add a credit card by visiting https://fly.io/trial.
```

and SIGINT'd the sidecar. Confirmed account-wide: `flyctl status`,
`flyctl machines list`, `flyctl ips list`, `flyctl machine status`
all now return `trial has ended, please add a credit card`. The
control-plane app `tex-center` is still serving (`/healthz` → 200,
verified) — the trial ban only blocks *new* machine creation past
the quota and read APIs; existing long-running machines keep
running. But every fresh per-project sidecar Machine created from
M7.1 onwards will hit the 5-min cap until billing is set up.

This is **the canonical upstream blocker the harness §7 calls
out** — agent cannot fix (only the human can add a credit card),
and it gates the entire remaining Fly track (M7.0.3 deploy
verification, M7.1 Machines API, M7.2 routing, M7.3 idle-stop,
M8.acceptance, future control-plane redeploys via CI). Recording
loudly in PLAN as the top risk; **not** halting because plenty of
non-Fly in-tree work remains (M7.5 daemon adoption, S3 adapter
when docker lands, refactors).
