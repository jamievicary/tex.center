## Goal for this iteration

Investigate why the gold-suite wallclock had been varying so widely
(recent iterations ranged from 5m 31s to 15m 02s) and parallelise it
"as far as reasonably possible". The starting hypothesis from
analysing `.autodev/iterate.out` and iter-302's
`last_gold_output.txt` was a three-pronged problem:

1. Per-test `testInfo.setTimeout` budgets sized at 5–7 min absorb
   silent hangs rather than catching real regressions.
2. The python case list under `tests_gold/cases/` runs strictly
   serial via `python -m unittest discover`.
3. The Playwright config has `workers: 1`, so no live-spec parallelism
   at all.

Driven as a main-session collaboration with the user rather than as
an autodev-engineer iteration, hence the extended discussion thread
and several rounds of "tried it / reverted it". Outcome at the end of
the session: a partial set of changes landed and shipped; the
workers > 1 attempt was unwound back to `workers: 1`.

## Actions

### Test budget tightening

Replaced 5–7 min `testInfo.setTimeout` ceilings with values
calibrated to roughly 1.5× the worst observed runtime across recent
runs. Affected specs:

- `verifyLiveGt6LiveEditableState`: 420 s → 40 s
- `verifyLiveGt6LiveEditableStateStopped`: 420 s → 60 s
  (the test was previously masking its own measurement under a 7 min
  ceiling; 60 s is the honest envelope for the 1 s product invariant
  it pins)
- `verifyLiveGt8ColdProjectNewpageDaemonCrash`: 360 s → 60 s
- `verifyLivePdfMultiPage` (both cases): 360 s → 60 s
- `verifyLivePdfMultiPageSeeded`: 360 s → 75 s
- `verifyLiveFullPipeline`: 300 s → 40 s
- `verifyLiveFullPipelineReused`: 300 s → 150 s (variance-prone —
  warm vs cold-restart path differ by ~90 s)
- `verifyLiveDeleteProject`: 300 s → 30 s
- `verifyLiveGt6FastContentAppearance`: 120 s → 30 s
- `verifyLiveWsUpgrade`: 5 min → 90 s
- `verifyLiveGt4SustainedTyping`: added explicit 60 s override
  (was inheriting the 45 s default, brushing its own typical
  ~36 s runtime)
- `bootstrapLiveProject` warmup budget: 240 s → 90 s

### Parallel python runner

Added `tests_gold/run_parallel_cases.py` and wired it into
`tests_gold/run_tests.sh`. Discovers `test_*.py` under
`tests_gold/cases/`, dispatches each as its own `python -m unittest`
subprocess across `TEXCENTER_GOLD_PARALLEL` workers (default 4), and
streams each module's output atomically so the iterator's awk parser
still sees per-test ✓ / ✘ / skipped lines.

Each case shells out to its own toolchain (pnpm/tsx/supertex/
lualatex/pglite) so they're already process-isolated. The only
shared external state is the live Fly app, and the only python case
that touches it write-wise is `test_playwright` itself (creates +
reaps uniquely-named projects). No collisions observed in any of
the seven gold runs done in this session.

### Fly health checks (deployed live)

Added native Fly TCP checks on the sidecar's listen port (3001):

- `apps/sidecar/fly.toml`: new `[checks]` block.
- `apps/web/src/lib/server/upstreamFromEnv.ts`: matching
  `checks.sidecar-tcp` on the per-project Machine config sent to the
  Machines API.

Complements the resolver's existing `waitForUpstreamReady` TCP
probe: Fly's edge proxy now also gates routing on the check
passing, closing the race window between Fly flipping `state` to
`started` and the sidecar Node process actually binding 3001.

### Per-worker isolation machinery (kept, currently vestigial at workers=1)

Built out a full per-worker test stack to support `workers > 1`:

- `tests_gold/playwright/fixtures/localStack.ts`: per-worker
  PGlite-over-TCP + SvelteKit dev server spawn on `3000 + workerIndex`.
- `tests_gold/playwright/globalSetup.ts`: stripped to only do the
  live-project bootstrap. PGlite and dev-server spawn moved out.
- `tests_gold/playwright/fixtures/authedPage.ts`: refactored to
  use a worker-scoped `localStack` fixture, per-worker DB user
  (random UUID; deleted at worker teardown; FK cascade reaps
  forgotten afterEach state), per-worker fly proxy port
  (`localPort = base + workerIndex`), and a `baseURL` fixture
  override that returns the right URL per worker.
- `tests_gold/playwright/fixtures/sharedProjectMutex.ts`:
  mkdir-based cross-process mutex around the bootstrapped live
  project, so the 7 shared-project specs (GT-A..GT-D, GT-7,
  PdfNoFlash) never run concurrently across workers.

Required to fix two genuine bugs that surface at workers > 1:

- PGlite-over-TCP server-side unnamed-prepared-statement isolation
  bug under concurrent connections (separate from production
  Postgres). Each worker now owns its own PGlite instance.
- Empty-state race in `projects.spec.ts`: workers used to share one
  seed user, so worker A creating a project for `editor.spec.ts`
  would race worker B's empty-state assertion.
- Cross-worker shared-project state corruption (GT-3 counting
  pdf-segments while GT-4 was typing into the same project).

Plus the `baseURL` precedence bug discovered late: the playwright
project config's `use: { baseURL }` overrides fixture defaults, so
the fixture override silently no-op'd and every worker's tests
collapsed onto port 3000. Fixed by removing the local project's
`baseURL` from the config (the fixture is now authoritative).
Live's `baseURL` stays in config — it's worker-invariant
(`https://tex.center`) and specs that import `test` from
`@playwright/test` directly (e.g. `verifyLive.spec.ts`'s
`request.get("/healthz")`) need it.

### Workers scan + revert

Tested workers=2, 3, 4 and recorded:

| workers | wallclock | failures | notes |
|---|---|---|---|
| 1 (iter 302) | 12 m 38 s | 1 | baseline |
| 2 (gold #7) | 3 m 44 s | 4 | clean |
| 3 | 2 m 52 s – 3 m 39 s | 5 | best |
| 4 (baseURL bug) | 3 m 32 s | 12 | all on port 3000 |
| 4 (fixed)      | 3 m 48 s | 12 | real ceiling |

CPU sampler (`/tmp/cpu_sampler.py`) confirmed the system was 80%
idle even at workers=4 (max load1 = 5.5 of 8 cores) — so CPU
contention is not the explanation for the workers=4 regression. The
real driver of the per-test slowdown across the session turned out
to be **sidecar image-pull tax** after the deploy in this session
changed the image SHA: every fresh per-project Machine on a host
without the new image cached pays ~5–10 s pulling 5 GB. As Machines
accumulated through the day, Fly scheduled new ones onto colder
hosts, pushing individual specs further over their tightened
budgets.

**Reverted to `workers: 1`** at the user's call: the architectural
work to enable workers > 1 is sound, but the gain it would deliver
is masked by the post-deploy image-pull variance. Per-worker
machinery (`localStack`, `sharedProjectMutex`, per-worker user,
`baseURL` fixture, per-worker fly proxy port) is left in place —
functionally identical at workers=1, just adds some constant
startup overhead. Cleaning it back out is a separate decision.

## Notes for future iterations

- **Image-pull tax is the dominant per-test cost variance.** Every
  fresh per-project Machine on a host that doesn't have the latest
  sidecar image SHA pays ~5–10 s pulling 5 GB. Hosts cache
  independently, so the tax repeats across hosts in the region.
  The single biggest lever to make gold faster (and less variable)
  is to **shrink the sidecar image**. ~5 GB is currently almost all
  texlive-full; auditing what's actually used by `supertex` +
  `lualatex` and dropping the rest could plausibly hit <500 MB. See
  `apps/sidecar/Dockerfile`.

- **`workers > 1` is now wired up but unused.** All the per-worker
  isolation primitives (PGlite, dev server, user, fly proxy, shared-
  project mutex, baseURL override) are in place. Re-enabling it is a
  one-line change in `tests_gold/playwright.config.ts`, but only
  worth it once the image-pull tax is gone — otherwise the gain is
  eaten by Fly-side variance.

- **Per-test setTimeout philosophy locked in.** Budgets are now
  ~1.5× max-observed runtime per spec, comments cite the observed
  values they were derived from. When a spec genuinely regresses,
  the test surfaces it within minutes rather than hiding under a
  multi-minute ceiling. **`verifyLiveGt6LiveEditableState`'s 40 s
  budget** is the user-locked one — exceeding it is a real signal
  about cold-from-suspended editable latency.

- **Known-RED tests still RED.**
  `verifyLiveGt6LiveEditableStateStopped` (M13.2(b).4) — pins a 1 s
  product invariant on cold-from-stopped that the architecture
  doesn't currently deliver. M13.2(b).5 lands the fix (PLAN.md).

- **Fly health checks are deployed live** but not yet wired into
  any code path that depends on them. The resolver still uses
  `waitForUpstreamReady`'s own TCP probe. Future iteration could
  switch the resolver to poll `checks[]` instead, saving the
  500-ms-poll-interval cost.

- **`test_sidecar_machine_count`** tripped during the session
  (12–14 non-shared Machines vs threshold 10). The orphan-sweep
  reaps stale `pw-*` machines older than 10 min; mid-suite
  accumulation under workers > 1 outpaced it. Either raise the
  threshold or accelerate cleanup if re-enabling workers > 1.

- **Live timing variance** observed across runs:
  globalSetup live-warmup ranged 11.9 s → 21.3 s for the same path;
  `verifyLiveFullPipeline` 17.9 s → 42.1 s; etc. The variance is
  almost entirely host-cache-state, not test-logic.
