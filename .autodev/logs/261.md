## Goal for this iteration

M11.1 rendering substrate. The PLAN's `M11.file-tree` milestone
has been open since the file-tree work began; it sits behind the
two larger M13.2 follow-ups (cold-stopped seed widening,
shared-R2 blob store) which both require multi-iteration substrate
changes. A small, focused slice that lands the path-tree
rendering layer without touching the validator or sidecar
persistence is strictly progress on M11 with zero behavioural risk
to today's flat-name flow.

## Approach

Pure separation between grouping and rendering:

1. **`apps/web/src/lib/fileTree.ts`** — `buildFileTree(files)` —
   pure function. Splits each input on `/`, walks an
   insertion-ordered child map per folder, freezes into a
   `FileTreeNode` discriminated-union forest. Folders sorted
   before files, alphabetic within. Defensive on malformed
   input (empty string, empty segment, file-vs-folder collision
   on identical prefix).
2. **`apps/web/src/lib/FileTreeNode.svelte`** — self-recursive
   collapsible node. Folder header is a button with `▾`/`▸`
   chevron and `aria-expanded`; files use the existing
   `row.active` highlight. Rename/delete buttons are still
   gated on `node.path !== MAIN_DOC_NAME`.
3. **`apps/web/src/lib/FileTree.svelte`** — drops the inline
   `<ul>{#each}` over the flat list, renders the derived forest
   via `<FileTreeNode>` instead. Per-folder collapse state lives
   in a `Map<string, boolean>` in `$state` (default expanded).
   Existing inline create / upload form and `serverError`
   surfacing are unchanged.
4. **Unit test** at `apps/web/test/fileTree.test.mjs`, registered
   in `test_node_suites.py` (`test_web_file_tree`). Nine cases
   covering empty, flat, sorted, single nested, mixed
   flat+nested, deep nesting, malformed inputs, duplicates,
   and file-vs-folder collision.

Behavioural invariance under today's validator: every input is a
single segment, so the forest is a flat list of file nodes — the
rendered DOM is structurally identical to the previous `<ul>` of
file rows.

Why landing the substrate now rather than bundled with M11.1b
(validator + sidecar persistence): keeping the slices separate
means the rendering change ships under exactly its own test
surface (a pure-function unit suite that covers every path-shape
the future validator will permit), and the validator/persistence
slice — which is the load-bearing protocol+disk-layout change —
gets its own focused iteration with sidecar tests for `mkdir -p`
and empty-parent reap.
