## Approach

- `flyctl postgres create --name tex-center-db -o personal -r fra
  --initial-cluster-size 1 --vm-size shared-cpu-1x --volume-size 1
  --flex` — unmanaged Fly Postgres, single node, smallest size in
  `fra` (same region as `tex-center` and the sidecar).
- `flyctl postgres attach tex-center-db --app tex-center` — sets
  `DATABASE_URL` as a secret on the control plane and creates a
  per-app role.
- `flyctl tokens create deploy -a tex-center-sidecar -x 720h` —
  deploy-scoped token for the control plane to wake/spawn sidecar
  Machines via the Machines API. Narrower than the personal token
  in `creds/fly.token`.
- `flyctl secrets set -a tex-center RUN_MIGRATIONS_ON_BOOT=1
  FLY_API_TOKEN=<deploy-token>
  SIDECAR_APP_NAME=tex-center-sidecar
  SIDECAR_IMAGE=registry.fly.io/tex-center-sidecar@sha256:f31ef…`
  — flips boot migrations on, wires the upstream resolver.
- Trigger a redeploy (the secret-set should re-roll automatically;
  if not, `flyctl deploy --remote-only -a tex-center`).
- Probe `/healthz` and assert `db.state: ok` + record the deploy
  state into `deploy/README.md`.
