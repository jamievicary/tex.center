## Goal for this iteration

Land the **server half of M2**: define the wire protocol in
`packages/protocol`, build a Fastify + `ws` sidecar that
maintains an in-memory Yjs document, accepts a "viewing page
N" channel, and runs a stub compile loop that ships the
fixture PDF back to the client on doc change. Defer wiring
the browser to the WebSocket (PdfViewer Uint8Array path,
reconnect logic, dev proxy) to the next iteration — that's
genuinely separate concerns (UI plumbing) and would bloat
this slice past coherent. The tree stays sensible: protocol
defined, server runnable + unit-tested, browser still loads
the static fixture as in M1.

## Approach

Wire format: each WebSocket frame is binary, with a 1-byte
tag prefix:
- `0x00` Yjs doc-update (rest of frame is the Yjs update)
- `0x01` Yjs awareness-update
- `0x10` control JSON (UTF-8 after the tag)
- `0x20` PDF segment (4-byte BE total length + 4-byte BE
  offset + 4-byte BE seg length + bytes)

Yjs sync chosen as raw doc-updates (`Y.encodeStateAsUpdate`
on connect; `Y.applyUpdate` on receive) rather than the
multi-message `y-protocols/sync` exchange — sufficient for
single-user MVP, simpler to test, no extra dep. Awareness
deferred until the collab pass.

Sidecar runs one `Y.Doc` per project, holding a `Y.Text`
named `main.tex`. On `observe`, debounced 100 ms, the
"compile loop" (just reads the fixture PDF off disk for
now) emits a `0x20` segment of the full PDF. Browsers
send `0x10 {"type":"view","page":N}` to update the
target page; server logs it (the stub doesn't act on it
yet but the protocol exposes the channel for M3).

Tests in `tests_normal/cases/test_protocol_codec.py` —
spawns Node with a tiny throwaway script under
`packages/protocol/src/__test_runner__.ts` (compiled via
ts-node? no — use `tsx` since we already run with Node 20
and `tsx` is a pure-JS transpiler with one dep). Actually,
simpler: write a `.mjs` shim that imports the compiled
output. Even simpler: keep protocol package pure
TS-with-no-runtime-deps and unit-test it from Node by
importing the `.ts` via `tsx`. Going with `tsx` invocation.

## Notes for future iterations

- **Browser wiring (M2.b).** `PdfViewer.svelte` should accept
  `Uint8Array | string` (or split into two components).
  Replace `src="/fixture.pdf"` with a sidecar-fed buffer once
  the WS client lands. Use Vite's `server.proxy` for
  `/ws/* → ws://127.0.0.1:3001` in dev.
- **Yjs ↔ CodeMirror binding.** `y-codemirror.next` is the
  canonical extension. It pulls in `y-protocols` which we
  currently avoid on the wire — that's fine, Yjs's
  client-side state machinery is independent of how updates
  travel over the wire. Just call `Y.applyUpdate(doc,
  bytesFromServer)` and ship `update` events back.
- **Awareness (`0x01`)** is reserved but unused. Wire it in
  when collab lands; no MVP value.
- **Compile-status surface.** Browser should show a small
  spinner on `running`, error toast on `error`. UI not in
  scope for M2.b but the channel is already there.
- **Fixture PDF duplication.** Both `apps/web/static/fixture.pdf`
  and `apps/sidecar/fixtures/hello.pdf` exist. Once the
  browser pulls PDFs via WS only, the static one can be
  deleted. After M3 the sidecar fixture goes too.
- **`view` channel is a no-op today.** Sidecar logs page
  number into `client.viewingPage` but the stub compile
  doesn't gate on it. M3 wires this through to supertex's
  `target_page=N`.
