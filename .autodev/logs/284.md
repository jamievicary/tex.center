### Reactivity design notes

The tree instance is `$derived.by(() => { … untrack(…) … })` so
it rebuilds only when `forest` changes (i.e. when the `files`
prop changes). User expand/collapse calls mutate the instance
in place via `item.expand()` / `item.collapse()`; this fires
the adapter's `onStateChange`, which (a) updates
`collapsed: Set<string>` to reflect the difference between
`folderPaths` and the new `expandedItems`, and (b) bumps
`tick`. The flat-row `$derived` reads `tick`, so Svelte
re-evaluates the row list on every state mutation without
having to recreate the headless instance under the user's
hand. When `files` does change (e.g. a remote create / delete),
the `untrack` block reads the latest `collapsed` set to compute
`initialExpanded` for the fresh instance, so collapsed state
survives `files` churn.

The `untrack` matters: without it, every `collapsed = next`
inside `onStateChange` would invalidate `tree`, forcing a fresh
TreeInstance per click and possibly losing focus / mid-action
state. With `untrack`, only `forest` triggers a rebuild.

### Behaviour preserved vs changed

- **All-expanded default** — pre-cutover, `collapsed` was a
  `Map<string, boolean>` defaulting empty, where a missing
  entry meant "not collapsed". After cutover, `initialExpanded`
  is seeded with every folder path, so the first paint shows
  every folder open. Matches the prior visual.
- **Single-select via parent `bindable`** — unchanged. We
  intentionally don't surface `selectionFeature`'s
  multi-selection through the UI; that's an M11.2 question.
- **Rename / delete buttons on file rows** — unchanged shapes
  and aria-labels.
- **`role="tree"` on `<ul>` + `role="treeitem"` per `<li>`** —
  added (the adapter expects ARIA-tree semantics; the spec
  asserts `[role=treeitem]` directly). Earlier markup used
  plain `<li>` without explicit roles.
