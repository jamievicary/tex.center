## Diagnosis

`tests_gold/lib/src/authedCookie.ts:76` defaults the database
name to `"postgres"` when `TEXCENTER_LIVE_DB_NAME` is unset. The
actual app DB on Fly is `tex_center` (per
`creds/fly-postgres.txt`: the per-app role and `DATABASE_URL`
secret both target `tex_center`). The Fly Postgres cluster also
ships a default `postgres` system DB which is empty of app
tables â€” that's what the live-pipeline job has been connecting
to. The fixture's connection succeeds (superuser auth works
against any DB on the cluster), but every `INSERT INTO sessions
â€¦` then errors with `relation "sessions" does not exist`.

This default has been wrong since the helper was introduced;
it stayed invisible because the live-pipeline job only started
running this iteration. Iter 157's manual probe used a
hand-set `DATABASE_URL` and exercised the WS path, not
`mintSession`, so it never hit the table-resolution failure.

## Fix

`tests_gold/lib/src/authedCookie.ts:76` â€” change the default from
`"postgres"` to `"tex_center"`. The happy-path unit test
`tests_gold/lib/test/authedCookie.test.mjs:51` flipped to match,
locking in the new default.

Minimal targeted change: do NOT add an env var to the workflow
or change schema/user defaults. The default in code is the right
place â€” local invocations of the same helper would silently
mis-target the empty `postgres` DB otherwise.
