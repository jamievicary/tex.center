## Goal for this iteration

M13.2(a) first slice. Iter 237 retargeted the `first-text-paint`
mark and chose the SSR-seed direction; iter 237's notes named iter
238 as the implementation. The iter-236 timeline pinned ~11.5 s in
the `route-mounted → ws-open` interval (per-project Machine cold
start gates the WS upgrade). For a fresh cold project the seed is
just the canonical `MAIN_DOC_HELLO_WORLD` template, so the control
plane can render it without waiting on the sidecar.

## Approach

Visual seed only (no Y.Doc insert). Two reasons:

1. **CRDT duplication risk.** A local-side `Y.Text.insert(0, …)`
   signs the insert with the client's `Y.Doc#clientID`; the
   sidecar's identical seed insert (`apps/sidecar/src/persistence.ts:284–294`)
   signs with a different `clientID`. On initial sync, Yjs has no
   way to dedupe two independent inserts at position 0 — they
   merge as a concatenation. Visual-only seeding sidesteps this:
   the local Y.Doc stays empty, the sidecar's first `doc-update`
   establishes the authoritative state, and the placeholder is
   replaced by the real CodeMirror when `snapshot.hydrated` flips.
2. **Live-spec compatibility.** Several live specs
   (`verifyLiveFullPipeline.spec.ts`, GT-5, etc.) wait for
   `.cm-content` and then click/type into it. If the placeholder
   carried `.cm-content`, typing would go to a `<pre>` and be
   silently dropped before the real CodeMirror mounts. Using a
   distinct `editor-seed` class preserves those specs unchanged —
   they continue to wait for the real mount.

GT-6 was the only spec that needed updating: its user-visible
contract is "the source is *visible* in the editor pane quickly",
not "the source lives in `.cm-content` quickly". Polling `.editor`
textContent expresses the contract without overconstraining the
implementation.

Gate: presence of a `machine_assignments` row for the project. If
absent, no WS has ever upgraded → sidecar has not diverged from
the template → seed is safe. Once a row exists, the gate flips off
and the page falls back to the existing `.editor-placeholder` until
hydrate.

## Notes for future iterations

- **Next M13.2 step:** widen the seed gate to non-fresh projects.
  For a project with a `machine_assignments` row, the sidecar's
  authoritative source may differ from the template. SSR needs to
  fetch the current `main.tex` from the blob store. Currently
  blocked on the blob store being per-Machine; lift it to a shared
  R2 bucket (schedule alongside M11.5). Until then the placeholder
  remains "cold-fresh-project only".
- **Possible refinement:** also surface the file list in SSR so
  the FileTree paints with the real list before WS hydrate (cuts
  the "tree shows only main.tex" period for projects with
  multiple files). Low-priority; the FileTree's pre-hydrate
  default of `[MAIN_DOC_NAME]` matches every cold-fresh project,
  which is the only case currently seeded.
- **Cleanup-then-reopen edge:** if a test (or a future idle-reap
  path) deletes the `machine_assignments` row, the SSR gate
  re-fires and shows the template again. Benign while the blob
  store is per-Machine (the user's edits are gone too); once
  shared, the gate must flip from "no assignment" to
  "no persisted blob".
