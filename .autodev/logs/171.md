## Goal for this iteration

FREEZE-lift criterion (b) per `162_answer.md` and the iter-171+
slot in PLAN: add a reused-pre-existing-project sibling to
M8.pw.4. Iter 170 confirmed (a) — `verifyLiveFullPipeline.spec.ts`
green on the fresh-seed path — but the iter-162 user report
demonstrated edit→preview can be broken on the reused path while
the fresh-seed spec stays green. Those two lifecycles diverge in
`upstreamResolver` (existing `machine_assignments` row vs.
fresh insert), in the sidecar's `persistence.ts`
(existing main.tex blob vs. seed-on-first-compile), and in Machine
state (long-running/idle-stopped vs. cold create). Per-iter gold
needs both.

## Approach

New Playwright spec file under `tests_gold/playwright/`. Uses the
same `authedPage` + `db` fixtures as the fresh-seed sibling so
auth/cookie/DB plumbing is shared. Differences vs. the sibling:

- Fixed-UUID project row (`00000000-0000-4000-8000-000000000001`)
  inserted idempotently via Drizzle's `.onConflictDoNothing()`.
  The all-zero prefix makes it obvious in DB inspection that
  it's a fixture row, not real user data.
- No `afterEach` deleting the project, no `cleanupProjectMachine`
  call. The whole point is the row, its Yjs doc state, and its
  per-project Machine persist across iterations.
- `Ctrl+A` + `Backspace` before typing, to overwrite accumulated
  source from prior iterations. Without this the Y.Doc grows
  unbounded and the typed-LaTeX stops being a valid compile
  unit after enough runs.
- Same assertions as the sibling: pdf-segment binary frame
  (wire-level proof) plus non-near-white PDF.js canvas pixel
  (render-level proof).

`tests_normal` lock-in guards the contract so it can't quietly
drift back into a fresh-seed spec or grow a teardown that erases
the persistence the spec exists to exercise.

## Notes for future iterations

- **If the new spec fails on the harness's first gold run,** the
  diagnostic flowchart in PLAN's "Iter 170+ — Old slot, kept for
  the diagnostic flowchart" section applies as-is — same
  `ws_proxy` + sidecar log shape. Branch (a) of `162_answer.md`
  (Machine OOM'd / running a stale image / etc.) is the most
  likely class because the long-lived per-project Machine for
  this fixture project may have stale state from previous
  iterations' typing.
- **If green:** both FREEZE-lift criteria are satisfied. The next
  iteration can lift the FREEZE block at the top of PLAN with an
  explicit edit, then unblock iter-166+ save-feedback UI work and
  the resumption of N%10 maintenance cycles.
- **Doc-size drift risk:** Yjs ops accumulate even with
  select-all + type-over (the history grows). For a single
  fixture project over hundreds of iterations this is probably
  fine, but if the Y.Doc ever gets large enough to slow the
  spec's first-frame latency a future iteration could reset it
  by deleting the row once (forcing fresh seed) or by adding a
  sidecar-side admin endpoint to truncate the Y.Doc.
- **Yjs select-all behaviour:** the spec relies on CodeMirror's
  default `Control+a` binding selecting all. If a future
  `apps/web` change rebinds it (unlikely — it's the standard
  cm6 default), the type-over becomes append-and-this-spec
  drifts back into the unbounded-growth failure mode. Not
  worth a separate lock-in test today, but worth knowing.
