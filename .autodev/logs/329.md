## Goal for this iteration

PLAN priority 1 was "scrape `flyctl logs` to find the dominant
cold-start phase". Orienting against live state revealed a far more
urgent blocker: **every deploy since iter 323 has been failing** in
the `runtime image smoke` step with
`[commonjs--resolver] Failed to resolve entry for package
"@tex-center/blobs"`. Iter 322 added `apps/web/src/lib/server/
blobStore.ts` as dark code; iter 323's cold-storage cutover wired
`coldSourceFor` into `+page.server.ts` (the SSR seed-doc chain),
making the new `@tex-center/blobs` import reachable from
`apps/web`'s vite build graph. The Dockerfile at
`apps/web/Dockerfile:46` says "the sidecar and blobs package are
not in apps/web's import graph" — true at iter 322, stale by iter
323 — and accordingly does not `COPY packages/blobs/` into the
builder stage. `pnpm install` resolves the workspace symlink
(because `packages/blobs/package.json` *is* copied for the manifest
pass), but the symlink target is an empty directory, so vite finds
no entry to resolve.

Net effect: prod is frozen at iter 322 since 2026-05-16T00:51Z.
All instrumentation since (iter 323's seed cutover, iter 328's
per-phase cold-start log, the `BLOB_STORE=s3` cutover the live
secrets are configured for) is dark. The cold-start scrape this
iteration was meant to do reads pre-instrumentation logs — there
are no `phases` fields to find. Fix the deploy first; let the next
iteration land the scrape against the actually-instrumented
sidecar image.

## Approach

- `apps/web/Dockerfile`: add `COPY packages/blobs/ packages/blobs/`
  to the source-copy block alongside `auth/`, `db/`, `protocol/`.
  Update the comment above the block to reflect that `blobs` is
  now in the import graph (via `coldSourceFor`).
- No source changes; this is a build-graph fix only.

## Actions

- `apps/web/Dockerfile`: added `COPY packages/blobs/ packages/blobs/`
  to the source-copy block. Updated the comment to record that
  blobs joined the web import graph in iter 323 via `coldSourceFor`
  and the seed-doc resolver.
- `tests_normal/cases/test_web_dockerfile.py`: new
  `test_source_copy_covers_web_workspace_deps`. Parses
  `apps/web/package.json` for `workspace:*` deps, maps each via
  `name → directory` from each workspace package's package.json,
  truncates the Dockerfile at the first `RUN pnpm ... build` line,
  and asserts each web workspace dep has a `COPY <pkg>/ <pkg>/`
  before the build step. Pinned against the deleted-blobs negative
  case to confirm it catches the iter-323 → iter-328 regression
  class. The existing `test_install_layer_covers_every_workspace_package`
  only checks manifest-COPY, which is satisfied even when the
  source is missing (pnpm install succeeds; vite build fails).

## Notes for future iterations

- **Watch the next deploy.** Iter 329's commit triggers the
  GitHub Actions `Deploy to Fly` workflow; the iter-328 instrumentation
  (`sidecar-listening` + `phases`) only becomes scrape-able after
  that succeeds. Iter 330 should `gh run list -R
  jamievicary/tex.center --workflow=deploy.yml --limit 5` as the
  first sanity check and only then proceed to the cold-start
  scrape.
- **Cold-start scrape recipe (deferred from iter 328).** Once
  iter 329 deploy goes green, run a single fresh cold start
  (`fly machine stop` the per-project Machine, click into the
  editor, wait for `compile ok`), then `flyctl logs -a
  tex-center-sidecar --since=10m` filtered to that project's
  Machine. First `compile ok` reveals the dominant phase. Likely
  suspects in descending order: supertex daemon spawn + format
  load, Tigris hydrate, checkpoint restore, Fly Machine boot.
- **Why this happened.** The Dockerfile comment said the blobs
  package was not in the web import graph. That was true in iter
  322 when blobs was added as dark code, became false in iter 323
  when `coldSourceFor` was wired into the SSR resolver chain, and
  rotted unnoticed for 5 iterations. The lesson is that comments
  asserting "X is not used" are tripwires: when X *becomes* used,
  the comment is the documentation that misleads the next
  reviewer. The new test removes the trip wire entirely.
