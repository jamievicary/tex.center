## Goal for this iteration

M15 **Step A** — instrument the sidecar with structured logging
of (a) per-compile source content and (b) per-round daemon
stdin / stderr lines, so M15 Step C can scrape `flyctl logs` and
distinguish "supertex misbehaves" from "client cursor lands past
`\end{document}`" outcomes for `verifyLivePdfMultiPage`. This is
the head of the active priority queue per iter-285 PLAN reset
(see `.autodev/discussion/284_answer.md`).

Iter 285 was discussion mode, so this iteration also discharges
the maintenance obligation status check: iter 280 was the last
N%10==0 refactor, iter 281 the last N%10==1 plan-review — both
within the 3-iteration deferral grace, so no maintenance carry
debt forced a different choice.

## Approach

1. Add a structured-log sink type `CompileDebugLog` (shape mirrors
   pino's `info(fields, msg)`) and thread it as an optional
   `SidecarOptions.compileDebugLog`.
2. Default-wire the sink to `app.log.info` whenever
   `DEBUG_COMPILE_LOG` env is unset or anything other than `0` /
   `false` — the spec calls for default-ON while M15 is open.
3. Extend `CompilerContext` with an optional `log: CompileDebugLog`
   so the per-project log sink reaches the daemon factory without a
   new singleton.
4. In `runCompile`, after the `p.text.toString()` snapshot, emit a
   `compile-source` record with `sourceLen`, `sourceBytes`,
   `sourceSha256`, `sourceHead` (first 80 utf8 bytes), `sourceTail`
   (last 80 utf8 bytes), and `endDocPos` (byte offset of
   `\end{document}` or -1).
5. In `SupertexDaemonCompiler`, accept a `log` + `projectId` option;
   emit `daemon-stdin` once per `recompile,…` write (with
   `target`, `round`, `sourceLen`) and a `daemon-stderr` per line
   in `forwardStderrLines`.
6. Lock the contract end-to-end with a recording-logger test that
   exercises both the server compile-source path and the daemon
   stdin/stderr path against the existing fake-daemon shell.

### Design notes

- The sink type is intentionally a callback rather than a Pino
  instance. Tests record into an array; production binds to
  `app.log.info`. No fastify-logger plumbing required, no pino
  test transport, no JSON-roundtrip in the test path.
- `endDocPos` is computed against the utf8 bytes (not the
  string) so it survives any future multibyte content. Today
  source is ASCII so the values would coincide; the byte-offset
  shape future-proofs Step C log-scrape logic that may want to
  index into the source by byte.
- The daemon's existing `process.stderr.write("[supertex-daemon
  stderr] LINE\n")` and `[supertex-daemon event] EV` writes are
  retained alongside the new structured records. They're useful
  in raw `flyctl logs` (no JSON-shape parser needed) and the
  cost is one extra line per stderr line. If they prove noisy
  in Step C analysis, deleting them is mechanical.
- Default-on rationale: the explicit M15 Step C plan needs these
  records on the next deploy; a deploy-time env-flag flip is one
  more thing to forget. If the volume turns out to be too high
  in production, the trace volume per compile is bounded by
  source size for `compile-source` and one + N-stderr-lines for
  the daemon, so it's manageable.
