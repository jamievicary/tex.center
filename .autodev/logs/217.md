## Goal for this iteration

Execute the probe iter 216 queued: drive `supertex --daemon`
directly with an oversize `T` parameter against a short
document and watch for SIGABRT. Independent of browser path,
cheap, proves-or-kills the strongest current hypothesis behind
the GT-7 daemon-crash regression.

## Approach

Added a new gold case `test_supertex_oversize_target`
(`tests_gold/cases/test_supertex_oversize_target.py` +
`tests_gold/lib/test/supertexOversizeTarget.test.mjs`) following
the existing `test_supertex_daemon_real` shape. Two probes in
the same script:
1. Issue `recompile,T` rounds with T ∈ {3, 5, 10, 100} against
   a 2-page fixture, asserting the daemon survives each round
   (no protocol violation / child-exited / stdin-not-writable
   error). Includes a baseline T=1 round before, and an edited
   liveness round after.
2. Apply a single large-paste edit growing the source by 30
   `\newpage X` lines and issue `recompile,1`, asserting the
   same survival shape.

## Findings

- **The oversize-T hypothesis is wrong.** `supertex --daemon`
  silently accepts T values far past the document's actual
  page count (tested up to 100 against a 2-page doc) and
  completes the round normally.
- **The large-paste-of-newpages hypothesis is wrong.**
  Growing the source by 30 `\newpage X` lines in a single
  edit followed by a single `recompile,1` runs to completion
  with all 32 shipouts emitted; no crash.
- **Unexpected observation:** the daemon's stderr includes
  lines like `supertex: edit detected at .../main.tex:49`,
  i.e. it watches the source file in addition to processing
  stdin commands. The iter-215 claim ("stdin-driven only, does
  not auto-reload on disk edits", `214_answer.md`) is
  therefore wrong. This re-opens the iter-213-era hypothesis
  that the crash involves a race between the sidecar's
  `writeMain(source)` and the subsequent stdin
  `recompile,T\n` — the file-watcher may already have
  scheduled a round by the time the stdin command arrives.

## Notes for future iterations

- Next probe: drive the daemon directly with rapid back-to-back
  source-file writes (no intervening stdin commands), to see
  whether the file-watcher re-enters and trips an assert.
  Then a follow-on probe interleaving a disk write with a
  near-simultaneous stdin `recompile`.
- The new gold case stays in the tree as a regression-lock
  on oversize-T tolerance and large-paste tolerance. If those
  invariants later regress (e.g. after a supertex upstream
  bump) we'd catch it before users do.
- The coalescer non-overlap unit test remains a worthwhile
  confidence follow-up but is now ranked behind the
  file-watcher race probe.
