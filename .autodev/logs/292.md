   false-negative on the new column.
2. **createProject.** `CreateProjectInput.seedMainDoc?: string`;
   INSERT writes to `seed_doc`. New helper
   `getProjectSeedDoc(db, id): Promise<string | null>`.
   `projects-pglite.test.mjs` adds a round-trip case at the END
   of the file so the existing `listProjectsByOwnerId` /
   `listAllProjectIds` invariants aren't perturbed.
3. **Sidecar.** `createProjectPersistence` accepts
   `seedMainDoc?: string`, used in place of
   `MAIN_DOC_HELLO_WORLD` for both in-memory and blob-store
   first-hydration paths. The override is consulted exactly once
   (when no `main.tex` blob exists); existing-blob hydration is
   untouched, so an iter-167-style "trim trailing newline"
   regression of the canonical template would still surface via
   the existing locks (cases 1‚Äì5 of `persistenceSeed.test.mjs`).
   `apps/sidecar/src/server.ts` decodes `SEED_MAIN_DOC_B64` once
   at boot and passes it to every `createProjectPersistence`
   call. Cases 6/7/8 in `persistenceSeed.test.mjs` pin the
   override behaviour.
4. **Upstream resolver.** `UpstreamResolverOptions` gains
   optional `seedDocFor: (id) => Promise<string|null>` plus
   `onSeedDocError` reporter. Called from `ensureMachineId`
   strictly on first `createMachine` (no cached assignment) ‚Äî
   the seed is a one-shot input baked into immutable Machine env,
   so a cached/started Machine never re-consults it. A thrown
   `seedDocFor` is logged and treated as "no seed"; an upgrade
   must not be blocked by a transient seed-lookup DB error.
   `upstreamResolver.test.mjs` cases 12/13/14/15 pin:
   non-null ‚Üí env populated; null ‚Üí no env entry; throw ‚Üí
   onSeedDocError fires + no env entry + upgrade still succeeds;
   cached Machine ‚Üí seedDocFor not called.
5. **Wiring.** `upstreamFromEnv.ts` propagates `seedDocFor` from
   `UpstreamFromEnvDeps`. The production entry
   `apps/web/src/server.ts` supplies it as a thin closure over
   `getProjectSeedDoc(getDb().db, id)`.
6. **Gold spec.** `verifyLivePdfMultiPageSeeded.spec.ts` creates
   a project with `seedMainDoc: STATIC_TWO_PAGE` (the same
   5-line 2-page LaTeX used by the existing
   `verifyLivePdfMultiPage` static case), opens it, waits for
   the initial pdf-segment, asserts ‚â•2 `.pdf-page` wrappers OR a
   single canvas > 1.8√ó viewport height. Mirrors the
   viewer-agnostic assertion shape of the existing M15 pins so
   a single eventual fix is likely to green both. No editing
   keystrokes at any point.

**Encoding choice (base64).** Env vars are technically allowed
to carry newlines via the Fly Machines API JSON shape, but the
encoding has historically been ambiguous across runtimes. Base64
is cheap (~33% expansion on a ~150 byte seed = ~200 bytes of
env), unambiguous, and aligns with Fly secrets conventions. The
sidecar's decode is a single `Buffer.from(raw, "base64")
.toString("utf8")` with a try/catch that falls through to the
default seed on failure rather than crashing.

**Why env var, not sidecar‚Üíweb pull.** The seed is needed at
the exact moment of first hydration, before the sidecar has any
viewers (and thus before any HTTP round-trip can be issued
correlated with a viewer connection). Env at Machine creation
time means the bytes are sitting in `process.env` by the time
`getProject()` first runs. The alternative ‚Äî sidecar fetches
seed from web via internal HTTP ‚Äî adds a round-trip on the
cold-start critical path and a new internal-auth surface. Env
is strictly simpler.

**Immutability is fine.** Fly Machine env is immutable for the
Machine's life. But the seed is a one-shot first-hydration
input: once the sidecar persists the seeded `main.tex` blob,
every subsequent hydration (including on the same Machine,
across suspend/resume) sees `hasMainBlob === true` and ignores
the env entirely. So the "env baked at create time" property
matches the "seed used once" semantics exactly.
