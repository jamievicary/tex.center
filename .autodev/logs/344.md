## Finding

The iter-343 sidecar deploy succeeded (workflow `25956320984`,
finished ~07:38 Z) and `SIDECAR_IMAGE` was pinned to the iter-343
digest, which triggered a `tex-center` redeploy that settled at
07:38:21 Z — well before the iter-344 gold pass at 08:28 Z. So the
iter-343 fix was unambiguously on the wire when gold ran. Confirmed
via `flyctl status -a tex-center-sidecar` (Image
`deployment-01KRQVEM1D1JH53C8432QZK75H`) and `flyctl secrets list
-a tex-center` (`SIDECAR_IMAGE` digest current).

`last_gold_output.txt` shows the iter-344 gold pass still RED on
GT-9 + GT-6-stopped + GT-6-suspended, all with the same failure
shape:

- `verifyLiveGt9StoppedPreservesEdits` — only two phase log lines
  emitted (`phase=project-created`, `phase=1-cold-start:goto`),
  then 60 s `cmContent.waitFor({state:"visible"})` timeout.
- `verifyLiveGt6LiveEditableStateStopped` — 60 s timeout, same
  shape.
- `verifyLiveGt6LiveEditableState` — 40 s test-level timeout,
  same shape.

So the iter-343 hypothesis (transient cold-reopen WS open-then-close
→ 5 s suspend self-fire) was either wrong or insufficient. The
fresh-project cold-start (`verifyLiveGt6FastContentAppearance`)
*did* pass — but that spec waits on the SSR-rendered `.editor`
pane, not `.cm-content`. The `.cm-content` element only mounts
once Yjs has hydrated, so the GT-9/GT-6 failures specifically
point at the WS handshake / Yjs sync path, not at image-pull or
SSR latency.

## Approach

We have zero signal beyond `phase=1-cold-start:goto`. Before
proposing another fix, instrument GT-9's phase-1 wait so the next
gold pass tells us *which* stage is hanging: WS upgrade, ws-open
mark, Yjs hydrate, CodeMirror mount, page errors, or something
else. The diagnostic is gold-only and gates nothing in production
code.
