## Goal for this iteration

M15 **Step B** — make `verifyLivePdfMultiPage.spec.ts`
shape-honest. After the keyboard sequence that inserts the
multipage body, immediately read the `.cm-content` source and
assert `\newpage` appears before `\end{document}`. If the
assertion fails the message names the cursor-past-`\end{document}`
failure mode directly and emits the full source. Also widen the
existing "no segment arrived" failure path to emit the full final
source (was: 40-byte tail). Per Step B of `284_answer.md`
§"Resolution / plan", queued as the head of the active priority
queue after iter 286 landed Step A. No upstream / vendor changes
this iteration — Step C (deploy sidecar + scrape `flyctl logs`)
follows.

## Approach

1. Inside the test, extract a small `readCmSource()` helper that
   joins `.cm-line` `textContent` with `\n` (CodeMirror renders one
   `.cm-line` per logical line; `.cm-content`'s direct `textContent`
   loses newlines).
2. After the keyboard sequence, poll up to 3 s for `\newpage` to
   appear in the joined text — `keyboard.type` resolves once the
   input events have dispatched, but `.cm-line` DOM may need one
   paint tick to materialise the new virtual lines.
3. Compute `newpageIdx` / `endDocIdx` and `expect(...).toBe(true)`
   with a failure message that explains the failure mode in plain
   English and dumps the full source.
4. In the existing post-typing "no segment arrived within
   `COMPILE_BUDGET_MS`" branch, replace the 40-byte
   `cmContentTail` with `finalSource=…` from the same helper. The
   tail couldn't disambiguate "body before vs after
   `\end{document}`"; the full source can.

### Design notes

- The `.cm-line` join with `\n` is the simplest robust way to
  reconstruct CodeMirror's logical text from the DOM. We don't
  reach into the `EditorView` instance — there's no public hook
  exposing it to the page test, and even if there were, the
  reconstruction-from-DOM path is what an end-user would see
  (and matches what the typing-shape question actually asks).
- For a 161-byte document with ~7 logical lines, CodeMirror's
  viewport virtualisation isn't a concern (all lines fit the
  viewport). If a future spec stresses scroll-virtualised content,
  the join will need to be supplemented by an EditorView readback.
- The pre-assert is intentionally placed *before* the long
  `COMPILE_BUDGET_MS` polling loop. Under outcome (β) per
  `284_answer.md` ("body lands past `\end{document}`"), the
  spec previously failed only after 180 s on a
  no-segment-arrived timeout with a 40-byte tail — useless. With
  the pre-assert, that scenario fails in <5 s with a clear
  diagnosis.
- The parallel "CodeMirror API positional-anchor" spec
  (`284_answer.md` Step B, second part — optional) is deliberately
  not added. It's only useful as a control once Step C names
  outcome (β); writing it now without knowing the outcome would
  pre-commit to the cursor-placement hypothesis.
