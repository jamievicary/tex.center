## Goal for this iteration

M23.5 — in-place `Y.Text` edit mirror for non-main project files.
M23.2 (iter 314) carried structural ops + cold-boot rehydration to
disk; M23.4 (iter 315) pinned the cold-boot 2-file `\input{sec1}`
compile end-to-end. The remaining hole is the "user opens sec1.tex
and types" path: a client doc-update mutates the Y.Text but no
persistence op fires, so the on-disk file stays stale and the next
compile reads pre-edit content. Iter 314's notes spelled out the
fix shape (per-file `Y.Text.observe`, unsubscribe on
delete/rename-old), and the M23.2 "(A)"-race write-up identified the
mitigation: the observe handler filters local-origin transactions
and the unsubscribe + flush sequence runs before the workspace
structural op so a late observer-write can't resurrect a deleted
path.

## Approach

- `apps/sidecar/src/persistence.ts`:
  - Track three per-name maps: `observers` (unsubscribe thunks),
    `writeInFlight` (current drain promise), `pendingObserveWrite`
    (set of dirty names).
  - `subscribeObserve(name)` attaches a `text.observe(handler)`
    that early-returns on `event.transaction.local` (our own
    `doc.transact` calls always set this true; client
    `Y.applyUpdate(doc, update, client)` sets it false — verified
    empirically against `yjs@13` before coding).
  - `scheduleObserveWrite(name)` adds to `pendingObserveWrite` and,
    if no chain is in flight, kicks a `run()` loop that drains
    while-pending. Each iteration re-reads
    `doc.getText(name).toString()` so a burst of 10 keystrokes
    collapses into one in-flight + one final-state write.
  - `unsubscribeObserve(name)` + `flushObserveWrites(name)` run
    before the workspace structural op in `deleteFile` and
    `renameFile` (old name); `subscribeObserve` runs after
    successful `addFile`, after successful rename (new name), and
    inside the hydration block for each rehydrated non-main file.
- `apps/sidecar/src/server.ts` `runCompile`: comment updated to
  point at M23.5 covering in-place edits — no code change required,
  the existing `writeMain(source)` and the persistence-owned
  non-main mirror together cover every reachable file shape now.
- Locks: new `apps/sidecar/test/serverObserveMirror.test.mjs` with
  4 ordered sub-cases on a single boot:
  1. Hydrate-then-edit — pre-seeded `sec1.tex` blob reaches disk
     via hydration; client edit lands at correct path.
  2. Add-then-edit — `create-file` → empty disk file → client
     edit body → disk reflects body.
  3. Rename-resubscribe — `rename-file` → old path gone, new path
     with body → client edit at new name → disk converges; old
     path stays absent.
  4. Delete-no-resurrect — schedule an edit then `delete-file`;
     after a 50 ms settle, file stays gone (proves the flush
     ordering keeps observer-writes from out-racing the delete).
- Registered the new test in `tests_normal/cases/test_node_suites.py`.

## Notes for future iterations

- The observe handler does not coalesce across files. A user
  rapidly switching files mid-burst gets one in-flight write per
  file. That's fine for typical interactive cadence; if a future
  pathology shows up (many-file rapid burst saturating disk
  bandwidth) a global semaphore on `writeInFlight` would cap
  concurrency without changing the per-file invariant.
- The M23.2 hand-off comment in `runCompile` is now load-bearing:
  any future "mirror non-main from compile time" temptation should
  be rejected with a pointer to the (A) race (iter 314 problem
  notes) and a reminder that persistence-level observe-mirror
  preserves disk-level write→delete ordering, which a compile-time
  loop does not.
- M23 is closed; the natural next pick-up per PLAN priority queue
  is **M22.2** (GT-F local Playwright cases for debug toasts) or
  **M22.4b** (wire-shipoutPage batch — protocol header 13→17,
  segment toast prefix). M22.4b touches the binary protocol header
  and is the higher-coordination slice.
