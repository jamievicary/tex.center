## Goal for this iteration

PLAN priority 1: cold-start latency reduction toward closing
`verifyLiveGt6LiveEditableStateStopped`. Iter 327 measured ~89 s from
dashboard click to first pdf-segment on a fresh cold start; the 60 s
test budget is unreachable until the dominant term is identified and
closed. This iteration lands the instrumentation pre-requisite: log
per-phase elapsed inside the sidecar's `runCompile` (hydrate / restore
/ writeMain / persist / compile) and a one-shot `sidecar-listening`
record at process startup. With these on every Fly Machine, the next
iteration can scrape `flyctl logs` for a single fresh cold start and
see exactly which phase dominates.

Pure instrumentation: no behaviour change, no surface-area shift on
the wire, no test-fixture churn. The existing `compile ok` log gains
a `phases` field; tests don't assert on log content.

## Approach

- `apps/sidecar/src/index.ts`: log `sidecar-listening` once `app.listen`
  returns, carrying `port`, `host`, `bootElapsedMs` (since module
  load via `performance.now()`). This pin gives an absolute marker
  for "Fly machine up + Fastify accepting connections" that pairs
  with the Machine-create / image-pull window visible in
  `flyctl machine status` history.
- `apps/sidecar/src/server.ts` (`runCompile`): instrument each await
  with `Date.now()` bookends, fold the resulting `{ hydrate,
  restore, writeMain, persist, compile }` map into the existing
  `compile ok` record under a `phases` key. Failure paths keep their
  existing log shape — only the success branch carries phases. The
  hydrate and restore promises are one-shot per project, so the
  *first* compile shows their real numbers and subsequent compiles
  show ~0 — exactly the slicing we want.
- No persistence-interface change. `PersistenceLogger` stays
  `{ warn }` so the existing test fixtures (which throw on any
  unexpected `log.warn`) need no edits.

## Actions

- `apps/sidecar/src/index.ts`: captured `boot start` at module load
  (`performance.now()`) and logged `sidecar-listening` after
  `app.listen()` with `bootElapsedMs`.
- `apps/sidecar/src/server.ts`: re-shaped `runCompile` to bookend
  each phase, log `phases: { hydrateMs, restoreMs, writeMainMs,
  persistMs, compileMs }` alongside the existing `elapsedMs` on
  the success branch.

## Tests

- `bash tests_normal/run_tests.sh` → 209 OK (~71 s). No test

## Notes for future iterations

- **Cold-start scrape.** Next iteration should run a single fresh
  cold start (`fly machine stop` the per-project Machine, click into
  the editor, wait for `compile ok`), then `flyctl logs -a
  tex-center-sidecar --since=10m` filtered to that project's
  Machine — first `compile ok` reveals the dominant phase. Likely
  suspects in descending order: (a) supertex daemon spawn + format
  load on first compile, (b) Tigris hydrate (list + N gets), (c)
  checkpoint restore, (d) Fly Machine boot itself (visible only in
  the `sidecar-listening` boot timestamp relative to the click).
- **Machine-boot lower bound.** The boot timestamp from
  `sidecar-listening` minus the click time isn't available from
  inside the sidecar — the Fly proxy logs the cold-start time
  separately. For the next iteration's analysis, capture both
  numbers (proxy "cold-start replay" log line + sidecar
  `sidecar-listening` log line) to bracket the boot phase.
