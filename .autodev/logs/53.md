## Goal for this iteration

UX glitch: an authenticated user navigating to `/` (a bookmark, a
post-deploy refresh, a stale tab) sees the "Sign in with Google"
widget — the page the white-sign-in-page constraint (GOAL #1)
demands for unauth visitors. For someone already signed in there
is nothing to do on `/` and the affordance is misleading.

Symmetric to the existing unauth→`/` redirect off `/editor`, mint
the inverse: authed→`/editor` off `/`. Lives in `hooks.server.ts`
so it intercepts before SvelteKit serves the prerendered HTML.

## Approach

- Add a `SIGN_IN_PAGE_PATHS` set (just `/` today; pluralised to
  leave the door open for future variants like `/signin`).
- After session resolution, if `session !== null` and the request
  is a `GET` for one of those paths, return a 302 to
  `/editor` (the canonical signed-in home, same constant as
  callback success in `auth/google/callback`).
- `GET`-only so `POST`s to root (none today, but the constraint
  costs nothing and avoids surprising redirect-on-POST behaviour
  if a future form ever posts to `/`).

GOAL #1 still holds: the unauthenticated white page is unchanged,
served exactly as before. The redirect only fires after a session
is resolved.
