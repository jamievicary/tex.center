## Goal for this iteration

M4.2.2 — wire `@tex-center/db` into the sidecar so the running
server holds a `DbHandle` keyed off `DATABASE_URL`, with the
connection pool tied to Fastify's start/stop lifecycle. The web
app uses `adapter-static` and has no SSR server, so it gets no
runtime DB seam (a future iteration adds DB access on the
control-plane side once that exists). This iteration adds the
plumbing only; no query code consumes the handle yet.

M4.2.1 (docker-compose + apply integration) is deferred: the
harness host has no Docker available (`which docker` → not
found), so a docker-backed gold case would either no-op locally
or be skipped behind a guard. M4.2.2 doesn't require Docker and
unblocks downstream auth/session work that needs `app.db`. PLAN
is updated accordingly.

## Approach

1. `apps/sidecar/package.json` — add `@tex-center/db: workspace:*`
   dep so the sidecar can `import { createDb, closeDb } from
   '@tex-center/db'`.
2. `apps/sidecar/src/server.ts`:
   - Extend `SidecarOptions` with `db?: DbHandle` (caller-owned,
     not closed on shutdown) and
     `dbFactory?: (cs: string) => DbHandle` (test seam,
     defaults to `createDb`).
   - At buildServer entry: if `opts.db` set → use it
     (ownedDb=false); else if `process.env.DATABASE_URL` set →
     `dbFactory(DATABASE_URL)` (ownedDb=true); else → null.
   - `app.decorate('db', dbHandle)` so future routes/plugins
     can read `app.db`. Module augmentation declares
     `FastifyInstance.db: DbHandle | null`.
   - Augment the existing `onClose` hook to call `closeDb` when
     ownedDb=true.
3. New test `apps/sidecar/test/serverDb.test.mjs`:
   - No env, no inject → `app.db === null`, `app.close()` clean.
   - Inject fake handle → `app.db === fake`, fake.client.end
     NOT called after close.
   - `DATABASE_URL` set + fake factory → factory called once
     with the URL; `app.db === factory result`; fake.client.end
     IS called once after close.
4. Register the new test in `tests_normal/cases/test_node_suites.py`.
5. Run `pnpm install` to update the lockfile.
6. PLAN: tick M4.2.2 done, note M4.2.1 deferred (no Docker on
   harness host); refresh Current focus.

## Notes for future iterations

- The decorator's module augmentation lives in `server.ts` for
  now because the sidecar is the only consumer. If a future
  package needs `app.db` typed too (e.g., a shared route
  plugin) it should move to a `types/fastify.d.ts` at the
  sidecar root and be picked up via `tsconfig.json`.
- `dbFactory` is intentionally a no-frills `(cs) => DbHandle`
  rather than full `Options` plumbing — production wires
  `createDb` directly. Adding pool/onnotice tuning later means
  shipping it on `createDb`'s options, not on this seam.
- The DATABASE_URL path is unit-tested via the factory spy, so
  the no-Docker constraint doesn't shrink coverage of the
  wiring itself. The remaining hole is `postgres-js` actually
  speaking to a real DB — that's M4.2.1's job.
