## Goal for this iteration

Read the iter-344 GT-9 probe transcript and act on the finding.

## Finding — root cause for GT-9 RED

The probe transcript pins a **persistence-on-disconnect data loss bug**,
not a connection / hydrate / mount problem.

The phase-1 cold-start probe shows the editor mounting fine
(`cm-visible` at 21 s, first pdf-segment at 23 s). The probes also
correctly show `editor:route-mounted` but null marks for `ws-open` /
`yjs-hydrated` — that turns out to be a side issue (the marks aren't
wired in `+page.svelte` yet; the cm-content does still mount once
the WS sync completes); it's misleading but not the GT-9 failure
cause.

The real failure shape is in phase 7. After typing
`preserve-45f334e2-d5c9-4215-bde7-5a4570d9c382` (46 chars), force-
stopping the Machine, and reopening, `.cm-content` rehydrates with
**only** `\documentclass{article}\begin{document}Hello, world!
preserve-45\end{document}` — the sentinel is truncated to its first
2 UUID chars, i.e. only the prefix `preserve-45` (~12 typed chars)
survives the round-trip. Same value at every probe iteration through
the full 180 s budget.

**Mechanism (verified against `apps/sidecar/src/server.ts` +
`compileCoalescer.ts` + `persistence.ts`):**

1. Client types 46 chars at ~12 ms/char → Yjs ops stream to sidecar.
2. First few ops arrive at sidecar; `onTextChange` kicks the
   coalescer; debounce timer (100 ms) fires; `runCompile` starts.
3. `runCompile` reads `source = p.text.toString()` (≈12 chars in)
   then calls `maybePersist()` which captures the **current** doc
   state and PUTs it to the blob (`Hello, world! preserve-45`).
4. While that compile is in flight, the remaining 34 chars arrive at
   the sidecar's Y.Doc. Each kick sets `coalescer.pending = true`.
5. The compile finishes; `.finally` schedules a second compile via
   another 100 ms debounce.
6. The test sees the pdf-segment shipped from step 3 and immediately
   navigates to `/projects`. The WS closes ~1 ms later.
7. `socket.on("close")` → `viewers.size === 0` →
   `coalescer.cancel()` clears the pending timer before it fires.
   **No second compile ever runs. The 34 trailing chars in the
   Y.Doc are never persisted.**
8. Test force-stops the Machine via Fly API ~150 ms later. Process
   dies. In-memory Y.Doc destroyed. Blob still holds only the
   partial source.
9. Test reopens → cold-boot hydration reads the partial blob → the
   editor displays `Hello, world! preserve-45`.

This is a **real product data-loss bug**, not a test artefact: any
user who closes the tab during a typing burst, before the trailing
debounce fires, loses everything typed after the last
debounce-fired compile.

## Fix

Add a final `persistence.maybePersist()` flush on the three exit
paths from "doc has unflushed edits":

1. **Last-viewer WS close** — fire `void persistence.maybePersist()`
   before `coalescer.cancel()`. Handles "user closes tab" and the
   GT-9 force-stop scenario (Machine continues running, persist
   completes in async I/O before any subsequent kill).
2. **Idle-fire (suspend/stop stage)** — call
   `await persistence.maybePersist()` on every project before the
   user `onSuspend`/`onStop` handler runs. Symmetric with the
   existing `persistAllCheckpoints` for compiler state. Handles the
   soft-shutdown path.
3. **Fastify `onClose`** — `await persistence.maybePersist()` for
   every project before destroying the Y.Doc. Belt-and-suspenders
   for `app.close()` callers, including `createStopHandler`'s
   normal path.

Coverage of the GT-9 test scenario: path (1) fires when the test
navigates to `/projects`, kicks off `maybePersist` which captures
the full sentinel state (all 46 chars are already in the sidecar's
Y.Doc by then). Fly's POST /stop sends SIGINT ~150 ms later;
default Fly kill-timeout is 5 s before SIGKILL. Tigris PUT round-
trip is well under that.

## Actions

- `apps/sidecar/src/server.ts`:
  - In `socket.on("close")`, when `viewers.size === 0`: fire-and-
    forget `persistence.maybePersist()` before `coalescer.cancel()`,
    with a `.catch` to log any error.
  - In `createIdleStage` idle-fire: add a project-loop persist
    flush before invoking the handler.
  - In Fastify `onClose` hook: add a project-loop persist flush
    before destroying each project's doc.
- `apps/sidecar/test/serverPersistOnViewerDisconnect.test.mjs`
  (new normal test): client connects, types content via Yjs,
  disconnects within the debounce window; assert blob holds the
  typed content after the disconnect (with a poll budget). Pins
  the fix at the WS-close path.
