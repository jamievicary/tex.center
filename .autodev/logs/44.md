## Approach

- Single workflow file `.github/workflows/deploy.yml`, single job
  `deploy`. Steps: `actions/checkout@v4` â†’
  `superfly/flyctl-actions/setup-flyctl@master` (the action
  Fly's own docs recommend) â†’ `flyctl deploy --remote-only`.
- `--remote-only` because the runner doesn't run Docker locally
  here â€” Fly's builder builds the image. `--config fly.toml
  --dockerfile apps/web/Dockerfile` makes the build context
  explicit even though both are defaults; cheap defense against
  the manifest ever moving.
- `concurrency: group=fly-deploy, cancel-in-progress=false`. A
  rapid double-push must queue, not cancel â€” cancelling mid-
  rollout strands a partially-updated Machine fleet. Queueing
  trades a few minutes of latency for atomicity.
- `timeout-minutes: 20`. Fly remote builds for a Node image are
  typically 2â€“5 min; 20 is a generous ceiling that still bounds
  a stuck job.
- `FLY_API_TOKEN` plumbed through `env:` on the deploy step, not
  the job, so the secret isn't injected into the checkout step.
- Test strategy: parse the YAML and check structural invariants,
  same shape as `test_fly_toml.py` and `test_web_dockerfile.py`.
  We can't shell out to `flyctl` (no creds, no binary on the
  runner) â€” the value the test buys is "drift between
  Dockerfile path / token name / branch trigger and prod
  doesn't ship silently".

## PyYAML quirk

YAML 1.1 parses the bare key `on:` as boolean `True`. PyYAML
follows 1.1. The test accepts either `doc["on"]` or `doc[True]`
so the workflow stays human-readable (quoting `"on":` would also
work but reads oddly). GitHub Actions itself parses with YAML 1.2
semantics where `on` is a string, so the file on disk is correct.
