## Goal for this iteration

**Discussion mode.** Answers `.autodev/discussion/354_question.md`
("`verifyLiveFullPipelineReused` is RED — primary focus"). Full
diagnosis lives in `354_answer.md`. Short version:

- The iter-354 gold transcript shows pdf-segment frames **do**
  arrive (×2) on the reused project. The WS / sidecar layer is
  healthy. **Not Bug B.**
- The failure is `expectPreviewCanvasPainted`'s
  `canvas.waitFor({ state: "attached" })` timing out — the
  `.preview canvas` is never created.
- Locus: `apps/web/src/lib/PdfViewer.svelte`'s `render()` invoked
  as `void render(...)` from the `$effect`, with no `try/catch`.
  PDF.js rejects somewhere in the parse/render chain and the
  promise tail is silently discarded — no log, no UI signal.

The question's directive on "no fresh evidence available" rules
out punting. The instrumentation gap is concrete and the question
explicitly permits landing a probe to capture it on the next
gold pass.

## Approach

Two minimal changes, both in the "landed alongside the answer"
class (`§7 step 5` of `autodev/engineer.md`):

1. **`apps/web/src/lib/PdfViewer.svelte`** — attach a `.catch()`
   to the `render()` call inside `$effect`. On rejection,
   `console.error("[PdfViewer] render failed:", message)` and
   write `host.dataset.pdfError = message`. Clear the dataset
   key on each fresh render so a recovered failure doesn't leave
   stale state.
2. **`tests_gold/playwright/fixtures/authedPage.ts`** — capture
   `page.on('console')` (error-level only) and
   `page.on('pageerror')` into a per-test
   `browserDiagnostics: string[]`. Dump on teardown via the same
   `TEXCENTER_DUMP_WIRE_TIMELINE`-gated `console.log` channel as
   the wire timeline, prefixed `[<spec>] browser diagnostics:`
   so it's greppable per-spec.

No PLAN.md re-prioritisation yet. The decision tree in
`354_answer.md` calls for a follow-up iteration that reads the
captured PDF.js error from the next gold transcript, then either
roots in the FE assembler (`pdfBuffer.ts`) or in sidecar
persisted-cache state. Re-prioritisation belongs in the
fix-shaped follow-up, not in this evidence-gathering probe — the
question's option (b) requires "a concrete next-step plan", which
the answer supplies and the next iteration executes.
