## Approach

The user-reported scenario: scroll so page N is fully visible; page
N+1's top edge sits just below the fold producing a tiny
`intersectionRatio` (5% or less). Today `pickMaxVisible` uses a
strict `> 0` predicate, so the sliver promotes max-visible to N+1.
Promote that predicate to "ratio above a documented threshold":

- Add `MAX_VISIBLE_RATIO_THRESHOLD = 0.1` exported from
  `apps/web/src/lib/pageTracker.ts`. 0.1 = at least 10% of the page
  area in the viewport — matches the IO threshold-step granularity
  the PdfViewer already uses (`Array.from({length: 11}, (_, i) =>
  i/10)`) so the IO callback fires at well-defined crossings.
- Widen `pickMaxVisible(items, minRatio = MAX_VISIBLE_RATIO_THRESHOLD)`
  to filter `ratio <= minRatio`. The `minRatio` parameter recovers
  the prior strict-`>0` semantics if a caller ever needs them
  (callers that explicitly pass `0`); the default flips behaviour.
- `pickMostVisible` is unchanged — it's the dominant-page picker and
  its strict `>0` predicate is still correct (you always want to
  know which page is most-visible, however small the candidate set).
- `PageTracker` calls `pickMaxVisible(items)` with no argument, so
  the default propagates automatically. No constructor change.

Test coverage lock:

- Update the two existing `pickMaxVisible` cases that depended on
  sub-threshold slivers winning (`ratio: 0.05` → 2; `ratio: 0.01` →
  5) — those expectations were the exact bug. Replace with
  above-threshold candidates that exercise the same shape.
- New sliver-suppression cases: 5% intrusion of page 2 → max-visible
  stays at page 1; an all-sliver frame → null; boundary case
  (`ratio === threshold`) → does not count (strict `>` predicate).
- Explicit-override cases: `minRatio = 0` recovers strict-`>0`;
  `minRatio = 0.5` demands half-visible (custom threshold).
- New PageTracker case: walking page 2 from 0.05 → 0.4 → 0.05 →
  asserts maxVisible stays at 1 across the sliver, flips to 2 when
  it climbs above threshold, and falls back to 1 when it retreats.
  This is the exact user-reported scroll sequence, locked in the
  tracker (not just the helper).
- `MAX_VISIBLE_RATIO_THRESHOLD === 0.1` is itself asserted at the
  top of the test so a future iteration that changes the constant
  without changing the test surface gets caught.

PdfViewer.svelte is unchanged: it constructs `new PageTracker()` and
reads `maxVisible` off the update return. The default threshold now
fires correctly under it.

## Notes for future iterations

- **M21.3c remains open.** Today's hardcoded `targetPage: 0` at
  `server.ts:528` means the threshold tightening has no observable
  effect on which segments ship — that wire is a no-op on compile
  pacing. M21.3c is still the right place to capture the
  `daemon-stdin` + `daemon-round-done` transcript for the
  user-reported "edit on hidden page N+2 ships nothing" repro.
  When (and if) we re-enable the per-compile target-page gate, the
  threshold here will start mattering directly. Until then the
  M21.3a fix matters only for the `outgoing-viewing-page` debug
  toast — it'll now stop lighting up N+1 when only N is intended.
- **Threshold choice is on PageTracker, not in PdfViewer.** If a
  future consumer wants a different sensitivity (e.g. an AOI tracker
  that should count any intersection), pass `0` explicitly to
  `pickMaxVisible` rather than re-introducing global config. The
  default is now the load-bearing value for the compile-pacing path.
- **No hysteresis here.** A page oscillating around `ratio === 0.1`
  could in principle flap maxVisible. Not addressing in this slice —
  IO fires at the threshold-step crossings, so flapping would imply
  the user is scrolling exactly through that boundary. If it surfaces
  as a real-world bug, the fix is a separate slice (enter-at-0.1,
  leave-at-0.05 style hysteresis around the threshold).
- **M20.2(d) is the next ordinary slice.** S3 adapter + request-
  recording HTTP stub + `envSelect` cutover. Sketch in `323_log`.
