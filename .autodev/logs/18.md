## Goal for this iteration

Begin M4.2 — Local Postgres bring-up. Pre-docker-compose slice
that lands the runtime seam from the application side: a
`postgres-js` driver dependency in `packages/db`, a
`createDb(connectionString)` factory returning the
`drizzle(client, { schema })` handle that's been promised since
M4.1, a hand-rolled SQL migration loader/applier
(`loadMigrations` + `applyMigrations`) that consumes the
existing `src/migrations/*.sql` files in lexicographic order, and
a `pnpm --filter @tex-center/db db:migrate` CLI that takes
`DATABASE_URL` from env. The migration-loader is testable
hermetically (just reads files); the applier and createDb get
their integration coverage when M4.2's docker-compose lands.
This split keeps the iteration coherent without depending on
docker in `tests_normal/`.

## Notes for future iterations

- `applyMigrations` uses `sql.begin(async (tx) => …)` per
  migration. postgres-js wraps the callback in a transaction
  with a SAVEPOINT-style nested model; that's the documented
  way to get atomicity. The bookkeeping INSERT shares the
  transaction so a partial DDL never leaves a poisoned row.
- `createDb` accepts the postgres-js `Options` directly. The
  CLI passes `{ max: 1, onnotice: () => {} }` because a
  one-shot migrator doesn't need a pool and we don't want the
  notices spam from `CREATE TABLE IF NOT EXISTS` on a fresh DB.
- The `(name PK)` `schema_migrations` table is intentionally
  minimal — no version vector, no `down`. Forward-only by design.
- M4.2.1 is the natural next step: `docker-compose.yml` with
  `postgres:16-alpine`, a gold test that spins it up, runs
  `db:migrate` twice, and asserts (a) every spec table exists,
  (b) the second run reports `skipped` for `0001_initial`. That
  needs the gold runner to start handling docker; today
  `tests_gold/run_tests.sh` is empty/passthrough — adding docker
  orchestration there is the bulk of M4.2.1.
