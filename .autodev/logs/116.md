## Goal for this iteration

Unblock M7.1.3.2.b by landing the **teardown half** as a small,
self-contained wedge. The blocker recorded since iter 114 was "no
cleanup path that destroys the spawned Machine + deletes its
`machine_assignments` row at probe end" â€” without that, a live
WS-upgrade probe would leak a Machine per run.

The destroy + row-delete primitives already exist
(`MachinesClient.destroyMachine` in
`apps/web/src/lib/server/flyMachines.ts:245`, `deleteMachineAssignment`
in `packages/db/src/machineAssignments.ts:77`). What was missing was
the orchestration: look up the assignment, destroy the Machine
(tolerating 404 "already gone"), then delete the row. That fits
cleanly in one iteration with full unit coverage, and the next
ordinary iteration can drop a Playwright spec on top.

Picked over starting the live spec directly because the spec needs
real flyctl-proxied Postgres + a real per-project Machine spawn â€”
much larger surface, and the helper is what would crash repeatedly
during that work if it had bugs. Landing it first with stub-driven
tests pays for itself.

## Approach

`tests_gold/lib/src/cleanupProjectMachine.ts` exposes one async
function over two minimal duck-type interfaces (`MachineDestroyer`,
`AssignmentStore`). Choices:

- **Duck-typing rather than importing `MachinesClient` /
  `flyMachines.ts`.** Keeps `tests_gold/lib/` free of an
  `apps/web/src/lib/server/` import; the real probe wires the
  classes at the call site.
- **404 â†’ success.** "Machine already gone" is the desired
  post-condition. Structural check on `err.status === 404` rather
  than `instanceof FlyApiError`, again to avoid the import.
- **Non-404 â†’ throw and preserve the row.** Keeps the next
  iteration's retry idempotent against a real `machineId` instead
  of an orphan.
- **No assignment â†’ no destroy, but `deleteAssignment` still runs.**
  Cheap idempotency: the helper's contract is "leave a clean
  slate," not "describe what was there."
