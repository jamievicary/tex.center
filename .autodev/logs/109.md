## Problems encountered

- **`authedHome.spec.ts` fails on live.** First attempt at
  exercising the new key ran `authedHome` (authed GET `/` â†’
  expects redirect to `/projects`); live returned 200 with the
  sign-in HTML instead. Root cause: `apps/web/src/routes/
  +layout.ts` sets `prerender = true; ssr = false`, so the `/`
  route is served as a static file in production â€” the server
  hook (and therefore `routeRedirect`) never runs. Local Vite
  dev SSRs everything and masks the bug. The HTTP-level probe
  `GET /projects` is unaffected (that route is non-prerendered)
  and was the right surface for the authed-cookie verification
  this iteration. Logged as M7.1.3.2.c.
- **`flyctl tokens create deploy` denied** (rediscovered from
  iter 106). The personal token's scope can't self-issue. Not
  blocking; the existing `creds/fly.token` works for the
  `flyctl secrets set` rotation.
- **Path resolution under `pnpm --filter exec tsx`.** The seed
  script needed running via `pnpm exec tsx scripts/â€¦` from the
  repo root (not `pnpm --filter @tex-center/db exec tsx ../../
  scripts/â€¦`, which changes cwd to `packages/db/` and breaks
  workspace-relative resolution).
- **Per-app role mismatch.** `authedCookie.ts` defaults
  `TEXCENTER_LIVE_DB_USER=postgres` + `TEXCENTER_LIVE_DB_NAME=
  postgres`, but `flyctl postgres attach` creates a per-app role
  `tex_center` + database `tex_center` with a separate password.
  First playwright run failed at `insertSession` with `password
  authentication failed for user "postgres"`. Setting both env
  vars to `tex_center` (and the password to the per-app role's)
  fixed it. Recorded in `deploy/README.md`.
