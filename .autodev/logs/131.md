## Goal for this iteration

Discussion mode. `131_question.md` reports a second production-down
OAuth-callback bug: after iter 129's `jose` fix, the callback now
reaches `findOrCreateUserByGoogleSub` and the user upsert throws.
Drizzle's `Failed query:` wrapper masks the underlying PG error;
the question asks for the real cause + a targeted fix, and
promotes a verification-surface priority block (M8.smoke.0 →
M8.pw.3 → M8.pw.4) above the remaining M7.x work.

See `.autodev/discussion/131_answer.md` for the diagnosis and
ordering rationale. Summary:

- **Root cause:** iter 109's deploy-verification seed inserted a
  row with `google_sub = 'probe-jamievicary-livefix'` and
  `email = 'jamievicary@gmail.com'`. The first real OAuth callback
  carried the same email but a different google_sub, so
  `ON CONFLICT (google_sub) DO UPDATE` couldn't fire and PG raised
  `users_email_key (UNIQUE email)` first.

- **Discovery path:** `flyctl logs --no-tail` only surfaces ~100
  lines and the window around 07:57:37Z had already rolled past.
  Proxied into `tex-center-db` via `flyctl proxy 5432:5432` and
  read `users` directly — one row, with the placeholder google_sub
  intact. That made the failure mode unambiguous.

- **Live fix (one-shot):** UPDATE'd the seed row's google_sub to
  the real value (`114596996082930464265`). Next OAuth attempt
  hits the conflict-DO-UPDATE branch cleanly; the same user id
  (`7d7a970e-…`) is retained, so `TEXCENTER_LIVE_USER_ID` stays
  valid for deploy-verification mintSession.

- **Structural fix:** `0002_drop_users_email_unique.sql` removes
  the redundant `UNIQUE (email)` constraint; `drizzle.ts:24` lost
  `.unique()`; `schema.ts:39` comment updated. Will apply to live
  on next deploy via `RUN_MIGRATIONS_ON_BOOT=1`.

- **Regression test:** extended
  `users-sessions-pglite.test.mjs` with a third insert that shares
  an email with an existing row but uses a fresh google_sub.
  Asserts a new row is created (pre-migration this raised
  `users_email_key`).

- **PLAN updates:** priority block (M8.smoke.0 / M8.pw.3 /
  M8.pw.4) promoted ahead of M7.4.2 and M7.2; "Current focus"
  rewritten. `FUTURE_IDEAS.md` records the cosmetic OAuth-scope
  broadening as a separate follow-up so it doesn't get bundled.
