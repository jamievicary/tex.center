## Goal for this iteration

Fix the iter-189 gold-test regression in
`tests_gold/lib/test/supertexDaemonReal.test.mjs`. Iter 189
correctly removed `SupertexDaemonCompiler.assembleSegment`'s
`maxShipout < 0` directory-scan fallback, so a no-op recompile
(round-done with no shipouts) now returns
`{ ok: true, segments: [] }` rather than re-shipping stale
chunks. The test, however, was driving the second compile with
the same source — exercising exactly that no-op path — and
asserting `segments.length === 1`. The right fix is to edit the
source between rounds, which is also a better test of "persistent
process survives a second compile" (a real edit, not a no-op).

Also defer the N%10==0 refactor cron one iteration (first defer
since iter-183 took the previous deferred cron) to land this
regression fix; record in PLAN.

While verifying I caught a transient flake in
`apps/sidecar/test/serverBlobs.test.mjs`'s failing-compile case:
the test waited for the blob to be persisted, then synchronously
asserted a `compile-status:error` frame was in the captured
frames array. Blob persistence and the error broadcast are
independent code paths on the sidecar — either ordering is legal
at the wire level. Fixed by folding both signals into a single
`waitFor` predicate. In scope as a co-located anti-flake fix on
the sidecar test surface.
