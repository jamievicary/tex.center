## Goal for this iteration

N=300 → refactor iteration (`N % 10 == 0`). Survey turned up two
related candidates flagged in the iter-299 tail note:

1. `apps/web/src/lib/PdfViewer.svelte` carries a ~100-line inline
   `makeAdapter` function that builds the `FadeAdapter` the
   controller drives. It is DOM-side glue and is the largest
   organic-grown block in the file (275 lines today; this block
   is ~40% of the script section). It already imports the strategy
   constants from `pdfCrossFade.ts`, so its dependencies are clean
   — there is no good reason for it to live inline next to the
   pdfjs render loop.
2. The adapter still adds `pdf-page--enter` / `pdf-page--leave`
   classes to wrappers in `fadeInWrapper` / `fadeOutAndRemoveWrapper`.
   No CSS selector targets either class; nothing in JS reads them.
   Same shape as the iter-299 `pdf-canvas--enter` / `pdf-canvas--
   leave` dead-class removal (left behind by an earlier design
   that never shipped).

and dropping the dead class adds in the same pass:

  pdfjs render loop, observer setup, controller lifecycle, and

  `classList.remove` inside the transitionend handler in

## Approach

- **New file `apps/web/src/lib/pdfFadeAdapter.ts`.** Exports
  `createFadeAdapter(opts)` returning a `FadeAdapter`. Opts:
  `{ target: HTMLElement; observer: IntersectionObserver | null;
  onFadeEnd: (pageIndex: number) => void }`. The `onFadeEnd`
  callback decouples the adapter from the controller's lifetime —
  today's inline closure captures `controller` (set lazily after
  the adapter is built), which is fragile. The new shape lets
  `PdfViewer.svelte` pass a stable lambda
  `(idx) => controller?.onFadeEnd(idx)`.
- **`PdfViewer.svelte` slimmed.** `makeAdapter` deleted; the
  controller construction site becomes
  `new PdfFadeController(createFadeAdapter({ target, observer:
  ensureObserver(), onFadeEnd: (idx) => controller?.onFadeEnd(idx)
  }))`.
- **Dead classes removed inside the new file.** `fadeInWrapper`
  no longer adds `pdf-page--enter` (or removes it on
  transitionend); `fadeOutAndRemoveWrapper` no longer adds
  `pdf-page--leave`.

## Actions

- `apps/web/src/lib/pdfFadeAdapter.ts` (NEW, 121 lines). Exports
  `CreateFadeAdapterOptions` + `createFadeAdapter(opts)` returning
  a `FadeAdapter`. Imports `FadeAdapter` type from
  `./pdfFadeController` (no value imports — keeps the controller
  free of adapter-implementation dependencies). Imports
  `CROSS_FADE_STRATEGY` from `./pdfCrossFade`. Same `void
  leave.offsetWidth` reflow trick before the leaving-canvas
  opacity transition; same `void w.offsetWidth` at the end of
  `commitFadeImmediately`. `onFadeEnd` is the caller-supplied
  lambda — replaces the prior pattern of closing over a
  `controller` variable that was nullable at adapter-construction
  time.
- `apps/web/src/lib/PdfViewer.svelte` (275 → 168 lines).
  - Imports `PdfFadeController` from `./pdfFadeController`
    (type-only import removed; no longer needs `FadeAdapter`).
  - Imports `createFadeAdapter` from `./pdfFadeAdapter`.
  - `CROSS_FADE_STRATEGY` import deleted (no longer used here).
  - Inline `makeAdapter(target)` deleted entirely (~100 lines).
  - Controller construction now reads:
    `new PdfFadeController(createFadeAdapter({ target, observer:
    ensureObserver(), onFadeEnd: (idx) => controller?.onFadeEnd(\
    idx) }))`. The `controller?.onFadeEnd` indirection through a
    lambda is necessary because the local `controller` binding
    is assigned in the same statement; the arrow defers the
    `controller?` read until the adapter actually fires.

  jsdom target, drive it through the adapter API, assert on the
  observable DOM state (inline-style writes, classList, child
  count, event listeners). No new infra needed yet.

  the adapter.** `fadeOutAndRemoveWrapper` runs when a commit
  shrinks the page count (e.g. final page lost on recompile).
  The current Playwright pins exercise commit-with-same-count and
  commit-with-more-pages, never commit-with-fewer-pages. A
  live-side pin (typing `\end{document}` mid-doc to drop the
  trailing pages, then asserting wrapper count converges) would
  close the coverage gap; not on the active queue.
