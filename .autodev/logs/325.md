## Approach

Three new files under `packages/blobs/src` + an envSelect cutover +
a hermetic round-trip test:

1. `sigv4.ts` ‚ÄĒ AWS Signature V4 signing helper. Pure function from
   `{method, url, headers, body, region, service, credentials, now?}`
   to a fully-signed headers map. Uses `node:crypto` (`createHmac`,
   `createHash`) ‚ÄĒ no external deps. Deterministic when `now` is
   passed; otherwise reads `new Date()`.
2. `s3.ts` ‚ÄĒ `S3BlobStore implements BlobStore`. Path-style
   addressing (`<endpoint>/<bucket>/<key>`), matches Tigris's S3
   profile. `put` ‚Üí signed PUT; `get` ‚Üí signed GET (404 ‚Üí `null`);
   `delete` ‚Üí signed DELETE (204 or 404 both succeed, mirroring the
   local-fs idempotent shape); `list(prefix)` ‚Üí paginated
   `?list-type=2&prefix=...` with `continuation-token`, returning
   keys in lex order; `health` ‚Üí signed HEAD on the bucket.
3. `envSelect.ts` ‚ÄĒ recognise `BLOB_STORE=s3`, read
   `BLOB_STORE_S3_ENDPOINT`, `BLOB_STORE_S3_REGION`,
   `BLOB_STORE_S3_BUCKET`, `BLOB_STORE_S3_ACCESS_KEY_ID`,
   `BLOB_STORE_S3_SECRET_ACCESS_KEY`. Fail fast at boot on any
   missing field, naming the field ‚ÄĒ this is the same error shape
   as `BLOB_STORE=local requires BLOB_STORE_LOCAL_DIR`.
4. `test/s3.test.mjs` ‚ÄĒ in-process `http.Server` that emulates the
   tiny subset of S3 we use: PUT/GET/DELETE/HEAD on
   `/<bucket>/<key>`, GET on `/<bucket>?list-type=2&prefix=...`
   returning a minimal `<ListBucketResult>` XML with paginated
   `NextContinuationToken`. The stub also records every request so
   tests can assert canonical request shape (path-style bucket
   prefix, `x-amz-content-sha256` body hash, signed `Authorization`
   header). Round-trip cases mirror `localFs.test.mjs` (put/get,
   missing‚Üínull, overwrite, list prefix, list pagination, delete
   idempotence, health) so the two adapters are interchangeable
   behind the protocol.

Cutover is dark code until production sets `BLOB_STORE=s3`; the
sidecar Dockerfile + Fly secrets stays as a follow-up slice (next
iteration). Today's TS typecheck + unit tests are the gate.

## Actions

- `packages/blobs/src/sigv4.ts` (new). Pure `signRequest({method,
  url, headers?, body?, region, service, accessKeyId,
  secretAccessKey, now?})` returning the headers map with `host`,
  `x-amz-date`, `x-amz-content-sha256`, and `Authorization`
  populated. Uses `node:crypto`; canonical headers sorted
  case-insensitively; canonical query string sorted by key then
  value with RFC-3986-strict percent encoding.
- `packages/blobs/src/s3.ts` (new). `S3BlobStore implements
  BlobStore` with constructor `{endpoint, region, bucket,
  accessKeyId, secretAccessKey, fetchImpl?}`. Path-style
  addressing throughout. `put/get/delete/list/health` route through
  a shared `request(method, key, body?, query?)` helper that calls
  `signRequest` then `fetch`. `list` paginates via
  `?continuation-token=` until `<IsTruncated>false</IsTruncated>`,
  defensively sorts the final result. `delete` swallows 204/200/404
  (mirroring `LocalFsBlobStore.delete`'s idempotent contract).

  implemented yet" throw with full wiring. Five required env vars:
  `BLOB_STORE_S3_{ENDPOINT,REGION,BUCKET,ACCESS_KEY_ID,SECRET_ACCESS_KEY}`.
  Each missing field's error names that field, so a misconfigured
  deploy fails fast at boot with a precise pointer.

## Notes for future iterations

- **M20.3 gold spec is now unblocked.** Production deploy needs Fly
  Tigris creds bound to `BLOB_STORE_S3_*` secrets on both the web
  and sidecar apps; the same `defaultBlobStoreFromEnv` switch wires
  both sides. The gold spec ‚ÄĒ `verifyLiveGt6LiveEditableStateStopped`
  is the existing red ‚ÄĒ should green once Tigris is wired in and a
  stopped Machine's source is reseeded from the blob on cold boot.
  No code change should be needed beyond the env-secret push.
- **List pagination cap.** S3's ListObjectsV2 caps at 1000 keys per
  page by default; `S3BlobStore.list` follows continuation tokens
  unbounded. For projects with thousands of files this could be slow
  ‚ÄĒ not addressing today (the current largest project has tens of
  files), but flag as a paging-knob FUTURE_IDEAS bullet if it
  surfaces.
- **SigV4 chunked uploads / streaming.** This adapter buffers the
  entire body in memory and hashes it for `x-amz-content-sha256`.
  Fine for source files (single-digit MiB) but a non-starter for
  large checkpoint blobs or workspace tarballs. If the checkpoint
  surface needs >100 MiB blobs, switch to `UNSIGNED-PAYLOAD` mode
  or implement chunked SigV4 ‚ÄĒ both are well-documented but
  separate concerns from M20.2(d).
- **AWS_* env-var compat (deferred).** Standard AWS SDK reads
  `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`/`AWS_REGION`/
  `AWS_ENDPOINT_URL_S3`. Today we use `BLOB_STORE_S3_*` to keep
  the namespace under our control and avoid colliding with any
  generic AWS-SDK env that might appear in deploy environments.
  If we ever adopt a real SDK, the env-shim is a one-liner.
- **`fetchImpl` override.** `S3BlobStore` defaults to global
  `fetch`. Provided as `fetchImpl` only for the unlikely test that
  needs to assert request shape without binding a real TCP socket;
  in practice the in-process stub server is the more direct path.
