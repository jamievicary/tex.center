## Findings (load-bearing — please read)

**The probe reproduced the bug on its first live run.** Captured
frame:

```
{"type":"compile-status","state":"error","detail":
 "supertex-daemon: protocol violation: child exited
  (code=134 signal=null) stderr=supertex: watching
  (daemon mode; stdin event-loop) supertex: daemon ready
  supertex: edit detected at /tmp/.../main.tex:56
  supertex: edit detected at /tmp/.../main.tex:163
  supertex: edit detected at /tmp/.../main.tex:187 ..."}
```

This is the **iter-213 protocol-violation daemon-crash shape**
(SIGABRT, code 134), NOT the iter-220
`already in flight` coalescer-defect shape. The iter-220 diagnosis
was wrong — it conflated an unrelated `already in flight` line
seen in a different live session with the user's actual repro. The
captured GT-8 transcript contains zero `already in flight` frames.

Three `edit detected` lines in stderr before the abort prove the
daemon successfully processed three `recompile,T` rounds; the
crash is on a subsequent round. Line numbers 56→163→187 grow
monotonically with the document, consistent with `\newpage XX`
being appended. The sidecar's `SupertexDaemonCompiler` does
recover (subsequent `running`/`idle` frames in the transcript) —
the user-visible toast is the single error frame.
