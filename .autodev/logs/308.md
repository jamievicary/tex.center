## Approach

- Read each point, mapped to the current code:
  - `toastStore.ts` DEFAULT_TTL_MS, `settingsStore.ts`,
    `debugToasts.ts`, `Toasts.svelte`, `wsClient.ts`,
    `compiler/supertexDaemon.ts`, `compiler/daemonProtocol.ts`,
    `packages/protocol/src/index.ts`.
- Identified one factual error in the user's framing: the wire
  emits exactly one `pdf-segment` per compile round, not "one or
  more" — flagged in the answer.
- For point 8 (page name on `pdf-segment` toast): the wire frame
  does not currently carry shipout-page info. Picked option (A) —
  extend the binary header by one `uint32` for `shipoutPage` (0
  sentinel = unknown). Sidecar already has the value
  (`events.maxShipout` in `supertexDaemon.ts:177`); the plumbing
  cost is small and the alternative (separate control message
  before the segment) couples two frames the client must
  correlate.
- Picked single values rather than asking back: animation duration
  500 ms cubic-out (midpoint of the 400–600 ms range); debug-\*
  TTLs uniformly 10 s; info/success/error TTLs unchanged (10 s
  success would be annoying — a different ergonomics class than
  debug spam).
- Decided to **split the batch into M22.4a (UI-only) and M22.4b
  (wire change)** rather than one big iteration. Each is green-
  on-its-own and revertable independently. The user's "all in one
  iteration" framing is fine for human-visible shipping cadence
  but mechanically the wire-format bump deserves a clean slice.
