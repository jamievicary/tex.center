## Goal for this iteration

M15.multipage-preview — pin RED on live. PLAN priority queue
post-iter-267 named M13.2(b).5 R1 first, but R1 is blocked on a
shared `BLOB_STORE` binding that doesn't exist yet (Cloudflare R2
or D1 + sidecar-writes-blob-on-edit signal — multi-iter
infrastructure work). R2 deploy is operator-cadence work. M15 is
the next ready slice: pin-RED-first is a self-contained iteration
that gives M15 diagnose-and-fix a concrete target on the next go.

Per `.autodev/discussion/241_answer.md`, the spec is deliberately
viewer-agnostic: ≥2 `canvas[data-page]` children **or** a single
canvas > 1.8 × viewport height. Both encode "more than one page
of rendered PDF" without prejudicing the diagnosis between the
three candidate root-causes (wire-format `totalLength` cap,
`PdfViewer` snapshot short-circuit, CSS overflow).

## Actions

1. New `tests_gold/playwright/verifyLivePdfMultiPage.spec.ts`.
   Pattern mirrors GT-8 (per-invocation `createProject` + final
   `cleanupLiveProjectMachine`) rather than the worker-scoped
   `liveProject` fixture, so the seed window stays clean and the
   shared GT-A/B/C/D body isn't mutated. Sequence:
   - Goto `/editor/${project.id}`, wait for `.cm-content`.
   - Wait for the initial seeded-template `pdf-segment` and the
     first canvas paint via `expectPreviewCanvasPainted`.
   - Position before `\end{document}`, insert four `\newpage` body
     fragments. `\newpage` is an unconditional page-break in
     `article`, so the resulting PDF is 5 pages irrespective of
     font metrics / fill behaviour.
   - Bounded poll up to 180 s for the post-edit `pdf-segment` to
     land, then 5 s drain.
   - One `page.evaluate` reads `canvasCount`, `pagedCanvasCount`,
     `tallestPx`, `viewportH`, `hostScrollH`. Single assertion:
     `pagedCanvasCount >= 2 || tallestPx > viewportH * 1.8`. The
     failure message inlines every measurement so the next
     iteration can diagnose without re-running the spec.
2. `.autodev/PLAN.md`:
   - M15 section: appended "Pin landed iter 268" sub-bullet and a
     "Next iter: diagnose + fix" plan (log `pdf.numPages` +
     `pdfBytes.length` at first segment arrival to discriminate
     wire-format-cap vs viewer-snapshot-short-circuit).
   - Priority queue: leads with M15 diagnose-and-fix now that
     R1's blob-store work is acknowledged as multi-iter, before
     R1 + R2-deploy + M11.1c + M16.

## Notes for future iterations

- **Diagnosis path.** Quick discriminator: temporarily log
  `pdf.numPages` and `pdf.getData()`-byte-length inside
  `PdfViewer.render` on the first arriving segment of the
  multipage spec. If `numPages === 1`, the PDF bytes themselves
  terminate after page 1 — chase `assembleSegment`'s `totalLength`
  in `apps/sidecar/src/server.ts` / `apps/sidecar/src/assembleSegment.ts`
  and the upstream supertex emit. If `numPages > 1` but only one
  canvas mounts, the viewer-side `$effect` is short-circuiting on
  a stale snapshot — start in `PdfViewer.svelte`.
- **Diagnostic instrumentation.** The first-arrival log line
  should be cheap to add behind the existing debug-mode toggle
  (`apps/web/src/lib/wsDebug.ts` or the Ctrl+Shift+D path) rather
  than unconditionally polluting console output.
- **Per-iter cold-start cost.** The spec creates a fresh project
  each run → cold sidecar Machine. 360 s `testInfo.setTimeout`
  matches GT-8's budget. If the gold runner gets contention
  pressure from this + GT-8 + the `.3`/`.4` cold-start specs all
  spinning new Machines, consider a worker-scoped multipage
  fixture later; for now per-invocation keeps the diagnosis
  simple.
- **R1 still owed.** PLAN priority moves M15 ahead of R1 only
  because R1 needs blob-store infrastructure. The 20 s+
  cold-stopped UX remains user-visible until R1 lands.
