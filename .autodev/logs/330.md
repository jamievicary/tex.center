| T (s) | Event                                          |
|-------|------------------------------------------------|

| ~4.75 | sidecar process starts                         |
| 4.99  | `sidecar-listening` `bootElapsedMs:247`        |
| 6.04  | incoming WS request (`/ws/project/825d8ecd…`)  |
| 6.61  | sidecar `compile start` (hydrate complete)     |
| 10.81 | supertex `watching` + `daemon ready`           |
| 10.87 | `compile ok elapsedMs:4263 phases:{hydrateMs:0, restoreMs:273, writeMainMs:3, persistMs:0, compileMs:4258}` |

Total: wall-clock ~10.9 s from "Pulling container" to first
`compile ok`. Phase shares of the in-sidecar 4.3 s `runCompile`:

- **`compileMs:4258` — 99% of in-sidecar work, 39% of full wall-clock.**
  This is the `SupertexDaemonCompiler.compile()` self-time: the
  daemon child is spawned lazily inside `ensureReady()` (private,
  one-shot), `supertex` reads its precompiled `.fmt`, prints
  `supertex: daemon ready` on stderr, then runs the first
  `recompile,end` round to first shipout. The `daemon ready`
  stderr marker arrived at T+10.81, only ~50 ms before the
  `compile ok` log — so essentially the whole 4.26 s is daemon
  startup (format load) rather than the first round itself.
- `restoreMs:273` — one Tigris GET to `projects/<id>/checkpoint.bin`
  returning null (compilers all return `null` from `snapshot()`
  today, so this is a wasted round-trip on every project boot but
  individually cheap).
- `hydrateMs:0` — fresh `Y.Doc` had hydration already complete by
  the time `runCompile` ran (the WS handshake awaits
  `persistence.awaitHydrated()` before kicking the coalescer, so
  the awaitHydrated inside `runCompile` is a no-op here).
- `writeMainMs:3` — disk write, negligible.
- `persistMs:0` — no diff vs last-persisted (none was written
  yet — the seed-on-empty-blob path inside hydration already
  wrote the canonical `main.tex` to Tigris).

The earlier 89 s figure in PLAN was measured during
`globalSetup` warmup (`bootstrapLiveProject` → end-to-end first
segment). That path includes Machine-create API, full editor
SSR, WS upgrade dial, and may have hit a true cold image-pull —
plus it was measured against the pre-iter-329 image where the
instrumentation was dark, so per-phase ms wasn't captured. The
10.9 s above is a stopped-→-started Machine where the image was
already on the Fly host's local cache. Worst-case image-pull
cold-starts could be substantially longer; capturing one is
the next data point.

## Dominant phase + next slice

**The single biggest term is the supertex daemon's first-time
format-load (`compileMs:4258 ≈ 4.3 s`).** Two things follow:

1. The daemon's `ensureReady()` does not need the source to be
   written to start loading the `.fmt`. It is invoked from inside
   `compile()` today, *after* `writeMain` + `awaitHydrated` +
   `ensureRestored`. Moving daemon-warmup earlier — to project-
   state creation, in parallel with WS handshake / hydrate /
   restore — would let the daemon's 4.3 s overlap with the
   ~1 s of pre-compile setup, expected net win ~3 s on cold
   start.
2. The `restore()` round-trip is a free 0.27 s today (compilers
   return null snapshot, restore is a no-op): we can skip the
   `loadCheckpoint` call entirely until upstream supertex
   exposes serialise/restore. Small win, but it's free.

I'm scheduling (1) for the next ordinary iteration (see
PLAN.md M20.3(a) update) and (2) as a follow-up.
