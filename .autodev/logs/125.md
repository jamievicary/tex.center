## Goal for this iteration

Pick up the FUTURE_IDEAS item "Unify file-op persistence return
types". `apps/sidecar/src/persistence.ts` exposed three distinct
result shapes (`{ added }`, `{ deleted }`, `{ renamed }`), forcing
`handleFileOp` in `server.ts` to take
`Promise<{ reason: string } | Record<string, unknown>>` and
runtime-cast to `{ reason?: string }`. M7.4.2 (upstream
serialise/restore) is the nominal next focus but lives in the
supertex repo, out of scope; M7.2 is effectively done already
(`/ws/project/<id>` is wired end-to-end through `wsProxy.ts` +
`upstreamResolver.ts`). This local cleanup tightens a typed
boundary without touching any deploy path.

## Approach

- Introduce a shared `FileOpResult = { ok: true } | { ok: false;
  reason: string }` exported from `persistence.ts`.
- Replace all three per-op shapes in the `ProjectPersistence`
  interface and both implementations (memory branch + blob-store
  branch).
- In `server.ts`, import `FileOpResult`, narrow `handleFileOp`'s
  parameter type to `Promise<FileOpResult>`, and switch the
  rejection check from `(res as { reason?: string }).reason !==
  undefined` to `!res.ok` â€” the cast is gone.
- No call sites outside `server.ts` consume the persistence return
  types (verified by grep across `apps/`, `packages/`,
  `tests_normal/`, `tests_gold/`). The protocol-level `file-op-error`
  frame the client sees is untouched.
