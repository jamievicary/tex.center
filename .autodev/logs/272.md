## Goal for this iteration

Self-heal the two coupled gold-runner leaks that wedged iters 270
and 271 (and would have wedged 272 unaltered): leaked dev server
on port 3000 and accumulating orphan-tagged per-project Fly
Machines. Both stem from a single root: when the gold playwright
invocation dies mid-run (per-spec timeout, crash, SIGKILL),
`globalSetup`'s teardown closure doesn't fire, so the detached
dev-server orphans AND `bootstrapLiveProject`'s teardown sweep
doesn't run. Next iteration sees port-in-use + orphan overflow,
fails both `test_playwright` and `test_machine_count_under_threshold`
again — leak-loop. iter 271's notes flagged this as the
recommended slice if iter 272 reproduced the breach. It did.

## Diagnosis

- `ss -tlnp` at iteration start showed pid 4522 still listening
  on 3000 — a `pnpm --filter @tex-center/web dev --port 3000`
  from a prior iteration, reparented to /init (pgid 4522 ≠
  harness pgid, because globalSetup spawns with `detached: true`).
  The harness's iteration-end process-group SIGKILL doesn't catch
  it.
- iter 271's gold output listed 7 non-shared sidecars (threshold
  5); 6 tagged with random project UUIDs that have no `projects`
  row, 1 the legit `0000…0001` fixture. Sweep code exists
  (`sweepOrphanedSidecarMachines.ts`) but is wired only into
  `bootstrapLiveProject.teardown`, which didn't fire for the
  same reason.

## Approach

Two minimal changes in `tests_gold/playwright/`:

1. `globalSetup.ts` `spawnDevServer` — on port-in-use, parse
   `ss -lntpH 'sport = :3000'` to extract listener pids, SIGTERM
   them, poll for port-free with 5s deadline, SIGKILL survivors.
   Re-check port; throw only if still in use after self-heal.
2. `liveProjectBootstrap.ts` — run the orphan sweep at startup
   (immediately after the DB handle is open, before the
   bootstrap creates its own project), in addition to the
   existing teardown sweep. Costs one extra Fly list+destroy
   round per gold run, gives every run a clean starting point
   regardless of prior-run exit state.

Coherent slice: same upstream defect, both fixes touch the same
file pair, both are pure self-heal with no behavioural change on
the happy path.

## Notes for future iterations

- **`detached: true` is still wrong** but harder to remove than
  to work around: without detached, `process.kill(-pid, sig)`
  can't address the whole pgid (pnpm forks vite which forks
  esbuild, all inheriting the launcher pgid — same as the
  python harness's pgid for `test_playwright`, so SIGTERM to the
  group would also kill the test runner). A future cleanup:
  spawn pnpm via `setsid` so it gets its own pgid without
  `detached: true`'s "survives parent exit" semantic. Closes the
  leak at root; not load-bearing now that self-heal exists.
- **Startup sweep is best-effort.** If the sweep itself throws,
  bootstrap continues — the count-guardrail then flags any
  residual leak on the same run, which is the correct signal.
- **M17 Playwright pin** remains the head of the priority
  queue.
- **Leak-loop pattern.** Second time (iter 174 chain → 175 fix,
  now iter 270 chain → 272 fix) a leaked subprocess has caused
  multi-iteration regressions. Lesson: when a feature spawns
  *any* long-lived helper, pair it with a startup self-heal,
  not just a teardown cleanup — teardown is a happy-path
  notion.
