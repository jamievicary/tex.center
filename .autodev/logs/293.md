## Goal for this iteration

Fix `test_machine_count_under_threshold` (RED since iter 290).
PLAN had diagnosed this as "spec creates Machines without sweepable
tag, or sweep's tag match too narrow". Empirical investigation
this iteration found that diagnosis is **wrong**: every per-project
sidecar Machine carries `texcenter_project=<uuid>`, the sweep
recognises that tag, and `getKnownProjectIds()` correctly excludes
their tags from the destroy set.

The real cause: the human user has used the live deploy to create
manual test projects ("Test", "Test3", "ererg", "Tes10", "Test11",
"Xxdsz", "Test12", "Test13" — 8 projects, 5 still with running
Machines as of 11:48Z). These are not Playwright artifacts; they
own the user's `owner_id` (same as the live-test sentinel user).
Plus the persistent `pw4-reused-fixture` (00000000-…001) Machine.
Plus one historic Playwright bootstrap leak from yesterday
(`pw-gt-shared-1778797048496`, 5-14 22:17). Total: 7 per-project
Machines. Threshold 5. **The threshold doesn't accommodate normal
user growth.**

Two-part fix:

1. **Bump `DEFAULT_MAX` from 5 to 10**, enough headroom for the
   human's ongoing manual testing without losing the leak signal
   (a runaway leak still produces 10+ Machines fast — see
   pre-iter-243 incidents where 30+ accumulated overnight).
2. **Auto-cleanup of stale `pw-*` DB rows in globalSetup startup.**
   New helper `cleanupOldPlaywrightProjects` deletes
   `machine_assignments` + `projects` rows whose name matches `pw-%`
   and whose `created_at` is older than 10 minutes. The existing
   orphan-machine sweep then destroys their (now-orphan-tagged)
   Fly Machines if any survive. Specifically scoped to `pw-*`
   names so user-created projects (`Test`, etc.) are never
   touched. 10-minute cutoff is well above any single live spec's
   runtime (longest is GT-7 sustained-typing at ~60s), so an
   active project is never collateral.

While here: also fix iter 292's other observation — the live
sidecar's `seed_doc` column **was** successfully applied on
production at 11:46:54Z (verified via `flyctl logs`), so the
iter-292 globalSetup failure is now self-resolved. The seeded
`verifyLivePdfMultiPageSeeded.spec.ts` should run cleanly on this
iteration's gold pass.

## Approach

- `tests_gold/lib/src/cleanupOldPlaywrightProjects.ts`: new pure
  helper. Inputs are `nowMs`, `maxAgeMs`, `db`, `flyToken`,
  `appName`. Lists projects where `name LIKE 'pw-%'` AND
  `created_at < (nowMs - maxAgeMs)`. For each, fetches the
  associated `machine_assignments` row (if any), destroys the Fly
  Machine (force=true; 404 is treated as success), deletes
  assignment row, deletes project row. Idempotent — re-run after
  partial failure proceeds from wherever it stopped.
- `tests_gold/lib/test/cleanupOldPlaywrightProjects.test.mjs`:
  pglite-backed unit test. Cases: (1) old `pw-*` cleaned, (2)
  recent `pw-*` preserved, (3) non-`pw-*` of any age preserved,
  (4) idempotent on no-op input, (5) Machine destroy 404 doesn't
  break the run, (6) missing assignment row is handled (Machine
  was already destroyed but project row stayed).
- `tests_gold/cases/test_cleanup_old_playwright_projects.py`: new
  gold case wrapping the .mjs script.
- `tests_gold/playwright/fixtures/liveProjectBootstrap.ts`: at
  globalSetup startup, call `cleanupOldPlaywrightProjects` BEFORE
  the existing `sweepOrphanedSidecarMachines` step. Sweep then
  catches anything left over (e.g. Fly Machine ID was wrong in DB
  but the actual Machine still tagged the project).
- `tests_gold/cases/test_sidecar_machine_count.py`: bump
  `DEFAULT_MAX` 5 → 10. Update docstring with the new rationale
  (the threshold is for leak detection, not Fly resource budget;
  10 is generous for normal user growth).
- `.autodev/PLAN.md`: replace the "investigate which spec creates
  Machines without sweepable tag" entry in §3 with the corrected
  diagnosis + landed fix.

Empirical findings driving this iteration (not previously known):

1. **Iter-292 gold suite failure was a deploy-timing issue, not a
   regression.** The new `seed_doc` column referenced by
   `createProject` couldn't be inserted because the live deploy +
   migration ran AFTER the iter-292 commit was pushed but BEFORE
   the gold runner kicked off. Confirmed via `flyctl logs -a tex-
   center --no-tail`: `migrations: 1 applied, 2 already present
   (0003_add_projects_seed_doc)` at 2026-05-15T11:46:54Z. The next
   gold pass (iter 293) will exercise both the seeded spec and
   the new stale-pw sweep against a live DB with `seed_doc`
   present.

2. **The PLAN's diagnosis of `test_machine_count_under_threshold`
   was wrong.** A `flyctl proxy` + DB query revealed the live DB
   has 20 `projects` rows and 20 `machine_assignments` rows;
   Fly Machines API reports 7 per-project (non-shared) Machines.
   Of those 7:
   - 5 are user-named (`Test`, `Test11`, `Test12`, `Test13`,
     `Tes10`, `Xxdsz` — the human's manual testing).
   - 1 is `pw4-reused-fixture` (00000000-…001), intentionally
     persistent for `verifyLiveFullPipelineReused.spec.ts`.
   - 1 is `pw-gt-shared-1778797048496` (5-14 22:17), an orphaned
     Playwright bootstrap project from yesterday whose teardown
     didn't fire.

   Every counted Machine carried `texcenter_project=<uuid>`, so
   the sweep recognised the tag — but its definition of "orphan"
   ("not in `projects.id`") preserved all of these because every
   one had a corresponding row.
