## Goal for this iteration

N=220 (`N%10==0`) → refactor iteration. Survey turned up one clear
duplicated-template smell in `apps/sidecar/src/persistence.ts`: six
sites inline `${projectFilesDir(projectId)}/${name}` (or
`${dir}/${name}` via a local) to build a blob key for a named project
file, while a paired helper `mainTexKey(projectId)` already exists
for `main.tex`. Introduce `projectFileKey(projectId, name)` and apply
it consistently; redefine `mainTexKey` in terms of it.

## Approach

- Added `projectFileKey(projectId, name)` next to `projectFilesDir` /
  `mainTexKey`. Doc-comment notes callers must pre-validate the name
  segment.
- Rewrote `mainTexKey(projectId)` to delegate to
  `projectFileKey(projectId, MAIN_DOC_NAME)`.
- Replaced every `${projectFilesDir(projectId)}/${name}` and
  `${dir}/${name}` site in `createProjectPersistence` (hydrate / add /
  delete / rename / `maybePersist`) with the new helper. Dropped the
  now-unused `const dir = projectFilesDir(projectId)` locals.
- Dropped a tiny dead branch in `addFile`:
  `body.length === 0 ? new Uint8Array(0) : new TextEncoder().encode(body)`
  is equivalent to `new TextEncoder().encode(body)` (the encoder
  already returns a zero-length array for the empty string).

No behaviour change. Public exports gain `projectFileKey`; nothing is
removed. Test surface (`apps/sidecar/test/persistenceSeed.test.mjs`,
`apps/sidecar/test/listProjectFiles.test.mjs`) only references
`mainTexKey` / `listProjectFiles`, so it is unaffected.
