## Goal for this iteration

Land FUTURE_IDEAS "M20.3(a)3" — empty `main.tex` placeholder in
`ProjectWorkspace.init()` so the iter-331 `compiler.warmup()` overlap
actually succeeds on a fresh-project cold start. Without the
placeholder, the supertex daemon child's spawn errors out
immediately on `main.tex: no such file`, the warmup `.catch`
swallows it, and the first `runCompile` pays the full ~4 s
`.fmt`-load through the dead-child detect/respawn fallback.

Active priority queue is otherwise data-starved: priorities #1
(GT-6-stopped) and #2 (Bug B) both wait on fresh evidence;
priority #5 step 5 is blocked on the same. This perf slice is
the readiest concrete work that meaningfully advances M20.3 — it
makes the iter-331 overlap correct on the cold path that GT-6-
stopped exercises every gold pass, even if the slice alone is
unlikely to flip the spec GREEN (compile cycle is ~4.7 s and the
spec budget is 1 s).

## Approach

Three fix candidates were listed in FUTURE_IDEAS; the empirically
cleanest is "ensure `main.tex` exists at `workspace.init()`":

- **Empty placeholder is sufficient.** Confirmed via a smoke run:
  `vendor/supertex/build/supertex --daemon ./chunks ./main.tex`
  against an empty `main.tex` prints `supertex: daemon ready` on
  stderr and waits on stdin. The daemon's startup only does the
  format-file (`.fmt`) load — independent of document content —
  before parking on the event loop; the actual document is re-read
  on each `recompile,N`, by which point `runCompile.writeMain`'s
  tmp+rename has materialised the real source.
- **`flag: 'wx'` (O_CREAT|O_EXCL) preserves prior persisted source.**
  On a stopped-Machine cold-restart the prior session's `main.tex`
  is already on disk from `persistence.maybePersist()`. EEXIST is
  the expected no-op path and is swallowed.
- **Sequence the warmup behind `workspace.init()`.** `getProject()`
  used to fire-and-forget `state.compiler.warmup()` directly; now
  it chains `state.workspace.init().then(() => state.compiler.warmup())`
  so the placeholder is on disk before the daemon spawn reads
  `main.tex`. Error handling unchanged (single `.catch` covers
  both init and warmup failure as non-fatal).

## Notes for future iterations

- **What this slice does and doesn't fix.** Iter-331's warmup
  overlap targets ~4 s of `.fmt`-load latency that previously
  serialised behind WS-handshake + Yjs-hydrate + checkpoint-
  restore. The bug was that the overlap was *forfeit* on every
  cold-start of a per-project Machine because the spawn died on
  missing `main.tex`: `scratchRoot` defaults to a freshly-minted
  `mkdtempSync` under `tmpdir()` on each process boot, so even a
  stopped→start cycle gets an empty workspace dir (persisted
  source is restored from blob store via `persistence` hydration,
  which then writes via `workspace.writeMain` later). This
  iteration restores the overlap on *every* cold start, including
  the GT-6-stopped / GT-9 paths.
- **The EEXIST branch is defensive.** Under the current
  `scratchRoot` regime, init() always sees a missing `main.tex` on
  first call; EEXIST is only possible if a future iteration makes
  `scratchRoot` persistent across reboots (e.g. via a Fly volume
  mount). The test in `workspace.test.mjs` pins the preserve-
  existing-content contract so that change is safe whenever it
  lands.
- **Empirical signal to watch.** Next gold pass on fresh-cold
  paths (GT-7, GT-8) should see `compileMs` drop by ~3.5–4 s in
  the `compile ok` log line. The wire-timeline fixture lands
  `compile-cycles` and `mean-cycle` per spec — GT-7's
  zero-delay-typing summary's `mean-cycle` is the cleanest single
  number to compare against iter 352's baseline (the GT-7 summary
  in iter-352 gold output shows `mean-cycle` at sub-second; we
  expect that to stay sub-second but the FIRST cycle to shrink
  substantially).
- **Maintenance cadence.** N=353, N%10==3, ordinary iteration.
  Next refactor N=360, next plan-review N=361. No deferral carry.
- **EEXIST swallow is narrow.** Only the `code === 'EEXIST'`
  branch is silenced. Anything else (EACCES, ENOSPC, etc.) still
  bubbles, so an actually broken workspace will still surface
  loudly through the `app.log.warn` in `getProject`'s `.catch`.
