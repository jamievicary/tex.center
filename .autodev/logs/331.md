## Goal for this iteration

PLAN priority #1, slice M20.3(a): the iter-330 cold-start data
pinned the dominant cold-boot term as `compileMs:4258` ≈ 4.3 s
spent in `SupertexDaemonCompiler.ensureReady()` (supertex daemon
spawn + `.fmt` load), and showed that `daemon ready` arrives
~50 ms before `compile ok` — so essentially the whole 4.3 s is
startup, not the first compile round.

Add a `Compiler.warmup()` hook on the interface; concrete impls
no-op except `SupertexDaemonCompiler.warmup()` which calls the
existing `ensureReady()`. Wire it from `getProject` immediately
after compiler construction so the daemon's format load overlaps
WS handshake + hydrate + restore. Pin behaviour with a unit
test: with a fake daemon that delays its `daemon ready` marker by
200 ms, an *unwarmed* `compile()` pays the full delay, while a
warmed compile completes well under it (the new caller-paid
overlap that production gets for free against the ~1 s of
pre-compile setup).

## Approach

1. Add `warmup(): Promise<void>` to the `Compiler` interface.
2. `FixtureCompiler.warmup` / `SupertexOnceCompiler.warmup` —
   no-op (resolved promise).
3. `SupertexDaemonCompiler.warmup` — delegate to `ensureReady()`,
   which is already idempotent (cached `readyPromise`).
4. In `server.ts getProject`, after constructing the compiler,
   call `state.compiler.warmup()` fire-and-forget with an error
   log handler. The existing `compile()` path keeps calling
   `ensureReady()` and will hit the cache.
5. Extend the fake daemon test rig with `FAKE_DELAY_READY_MS`,
   then add a unit lock asserting compileMs ≪ delay when warmup
   was awaited beforehand, and ≥ delay when it was not.
