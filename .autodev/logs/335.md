## Goal for this iteration

PLAN priority #1 (M20.3) still needs GT-9 to be a truthful, useful
signal. The iter-334 typing-strategy fix should have unblocked the
post-edit `pdf-segment` wait, but the iter-334 gold pass timed out
GT-9 at the 5-minute test wall (`Test timeout of 300000ms exceeded`)
with **zero diagnostic output** — no `[verifyLiveGt9...]` log line
of any kind. Two compounding problems:

1. The spec's seven phase budgets sum to ~365 s
   (`cmContent.waitFor` 60 s + cold-start 90 s + post-edit 30 s +
   stop settle 60 s + `waitForURL` 30 s + sentinel visible 90 s +
   small navs), so `testInfo.setTimeout(5 * 60_000)` is structurally
   below the worst-case sum. Any single slow phase can blow the wall
   without producing an assertion-style failure.
2. The spec emits its diagnostic `cmTextPrefix` log only at the very
   end (line 251), after the sentinel polling loop. On a test-level
   timeout firing mid-phase, that line is never reached, so we have
   no idea which phase was the slow one. The footer comment in iter
   334's log explicitly anticipated this informative-failure mode
   but it isn't wired.

Fix in this iteration: add a per-phase `[verifyLiveGt9
phase=<N> ...]` console.log at every phase boundary including the
ones inside the deadline-poll loops (so partial progress is
captured), and raise `testInfo.setTimeout` to 8 min (480 s) — a
single safety margin over the legitimate worst-case sum, not a
flake-paperer. No behavioural change otherwise; the test still
fails exactly when persistence is actually broken.
