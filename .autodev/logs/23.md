## Goal for this iteration

Unblock **M4.2.1** by adopting in-process PGlite (already pre-approved
in `FUTURE_IDEAS.md`) instead of waiting for a Docker-capable host.
The acceptance criterion from the milestone — "gold test runs
`db:migrate` against a real Postgres, asserts every spec table
exists and re-running is a no-op (`skipped` only)" — is met against
PGlite's in-process Postgres engine, with no host services required.

To make `applyMigrations` usable from both `postgres-js` (prod path)
and PGlite (test path) without duplicating the bookkeeping logic,
refactor it to accept a small `MigrationsDriver` interface; ship
both adapters from `@tex-center/db`.

## Approach

1. `packages/db/src/migrations.ts` — introduce `MigrationsDriver`
   with three methods (`ensureMigrationsTable`,
   `loadAppliedRows`, `applyOne`). Rewrite `applyMigrations` to
   call those. Add `postgresJsDriver(sql)` adapter that wraps a
   postgres-js `Sql` (the existing prod behaviour).
2. `packages/db/scripts/migrate.ts` — pass
   `postgresJsDriver(handle.client)` instead of the raw client.
3. `packages/db/package.json` — add `@electric-sql/pglite` as a
   devDep (test-only, never imported from `src/`).
4. New `packages/db/test/migrations-pglite.test.mjs` — boots a
   memory PGlite, applies via `applyMigrations(pgliteDriver(pg),
   migrations)`, asserts the round-trip and tables/columns.
   The `pgliteDriver` adapter lives in the test file.
5. `tests_gold/run_tests.sh` — replace the placeholder with a
   real `unittest` discovery.
6. New `tests_gold/cases/test_pglite_migrations.py` — shells out
   to `pnpm exec tsx packages/db/test/migrations-pglite.test.mjs`.
7. `.autodev/PLAN.md` — tick M4.2.1 done.
8. `.autodev/FUTURE_IDEAS.md` — drop the PGlite item; replace it
   with a "docker-compose for Postgres+MinIO" note still wanted
   for the M4.3 file-blob side.

## Notes for future iterations

- `pnpm install` shows `Packages: +2 -80` after adding PGlite —
  the `-80` is pnpm rebalancing the dedup graph, not a real
  removal (re-run is a no-op). Worth a glance at `pnpm-lock.yaml`
  diff in the iteration commit to make sure nothing unexpected
  vanished.
- The PGlite test holds the engine open for the whole run; if
  future gold cases want their own PGlite instance, prefer
  per-test isolation (`new PGlite()` is cheap, ~tens of ms).
- `pgliteDriver` lives in the test file deliberately — moving it
  to `src/` would force PGlite into prod deps. If a second
  PGlite-backed test arrives, lift it to a shared helper under
  `packages/db/test/_helpers/` rather than into `src/`.
