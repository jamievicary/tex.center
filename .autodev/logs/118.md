## Goal for this iteration

Land **M7.1.4** ‚Äî idle-stop wiring on the per-project Machine
side. Sidecar self-exits cleanly after a configurable idle
timeout (no WS viewers); combined with the existing per-project
Fly Machine restart policy `on-failure` set in
`upstreamFromEnv.ts`, a clean `process.exit(0)` leaves the
Machine in `stopped` state (not destroyed, not restarted). The
control plane's `upstreamResolver.ts` already wakes stopped
machines on the next WS upgrade (lines 139-141), so this single
slice closes both M7.1.4 and M7.3.

## Approach

Track viewer count globally in `buildServer`. On the
`socket.add`/`viewers.delete` paths, increment/decrement; when
count hits zero, arm a `setTimeout(idleTimeoutMs)` that calls the
caller-supplied `onIdle` hook. New connection cancels the timer.

- `idleTimeoutMs?: number` and `onIdle?: () => void` on
  `SidecarOptions`. Both unset ‚Üí no idle-stop (preserves all
  existing tests' behaviour by default).
- Entry-point `main()` reads `SIDECAR_IDLE_TIMEOUT_MS`
  (default 600_000 = 10 min; `0` disables) and supplies
  `onIdle = () => app.close().then(() => process.exit(0))`.
- `onClose` clears any armed idle timer (test-side hygiene).

Why count viewers globally and not per-project: a per-project
Machine has one project. Even in a debug multi-project boot, "no
viewers anywhere" is the right idle predicate; per-project
gating would let one stale project pin the Machine.

Why not also gate on "no in-flight compile": compile only
fires from a viewer's edits. With viewers=0 the compileTimer
gets cleared in the close handler already (server.ts:402-405),
so the predicate is sufficient.

## Notes for future iterations

- M7.4 (checkpoint blob protocol) is the natural next step. The
  `onIdle` hook in `index.ts` is the obvious place to flush a
  checkpoint blob to Tigris before exit; the daemon-mode
  compiler (`SupertexDaemonCompiler`) is the side that needs to
  emit serialised checkpoints. The compiler `close()` contract
  already drains the daemon ‚Äî extend it to optionally produce a
  checkpoint blob, then `onIdle` becomes:
  `for (p of projects) await persistence.persistCheckpoint(p.compiler.snapshot())`
  before `app.close()`.
- The `unref()` on the idle timer is intentional for tests
  (otherwise `app.close()` after a timeout would race), but on a
  long-running production sidecar the live WS connections
  themselves keep the loop alive, so the unref is safe.
