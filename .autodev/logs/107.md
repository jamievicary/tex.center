## Goal for this iteration

M7.5.2 — implement `SupertexDaemonCompiler` next to the existing
`SupertexOnceCompiler`. One persistent `supertex --daemon DIR`
child per project: lazy spawn on first `compile()`, write
`recompile,N\n` to stdin, collect events via the parser landed
iter 91, read chunk files from DIR after `[round-done]` and
concatenate to a single PDF segment. `close()` shuts down via
stdin EOF → SIGTERM grace → SIGKILL. Plus the M7.5.0 Dockerfile
carry-over recorded in PLAN: switch `make -C vendor/supertex`
target to `all`, and point `SUPERTEX_BIN` at `build/supertex`.

The compiler stays gated behind a new `SIDECAR_COMPILER=
supertex-daemon` selector; the production default `supertex`
(once-compiler) is unchanged. M7.5.3 (error wire frame), M7.5.4
(stability watcher gating), and M7.5.5 (integration tests + flip
the default) are deliberately deferred.

## Approach

- Module: `apps/sidecar/src/compiler/supertexDaemon.ts`. Uses
  `DaemonLineBuffer` from `daemonProtocol.ts`. Lazy spawn from
  the first `compile()` (constructors stay synchronous; matches
  the once-compiler's contract).
- Spawn signature: `<bin> --daemon <workDir>/chunks <workDir>/
  <sourceName>` with piped stdio. Wait for the
  `supertex: daemon ready` marker on stderr (bounded poll, default
  60s) before the first recompile.
- `compile({targetPage})`: write `recompile,<N>\n` (or
  `recompile,end\n` when `targetPage <= 0` — the M3 server passes
  `maxViewingPage(p)` which defaults to 0 when no client has
  signalled). Wait for `[round-done]`. Collect `[N.out]` events
  to know which chunks to assemble, and any `[error <reason>]`
  → return `CompileFailure`. After round-done, read chunk files
  `0.out`..`max.out` from the chunks dir in numeric order,
  concatenate, return as a single segment.
- Protocol violation → kill child, return failure.
- `close()`: end stdin, wait `gracefulTimeoutMs` (default 5s),
  then SIGTERM with `killTimeoutMs` (default 2s) grace, then
  SIGKILL. Idempotent.
- Single in-flight compile at a time (the protocol explicitly
  requires waiting for `[round-done]` before sending another
  `recompile`). Enforced by an internal mutex; concurrent
  `compile()` calls reject.

## Notes for future iterations

- M7.5.3 next on the daemon track: `[error reason]` →
  `compile-status:error` wire frame in `packages/protocol`. The
  compiler today returns `CompileFailure` with the reason inline;
  the existing protocol failure path already plumbs that string,
  but a dedicated frame makes the client distinguish "engine
  said no" from "process crashed".
- M7.5.4 (`PdfStabilityWatcher` gating) should land alongside the
  default flip in M7.5.5. The watcher is dead code today on the
  daemon path — `[round-done]` *is* the stability signal — but
  flipping the default without gating leaves it ticking pointlessly.
- The `assembleSegment` happy path returns one segment with
  `offset=0` and the full concatenation. M7 wire-protocol work
  later wants per-shipout deltas exposed to the browser;
  `PdfSegment` already supports `offset` + `totalLength`, so the
  upgrade is local to `assembleSegment` (return one segment per
  chunk file) once the receiver patches in-place rather than
  replacing the buffer.
- The Dockerfile fix means the next sidecar deploy will rebuild
  the full supertex toolchain (~50 binaries) and pick up the
  correct `SUPERTEX_BIN` path; the iter-93 deploy was working
  only because the live workflow never actually exercised the
  once-compiler end-to-end. M7.1.3.2 (authed WS upgrade) will be
  the first time the real `SupertexOnceCompiler` runs in prod.
