## Goal for this iteration

PLAN's "next ordinary iteration" (M6.3.1) is the out-of-tree
one-shot needing live Fly + Cloudflare tokens — not runnable
inside autodev. Picking from the smaller-alternatives list:
land the listing primitive that the multi-file slice will need.
`packages/blobs` already exposes `list(prefix)`; what's missing
is a project-aware wrapper that strips the `projects/<id>/files/`
prefix and guards against the sibling-`files-meta` foot-gun.
Adding it now means the eventual multi-file hydration step (or a
"file tree from server" wire message) gets to bind to a tested,
typed function instead of open-coding the slice.

## Approach

- New `projectFilesDir(projectId)` helper in `persistence.ts` —
  no trailing slash, since `validateKey` rejects them. Existing
  `mainTexKey` redirected through it (so the path layout has one
  authoritative source).
- `listProjectFiles(blobStore, projectId)` calls
  `blobStore.list(projectFilesDir(id))`, then filters keys to
  those starting with `${dir}/` and slices the prefix off. The
  filter step is load-bearing: `LocalFsBlobStore.list("…/files")`
  walks any sibling whose name *begins with* `files`, so a
  `projects/<id>/files-meta` blob would otherwise leak in.
- Standalone unit test under `apps/sidecar/test/`, registered in
  `test_node_suites.py`. Empty project, multi-file project (incl.
  nested), sibling-`files-meta` exclusion, project-id isolation.

## Notes for future iterations

- The sibling-`files-meta` exclusion isn't hypothetical: any
  future "project metadata" blob (per-project compiler flags,
  bibliography settings, etc.) would naturally live next to
  `files/` and a startsWith-only filter would silently include
  it. The unit test pins the contract.
- Wiring this into the sidecar's WS handshake (so the FileTree
  shows real files instead of a hard-coded `["main.tex"]`) needs
  a new control-message type in `packages/protocol`. Coherent
  next slice; not bundled here because it touches three packages
  and the protocol decoder.
