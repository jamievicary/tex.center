## Goal for this iteration

M23.1 — extend `ProjectWorkspace` with `writeFile(name, content)`,
`deleteFile(name)`, `renameFile(oldName, newName)`. Pure workspace
primitives; no wiring into `persistence.ts` yet (that's M23.2). This
slice lands the on-disk plumbing behind `validateProjectFileName`
with the atomic write-tmp-rename + empty-parent-reap pattern used by
`LocalFsBlobStore`. Top-of-queue work in `.autodev/PLAN.md` since
iter 310; identified as the natural pick-up by iters 311 and 312.

## Approach

- Add three methods plus a private `pathFor(name)` that validates
  against `validateProjectFileName` (defence-in-depth — the wire
  layer already validates).
- `writeFile`: `mkdir -p dirname(target)` then write-to-tmp + rename
  (matches existing `writeMain` semantics).
- `deleteFile`: `rm` with `force: true` (ENOENT tolerant), then reap
  empty parents up to but not including `this.dir`.
- `renameFile`: `mkdir -p dirname(dst)` then `rename(src, dst)`,
  then reap old parents. Same-name short-circuits.
- No special handling for `main.tex` — callers can pass it and the
  effect is identical to `writeMain`. Keeps the API uniform; the
  persistence layer in M23.2 will route by name anyway.
- Reap-parents helper extracted as a private method (used by
  delete + rename). Mirrors `LocalFsBlobStore.delete`'s reap loop:
  stop on ENOTEMPTY / EEXIST / ENOENT.
- Lock with extensions to `apps/sidecar/test/workspace.test.mjs`.

## Actions

- `apps/sidecar/src/workspace.ts`:
  - Add `import { validateProjectFileName } from "@tex-center/protocol"`.
  - Add `rmdir` to the `node:fs/promises` imports (rename moved to
    place `rename` next to its peer `writeFile`).
  - New public methods: `writeFile(name, content)`,
    `deleteFile(name)`, `renameFile(oldName, newName)`.
  - New private helpers: `pathFor(name)` (validate + join across
    `/`-separated segments) and `reapEmptyParents(target)` (walk
    `dirname` upward, `rmdir` while empty, stop at `this.dir`).
  - Header comment refreshed to describe the M23.1 surface.
- `apps/sidecar/test/workspace.test.mjs`: added four blocks
  covering (a) flat-name write + overwrite + no stray `.tmp`,
  (b) slashed-name write with `mkdir -p` parents and reap-on-delete
  semantics (subdirectory reaps when last leaf removed, parent
  survives while another sibling lives), no-op on missing file,
  (c) rename across directory boundaries reaps the emptied source
  parent + same-name short-circuit, (d) validation rejects `..`,
  trailing `/`, double `/`, and empty name across all three
  methods.

## Notes for future iterations

- **M23.2** wires `persistence.ts` to call through on every file
  mutation. Recommended seam (per iter-310 notes): persistence's
  `addFile` / `deleteFile` / `renameFile` plus a `Y.Text.observe`
  subscription on each non-main file for in-place edits, debounced
  via the same coalescer that gates `writeMain`. The workspace
  primitive landed here is uniform across `main.tex` and
  non-main files — callers may route `main.tex` through
  `writeFile` too, but the existing `writeMain` call site remains
  load-bearing because it's on every compile-cycle (no Yjs
  observe), so don't bundle that refactor into M23.2.
- **M23.3** cold-boot rehydration: after persistence rehydrates
  from the blob store, iterate `persistence.files()` and call
  `workspace.writeFile(name, doc.getText(name).toString())` for
  each non-main entry *before* the first compile call. Should be
  a small slice on top of M23.2.
- **M23.4** gold spec: 2-file project with `\input{sec1}`. Local
  Playwright likely the cleanest fit — depends on whether the
  page-content assertion is easier via canvas pixel inspection or
  a sidecar-level integration test that exercises the daemon
  output stream directly. Either is fine; pick at slice time.
- Slashed-path semantics now match `LocalFsBlobStore.delete`
  parent-reap exactly, so cold-boot rehydration of nested files
  through M23.3 will produce an on-disk shape congruent with the
  blob-store shape from `slashedFileNames.test.mjs`. That parity
  is intentional — same mental model for the daemon's cwd and the
  cold-boot source.
