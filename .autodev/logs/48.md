## Goal for this iteration

Tiny but real: the landing page's "Sign in with Google" button still
mocks the auth path — it just runs `window.location.assign("/editor")`.
M5 landed real Google OAuth (iter 35–39); the only remaining gap on
the unauth side of the GOAL acceptance criteria #1/#2 is wiring the
button to the live flow. Switch to a styled anchor pointing at
`/auth/google/start` so the flow works without JS as well (the layout
sets `ssr = false`, but a static anchor is still the cheapest
progressive-enhancement story for the one widget on the page).

While here, drop the stale "adapter-static needs prerender enabled"
comment in `+layout.ts` — iter 34 switched the app to adapter-node;
the prerender default still makes sense (the marketing/landing page
ships as a static asset) but the *reason* is no longer about
adapter-static.

M6.3.1 (live cutover) remains out-of-tree.

## Approach

- `apps/web/src/routes/+page.svelte`: replace the `<button onclick>`
  with `<a href="/auth/google/start">`, keep visual styling identical.
- `apps/web/src/routes/+layout.ts`: replace the stale comment.
- New `tests_normal/cases/test_landing_sign_in.py` parses
  `+page.svelte` and asserts: there is exactly one interactive control
  on the page, its target is `/auth/google/start`, and there is no
  reference to `/editor` (the GOAL forbids hinting at what the
  service is — a sign-in widget pointing straight at `/editor` would
  also leak that hint to an unauth viewer of view-source).
