## Design notes

- **Why no browser.** The surface under test is the HTTP envelope
  of the test-callback route + the session-cookie hook. Driving
  Chrome would not add coverage but would double the surface area
  (Origin handling, mixed content, cookie attribute handling
  differences). `node:https.request` mirrors the existing
  `verifyLiveAuthed` / `verifyLiveWsUpgrade` style.

- **Why the `db` fixture.** Cleanup of the freshly-minted
  `sessions` row needs a live drizzle handle. Reusing the existing
  fixture is cheaper than threading another flyctl-proxy startup
  into this spec ‚Äî and the fixture already self-skips on missing
  live-DB env, so the gating composes correctly: if the worker
  can't reach the DB the test never runs anyway.

- **Why an explicit `verifySessionToken` step.** Without it, a
  hypothetical signing-key drift between the response cookie and
  the in-process `db.signingKey` would still produce a 302 ‚Üí / on
  step 5, indistinguishable from a "no cookie sent" failure. The
  intermediate assertion separates "callback minted a valid
  cookie" from "hook accepted that cookie" ‚Äî useful when triaging
  which side of the seam regressed.

- **Why TEST_OAUTH_BYPASS_KEY from env, not creds/.** The Fly
  secret is the source of truth (operator-managed, rotatable
  without a code edit); requiring the same env var locally keeps
  one credential surface. Adding a `creds/test-oauth-bypass-key.txt`
  fallback would just multiply ways for them to drift apart.

- **Activation is operator work.** Three steps gate the live
  exercise: (a) create the test OAuth client in Google Cloud with

  and export the same value in any shell that runs the live gold
  suite. Until those land the spec self-skips on every `live` run.

- **First post-activation run will exercise live OAuth surface.**
  That triggers a real `users` UPSERT against the existing
  `jamievicary@gmail.com` row (UPDATE branch) plus a fresh
  `sessions` INSERT + DELETE round-trip per run. Cheap. The
  refresh-token-grant call to `oauth2.googleapis.com` is the only
  egress; if Google ever rate-limits, the helper's bare `fetch`
  has no retry ‚Äî failures would surface as the test's first
  assertion.

- **Refresh-token rotation.** `mintGoogleIdToken` drops any
  rotated `refresh_token` Google occasionally returns on a
  refresh-token grant. If the live spec ever sees `invalid_grant`
  after long idle the helper will need to persist the rotated
  token back; recorded as a follow-up rather than pre-empted.
