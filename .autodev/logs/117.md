## Approach

`tests_gold/playwright/verifyLiveWsUpgrade.spec.ts`:

- Reuses the `db` worker fixture from `fixtures/authedPage.ts` so
  the live/local split (flyctl-proxied Postgres vs PGlite) is one
  code path. The fixture already self-skips on missing live env.
- Skips on the `local` project (the upgrade only makes sense
  against the deployed control plane) and additionally on missing
  `FLY_API_TOKEN` / `SIDECAR_APP_NAME` â€” the destroyer needs both
  and the resolver pins the destroy target to the latter app.
- Test body creates a real `projects` row for the seeded live
  user (project id must be a uuid, since `machine_assignments.
  project_id` FKs `projects.id`), mints a session via the existing
  `mintSession` helper, then issues `node:https.request` with
  `Connection: Upgrade`, `Upgrade: websocket`,
  `Sec-WebSocket-{Version,Key}` and `Cookie: tc_session=...`. The
  `'upgrade'` event fires with `res.statusCode === 101` on
  success; the `'response'` path is captured as the failure mode
  (401/404/502) with body for diagnostics.
- Cleanup is test-level try/finally, not `afterAll`. Three reasons:
  the project/session ids are local to the test scope, the
  iter-116 helper already idempotent-deletes the assignment row,
  and putting it in `finally` means a thrown 101-assertion still
  runs the destroy before failing. `cleanupProjectMachine` is fed
  duck-typed `MachineDestroyer` (inline `fetch` against
  `api.machines.dev`) + drizzle-backed `AssignmentStore`. No
  import from `apps/web/src/lib/server` â€” the helper's whole point
  is to keep tests-side teardown free of that.
- `test.setTimeout(5 * 60_000)` for the cold-start of a fresh
  Machine; default 30s is too tight.

Why no Playwright browser instance: the WS handshake is HTTP-level
and the spec only needs to inspect the upgrade status. Driving it
through Chromium would add Origin/mixed-content surface without
new coverage.
