## Goal for this iteration

Land **M5.1.1**: the `/auth/google/start` `+server.ts` route. This is
the first dynamic server endpoint in `apps/web`, made possible by the
iter-34 adapter-node swap. It:

1. Generates a fresh PKCE pair (`@tex-center/auth.generatePkce`).
2. Generates a random opaque `state` value.
3. Mints a signed cookie carrying `{state, verifier, exp}`.
4. 302-redirects to Google's authorize URL with `code_challenge`,
   `code_challenge_method=S256`, `scope=openid email`,
   `prompt=select_account`.

The callback (M5.1.2) verifies the cookie, exchanges `code +
verifier` for tokens, JWKS-verifies the ID token, allowlist-checks the
email, persists a session row, and mints the session cookie.

## Approach

**Refactor `packages/auth` to share signing primitives.** The
iter-32 `signSessionToken` payload type is `{sid, exp}`; the start
cookie wants `{state, verifier, exp}`. Iter-34's note flagged two
options — extend the payload type, or factor a generic signed-JSON
helper underneath. I picked the second: it's marginally more code
now (one file) but the *next* signed-payload case (cookie pre-empted
by CSRF, OIDC nonce, anything else) reuses it for free. And the
session/state types end up symmetric.

- `packages/auth/src/signed.ts` — new. Generic `signJsonString` /
  `verifySignedJson` operating on opaque payload strings. Two
  rejection reasons only: `malformed` and `bad-signature` (no
  expiry logic — payload-shape and expiry decisions belong with
  whoever knows what's inside).
- `packages/auth/src/session.ts` — rewritten to layer JSON-shape
  + expiry checks on top of `signed.ts`. Public surface
  (`signSessionToken`, `verifySessionToken`, `SessionPayload`,
  `VerifyResult`, `VerifyFailure`) is unchanged so the existing
  test passes without edits.
- `packages/auth/src/state.ts` — new. `signStateCookie` /
  `verifyStateCookie` for `{state, verifier, exp}`, same pattern.
  `verifier` validated as RFC 7636 §4.1, `state` as non-empty
  base64url (we generate it that way).
- `packages/auth/src/index.ts` — re-exports the new state surface.
- `packages/auth/test/state.test.mjs` — round-trip + every
  rejection branch (bad signature, expired, tampered payload,
  malformed cookie, bad-shape JSON, bad-verifier).
- `tests_normal/cases/test_node_suites.py` — wires the new state
  test.

**Pure builder for the start endpoint, thin route wrapper.**
The `/auth/google/start` route handler is a few lines of glue
around a pure function. Keeping the pure layer testable means we
don't need SvelteKit's `Request`/`Response` machinery in the unit
test, and the route stays trivial enough to read at a glance.

- `apps/web/src/lib/server/oauthStart.ts` — pure
  `buildGoogleAuthorizeRedirect({clientId, redirectUri,
  signingKey, pkce, state, nowSeconds, stateTtlSeconds})` returns
  `{location, stateCookie}`. Cookie is `Path=/auth; HttpOnly;
  SameSite=Lax; Secure; Max-Age=<ttl>` — `SameSite=Lax` is the
  right value for an OAuth redirect (the callback navigation is
  top-level, so `Lax` lets the cookie ride along; `Strict` would
  drop it). `Secure` always on in prod; the route opts out only
  for `http://localhost` in dev (see below).
- `apps/web/src/lib/server/oauthConfig.ts` — pure config loader.
  Reads `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_REDIRECT_URI`,
  `SESSION_SIGNING_KEY` (base64url, ≥32 bytes) from env. Falls
  back to `creds/google-oauth.json` for `client_id` (per GOAL.md
  the canonical source). If anything is missing or malformed,
  throws with the exact env var or file path the user needs to
  fix — silent fallback would emit an unhonourable redirect.
- `apps/web/src/routes/auth/google/start/+server.ts` —
  `RequestHandler` calling the builder. Catches the
  config-missing error and returns 500 with a one-line body
  ("Server misconfigured: <reason>") rather than letting the
  framework's error page leak.
- `apps/web/test/oauthStart.test.mjs` — unit test of the pure
  builder.

**`apps/web` dependency on `@tex-center/auth`.** Added to
`apps/web/package.json` as `workspace:*`. pnpm-lock regenerated.

## Decisions

- **Cookie name `tc_oauth_state`.** Short, namespaced
  (`tc_` for tex.center), distinct from the post-callback session
  cookie (`tc_session`, when M5.1.2 lands). No prior `tc_` cookies
  to collide with.
- **`stateTtlSeconds` default = 10 min.** Plenty for a sane
  Google round-trip; short enough that a forgotten browser tab
  isn't a stale-state hazard.
- **`state` is 32 random bytes b64u'd.** Same as the verifier
  (per `generatePkce`). Already CSRF-grade.
- **Redirect uses `302 Found` with `Cache-Control: no-store`.**
  An accidentally cached 30x to Google's authorize URL would be
  bad UX on subsequent sign-ins (different state each time).
- **No `nonce` parameter yet.** Google's ID token will carry one
  if we send it; the callback (M5.1.2) will need a `nonce` claim
  check anyway for OIDC robustness. M5.1.2 can add it in one
  place — the cookie payload extends to `{state, verifier, nonce,
  exp}` and the route forwards it as `&nonce=`. Not adding
  speculatively this iteration.

## Notes for future iterations

- **M5.1.2 (next ordinary iteration).** Add
  `apps/web/src/routes/auth/google/callback/+server.ts`:
  1. Read `tc_oauth_state` cookie; `verifyStateCookie` against
     `config.signingKey` and current epoch. On failure: 400 with a
     terse one-liner; clear the cookie (`Set-Cookie:
     tc_oauth_state=; Path=/auth; Max-Age=0`).
  2. Read `code` + `state` from query. `state` must equal the
     cookie payload's state (otherwise 400).
  3. POST to `https://oauth2.googleapis.com/token` with
     `grant_type=authorization_code`, `code`, `redirect_uri`,
     `client_id`, `client_secret`, `code_verifier`. Use
     `application/x-www-form-urlencoded`.
  4. JWKS-verify the returned `id_token`. Easiest: add a tiny
     `jose` dep (well-audited, tree-shakes to <10kB for the JWKS
     case). Caching of JWKS keys is fine in-memory in the control
     plane.
  5. Extract `email`, `email_verified` from the ID token claims.
     `isAllowedEmail(email)` + `email_verified === true` →
     proceed; else clear cookie and 302 back to `/`.
  6. Insert a session row into Postgres (uuid pk, user id, exp,
     created_at). `@tex-center/db` already has a sessions table per
     M4.0.
  7. `signSessionToken({sid, exp})` → `Set-Cookie: tc_session=…;
     Path=/; HttpOnly; SameSite=Lax; Secure; Max-Age=<TTL>`.
  8. 302 to `/editor`.

  Implementation tactic: same pure-builder + thin-route split that
  worked here. A `buildGoogleCallbackResolution({…})` returning a
  discriminated `{kind: "redirect", location, cookies}` /
  `{kind: "error", status, body}` keeps the unit test out of
  SvelteKit's machinery. The fetch + DB insert stay in the
  `+server.ts`, but can be parameter-injected if desired.

- **`SESSION_SIGNING_KEY` rotation.** Single key for now;
  rotation would require keyset support in `signed.ts`. Defer
  until there's a reason.

- **`hooks.server.ts` (M5.1.3)** reads `tc_session`, calls
  `verifySessionToken`, looks up the session row, populates
  `event.locals.session`. Protected pages (the `/editor`
  client-only routes — they ship as prerendered HTML, but the
  `+server.ts` channels they hit can guard themselves) check
  locals.

- **OAuth config readiness.** `loadOAuthConfig` is called per
  request from `+server.ts` but memoised; the first request after
  process boot reads `creds/google-oauth.json` once. If the file
  is rotated, the process must restart — fine for the per-deploy
  model.
