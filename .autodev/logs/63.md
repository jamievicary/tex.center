## Approach

- Protocol: `{ type: "rename-file"; oldName; newName }`.
- Persistence `renameFile(old, new)`:
  - Rejects `MAIN_DOC_NAME` on either side, unknown `oldName`,
    duplicate or invalid `newName`.
  - Blob-store path: PUT new key with old contents â†’ DELETE old.
    A DELETE failure after a successful PUT orphans the old blob
    rather than rolling back (rollback could itself fail and
    leave both keys populated, which is worse â€” in-memory truth
    has moved on).
  - Doc transact: insert old contents into new `Y.Text`, then
    clear old. Single transaction so observers see one update.
  - Updates `knownFiles` / `persistedByName` last so a PUT
    failure leaves prior state intact.
- Server: `rename-file` handler broadcasts a fresh `file-list`
  on success; logs reason on rejection.
- `WsClient.renameFile(oldName, newName)`.
- `FileTree`: per-row "âśŽ" button (hidden for `main.tex`) opens
  `window.prompt` for the new name. Lightweight UI â€” MVP.
- Editor page: when the renamed file is the selected one, point
  `selected` at `newName` before sending so the `$effect` rebinds
  CodeMirror to the new `Y.Text` (which already carries the old
  contents post-transact).

## Notes for future iterations

- File-tree CRUD is now complete: create / delete / rename all
  land via protocol verbs against the in-memory Y.Doc and the
  blob store. The remaining "natural" file-tree affordance is
  upload-from-disk, but the current `create-file` already gives
  the user a way to seed a `Y.Text` they can paste into.
- The "PUT new then DELETE old, orphan on DELETE failure"
  policy in `renameFile` is the simplest correct shape against a
  store with no atomic rename. Worth revisiting once the S3
  adapter lands (M4.3.1): S3 has no rename either, but a
  scheduled garbage-collection pass on `projects/<id>/files/`
  could pick up orphans.
