## Goal for this iteration

M9.resource-hygiene second slice: diagnose + fix idle-stop. Per
`flyctl machine list -a tex-center-sidecar` snapshot at iter
start, 10 machines exist; two per-project ones
(`7815104f060d28` withered-dawn-7366 and `28624d5c745998`
morning-haze-7391) are `started` 17+ minutes after creation,
past the 10-min idle window. `flyctl logs` for `7815104f060d28`
shows only `Server listening at http://[::]:3001` after boot —
**no viewer ever connected**.

Root cause in `apps/sidecar/src/server.ts`: the idle timer only
arms on the `viewerCount: 1→0` transition inside
`noteViewerRemoved`. A Fly Machine that boots (e.g. the control
plane proxy triggers wake but the WS handshake never completes,
or the user navigates away mid-cold-start) never sees 0→1→0, so
the timer never arms. The machine stays `started` indefinitely.

Fix: arm the idle timer at server startup. First
`noteViewerAdded` clears it; existing flows are unchanged.
Add a regression unit test for the "no viewer ever connects"
case.

Live diagnostics:
- `SIDECAR_IDLE_TIMEOUT_MS` is **not** set in
  `tex-center-sidecar` secrets — code default 600_000ms (10
  min) is in effect. Fine.
- Stopped per-project machines show stop times 10–12 min after
  start, confirming idle-stop *does* fire when a viewer
  connect+disconnect cycle happens. The bug is specifically
  the never-connected path.
