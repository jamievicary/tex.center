## Goal for this iteration

PLAN priority #1 (M20.3 closeout). Iter 342 framed the residual
GT-9 + GT-6-stopped RED failure as a disconnect-arm race: a
transient cold-reopen WS open-then-close cycle drives
`noteViewerRemoved` → 5 s `suspendStage.arm()` → self-suspend
mid-handshake, and the web proxy's 6PN TCP dial cannot auto-resume
a suspended Machine. Iter 342 added diagnostic logs (deployed at
07:17:21 Z, image `@sha256:aaedff…`) so the next gold pass would
record viewer-add/remove + idle-arm timing.

Iter 343's goal: apply the targeted fix without waiting for a
manual GT-9 transcript. The hypothesis is well-grounded (matches
iter 341's `flyState=starting → suspended` shape exactly, and is
the only remaining suspend-arm site), and the fix generalises the
iter-340 invariant rather than introducing a new mechanism.

## Approach

Generalise the iter-340 cold-boot invariant: `buildServer` arms
ONLY the stop stage on every idle entry, never the suspend stage.
The argument is symmetric. Iter 340 forbade suspend-on-cold-boot
because the 5 s suspend timer raced the 20–60 s drive-to-started
+ tcpProbe + WS upgrade chain. The disconnect path has the same
structural property: the proxy *might* be about to retry, the
client *might* be reopening, and the 5 s timer can win either
race. Until we have a definitive "no more reconnects coming"
signal — an explicit client→server tab-close frame — the safest
default is "never auto-suspend".

The cost is real but bounded: a tab-closed Machine stays in
`started` state (RAM allocated) until the 5-minute stop timer
fires, instead of suspending in 5 s. Acceptable for the user
counts at this stage; trades latency-and-correctness for memory.
Recorded in `FUTURE_IDEAS.md` so the optimisation is recoverable
when the explicit tab-close wire lands.

The suspend primitive is kept in the API surface
(`suspendTimeoutMs`/`onSuspend` still accepted by `buildServer`)
so the future wiring needs no plumbing changes — only an arm
site, gated on the explicit signal.
