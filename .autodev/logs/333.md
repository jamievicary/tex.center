## Goal for this iteration

PLAN priority #1, M20.3(b): add the real preservation gold spec
PLAN named as the next slice once M20.3(a) and (a)2 landed
(iter 331/332). The existing `verifyLiveGt6LiveEditableStateStopped`
only asserts the editor shows the `documentclass` sentinel after a
force-stop + cold reopen — `MAIN_DOC_HELLO_WORLD` (the in-sidecar
fallback that runs when the blob store is missing/empty) satisfies
that text, so a broken Tigris persistence path would still let it
pass. We need a spec that types a **unique** marker, lets one
post-edit compile run (which is what triggers
`persistence.maybePersist()` writing to the blob store — see
`apps/sidecar/src/server.ts:549`), force-stops the Machine, and on
cold reopen asserts the unique marker appears in `.cm-content`.

## Approach

New live-only Playwright spec
`tests_gold/playwright/verifyLiveGt9StoppedPreservesEdits.spec.ts`:

1. Create a fresh project; cold-start by navigating to
   `/editor/<id>`; wait for the first `pdf-segment` so the editor
   is fully hydrated and the daemon is ready.
2. Click `.cm-content`, press `Control+End`, type a comment line
   `% preserve-<randomUUID>`. LaTeX ignores anything after
   `\end{document}`, so the compile path stays green; the sentinel
   is a fresh UUID per run so two runs can never collide via stale
   blobs.
3. Wait for the next `pdf-segment` — proof of one
   `runCompile` end-to-end, which means `maybePersist()` has run
   and the source is in Tigris.
4. Leave editor → `/projects` (closes the WS).
5. Force-stop the Machine via Fly `POST /machines/{id}/stop`,
   poll `GET /machines/{id}` until `state === "stopped"`. This is
   the same state the sidecar's `process.exit(0)` idle handler
   leaves the Machine in (mirroring `verifyLiveGt6LiveEditable
   StateStopped`).
6. Click the dashboard link to reopen; the Fly proxy must
   cold-start the Machine; the sidecar's persistence layer must
   hydrate the Y.Doc from the persisted `main.tex` blob; the
   client must receive the hydrated doc.
7. Poll `.cm-content textContent` for the sentinel substring;
   `.cm-line` joins drop the inter-line separators but the
   sentinel is on a single line so the substring stays intact.

Per-phase budgets are chosen so that on a healthy production
deploy (cold-start ~10–15 s) the test completes well under its
5 min ceiling, while a real persistence regression surfaces as a
diagnostic timeout on the sentinel poll rather than the test
ceiling. Latency-agnostic by design — pins **byte preservation**,
not perf (a separate spec, GT-6-stopped, covers cold-start time).
