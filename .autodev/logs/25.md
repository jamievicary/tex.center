## Goal for this iteration

Land M4.3.2 (sidecar wiring of `BlobStore`) for the source-file
half: hydrate the project's `Y.Text` from a configured blob
store on project open, persist back after each successful
compile. Checkpoint persistence (the second half called out in
the plan) waits on M3.5/M7 ŌĆö compilers don't yet expose
checkpoint blobs, and upstream `supertex` doesn't serialise them
either, so wiring it now would be vapour.

PLAN flagged this and M3.5 as next-up; M3.5 is two upstream PRs
against `vendor/supertex`, not actionable from inside this
checkout, so M4.3.2-source-half it is.

## Approach

1. `apps/sidecar/package.json` ŌĆö add `@tex-center/blobs` as a
   workspace dep.
2. `apps/sidecar/src/server.ts`:
   - `SidecarOptions.blobStore?: BlobStore`. Default factory
     keys off `BLOB_STORE` env (`unset`/`none` ŌåÆ no persistence;
     `local` requires `BLOB_STORE_LOCAL_DIR`; `s3` errors with a
     "M4.3.1 reserved" message).
   - `ProjectState` gains `id`, `persistedSource`, and a
     `hydrated: Promise<void>`.
   - On first `getProject(id)`, kick off
     `blobStore.get('projects/<id>/files/main.tex')`; if non-
     empty bytes come back, `text.insert(0, source)` and seed
     `persistedSource`.
   - The WS handler's initial state encode + first compile schedule
     are deferred until `project.hydrated` resolves, so the first
     client sees the persisted source. The compile pipeline
     `await`s hydration too (defensive ŌĆö a third actor could
     trigger it).
   - After a successful compile, if the source differs from
     `persistedSource`, write it back via `blobStore.put` and
     update `persistedSource`. Failures log-and-continue
     (avoids blob outages stalling the editor; the next compile
     retries).
3. `apps/sidecar/test/serverBlobs.test.mjs` ŌĆö boots a server
   against a `LocalFsBlobStore`, pre-populates `main.tex`,
   asserts the first client sees it; sends an edit, asserts the
   blob picks up the change; restarts the server against the
   same root, asserts the new content rehydrates.
4. `tests_normal/cases/test_node_suites.py` ŌĆö register the new
   case.
5. `.autodev/PLAN.md` ŌĆö mark M4.3.2 `[~]` (source half done,
   checkpoint half deferred); update Current focus.

## Problems encountered

- First test cut waited for the local `Y.Text` to reflect our
  edit by listening for the server's echo. The server's
  origin-tagged update path explicitly **does not** echo back to
  the originator (correct: prevents infinite loops). Switched
  the assertion to "wait for the blob to update", which is the
  invariant we actually care about anyway. Frames inspection
  (`["control","doc-update","doc-update","control","pdf-segment","control"]`)
  pinned the missing echo before I went hunting in the wrong
  place.
- An async predicate was passed to a sync `setInterval`-based
  `waitFor`, which silently truthy-evaluated the Promise.
  Rewrote the helper to `await check()` in a bounded loop.

## Notes for future iterations

- Single-file: today only `main.tex` is hydrated/persisted.
  Multi-file projects will need a `list("projects/<id>/files/")`
  walk on hydration and per-edit tracking of which file changed
  ŌĆö this lives behind whatever frontend story replaces the
  current single Y.Text model.
- Checkpoint persistence (the deferred half of M4.3.2): when
  `Compiler` grows a `checkpoint(): Promise<Uint8Array | null>`,
  the cleanest spot to call it is the existing `app.close()`
  hook, just before `compiler.close()`. Key:
  `projects/<id>/checkpoint.bin`. Do NOT persist incrementally
  ŌĆö the whole point is rehydration on cold-start.
- Persist failures currently warn-and-continue. If a real
  outage on Tigris drags on, the editor keeps working but the
  source diverges silently. Worth surfacing as a "compile-status"
  warning (or new control message) once we have a UI affordance
  for "your edits aren't being saved right now".
