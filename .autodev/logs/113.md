## Goal for this iteration

Close out two gaps in `apps/sidecar/test/supertexDaemonCompiler.test.mjs`
that were explicitly deferred to M7.5.5 when the daemon compiler
landed in iter 107: a **rollback** case and an **error→ok recovery**
case on a single daemon process. Both are tractable against the
existing fake-daemon harness (no real `supertex` ELF needed), and
they're the only items in the M7.5.5 test list that weren't already
exercised. M7.1.3.2.b (the WS-upgrade-with-cookie live probe) is the
"next ordinary" pick in PLAN, but it needs a Machine-cleanup hook
that the existing Playwright fixtures don't have — bigger slice. The
two daemon cases are independent and let M7.5.5 cross most of the
finish line.

## Approach

Extend the fake-daemon's mode switches with two new behaviours,
keyed off env vars passed through `makeSpawnFn`:

- `FAKE_MODE=rollback` (with `FAKE_ROLLBACK_K`): emit chunks
  `0..total-1`, then unlink chunks > K from `DIR` (mirroring the
  real daemon's behaviour on rollback), then emit `[rollback K]`,
  then `[round-done]`. The compiler's `collectRound` should set
  `maxShipout = K`, and `assembleSegment` should produce just the
  surviving chunks.
- `FAKE_MODE=error-then-ok` (with `FAKE_ERROR_ROUNDS`): emit
  `[error simulated failure]` on rounds 1..N, then succeed
  thereafter. Lets us exercise that the daemon process is reused
  across a failed compile and that the next compile returns ok.

Two new test blocks (#10, #11) drive these. The rollback case also
asserts `readdir(chunksDir)` afterwards, so a future regression
where the compiler tracks rollback in memory but the assembler
re-globs the directory would be caught.
