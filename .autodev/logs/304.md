## Goal for this iteration

M22.1 (active queue #1) — extend `wsClient.ts`'s debug-event
fan-out to cover every outbound control frame, not just
`outgoing-doc-update`. Today the only wire send that surfaces as a
debug toast is the Yjs op; the other five sends (`setViewingPage`,
`createFile`, `uploadFile`, `deleteFile`, `renameFile`) are silent.
The PLAN names this slice as the prerequisite for closing GT-F.
`recompile-request` is listed there too but `wsClient.ts` never
sends one (recompile is server-driven from a `doc-update`), so I
omit that variant and note it.

## Approach

- New `WsDebugEvent` variants, all symmetrical with
  `outgoing-doc-update`:
  - `outgoing-viewing-page` (`page: number`)
  - `outgoing-create-file` (`name: string`)
  - `outgoing-upload-file` (`name: string`, `bytes: number`)
  - `outgoing-delete-file` (`name: string`)
  - `outgoing-rename-file` (`oldName: string`, `newName: string`)
- Emit per send, *only if* `this.send()` returned `true` (socket
  open). Same convention as the existing `outgoing-doc-update`
  guard — pre-open writes are no-ops on the wire and so must be
  no-ops on the debug stream too.
- All outgoing-* events map to `debug-green` (consistent with
  the existing Yjs-op category). Shared aggregateKey per kind
  (e.g. `debug:viewing-page`, `debug:create-file`, …) so bursts
  coalesce — page-scroll spam should collapse to one `×N`. The
  `text` carries the per-call detail (file name, page number,
  byte count) so the latest pumped wins in the aggregation
  window.
- Test extends `wsClientDebugEvents.test.mjs`: a new case 9
  exercises each of the five sends after `sock.open()` and
  asserts one event per call with the right shape; pre-open
  sends remain silent. Case 7 (the `debugEventToToast` mapping
  matrix) gains the five new kinds with `debug-green` +
  non-empty `aggregateKey`.


## Actions

- `apps/web/src/lib/wsClient.ts`: `WsDebugEvent` union grew five
  outgoing variants. Each public send method
  (`setViewingPage` / `createFile` / `uploadFile` / `deleteFile` /
  `renameFile`) captures the `send()` return value and fires
  `onDebugEvent` only on `true`. `uploadFile` reports the
  UTF-8-encoded byte length of `content` to match the wire payload
  size users care about, not the JS string length.
- `apps/web/src/lib/debugToasts.ts`: five new `case` arms in
  `debugEventToToast`. All `debug-green`. aggregateKey is the
  fixed per-kind string `debug:outgoing-<verb>`; toast `text`
  carries the per-call detail (page number / file name / byte
  count / rename pair) so bursts coalesce on count + show the
  latest payload.
- `apps/web/test/wsClientDebugEvents.test.mjs`: case 7 mapping
  matrix gains five rows (all `debug-green`); new case 9 drives
  each of the five sends pre- and post-open and asserts (a)
  pre-open silence, (b) one event per call post-open, (c)
  per-event shape.
- `.autodev/PLAN.md`: M22 active-queue entry updated; M22.1
  marked closed with lock-file pointer; explicit note that
  `recompile-request` is not in scope.

## Notes for future iterations

- **M22.2 next.** With M22.1 in place, the GT-F local Playwright
  spec can be written end-to-end: enable `?debug=1` on a local
  project, drive a keystroke + `setViewingPage(2)` etc, assert the
  Toasts.svelte renders the expected coloured chips with `×N`
  badges. The toast store + Toasts.svelte aggregation is already
  unit-tested; this is the integration pin.
- **`outgoing-doc-update` is the only outgoing event keyed on
  byte count alone.** The five new outgoing events carry richer
  payload (file name, page number) in their `text`. If a future
  iteration adds a UI surface that wants to display the *latest*
  affected filename per category (e.g. a status-bar pill), the
  store already retains the most-recent `text` across an
  aggregation window — no extra wiring needed.
- **`recompile-request` left out.** The web client never sends
  one. If a future protocol revision lets the user request a
  forced recompile from the UI, add an `outgoing-recompile-request`
  variant then; not before.
