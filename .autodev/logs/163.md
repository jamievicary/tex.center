## Goal for this iteration

Per PLAN iter-163: read the iter-162 deploy live-pipeline
result and diagnose. Result: **branch (b)** — M8.pw.4 failed on
`pdf-segment` poll timeout (240s, 0 frames). Combined with
sidecar logs (per-project Machines `e829704ae25658` and
`d8d329da16e738` were spawned, "Server listening at
http://[::]:3001", then **no `/ws/project/...` incoming requests
ever logged**), the dial almost certainly didn't reach the
Machine — or did and a subsequent step failed silently.

Found the observability gap: `apps/web/src/server.ts:80` calls
`boot(...)` without an `onEvent` callback, and `boot.ts` doesn't
even forward one to `attachWsProxy`. So every `resolve-error` /
`upstream-error` / `auth-error` / `closed` event the WS proxy
emits goes to a noop. The production tex-center logs have zero
information about WS-proxy outcomes — we cannot tell from logs
whether dial succeeded, timed out, or hit a 502.

This iteration's slice: wire `onEvent` plumbing through
`boot()` → `attachWsProxy`, log structured events to stdout in
`server.ts`, and add a regression locking the wiring. Also add
info-level logs in the sidecar on Yjs `doc-update` arrival and
compile result, so the next deploy gives us a diagnosable run on
both sides of the proxy.

## Changes

- `apps/web/src/lib/server/boot.ts`: new `onWsProxyEvent` field
  on `BootOptions`, forwarded to `attachWsProxy` as `onEvent`.
- `apps/web/src/server.ts:80`: supplies the shim
  `(event) => console.log(JSON.stringify({ ws_proxy: event }))`
  so every WS-proxy lifecycle event lands in `flyctl logs -a
  tex-center` as a single greppable JSON line.
- `apps/sidecar/src/server.ts`: pino info logs at three points:
  client `doc-update` arrival (updateBytes), compile start
  (sourceLen), compile ok (elapsedMs / segments / bytesShipped)
  / compile error (elapsedMs / error).
- `apps/web/test/boot.test.mjs`: regression — supplies
  `onWsProxyEvent` to `boot()` and asserts the captured kinds
  include both `no-match` (from the /nope upgrade) and
  `upstream-connect`(ed) (from the /ws/project/proj1 upgrade).
