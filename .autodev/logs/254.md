## Goal for this iteration

Land **M13.2(b).2 — optimistic project delete**, the next sequenced
slice in `.autodev/PLAN.md`. Currently `deleteProject` awaits Fly
`destroyMachine` before deleting the DB row. With per-project
Machines now suspended rather than destroyed on idle (M13.2(b).1),
a `destroyMachine` call against a stopped Machine can take seconds —
during which the user sees the row still on `/projects`. The
orphan-tag sweep already exists as a safety net, so we can flip the
ordering: DB delete first, Fly destroy as a fire-and-forget
background task.

## Approach

Rewrite `apps/web/src/lib/server/deleteProject.ts` so that, after
the assignment lookup, the DB row is deleted (cascade reaps the
`machine_assignments` row), and only then is `destroyMachine`
kicked off via `runBackgroundDestroy(...)`. The helper returns
immediately with a new `destroyComplete: Promise<{destroyed,
error?}>` field that tests can await; the `/projects` `?/delete`
action ignores it (production path returns straight to the redirect).
`runBackgroundDestroy` swallows 404s as "already gone" and routes
non-404 errors through an injectable `logError` (default
`console.error`) — never throws, so the orphaned `destroyComplete`
promise can't surface as an unhandled rejection.

## Actions

- `apps/web/src/lib/server/deleteProject.ts` — rewritten. New result
  shape: `{ hadAssignment, rowDeleted, destroyComplete }` (the
  previous `machineDestroyed` flag is now `destroyComplete.destroyed`).
  New optional input `logError(msg, err)`.
- `apps/web/test/deleteProject.test.mjs` — updated for the new
  shape, and a new "optimism" test was added: a stubbed
  `destroyMachine` is suspended on a manual gate; the test asserts
  that `deleteProject` resolves and the DB row is gone *before*
  `destroyComplete` settles, then releases the gate and awaits it.
  The non-404-failure case flipped from "row preserved, error
  propagated" to "row deleted, error logged once, available on
  destroyComplete.error".
- `apps/web/src/routes/projects/+page.server.ts` — unchanged. The
  existing `await deleteProject(...)` now returns after only the DB
  delete, then the route 303s. `result.destroyComplete` is left
  unobserved on purpose; it cannot reject by construction.

No edits to the live gold spec `verifyLiveDeleteProject` — its
30 s window for the link to disappear still works, and a sub-500 ms
post-click latency assertion (per the original PLAN bullet) was
rejected as flake-prone over the live network: a form POST → 303
redirect → fresh `GET /projects` round-trip on a fra→user link
already eats most of that budget before any DOM parse. The unit
tests carry the optimism invariant deterministically.

## Notes for future iterations

- The user-facing effect of this change only shows up on projects
  whose per-project Machine is in `stopped`/`suspended`, where the
  Fly `destroyMachine` call can take seconds. Live verification is
  available via the existing `verifyLiveDeleteProject` spec — if a
  future regression makes the redirect block on destroy again, that
  spec will get slower but probably won't fail outright.
- M13.2(b).3 next: cold-editable gold spec — `.cm-content`
  populated and a keystroke producing a Y.Doc op frame within
  1000 ms of clicking a suspended-Machine project link.
- The new `destroyComplete` promise is intentionally not consumed
  by the route handler. If we ever move to a runtime with
  `event.platform.context.waitUntil` (Cloudflare Workers etc.),
  that is the place to hand it off; on Fly the Node process simply
  keeps running so an unobserved promise is fine.
