## Problems encountered

### 1. `flyctl deploy ../../` accidentally redeployed the web app

First attempt: `cd apps/sidecar && flyctl deploy --remote-only
--no-public-ips ../../ -c fly.toml`. flyctl treats the positional
as the project directory, then re-reads `fly.toml` from *there*
(the repo root), so it built `apps/web/Dockerfile` and pushed it
to the `tex-center` (control plane) app, taking the running
Machine from v18 â†’ v19. The image is functionally identical (no
apps/web source changed) and the post-deploy check is passing â€”
zero damage â€” but worth recording so future deploys remember to
always run with both `-a <app>` and `--config <path>` explicit.
Updated `deploy/README.md` workflow is implied (see PLAN follow-
up); the canonical sidecar deploy command is captured here.

### 2. `[build].dockerfile` is relative to fly.toml dir, not WD

Second attempt: `flyctl deploy ... --config apps/sidecar/fly.toml
.` with `[build] dockerfile = "apps/sidecar/Dockerfile"`. flyctl
resolved that to `apps/sidecar/apps/sidecar/Dockerfile` â€” i.e.
the path is joined to the config file's directory, not the build
context working directory. Web's fly.toml gets away with
`apps/web/Dockerfile` because it lives at the repo root, where
both interpretations coincide. Fixed by setting `dockerfile =
"Dockerfile"` in `apps/sidecar/fly.toml` and documenting why in
a comment. Test updated accordingly.

### 3. (Real M7.0.2 blocker.) lualatex.fmt dump fails in runtime stage

Third attempt got all the way through pnpm install / supertex
make / texlive-full apt-install, then crashed at the very last
runtime layer (the `lualatex.ini` â†’ `lualatex.fmt` dump):

```
warning: (kpathsea) kpathsea: configuration file texmf.cnf not
  found in these directories: /opt/engine:/opt/engine/share/...
This is LuaTeX, Version 1.25.9 (TeX Live 2027/dev)  (INITEX)
! I can't find file `lualatex.ini'.
```

The patched engine was compiled against TL 2027/dev, so its
hard-coded kpathsea search prefix is `/opt/engine/...`. The
runtime stage installs Debian-packaged `texlive-full` (TL 2022)
into `/usr/share/texlive/...`, which the engine binary has no
way to find on its own â€” none of the post-install RUN steps set
`TEXMFCNF` or symlink the system tex tree into `/opt/engine/`.

This is **a M7.0.1 Dockerfile bug**, not a fly.toml or M7.0.2
issue. iter 75 added the engine binary and the fmt-dump RUN
layer but the structural test never exercised an actual build,
so the missing-`TEXMFCNF` regressed silently. The maintainer's
local `setup_smoketest.sh` evidently runs inside a fuller env
where kpathsea finds texmf.cnf via `TEXMF*` env vars set in the
user's shell, not the binary's compiled-in defaults.

Likely fix (next iteration): prepend `TEXMFCNF` and/or `TEXMF`
to the RUN env so kpathsea sees the system texlive's `texmf.cnf`
(`/usr/share/texlive/texmf-dist/web2c/texmf.cnf`). Possibly also
need `TEXMFDIST` and `TEXMFMAIN` exported. Quick local repro:
`docker build apps/sidecar/` once docker is available, or
re-deploy with the env fix.
