## Goal for this iteration

Continue PLAN priority #1 (M20.3 GT-9). Iter 337 added an enriched
`7-sentinel:probe` log to the sentinel-poll loop and tightened the
`textContent` call to a 1 s per-iteration timeout. The iter-337 gold
pass then produced an extremely clean signal:

```
phase=6-reopen:click       elapsedMs=32506
phase=6-reopen:url-arrived elapsedMs=32558  (52 ms after click)
phase=7-sentinel:poll      elapsedMs=32558
phase=7-sentinel:probe iter=1  cmLen=0 cmPrefix=
phase=7-sentinel:probe iter=21 cmLen=0 cmPrefix=
phase=7-sentinel:probe iter=41 cmLen=0 cmPrefix=
phase=7-sentinel:probe iter=61 cmLen=0 cmPrefix=
phase=7-sentinel:done  iter=79 found=false cmLen=0
```

`.cm-content` is **completely absent** for the entire 90 s
post-reopen window. This is structurally not a sentinel-text
mismatch (cmLen would be ~50–200 for the canonical seed); it's
"editor element never mounts". Per `apps/web/src/routes/editor/
[projectId]/+page.svelte`, `.cm-content` is gated on
`snapshot.hydrated === true`, which only flips once the per-project
sidecar WS opens AND the Yjs initial-sync frame lands. So the
failure is one of three:

1. **WS never opens.** Fly proxy can't cold-start the Machine
   within 90 s, or the upgrade is failing.
2. **WS opens but Yjs initial sync never completes.** Sidecar
   accepts the WS but the hydration handshake never finishes —
   e.g. the persistence-rehydrate-from-blob step is broken or
   hangs on the post-stop reopen path.
3. **Page-level crash before mount.** SvelteKit client-side nav
   from `/projects` to `/editor/<id>` throws during hydration; the
   editor route's `onMount` never runs.

Note GT-6-stopped is RED for the same underlying reason
(`verifyLiveGt6LiveEditableStateStopped` line 167 of iter 337's
gold summary), so this is a single load-bearing bug, not a GT-9
test artefact. Fixing it unblocks **two** open gold cases.

Iter 338 enriches the diagnostic probe one more step so the next
gold pass discriminates which of (1)/(2)/(3) is happening. No
production code change; iteration is on the diagnostic shape only.
Cost is justified because the alternative — speculating which
subsystem is broken and shipping a fix — would burn iterations and
risk masking the real fault.

## Approach

Add the following telemetry to GT-9's spec body without touching
production code:

- **WS lifecycle counters** maintained across the whole test:
  `wsOpenCount`, `wsCloseCount`, `wsFrameCount`, `lastWsCloseInfo`.
  Listeners attached when `authedPage.on("websocket")` matches the
  project URL. Snapshots taken at entry to the sentinel-poll loop
  so the probe can show delta-after-reopen.
- **Page error capture**: `pageerror` listener (uncaught
  JavaScript exceptions in the page context) and
  `console.error` listener. Collected into `pageErrors[]`; probe
  surfaces count + last entry.
- **Per-probe `page.evaluate`** in a single round-trip per probe
  collects `location.href`, `document.body.innerText.length`,
  `.editor-seed` presence (SSR placeholder), `.cm-content`
  presence, `.cm-editor` presence, `document.title`.

The probe now logs roughly:

```
phase=7-sentinel:probe iter=… cmLen=… url=… title=… bodyLen=…
  seedPresent=… cmContentPresent=… cmEditorPresent=…
  wsOpens=… wsFrames=… wsCloses=… lastWsClose=…
  pdfSegments=… pageErrorsCount=… lastPageError=…
```

Decision matrix the next gold pass should drive:

- `bodyLen=0` ⇒ page is blank / errored before render. Look at
  `url`, `title`, `lastPageError`.
- `wsOpens=0` ⇒ client never even tried (or only tried before the
  reopen) — likely client-nav routing bug.
- `wsOpens>=1 && wsFrames=0` ⇒ Fly proxy isn't completing the
  upgrade to the stopped Machine; cold-start is failing or
  exceeding budget. Worth raising COLD_START_BUDGET_MS or
  investigating Fly proxy behavior on stopped + force-stopped
  Machines.
- `wsOpens>=1 && wsFrames>0 && pdfSegments=0 && cmEditorPresent
  =false` ⇒ WS open but Yjs initial sync stuck (hydration path
  broken) — investigate sidecar persistence rehydrate.
- `cmEditorPresent=true && cmContentPresent=false` ⇒ editor
  mounted but CodeMirror initialization broke.
- `seedPresent=true` ⇒ assignment was null at SSR (surprising for
  a re-opened project); SSR path is wrong.
- `lastPageError` non-empty ⇒ JS crash — name and stack tell us
  exactly where.

Production code intentionally unchanged. Behavioural assertions
unchanged. The PASS path is identical (first iteration finds the
sentinel ⇒ break ⇒ probe doesn't fire). The FAIL path gains the
diagnostic payload.
