## Wiring bug discovered + fixed

Iter-85's globalSetup-export-env pattern was untested: the only
playwright spec at the time (`landing.spec.ts`) doesn't read the
DB, so the env never needed to reach the dev server. Adding the
first authed spec revealed that Playwright's runner starts
`webServer` plugins *before* `globalSetup` runs
(`playwright/lib/runner/tasks.js:100` orders `pluginSetupTasks`
then `globalSetups`). The dev server therefore inherited
`process.env` from before globalSetup, with no `DATABASE_URL` or
`SESSION_SIGNING_KEY` — `hooks.server.ts` collapses to anonymous
on a missing signing key, so every authed redirect silently
bounced the user back to `/`.

Fix (in this iteration): own the dev-server spawn from
`globalSetup.ts` instead of relying on Playwright's `webServer`
block. The new code:

- Pre-checks port 3000 isn't already in use (else fails fast with
  a leak-diagnostic message — found one stale vite hanging
  around mid-iteration).
- Spawns `pnpm --filter @tex-center/web dev --port 3000` with
  `detached: true` after the env is set.
- Polls `http://127.0.0.1:3000/` with a child-exit short-circuit.
- Teardown signals the process group (`process.kill(-pid, ...)`)
  so vite (a grandchild of pnpm) dies cleanly; SIGTERM, 5s wait,
  SIGKILL fallback.

## Notes for future iterations

- The dev server is now under our control, not Playwright's.
  If a future spec needs `reuseExistingServer` semantics for
  iterative dev, that's a separate helper — but the gold runner
  is single-shot, so the boot cost is paid once per invocation.
- `apps/web/src/lib/server/db.ts`'s `createDb` call doesn't pass
  an `onnotice` handler, so the PGlite-over-TCP server's
  default NOTICE/DEBUG stream leaks to the dev server's stdout.
  Currently silenced at the test boundary by dropping dev stdout
  in `globalSetup.ts`. If anything else ever wants dev stdout,
  the right fix is to silence notices at the `getDb()` boundary
  (a one-line change in `db.ts`).
- pw.2 is the natural home for the deferred "404 for stranger's
  project" case: with `+layout.ts`'s `ssr=false`, the initial
  navigation always returns 200 and the 404 only surfaces on
  the SvelteKit data fetch. Either flip to SSR for `/editor/*`
  or add a CSR-aware assertion that waits for the error page.
- Stray file `packages/db/src/sessions.ts.tmp.3044.1778447037805`
  still untracked (carried over from iter 84).
