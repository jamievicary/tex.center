## Diagnosis (from live logs)

`flyctl logs -a tex-center` for the 14:18Z spec window showed the
classic cold-start race for the spec's freshly-created project
`ec9bab00-…8198ef`:

- 14:16:50Z resolver upserts `machine_assignments` row pointing at
  Machine `d8d30e0b14e408` (Postgres logged as a `resolve-error` —
  drizzle's error message is the query text, the real cause was
  drowned; unrelated to the fix but a separate observability gap).
- 14:18:18Z sidecar Machine `286ded5c751948` reaches `started`
  per Fly. Image pull took ~1m17s, exactly the population iter-164
  added cold-start API retries for.
- 14:18:21Z sidecar process logs `Server listening at http://[::]:3001`.
- 14:18:21Z web ws-proxy fires `upstream-connect` to
  `286ded5c751948.…internal:3001`, immediately followed by
  `upstream-error: connect ECONNREFUSED fdaa:74:…:3001`.

The dial raced the sidecar's `listen()` by sub-second. The browser
side never recovered within the spec's 240s — only one
`upstream-connect` for `ec9bab00-…` appears in the window. Same
race seen earlier for `81e0c061-…` and `7128ec25-…`.

## Approach

Close the gap inside the resolver, not the wsProxy: after Fly says
`started`, probe the upstream's TCP port until accept succeeds
(bounded by `tcpProbeTimeoutSec`, default 60s) before returning
the upstream. Anything past `started` that ECONNREFUSEs is exactly
this race; retrying at the connect layer eliminates the entire
class without needing wsProxy to know about Machine lifecycle.

## Actions

- `apps/web/src/lib/server/upstreamResolver.ts`
  - New `defaultTcpProbe(host, port)` exported helper —
    `net.connect`, resolve on `connect`, reject on `error`.
  - `UpstreamResolverOptions` gains optional `tcpProbe` and
    `tcpProbeTimeoutSec` (default 60s).
  - `resolveOnce` runs `waitForUpstreamReady(upstream)` after
    `driveToStarted` returns; retries the probe every 500ms until
    accept or deadline, then throws with the last probe error
    quoted.
- `apps/web/src/lib/server/upstreamFromEnv.ts`
  - Threads `tcpProbe` (defaulting to `defaultTcpProbe`) and
    `tcpProbeTimeoutSec: 60` into the production resolver.
  - `UpstreamFromEnvDeps` gains optional `tcpProbe` test injection
    point.
- `apps/web/test/upstreamResolver.test.mjs`
  - `baseOpts.tcpProbe = async () => {}` keeps cases 1–9 working
    without a real net.connect.
  - New case 10: probe fails ECONNREFUSED twice then ok → resolver
    succeeds and the probe was called three times.
  - New case 11: `tcpProbeTimeoutSec: 0` + always-failing probe →
    resolver rejects with the budget-exceeded message.
- `apps/web/test/upstreamFromEnv.test.mjs`
  - All three resolver-invoking cases now pass `tcpProbe: async
    () => {}` so the test's fake `m-123.vm.sidecar-app.internal`
    hostname is never actually dialled.
