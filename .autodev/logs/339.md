## Goal for this iteration

PLAN priority #1 (M20.3). Iter 338's enriched GT-9 probe produced a
clean diagnostic signature on the most recent gold pass:

```
phase=7-sentinel:poll       elapsedMs=21454 wsOpensBeforeReopen=1 wsFramesBeforeReopen=11
phase=7-sentinel:probe iter=1  cmLen=0 bodyLen=89 cmEditorPresent=false
                                wsOpens=2 wsFrames=11 wsCloses=1
                                lastWsClose=wasClean=true pageErrorsCount=0
phase=7-sentinel:probe iter=21 ...wsOpens=2 wsFrames=11 wsCloses=1...
phase=7-sentinel:probe iter=41 ...wsOpens=2 wsFrames=11 wsCloses=1...
phase=7-sentinel:probe iter=61 ...wsOpens=2 wsFrames=11 wsCloses=1...
phase=7-sentinel:done  iter=79 found=false cmLen=0 wsOpens=2 wsFrames=11
```

Read it: after the reopen click, the client initiated a 2nd
WebSocket (wsOpens 1→2) but received **zero frames** on it for the
full 90 s budget. wsCloses stays at 1 (the dashboard-nav close);
the 2nd WS is still pending after 90 s. No page errors. body
content unchanged for 90 s.

The 11 pre-reopen frames came from the first WS during phase-1 cold
start (`hello`, `file-list`, `compile-status` running/idle pairs,
PDF segments, etc.). The second WS getting zero means the WS
upgrade response never reached the client. That points squarely at
the web tier's WS proxy holding the upgrade while
`createUpstreamResolver` drives the stopped Machine to `started` —
the proxy calls `dialAndPipe` only after `options.upstream(projectId)`
resolves, and on a `stopped`-state cache hit `driveToStarted` does
`startMachine` + `waitForStartedWithRetry` (≤300 s) + `tcpProbe`
(≤60 s) before forwarding the upgrade. The 5 s `connectTimeoutMs`
in `attachWsProxy` is the TCP-dial timeout AFTER resolution, not a
cap on the resolver itself.

So one of three things is happening server-side:
1. Fly machine startup from `stopped` is taking >90 s for the
   force-stopped Machine (warm-image but cold runtime); the resolver
   is correctly waiting.
2. Fly reports `started` quickly but tcpProbe is wedged because the
   sidecar process is slow to bind port 3001 on cold boot.
3. The resolver hit an exception that should have produced a 502 but
   somehow the 502 write didn't close the WS (rare — would still
   destroy the socket).

To discriminate: enrich the GT-9 probe with a **per-probe Fly
Machine state fetch** (we already have `FLY_API_TOKEN` + the
machineId). And raise `SENTINEL_VISIBLE_BUDGET_MS` 90 s → 180 s so
the legitimate stopped→started path has time to complete and the
spec doesn't time out before the resolver does. 180 s is a real
budget — Fly's `/wait?state=started&timeout=60` plus the 60 s
tcpProbe loop plus per-iteration `getMachine` overhead sit ~140 s
worst case for a warm-image cold boot.

If the next gold pass shows the Fly state transitions `stopped` →
`starting` → `started` within the 180 s window and `.cm-content`
populates, the underlying primitive is fine and we close M20.3.
If state stays `stopped` past the 180 s, hypothesis (1) is the bug
(Fly auto-start regression or our resolver isn't issuing
`startMachine`). If state reaches `started` but cmContent stays
empty, hypothesis (2) is the bug (sidecar bind race or hydration
hang on cold storage).

Out of scope this iteration: any production code fix. The diagnostic
shape is the single thing changing.
