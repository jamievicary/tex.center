## Goal for this iteration

Land M3.4 ‚Äî per-shipout PDF byte-range deltas. Today
`SupertexWatchCompiler.compile()` returns one big segment containing
the full PDF on every round, defeating the purpose of GOAL.md's
"hundreds of ms edit-to-preview" promise as soon as the PDF gets
large. M3.4 chunks the PDF using the page‚ÜíPDF-offset map written by
supertex via `--live-shipouts`, emitting one segment per shipout that
was actually re-emitted in this round.

## Approach

Introduce a new `ShipoutSegmenter` (`apps/sidecar/src/compiler/pdfSegmenter.ts`):
state is the current per-page offset map plus the last-read byte
position in the shipouts file. Per compile, the segmenter reads the
delta lines added since the previous compile ‚Äî those entries are the
shipouts that supertex re-emitted in this round, and define the set of
segments we forward to the client. The next-offset of each emitted
segment comes from the *full* current per-page-offset map (sorted by
offset), so segment ranges cover exactly to the start of the next
shipout (or PDF EOF for the final shipout). Pages with stale offsets
beyond the current PDF length are dropped from the map.

This works with `--live-shipouts`'s append-only contract as it stands
today ‚Äî no upstream change needed for this milestone (M3.5 still owns
`--ready-marker` and `--target-page=N`). For first compile against an
empty/missing shipouts file we fall back to one whole-PDF segment.

## Notes for future iterations

- The "drop stale entries past EOF" rule has a known soft edge:
  a page whose stale offset is still < new totalLength survives in
  the per-page map and can land an unwanted boundary inside the
  current PDF. Would need a "round-end / now-N-pages" hint from
  supertex to fix exactly. Folded into M3.5 as a candidate
  upstream tweak alongside `--ready-marker` and `--target-page=N`.
- The fake watch binary re-emits both shipouts every round even
  though chunk-A's bytes are identical. That's intentional ‚Äî it
  matches what an append-only `--live-shipouts` looks like in
  practice (real supertex would only re-emit re-shipped pages,
  which we test against directly in `pdfSegmenter.test.mjs`).
- `SupertexOnceCompiler` still ships a single whole-PDF segment.
  Once-mode is fire-and-forget so there's no baseline to diff
  against; a richer once-mode segmenter buys nothing today.
