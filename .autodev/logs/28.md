## Goal for this iteration

Decouple blob persistence from compile success in
`apps/sidecar/src/server.ts`. Today persistence sits at the tail
of `runCompile`, gated on `result.ok` — if a TeX error makes the
compile fail (the common case while editing), the user's edits
are lost on restart even though `writeMain` already wrote them
to the workspace dir. Move the blob-put to immediately after
`writeMain` succeeds, so persistence is gated on "we have a
durable on-disk source", not on "the compiler liked it". Add a
regression test exercising a deliberately-failing compiler.

PLAN's "Current focus" listed three candidates: M3.5 (upstream
PRs, not actionable here), multi-file project support (large,
needs frontend slice too), or M5 auth scaffolding (large). This
is a small correctness fix that should land before either of the
larger slices.

## Approach

1. `apps/sidecar/src/server.ts` — move the `if (blobStore && source !== p.persistedSource) { ... }` block from after `compiler.compile(...)` to between `workspace.writeMain` and `compiler.compile`. Comment explains the invariant.
2. `apps/sidecar/test/serverBlobs.test.mjs` — add Run 1b: build a server with a `compilerFactory` that always returns `{ ok: false, error: "boom" }`; edit through the existing client; assert the blob picks up the change anyway. Also assert a `compile-status:error` frame was actually broadcast (sanity that the failing-compiler path ran).
3. Update Run 2 expectation to include both edits (the original "extra line" + the failing-compile line) since it shares the same blob root and runs after Run 1b.

## Notes for future iterations

- Persistence is still tied to the compile *trigger* (text change → debounced compile → writeMain → put). If a future change separates compile triggering from text-change observation, the put-call should follow the text-change path, not the compile path. Today they're co-located, so the simpler placement is fine.
- The failing-compiler path now hits the "edit while broken" use case explicitly. When real supertex lands (M3.5), this same test will keep us honest: a TeX syntax error must not cause silent data loss.
- Multi-file support remains the obvious next slice — `mainTexKey` is still the single hardcoded blob key, and `text` is still the single Y.Text on the project. Both will need to fan out together.
