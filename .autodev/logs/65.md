## Goal for this iteration

PLAN's "next ordinary iteration" (M6.3.1) is out-of-tree. File-tree
CRUD verbs landed iter 61–63 and client-side validators landed
iter 64, but rejections that bypass the client mirror (genuine
race conditions: two windows creating the same name; a delete of
a file that disappeared; a rename target that was just created)
are still log-only on the server. The client sees no change and
no feedback. Add a small `file-op-error` control message on the
wire so the originating client can surface server-side rejections
inline.

## Approach

- Protocol: new variant
  `{ type: "file-op-error"; op: "create-file"|"delete-file"|"rename-file"; reason: string }`.
- Server (`apps/sidecar/src/server.ts`): in each existing
  rejection branch (`!res.added` / `!res.deleted` / `!res.renamed`),
  send a `file-op-error` frame to the originating client only
  (`client.send(...)` rather than `broadcast`) — the other
  windows did not request the op and shouldn't see its failure.
  Keep the existing `app.log.warn`.
- Client (`apps/web/src/lib/wsClient.ts`): add `fileOpError`
  field on `WsClientSnapshot`; populate on receipt; auto-clear
  on the next `file-list` (i.e. some op succeeded since).
- UI (`apps/web/src/lib/FileTree.svelte` +
  `editor/+page.svelte`): pass `serverError` prop; render
  inline below the create form using the existing `.err` style.
- Tests: protocol codec round-trip; extend
  `serverRenameFile.test.mjs` to assert each rejected rename
  produces a `file-op-error` frame to the originator and the
  reason matches; do not weaken the existing "no new file-list"
  assertion.

## Actions

- `packages/protocol/src/index.ts`: new `file-op-error` variant
  on `ControlMessage`.
- `apps/sidecar/src/server.ts`: each rejected file-op
  (`create-file` / `delete-file` / `rename-file`) now `client.send`s
  a `file-op-error` to the originator alongside the existing
  `app.log.warn`. Other connected viewers don't see it (they didn't
  request the op).
- `apps/web/src/lib/wsClient.ts`: `WsClientSnapshot.fileOpError`
  populated on receipt; cleared on the next `file-list` (which
  only arrives after a successful op, so the banner clears the
  moment the user gets one through).
- `apps/web/src/lib/FileTree.svelte`: optional `serverError` prop;
  rendered as a second-priority `.err` paragraph (the local
  validator's `createError` wins when both are present, since
  it's about the live input).
- `apps/web/src/routes/editor/+page.svelte`: passes
  `serverError={snapshot.fileOpError}`; initial snapshot grew
  the new field.
- `packages/protocol/test/codec.test.mjs`: round-trip for the new
  variant across all three op values.
- `apps/sidecar/test/serverRenameFile.test.mjs`: the four-rejects
  block now `waitFor`s four `file-op-error` frames, and asserts
  each carries `op: "rename-file"` and a non-empty `reason`. The
  pre-existing "no new file-list" assertion is unchanged.

## Notes for future iterations

- The auto-clear-on-next-`file-list` policy assumes the next
  successful op invalidates the staleness of the prior error.
  Two race-rejected ops in a row will overwrite each other in
  the banner, which is fine for single-user MVP but a queue + ack
  would be cleaner if we ever surface per-attempt status (e.g. a
  spinner per pending op).
- No new wire shape is needed when the upload-from-disk affordance
  lands later: it can reuse `file-op-error` with
  `op: "create-file"`.
