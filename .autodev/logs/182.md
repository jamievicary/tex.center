## Goal for this iteration

Take the cold-start fresh-project flakiness investigation flagged
in PLAN §3 (iter 181 recommended this over the deferred refactor
cron, which now rolls onto iter 183). Iter 181 had two live-spec
failures and a knock-on machine-count breach:

- `verifyLiveFullPipelineReused` (reused/warm path) — `pdf-segment`
  frame arrived but the preview canvas was blank when sampled.
  Hypothesis (c) from PLAN §3: PdfViewer canvas paint racing the
  first `pdf-segment` arrival.
- `verifyLiveGt1NoFlashLoad` (GT-A, fresh project) — `.cm-content`
  not attached within 60s. Same cold-start envelope iter 180 hit
  on M8.pw.4 (fixed there by bumping 30s → 120s); GT-A still on
  60s.

Both have surgical, principled anti-flake fixes that don't fall
into the "timing slack tuned until CI is green" trap.

## Approach

Two coherent fixes:

1. **GT-A `.cm-content` timeout 60s → 120s.** Same justification
   as iter 180's M8.pw.4 bump: post-iter-177 hydration gating
   means `.cm-content` cannot attach until the per-project Fly
   Machine cold-starts + Yjs initial sync completes. 120s is
   the cold-start TCP-probe budget the WS-upgrade path already
   uses; if exceeded the failure is real, not a tight timeout.
2. **Single-shot canvas snapshot → bounded poll** in both
   `verifyLiveFullPipeline.spec.ts` and
   `verifyLiveFullPipelineReused.spec.ts`. PDF.js parse + render
   is async after the `pdf-segment` frame arrives — a one-shot
   `canvas.evaluate(nonBlank)` was racing the paint. The
   canonical anti-flake primitive for "wait until X" is a
   bounded poll with an explicit deadline (30s). The poll
   re-locates `.preview canvas` each tick so an incremental
   re-render replacing the canvas element is handled, and a
   thrown evaluate (detached canvas mid-replace) is caught as
   `false` so the next tick retries. Width/height==0 guard
   avoids spurious "blank" reads during initial mount.

Did NOT touch the `pdf-segment` arrival assertion itself — that
already uses `expect.poll`, and the iter 181 failure path got
past it (the bytes arrived; only the paint raced).

Did NOT investigate hypothesis (a) Fly cold-start tail latency
or (b) compile coalescer × first pdf-segment of a fresh Y.Doc:
both M8.pw.4 (fresh) and reused (warm) have been affected, so
the common factor is the post-frame paint race rather than
anything cold-start-specific. Pure-fresh-only flakes (GT-A
specifically, recurring across 177/180/181) get the cold-start
budget fix.

The machine-count guardrail breach in iter 181 is a knock-on of
the fresh-project spec failure (afterEach reaped, but the
`liveProject` worker fixture for GT-A may have left the machine
on a failure path); once GT-A stops timing out, the count
guardrail should clear on its own. If it persists across iter
182 that's a separate hygiene issue to take next.
