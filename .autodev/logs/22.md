## Goal for this iteration

Wire the sidecar's `/healthz` route to probe `app.db`. M4.2.2
left the DB handle decorated but unread, which was called out as
a live caveat in PLAN ("no route reads `app.db` yet"). Sidecars
deployed on Fly will need a real readiness signal that reflects
DB connectivity, and this is the first concrete consumer of the
DB seam ‚Äî small, scoped, eliminates the caveat.

## Approach

1. `apps/sidecar/src/server.ts`: replace the trivial `/healthz`
   handler with one that, when `app.db` is set, runs
   `app.db.client\`SELECT 1\`` and reports `db: {state:"up"}` or
   `db: {state:"down", error}`. When unset, `db: {state:"absent"}`.
   Top-level `ok` becomes `state !== "down"`. Shape:
   `{ ok, protocol, db: { state, error? } }`.
2. New test `apps/sidecar/test/serverHealth.test.mjs` injecting
   three fake handles (success, throw, none) and using
   `app.inject({method:'GET', url:'/healthz'})` to assert the
   payloads.
3. Register in `tests_normal/cases/test_node_suites.py`.
4. PLAN: drop the "no route reads `app.db` yet" caveat.

## Notes for future iterations

- The probe runs synchronously per request, so a wedged DB
  blocks `/healthz` until postgres-js's own connect timeout.
  If we ever put `/healthz` behind Fly's hard ~5 s liveness
  budget, swap the bare `await` for a `Promise.race` against a
  short timer and report `down` with `error: "probe timeout"`.
- The `db` field in the response shape is intentionally a
  nested object rather than a flat string so future state
  (e.g. last-applied-migration name, replica lag) can be
  attached without breaking existing callers.
