## Goal for this iteration

Close **M11.5a** ‚ÄĒ drop-text-files-onto-file-tree ‚Üí existing
`upload-file` text path. Active queue item #5; listed as
"unblocked, small". Today the file tree exposes upload only via
an `‚ÜĎ` button that opens a hidden `<input type="file">`; the
drop-zone affordance would be the natural way to grab a `.tex`
from the OS without traversing the picker. Iter 66 closed the
text upload path itself (file-pick ‚Üí `await file.text()` ‚Üí
`onUploadFile(name, content)`); reusing it for drop is a few
event handlers plus a dragover affordance.

## Approach

- **New pure helper `apps/web/src/lib/fileDropUpload.ts`** ‚ÄĒ
  `classifyDroppedNames(names, existing)`. Mirrors the
  inline `rejectionReason()` in `FileTree.svelte` exactly:
  trim, `validateProjectFileName`, `MAIN_DOC_NAME`, dedup
  against `existing`, dedup within the same drop. Returns
  `{ accepted: string[], rejected: { name, reason }[] }`.
  Async I/O (file-text read) stays in the `.svelte` handler;
  the helper is pure and unit-testable.
- **`apps/web/src/lib/FileTree.svelte`**:
  - `let isDragOver = $state(false)`; gates a `.dragover`
    border affordance on the tree root.
  - `onDragOver` / `onDragLeave` / `onDrop` handlers wrapping
    the `<ul class="root">`. Guard `dragover` on
    `e.dataTransfer?.types.includes("Files")` so non-file
    drags (e.g. text selection) don't activate the affordance.
  - `onDrop` calls `classifyDroppedNames` on the file list,
    `window.alert` per rejection (matching the existing picker
    UX), then `await file.text()` and `onUploadFile(name, content)`
    for each accepted entry. Sequential await is fine ‚ÄĒ uploads
    are small, the server serialises them anyway, and
    aggregate-error visibility is improved.
- **`apps/web/test/fileDropUpload.test.mjs`** ‚ÄĒ unit cases:
  - empty input ‚Üí empty accept/reject
  - clean drop of two new names ‚Üí both accepted, ordered
  - invalid name characters ‚Üí rejected with reason from validator
  - drop of `MAIN_DOC_NAME` ‚Üí rejected "name reserved"
  - drop of name already in `existing` ‚Üí "already exists"
  - duplicate within same drop ‚Üí second occurrence "already exists"

GT-F-style Playwright pin for drop deferred ‚ÄĒ Playwright DnD on
synthetic File payloads needs a fixture-level `DataTransfer`
construction which doesn't transfer cleanly through Page.evaluate
in our current setup. The pure-helper test plus the existing
`onUploadFile` mock-callable contract is the load-bearing
coverage; the drop wiring in the .svelte file is mechanical.

## Actions

- **New** `apps/web/src/lib/fileDropUpload.ts`. Exports
  `classifyDroppedNames(names, existing): { accepted, rejected }`.
  Implements the validation parity described above; the picker
  flow's `rejectionReason` in `FileTree.svelte` is unchanged but
  now has a tested twin for the drop entry-point.
- **`apps/web/src/lib/FileTree.svelte`**:
  - Imports `classifyDroppedNames`.
  - Adds `isDragOver` state + three handlers (`onDragOver`,
    `onDragLeave`, `onDrop`). `dragHasFiles` gates all three so
    text-selection drags don't activate.
  - `onDragLeave` uses `currentTarget.contains(relatedTarget)` to
    suppress flicker as the pointer crosses child elements.
  - `onDrop` reads `dataTransfer.files`, runs them through
    `classifyDroppedNames`, surfaces each rejection via
    `window.alert` (parity with picker-flow), and `await
    file.text()` / `onUploadFile(name, content)` for each accepted.
  - Wraps `<ul.root>` + the create/upload `<form>` in a new
    `<div class="ft-host" data-testid="filetree-host">` that
    carries the three drag handlers and the `.dragover` class.
  - CSS: `.ft-host { min-height: 100%; outline: 2px dashed
    transparent; outline-offset: -2px; }`; `.ft-host.dragover {
    outline-color: #2563eb; background: rgba(37,99,235,0.04); }`.
    The reserved transparent outline keeps children from shifting
    when the affordance lights up.

## Notes for future iterations

- **`classifyDroppedNames` and the inline `rejectionReason` are
  twin sources of truth.** They mirror today. If the picker-flow
  rejection rules ever change (e.g. an additional reserved name,
  a stricter extension whitelist), update both. A near-term
  refactor could extract `rejectionReason` into the same
  `fileDropUpload.ts` module ‚ÄĒ left to a `N%10==0` iteration so
  the diff stays focused.
- **Drop affordance is full-column.** The wrapping div has
  `min-height: 100%`, so a sparsely-populated tree still has a
  generous drop target. The aside container's `overflow: auto`
  remains the natural scroll boundary.
- **Binary-asset drop (M11.5b) still blocked** on the wire-protocol
  design for `upload-file` extending beyond UTF-8 text (see
  FUTURE_IDEAS). Once that lands, the same drop handler can branch
  on MIME / extension to dispatch text vs. binary uploads through
  separate wire frames; the classifier becomes the natural place
  to surface that branching decision.
