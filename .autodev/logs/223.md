## Goal for this iteration

Execute iter-222's "next iteration plan" step 1: add gated
structured trace logging to `CompileCoalescer` so the next live
deploy of the sidecar can scrape `flyctl logs` and pin which state
transition produces the iter-221 `compile-status state:"error"
detail:"…already in flight"` toast cluster on GT-7. The trace must
be off by default (no production overhead) and fully optional in
the constructor (no churn at the existing call sites and zero
behavioural change when un-wired).

The iter-222 sidecar-level gold case did not reproduce the bug
locally even with a 5 s forced first-compile delay, which rules out
the naive "slow cold-start × debounce kicks" model. The trace is
the cheapest remaining diagnostic — once the next live gold pass
captures the failing transition, the fix in iter 224+ should be a
small, targeted state-machine repair.

## Approach

- `CompileCoalescer` gains a `trace?: (event: CoalescerTraceEvent)
  => void` option. The event carries `{seq, event, inFlight,
  pending, hasTimer}`. Events emitted: `kick`, `timer-fire`,
  `maybeFire-skip-inflight`, `maybeFire-skip-empty`, `run-start`,
  `run-finally`, `cancel`, `kickForView-fire`, `kickForView-skip`.
  `seq` is a per-coalescer monotonic counter starting at 1.
- The class invariant for the gate (`inFlight=true → no overlap`)
  is unchanged. Trace emission happens at the same points the
  state changes; no new branches in the hot path.
- `server.ts` reads `SIDECAR_TRACE_COALESCER` at
  `buildServer()` time; when `"1"`, wires the trace sink to
  `app.log.info({ projectId, coalescer: event }, "coalescer-trace")`.
  Otherwise the option is not passed (clean under
  `exactOptionalPropertyTypes`).
- New focused unit test
  `apps/sidecar/test/coalescerTrace.test.mjs` pins the trace
  contract: optional, monotonic seq, canonical kick→timer→run
  shape, in-flight gate emits `maybeFire-skip-inflight`, cancel
  always emits, kickForView distinguishes skip vs fire. Drives
  the class directly, no Fastify boot.

## Notes for future iterations

- Operator step (out of this iteration): set
  `SIDECAR_TRACE_COALESCER=1` on the sidecar app (`flyctl secrets
  set -a tex-center-sidecar SIDECAR_TRACE_COALESCER=1` plus a
  redeploy, OR pass it through the project-machine spawn path if
  per-project Machines own the env). Run augmented GT-7
  (`verifyLiveGt7RapidTypingDaemonStable.spec.ts`) or wait for the
  bug to recur naturally, then scrape `flyctl logs` for
  `coalescer-trace` records around the failure. The event sequence
  immediately preceding the first `already in flight` frame is the
  answer.
- The trace will reveal one of three shapes:
  1. `run-start` while `inFlight=true` — impossible by the gate;
     would prove a missing `await` / re-entrancy bug in
     `maybeFire`.
  2. Two `run-start` events with no intervening `run-finally` —
     proves a second coalescer instance for the same project (state
     pollution, multiple `ProjectState`).
  3. Some `compile-status state:"error"` frame originating
     somewhere *other* than the coalescer-driven `runCompile`
     path — proves a caller bypassing the gate.
- If the answer is (3), the next probe widens to the WS message
  switch and the persistence/file-op paths, looking for a
  `compiler.compile()` call that doesn't go through the coalescer.
