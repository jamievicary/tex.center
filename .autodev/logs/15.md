## Goal for this iteration

M3.5 sidecar half — wire feature-detection of the upcoming
`--ready-marker <STRING>` and `--target-page=N` supertex flags into
the sidecar's compiler factory, and conditionally pass the flags to
the spawned binary based on what the engine advertises in `--help`.

This is the part of M3.5 that lands without waiting for the
upstream PRs to merge: sidecar grows the plumbing + capability
gating + tests, defaults stay safe (no new flags emitted unless
detected), and the moment upstream supertex ships the flags the
sidecar starts using them automatically. The two upstream PRs (the
flags themselves) remain open as work against
`github.com/jamievicary/supertex` — separate repo, separate iteration
loop.

## Approach

1. New `apps/sidecar/src/compiler/featureDetect.ts` —
   `detectSupertexFeatures(supertexBin, opts?)` runs
   `<bin> --help` with a short timeout, parses stdout+stderr for
   the flag names, returns `{ readyMarker, targetPage }`. Capture
   both streams because real supertex prints help to stdout but a
   future build might use stderr.
2. `SupertexOnceCompiler` and `SupertexWatchCompiler` accept an
   optional `features` field. When `features.targetPage` is true
   and the request carries a `targetPage`, append
   `--target-page=N` to the spawn args. When
   `features.readyMarker` is true, append
   `--ready-marker <readyMarker>` so the engine knows what string
   to emit. When false, the watch compiler still listens for the
   default marker (a fake test binary emits it unconditionally) —
   in production with a real supertex that lacks the flag, the
   compiler would just time out, which surfaces the missing
   capability cleanly.
3. `defaultCompilerFactory` in `server.ts` runs the feature
   detection once at startup (not per project) and passes the
   detected capabilities into each compiler instance it builds.
4. New `featureDetect.test.mjs` covers: flag detected from
   stdout, flag detected from stderr, flag absent, binary missing.
5. Update existing fake test binaries in `supertexOnceCompiler.test.mjs`
   and `supertexWatchCompiler.test.mjs` to accept (and ignore, or
   record) `--target-page=N` and `--ready-marker <STRING>` so
   tests can drive the new code paths without regression.
6. Add a "features advertised → flags passed" test case to each
   compiler test.

## Notes for future iterations

- Watch mode locks `--target-page=N` at spawn time. Mid-session
  page changes (the GOAL.md "user scrolled" path) would need a
  control channel — `supertex` already exposes
  `--control-fifo`; threading a "target-page=K" command through
  it could land alongside the upstream flag PR. Not required
  for M3.5.
- Real supertex isn't usable as `supertex-watch` until upstream
  ships `--ready-marker`. The sidecar will detect the absence
  cleanly and not pass the flag, but compile() will then time
  out on every round — a clear failure mode that makes the
  missing capability easy to diagnose.
- Feature detection is per-process, not per-project; if the
  binary changes under the sidecar without a restart we won't
  notice. Acceptable today (single binary baked into the per-
  project Machine image at deploy time).
