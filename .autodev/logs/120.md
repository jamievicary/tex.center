## Goal for this iteration

Close M7.4.1 — wire the M7.4.0 checkpoint primitives
(`Compiler.snapshot/restore`, `loadCheckpoint`/`persistCheckpoint`)
into `buildServer` so cold-start restore happens before the
first compile, and idle-stop snapshots+persists every project
before the user-supplied `onIdle` fires. Today every concrete
`snapshot()` returns null so end-to-end behaviour is unchanged;
the value is having the sidecar half ready the day upstream
supertex gains a serialise wire (M7.4.2).

## Approach

Two seams in `buildServer`:

- `ensureRestored(p)` — per-project lazy promise. First call
  triggers `loadCheckpoint` + `compiler.restore(blob)` (only
  when bytes are non-null); subsequent calls await the same
  promise. Errors are warned and swallowed: the documented
  fallback for any `restore()` impl is "clean rebuild", and the
  in-flight compile will do exactly that. Wired into
  `runCompile` after `awaitHydrated` and before `compile()`
  itself — that's the unique spot that must block until restore
  completes.
- `persistAllCheckpoints()` — walks every live project, awaits
  `snapshot()`, calls `persistCheckpoint`. Wrapped around the
  idle-timer fire so it runs **before** the user-supplied
  `onIdle` (the entry point's onIdle calls `app.close()`, which
  destroys per-project state, so any persist has to happen
  first). Per-project failures are warned and skipped — one
  wedged compiler must not block the others from persisting.

Per-project state gains one field, `restorePromise: Promise<void> | null`,
to dedup the cold-start work.

`serverCompileError.test.mjs`'s `StubErrorCompiler` does not
implement `snapshot`/`restore`; the worry from log 119 was that
M7.4.1 would force it to. It turns out the restore path is
gated on `blobStore` being present and that test boots without
a blob store, so neither method is ever called. No change
needed there.

## Notes for future iterations

- The `idleTimer = setTimeout(...)` fire path used to be sync
  fire-and-forget; it's now `void (async () => …)()` so the
  snapshot/persist work runs in the background of the timeout
  callback. The wrapping `try/catch` for `onIdle` is inside the
  async IIFE so it still catches user exceptions; no behaviour
  change for the case-3 idleStop test (which had no blob store
  and no projects → `persistAllCheckpoints` no-ops immediately).
- The "first call to `ensureRestored` after `awaitHydrated`"
  ordering is load-bearing only if a future `restore()` impl
  needs to look at the Y.Doc state. Today it doesn't, but
  awaitHydrated is cheap and the ordering removes a subtle
  surprise.
- When M7.4.2 lands and `SupertexDaemonCompiler.snapshot()`
  starts returning real bytes, watch the blob size and the
  idle-timeout budget — the snapshot happens inside the
  idle-timer callback before the Machine is allowed to exit,
  and a slow Tigris PUT on a 50 MB checkpoint could push
  Machine-stop latency past `restart: on-failure`'s patience.
  Likely the right move at that point is to push the persist
  earlier (e.g. after every compile) and treat the idle-stop
  snapshot as a fast tail-flush.
