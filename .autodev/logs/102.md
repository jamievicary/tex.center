## Notes for future iterations

- `MachineAssignmentRow.state` is still typed as `'starting' |
  'running' | 'stopped'` in `schema.ts`; Fly's actual states
  (per `flyMachines.ts:MachineState`) are a broader union with
  `'started'`, `'stopping'`, `'destroyed'`, etc. M7.1.2 is the
  natural place to widen this â€” either re-narrow to the subset
  the resolver actually caches, or just drop the union and use
  `string`. The drizzle column is `text` so no DB migration is
  needed either way.
- `app_name` deliberately omitted from the table. If a future
  slice needs multi-app routing (e.g. per-region sidecar apps),
  promote it to a column. For now the control plane reads a
  single `SIDECAR_APP_NAME` env var (to be wired in M7.1.2).
- `upsertMachineAssignment` preserves `created_at` across
  upserts (the `set` clause only touches mutable columns); the
  test asserts this. If a future resolver wants "row was
  recreated" semantics it should call `deleteMachineAssignment`
  + `upsertMachineAssignment` rather than relying on a single
  upsert to bump `created_at`.
