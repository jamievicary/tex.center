## Goal for this iteration

M13.1 diagnostic step: extend GT-6
(`verifyLiveGt6FastContentAppearance.spec.ts`) to read the five
editor lifecycle `performance.mark` entries via `page.evaluate`
and include them in the failure message. Iter 235 wired all five
marks (`editor:route-mounted`, `editor:ws-open`,
`editor:yjs-hydrated`, `editor:first-text-paint`,
`editor:first-pdf-segment`). The spec currently fails at ~500 ms
with the source appearing around 5 s; without the timeline we
can't tell which sub-step dominates. Iter 235 explicitly named
this as the cheaper next probe (option a in its notes) — purely
spec-side, no production-code change, immediately diagnostic, and
unblocks the fix iteration.

## Approach

The existing `catch` block already keeps polling past the 500 ms
bound to capture the true appearance time. After that polling
finishes, read `performance.getEntriesByType("mark")` via
`page.evaluate`, filter to the five `editor:*` names, compute
deltas relative to the earliest mark for readability, and append
the formatted timeline to the thrown error. Marks that never
fired show as `"(absent)"`.
