## Goal for this iteration

Open M4 with the smallest coherent persistence slice: a new
`packages/db` workspace package owning the canonical data model
(entity row types, per-table column specs, initial SQL migration)
plus a unit test that asserts the spec and the SQL stay in sync.
No new runtime dependencies, no DB connection ŌĆö this is pure
declaration. Subsequent iterations layer Drizzle on top of the
spec and bring up local Postgres.

M3.5's outstanding work is upstream PRs in a separate repo and
doesn't unblock anything inside this tree, so M4 is the next
useful milestone to advance.

## Approach

1. New `packages/db` package: `package.json` (`@tex-center/db`),
   `tsconfig.json` extending the shared base, `src/index.ts`
   re-exporting `src/schema.ts`.
2. `src/schema.ts` declares `UserRow`, `SessionRow`,
   `ProjectRow`, `ProjectFileRow`, `MachineAssignmentRow` plus a
   small in-house DSL (`ColumnSpec`, `TableSpec`) and one spec
   per table. `allTables` lists them in declaration order.
3. `src/migrations/0001_initial.sql` carries the initial DDL,
   hand-written to match the specs (PKs, FKs with cascade
   deletes, sensible unique constraints / indexes).
4. `test/schema.test.mjs` (run via `tsx`) asserts: every table
   has exactly one PK; all FK targets resolve to a table+column
   declared in the same module; the migration contains a
   `CREATE TABLE` for each table; every spec column shows up in
   the SQL with the right SQL type.
5. Register the suite in
   `tests_normal/cases/test_node_suites.py` as
   `test_db_schema`.

Deliberately deferred: pulling in `drizzle-orm` (network +
lockfile churn, no runtime user yet); migration runner; any
production-side wiring. M4.1+ will absorb those in turn.

## Notes for future iterations

- Spec/SQL drift is caught by the schema test by string-matching
  `<col> <type>` per column. False negatives are possible if a
  DDL line wraps weirdly ŌĆö fine for the hand-maintained file
  today; once Drizzle generates the SQL the duplication goes
  away and so does this test.
- `MachineAssignmentRow.state` is typed as a TS literal union
  but stored as `text` in SQL (no enum type yet). When M4.1
  introduces Drizzle, consider promoting it to a Postgres ENUM
  via a follow-up migration.
- `size` on `project_files` is a JS `number` mapped to SQL
  `bigint`. Files >2^53 bytes don't exist in practice; if that
  ever changes the type becomes `bigint` on both sides.
