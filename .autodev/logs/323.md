## Goal for this iteration

`N=323` ordinary iteration. Iter 322 landed M20.2(a) — the
`coldSourceFor(blobStore, projectId)` primitive in
`apps/web/src/lib/server/blobStore.ts` (dark code, no production
caller). The 322 notes explicitly named M20.2(c) as the next
ordinary slice: "one-line wiring change at
`apps/web/src/server.ts:86-89` (the `seedDocFor` lambda)". This
iteration lands that cutover plus the symmetric change in
`+page.server.ts` for the SSR seed placeholder.

The PLAN's M13.2(a) follow-up note also flagged that
`cleanupProjectMachine` re-arming the SSR seed gate while the blob
store may still hold edits is "benign while blob store is
per-Machine; once shared (M20.2), the gate must flip from 'no
machine assignment' to 'no persisted blob'". The cleanest reading:
the gate *condition* still fires when cold-start is about to be
paid (i.e. no Machine assigned), but the placeholder text should
reflect what cold-start will actually render — blob-content when a
blob exists, hello-world otherwise. The condition flip itself
would mis-fire when a machine is live (assignment present, blob
also present, no cold-start ahead) so the right shape is "same
gate, smarter text". Recorded in the PLAN follow-up.

## Approach

Two wiring changes, both backed by one helper:

1. New `createSeedDocFor({ blobStore, getDbSeedDoc, onBlobError })`
   in `apps/web/src/lib/server/blobStore.ts`. Returns the chain
   `(projectId) => blob → db → null`, reporting blob lookup
   failures via `onBlobError` and falling through to db rather
   than throwing (a transient blob-store outage must not pin a
   project to its db seed).
2. `apps/web/src/server.ts`: wrap the existing db-only
   `seedDocFor` lambda in `createSeedDocFor`. `blobStore =
   webBlobStoreFromEnv()` is captured at module load. `onBlobError`
   logs a `blob_seed_lookup_error` JSON event matching the
   surrounding `ws_proxy` log style.
3. `apps/web/src/routes/editor/[projectId]/+page.server.ts`: SSR
   seed text learns the same blob chain. When `assignment ===
   null`, render `{ name: MAIN_DOC_NAME, text: cold ??
   MAIN_DOC_HELLO_WORLD }`. The `coldSourceFor` call is wrapped in
   `.catch(() => null)` so an SSR blob-read failure just falls
   back to hello-world (the page must always render).

Cutover is *behind* today's per-Machine `LocalFsBlobStore` —
production deploys still set no `BLOB_STORE` env on the web tier
(the existing `webBlobStoreFromEnv` returns `undefined` and the
chain falls straight through to the db lookup), so observed
behaviour is unchanged until the S3/Tigris adapter lands. The
M20.2(d) entry in PLAN tracks that next slice.

## Notes for future iterations

- **M20.2(d) S3/Tigris adapter.** `packages/blobs/src/envSelect.ts`
  currently rejects `BLOB_STORE=s3` with "not implemented yet (M20.2
  cutover)". The natural shape: `S3BlobStore` in
  `packages/blobs/src/s3.ts` that implements the same `get` / `put`
  / `delete` interface as `LocalFsBlobStore`. Pair it with a
  request-recording HTTP stub (a tiny in-process http.Server that
  echoes request/response shapes) so adapter tests are deterministic
  — Tigris's eventual consistency makes live tests flaky. Once
  the adapter lands, the same `BLOB_STORE_LOCAL_DIR`→
  `BLOB_STORE_S3_BUCKET` env switch wires production end-to-end
  without code change.
- **Per-request blob store instantiation.** Both `server.ts` and
  `+page.server.ts` capture `blobStore` at module scope; the
  `LocalFsBlobStore` is stateless beyond a root-dir string so this
  is safe. If a future S3 adapter holds connection state (pooled
  HTTP, refresh tokens), revisit — it might want a singleton
  registry rather than per-module instantiation.
- **SSR seed via db `seed_doc`.** The SSR path renders blob OR
  hello-world; it does not consult the db `seed_doc` column the
  way `seedDocFor` does. Today this is the correct shape (every
  use of `seed_doc` is for gold-suite-seeded projects which open
  briefly, allocate a Machine, then close — by the time SSR runs
  again, `assignment` is non-null and SSR returns no seed). If a
  future flow leaves projects with `seed_doc` + no assignment +
  no blob for sustained periods, the SSR text should chain to
  `seed_doc` before hello-world; do that by lifting the
  three-step chain into `blobStore.ts` and consuming it from
  both call sites.
