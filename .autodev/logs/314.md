## Goal for this iteration

M23.2 â€” wire the M23.1 workspace primitives into the persistence layer
and the compile loop so non-main project files actually reach the
on-disk workspace the supertex daemon spawns in. Today
`apps/sidecar/src/server.ts` only calls `workspace.writeMain(source)`;
`addFile` / `deleteFile` / `renameFile` mutate Yjs + the blob store
but never touch disk, and cold-boot rehydration leaves the workspace
holding only `main.tex`. This is the categorical multi-file
regression that motivated M23 (see `309_answer.md` item 2).

This slice absorbs M23.3 too: the runCompile-time loop that mirrors
every non-main file before invoking the compiler covers both
"in-place Y.Text edit on `sec1.tex`" (which never fires a persistence
op) and "cold boot has `sec1.tex` in blob store but not on disk".
M23.4 (gold spec) is the next slice.

## Approach

- `apps/sidecar/src/persistence.ts`:
  - Accept an optional `workspace?: ProjectWorkspace` on
    `createProjectPersistence(...)`.
  - After every successful mutation (`addFile`, `deleteFile`,
    `renameFile`), call through to the corresponding `workspace`
    method, best-effort. Yjs remains the source of truth, so a
    workspace-mirror failure logs and is swallowed â€” does not
    fail the op.
- `apps/sidecar/src/server.ts`:
  - Pass the per-project `workspace` to `createProjectPersistence`.
  - In `runCompile`, after `writeMain(source)`, iterate
    `persistence.files()` and write each non-main file's current
    `Y.Text` contents to disk via `workspace.writeFile`. Wrapped
    in the same try/catch as `writeMain` so a write failure
    aborts the compile cleanly with a `compile-status:error` toast.
- Locks: extend `serverCreateFile`, `serverDeleteFile`,
  `serverRenameFile`, `serverUploadFile` tests to assert the
  workspace mirror state. Add a new test that exercises the
  cold-boot rehydration path via runCompile.

## Problems encountered

- **Race between runCompile mirror loop and concurrent delete.**
  First implementation added a `for (name of persistence.files())`
  loop inside `runCompile` that wrote every non-main file before
  invoking the compiler. Five-run flake check showed 4/5 failures:
  the loop's tmp+rename atomicity could resurrect a file that a
  concurrent `delete-file` op had just removed (delete enqueues
  `workspace.deleteFile`; runCompile holds a stale `files()` snapshot
  and `writeFile`'s rename completes after delete's `rm`).
  Anti-flake rules forbid retries / weakened assertions; the right
  fix is a structural one. Two candidate designs:
  - **(A)** Add a workspace-internal queue + per-write membership
    re-check inside runCompile. Race-free but adds machinery for a
    case the simpler design doesn't need.
  - **(B)** Remove the runCompile mirror loop. Do structural ops
    + cold-boot rehydration entirely from `persistence.ts`. Drop
    in-place `Y.Text`-edit mirroring from M23.2; defer to M23.5
    via per-file `Y.Text.observe` subscriptions.
  Chose (B): it's a smaller surface and produces an immediately
  shippable slice that closes the categorical regression for the
  typical "aux file is uploaded once, then read-only" case. The
  "edit aux file in the editor" path stays broken until M23.5 â€” a
  partial fix, but a strictly smaller broken surface than today.

## Notes for future iterations

- **M23.4 gold spec** is the natural next pick-up. A 2-file project
  (`main.tex` with `\input{sec1}` + `sec1.tex` body) exercises the
  cold-boot rehydration mirror end-to-end. Sidecar-level
  integration test is probably cleaner than Playwright here â€”
  asserting "the compile produced output that includes the body
  text" via the daemon protocol is more direct than canvas pixel
  inspection.
- **M23.5 in-place edit mirror.** Subscribe per-file
  `Y.Text.observe` during hydration / `addFile`; on observe,
  debounce-write to disk. Unsubscribe on `deleteFile` /
  `renameFile(old)`. The race that bit (A) above is *not* a
  concern for the observe-based design because each observe
  handler runs serially relative to other observe handlers for
  the same Y.Text, and `deleteFile`'s `unsubscribe` happens
  synchronously before the workspace mirror call â€” so an observe
  cannot fire after deleteFile starts.
- The `addFile` path with empty `body` writes an empty file on
  disk. This matches the blob-store semantics (empty PUT) and is
  what the gold spec assumes.
