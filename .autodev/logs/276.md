## Goal for this iteration

Pick up M15 diagnosis from iter 275. Iter-275 added
`compileStatusEvents` to `verifyLivePdfMultiPage`'s failure
diagnostic; the resulting gold run came back with
`running×11,idle×11 lastErrorDetail=null` — failure mode (c):
sidecar reached the daemon 11 times and got `ok` with zero
segments on every round. Goal: pinpoint *why* the daemon is
returning zero segments, given the sidecar-level
`test_supertex_multipage_emit` pin (SEED → MULTI, one compile) is
green.

## Approach

   see if it reproduces the live RED.
2. If it reproduces locally → iterate on `vendor/supertex/`.
   If it doesn't → escalate to reading live sidecar logs.

- Ran the pin locally: **PASSES** — every step emits a segment
  carrying the incremental page count (baseline=1, step-1..3=2
  pages, step-4..6=3, step-7..9=4, step-10..11=5). So the daemon
  handles this shape correctly in the controlled scenario; the
  live trigger needs additional state we haven't replicated.
- Read live sidecar logs via `FLY_API_TOKEN="$(cat creds/fly.token)"
  flyctl logs -a tex-center-sidecar --no-tail`. Smoking gun on
  every one of the 11 RED rounds:
  ```
  {projectId, sourceLen: 161, msg: "compile start"}
  [supertex-daemon stderr] supertex: edit /tmp/…/main.tex@85 past last consumed byte 70 — no recompile
  [supertex-daemon event] round-done
  {projectId, elapsedMs: 9, segments: 0, bytesShipped: 0, msg: "compile ok"}
  ```
  This is the iter-726 supertex short-circuit
  (`vendor/supertex/discussion/726_*`) misfiring. The 726 fix's
  predicate `edit_byte >= read_end_highwater` was designed for
  edits in trailing whitespace past `\end{document}`; here the
  daemon's recorded highwater stays pinned at byte 70 (end of the
  seeded `\end{document}\n`) while the user inserts new content
  *before* `\end{document}`. Each compile's first-byte-diff lands
  past 70 because the daemon's previous tracked state is whatever
  the prior no-op-skipped compile saw (not the original SEED), so
  the short-circuit fires again; highwater never grows; vicious
  cycle.
- Also confirmed the deployed `SIDECAR_IMAGE` is
  `sha256:ceca0de…` (iter 269, with the `targetPage=0` fix), via
  `flyctl ssh console -a tex-center -C "printenv SIDECAR_IMAGE"`
  on a temporarily-started control-plane Machine. Some leftover
  per-project Machines still run older images (`6138bbe…` =
  iter 255), but fresh test-project Machines spawn from the
  iter-269 SIDECAR_IMAGE secret — so the targetPage fix IS on the
  path. The bug is purely in supertex daemon's no-recompile
  short-circuit interacting with body-insertion edits.
- PLAN.md M15 section: appended the iter-276 diagnosis paragraph,
  the local-pin contrast result, and the two candidate close-out
  paths (extend local pin / escalate to supertex discussion).
