## Goal for this iteration

Land **M5.1.2a** — the `/auth/google/callback` `+server.ts` route, with
all OAuth I/O (token exchange, ID-token JWKS verification) and a
signed session cookie minted. Session-row DB persistence is deferred
to **M5.1.2b**: that requires `@tex-center/db` wiring in `apps/web`
(`DATABASE_URL` handling, user upsert helper, session insert
helper) — a coherent slice on its own. This iteration lands the
OAuth ↔ cookie round-trip with all the cryptographic steps real, so
M5.1.2b can focus on persistence and M5.1.3 on hooks.

## Approach

- New pure orchestrator `resolveGoogleCallback({…})` taking
  injected `exchangeCode` and `verifyIdToken` async functions plus
  a `mintSid()` callback. Returns a discriminated union
  (`redirect` / `error` / `signed-out`). Every branch is exercised
  by stubs in the unit test.
- Concrete I/O in `googleTokens.ts` using `fetch` + `jose`'s
  `createRemoteJWKSet` + `jwtVerify`. `jose` added to
  `apps/web` deps.
- Route file is a thin wrapper: parse `Cookie` header + query,
  load config, call resolver with real I/O, translate result into
  the SvelteKit `Response`.

## Decisions

- **Persistence deferred to M5.1.2b.** The route mints a valid
  signed `tc_session` cookie with `sid = crypto.randomUUID()`,
  but no users/sessions rows are written yet. Reason: wiring
  `@tex-center/db` into `apps/web` (DATABASE_URL handling, user
  upsert helper, session insert helper) is itself a coherent
  slice; bundling them would have produced a sprawling iteration
  with weak per-piece test coverage. The cookie is not yet
  useful end-to-end (M5.1.3 hooks lookup would miss), but every
  cryptographic step is real. Documented in PLAN.
- **`jose` over hand-rolled JWKS verify.** ~50kB, tree-shakes
  well, exposes exactly the primitives we want
  (`createRemoteJWKSet` + `jwtVerify` with `audience` / `issuer`
  / `algorithms`). The alternative (fetch JWKS, parse JWKs by
  hand, do RS256 with `node:crypto`) is ~150 lines of
  error-prone code with worse caching primitives.
- **`SameSite=Lax` + `Secure` on session cookie.** Same
  reasoning as the state cookie. `Strict` is too aggressive (an
  inbound link from a colleague's chat would hide the session
  on first navigation).
- **Session TTL = 30 days.** Matches typical Overleaf-like
  comfort; the cookie is a thin reference to a server-side row
  (when M5.1.2b lands), revocable per-row.
- **`signedOutPath = "/"`.** Per GOAL.md, non-allowed users land
  on the white sign-in page. State cookie cleared so the next
  click of "Sign in with Google" starts fresh.
- **Error bodies are short plain text.** Not user-facing in
  steady state — anyone seeing them is debugging. Google
  `?error=` codes echoed only if they match `[A-Za-z0-9_-]{1,64}`,
  otherwise `unknown` (mild XSS hygiene even though Content-Type
  is text/plain).
- **`exchangeCode`/`verifyIdToken` as injectable async fns.**
  Keeps the resolver pure and the unit test offline. Production
  closures the real `clientId`/`clientSecret`/`redirectUri` once
  per request. JWKS is fetched lazily and cached by `jose`'s
  internal store.

## Notes for future iterations

- **M5.1.2b (next ordinary iteration).** Wire DB into `apps/web`:
  - `apps/web/src/lib/server/db.ts` — copies the sidecar pattern:
    `getDb()` lazily creates a `DbHandle` from `DATABASE_URL`,
    throws with a clear message if absent. Process-lifetime
    singleton. Same `closeDb` on shutdown (SvelteKit's
    `handleShutdown` hook or a `process.on('SIGTERM', ...)`).
  - `packages/db/src/users.ts` — `findOrCreateUserByGoogleSub`
    upsert. Uses `INSERT ... ON CONFLICT (google_sub) DO UPDATE
    SET updated_at = now() RETURNING *`. Display-name backfill
    on first login from the `name` claim.
  - `packages/db/src/sessions.ts` — `insertSession({userId,
    expiresAt}) → {id}`. PK is `uuid` defaulted (or generated by
    caller; today the route uses `randomUUID`).
  - Route swap: `mintSid` no longer just generates a uuid; it
    inserts the row and returns its id. Wrap in try/catch and
    surface 500 on DB failure.
  - Tests: PGlite integration test for both helpers
    (`packages/db/test/users.test.mjs`,
    `sessions.test.mjs`) — same harness as the existing
    `migrations.test.mjs`. The route unit test grows a stub for
    the DB-backed mintSid.
- **M5.1.3 (after M5.1.2b).** `apps/web/src/hooks.server.ts`:
  read `tc_session`, `verifySessionToken`, look up the row,
  populate `event.locals.session`. Protected `+server.ts`
  endpoints (when they exist) check it.
- **JWKS clock-skew tolerance.** `jose.jwtVerify` defaults to a
  ±0 clockTolerance; consider `clockTolerance: 60` once we see
  a real Google issuance to be safe against minor server
  drift. Cheap to add later.
