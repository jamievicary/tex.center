## Goal for this iteration

M7.2 — `/ws/project/<id>` per-project access gate. Today the WS
upgrade authoriser only checks "valid session"; any signed-in user
can hit `/ws/project/<arbitrary-id>` and the resolver will spin up
a Machine for it. Single-user allowlist limits the practical blast
radius but the gap is real (DoS via arbitrary-id Machine spawn) and
preconditions multi-user later. This iteration enforces project
ownership at the WS upgrade boundary.

## Approach

- Extend `WsProxyOptions.authoriseUpgrade` signature to `(req,
  projectId) => boolean | Promise<boolean>`. The proxy already
  extracts and validates `projectId` before the auth call; pass it
  through. Existing zero-arg authoriser functions in tests still
  compose (extra args ignored).
- New `makeProjectAccessAuthoriser` in `wsAuth.ts`: combines the
  session lookup with a `lookupProjectOwner(projectId)` callback.
  Gate passes only when the session's user-id equals the project's
  owner-id.
- Wire it in `server.ts` against `getProjectById`.

## Actions

- `apps/web/src/lib/server/wsProxy.ts`: widened `authoriseUpgrade`
  signature to `(req, projectId)`; the proxy now passes the validated
  projectId.
- `apps/web/src/lib/server/wsAuth.ts`: added
  `LookupProjectOwnerFn`, `ProjectAccessAuthoriserOptions`, and
  `makeProjectAccessAuthoriser`. The new authoriser runs the same
  `resolveSessionHook` path as `makeSessionAuthoriser` (re-using the
  internal lookup-throw-→-null-session swallow), then resolves the
  project owner and admits only when `ownerId === session.user.id`.
  Missing project → false (so resolving an arbitrary projectId never
  reaches the upstream resolver / Machines API). Both injected
  lookups have outer try/catch as defence-in-depth even though the
  inner session-hook already swallows DB throws.
- `apps/web/src/server.ts`: switched the WS authoriser from
  `makeSessionAuthoriser` to `makeProjectAccessAuthoriser`, wiring
  `lookupProjectOwner` to a `getProjectById`-backed closure.
- `apps/web/test/wsAuth.test.mjs`: added six cases for the new
  authoriser (owned → true; other-owned → false; missing → false;
  no session → false; owner-lookup throws → false; session-lookup
  throws → false). Existing `makeSessionAuthoriser` cases retained
  unchanged — the inner function only references `req`, so passing
  a phantom second arg at the JS layer is a no-op.
- `.autodev/PLAN.md`: M7.2 split into substeps with M7.2.0 ticked.

## Notes for future iterations

- 401 vs 403: the proxy lumps "no session" and "wrong owner" both
  to `401 Unauthorized` via the boolean authoriser shape. A future
  slice could widen the authoriser return to a discriminated union
  (`{ kind: "allow" } | { kind: "deny-anon" } | { kind: "deny-acl" }`)
  so the proxy can distinguish 401 from 403. Not load-bearing for
  the single-user MVP; deferred.
- Per-project Machine resolver is now genuinely gated: a logged-in
  user can no longer cause `MachinesClient.createMachine` to fire
  for an arbitrary projectId.
