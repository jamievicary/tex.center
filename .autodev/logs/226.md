## Actions / probe variants

Three variants of probe3 tried, each smoke-run via a standalone
driver (logs at /tmp/probe3_*.mjs, not committed):

- **v1**: strict cadence, `await awaitRoundDone()` between iterations.
  20 iters in ~100s. **No crash.** Daemon stayed caught up; initial
  compile took only 300ms locally vs ~20s on live cold-Machine.
- **v2**: enqueue all 20 without waiting for round-done. Daemon
  emitted `WARN no usable rollback target` for byte-offset 92 and
  224 (skipped intermediate snapshots), then drained cleanly.
  **No crash.**
- **v3**: don't even await `daemon ready` — fire stdin from t=0.
  Same outcome as v2; local initial compile too fast to build
  the queue.

Removed probe3 from the test file since per the user's protocol I
refuse to commit a passing test mislabeled as a reproducer.

## Findings (written up in 225_answer.md)

The live SIGABRT requires the cold-start window where supertex's
initial lualatex compile blocks for 20–90s on a freshly-spawned Fly
Machine. During that window the sidecar's writes+recompile commands
queue, and by the time the initial compile completes the daemon
faces a backlog with the latest edit-byte far past any extant
checkpoint. My local lualatex compiles `Hello, world!` in ~300ms
with warm caches, so the cold-start race window never opens.

Three options for a real local reproducer proposed in
225_answer.md, recommendation (1):

1. PATH-shim `lualatex` with `sleep 30 && exec lualatex "$@"` to
   force a slow initial compile.
2. Pre-fill stdin with all 20 commands before spawning the daemon.
3. SSH against a remote sidecar Machine.
