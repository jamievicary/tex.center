## Goal for this iteration

Close **M22.3** — toast UX polish — the small final slice of the
M22 debug-toasts queue (M22.1 closed iter 304; M22.2 is the
Playwright spec that depends on the polished UX surface). Three
coherent edits land together:

1. Info default TTL 4 s → 5 s. The 4 s default barely allowed the
   message to register before disappearing; success stays at 3 s
   (deliberate brevity for confirmation pings); error stays 6 s.
2. Stack ordering: newest-on-top. Today new pushes append below
   existing toasts, so during a burst the user reads stale ones
   first. Reverse the render order so the most recent is at the
   top of the stack and older toasts drift downward.
3. Dismiss button for info and success. Today only `persistent`
   toasts expose a `×`; users have to wait out the TTL on
   transient messages. Add the affordance to info/success too.
   Error is left alone — per PLAN — the 6 s window is short and
   surfacing-then-vanishing is the deliberate UX.

Pure UI/UX polish — no wire-protocol or store-API changes.

## Approach

- `apps/web/src/lib/toastStore.ts`: `DEFAULT_TTL_MS.info`
  `4_000 → 5_000`. Other categories untouched.
- `apps/web/src/lib/Toasts.svelte`: render the items array in
  reverse so the last-pushed toast is at the top of the
  flex-column stack. Aggregating pushes still update the
  existing toast in place; the order constraint is purely
  newest-first across distinct toasts.
- `apps/web/src/lib/Toasts.svelte`: extend the dismiss-button
  guard from `t.persistent` to
  `t.persistent || t.category === "info" || t.category === "success"`.
- `apps/web/test/toastStore.test.mjs` case 2: update the info
  auto-dismiss expectation 4000ms → 5000ms, and the gap-to-error
  advance from `+2000` → `+1000`.

  no-merge assertion to also lock the store's oldest-first
  insertion order, so the renderer's `slice().reverse()` keeps
  a known contract to invert.

## Notes for future iterations

- **M22.2 still the open M22 slice.** GT-F as drafted in PLAN
  asserts `?debug=1` flips localStorage; single keystroke
  produces a green Yjs-op toast and (after compile) a blue
  pdf-segment toast; rapid typing aggregates into one green
  `×N`. The wire-event side of that is in place (M22.1 +
  current); the open work is the local Playwright spec. Note
  the **local-stack constraint**: the local SvelteKit
  webServer has no sidecar, so `outgoing-doc-update` (gated on
  `send() === true`) won't fire from a keystroke unless the spec
  either (a) boots a sidecar against the local stack or (b)
  exposes a way to inject toasts (or fire `onDebugEvent`)
  directly from the test. Option (b) is the smaller iteration:
  add a `?debug=1`-gated `window.__texCenter.toasts` accessor
  the spec can call and assert against.
- **Newest-on-top + aggregation interplay.** A burst toast (e.g.
  `Yjs op ×7`) stays at its original position even as it
  aggregates: only new distinct pushes appear above. That's
  intentional — moving an aggregating toast around mid-burst
  is visually jarring and the count badge already conveys
  freshness. If a future iteration wants "always-newest-on-top
  including aggregates", lift the existing toast to the end of
  the store array on each merge in `push()`, not in the
  renderer.
- **Error category stays auto-dismiss-only.** If a future
  failure mode pushes a `persistent: true` error, the dismiss
  guard already covers it via the existing `t.persistent`
  arm — no further change needed.
