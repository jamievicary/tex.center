## Goal for this iteration

Fix the M7.0.1 kpathsea regression diagnosed in iter 87: the
runtime stage's `lualatex.fmt` dump aborts because the patched
luatex binary (TL 2027/dev, compiled-in prefix `/opt/engine`)

## Problems encountered

### 1. TEXMFCNF before apt-install breaks the `context` postinst

First cut put the `TEXMFCNF` setting in the same ENV block as
`PORT`, `HOST`, etc., which means it's set *before* `apt-get
install texlive-full ...`. The `context` package's
post-installation script runs `luatools --make cont-en` against
its own texmf tree (separate from texlive's); with `TEXMFCNF`
pointing only at the texlive dirs, `luatools` can't find
context's `texmf.cnf`, fails with exit 1, and dpkg leaves both
`context` and (its reverse-dep) `texlive-full` unconfigured.
Resulting log signature:

```
luatools --make cont-en failed. Output has been stored in /tmp/luatools.…
dpkg: error processing package context (--configure):
 installed context package post-installation script subprocess returned error exit status 1
Errors were encountered while processing:
 context
 texlive-full
```

This was a genuinely silent regression vs. iter 87 (which got
all the way to the fmt-dump step on the same apt-install layer):
the env change above invalidates the apt-install cache layer
*and* poisons the postinst environment. Fix: split the ENV so
TEXMFCNF is set in its own `ENV` directive **after** the
apt-install RUN. Documented in the Dockerfile so future edits
don't merge them back.

### 2. Wallclock — deploy retry deferred

The deploy attempt above consumed ~15 minutes (8 min builder
stage + 7 min runtime apt-install before the context failure).
With the fix applied, the apt-install layer can't reuse cache
(ENV ordering changed). Budget for a fresh successful build is
~25 min — pushes the iteration into the 45-min hard limit
territory if anything else slips. Per harness §3 wallclock
rules: the slice is too big to also land the live deploy this
iter. Committing the Dockerfile + structural-test fix alone and
deferring the deploy to the next iteration is strictly better
than a panic-timeout half-state.
