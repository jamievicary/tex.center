## Goal for this iteration

Implement the topbar iteration indicator committed in iter 184's
discussion-mode answer (q/a 184). Single coherent slice: thread
`TEXCENTER_ITER` from `.autodev/logs/` count → CI build-arg →
Dockerfile ENV → SvelteKit public env → both topbar pages, plus
a `tests_normal/` regression lock for all four wires.

## Approach

One deviation from the 184_answer plan: used `$env/dynamic/public`
rather than `$env/static/public`. Reason: `$env/static/public`
types are generated by `svelte-kit sync` from env vars present at
sync time; with `PUBLIC_TEXCENTER_ITER` absent from `process.env`
during the normal-suite typecheck, svelte-check fails with
`"$env/static/public" has no exported member 'PUBLIC_TEXCENTER_ITER'`.
Workarounds (gitignored `.env`, a `define` in vite config) would
have been more invasive than the user's stated "pick whichever
is least invasive" allowance in 184_question. `$env/dynamic/public`
reads `process.env.PUBLIC_TEXCENTER_ITER` at SSR time and SvelteKit
injects it into the client bundle automatically.

The "build-time bake, not runtime fetch" constraint is still
satisfied: `Dockerfile` runtime stage declares
`ARG TEXCENTER_ITER=dev` + `ENV PUBLIC_TEXCENTER_ITER=$TEXCENTER_ITER`.
The env is baked into the image at `docker build` time and
cannot drift without a redeploy (no Fly secret involvement).
