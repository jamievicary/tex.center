## Goal for this iteration

Follow PLAN priorities #1 and #2 — the iter-361 plan-review
identified these explicitly as "iter 362's likely move". The
single issue across the three iter-360 red gold specs is that
their per-test budgets (`testInfo.setTimeout` /
`test.setTimeout`) are tighter than the actual cold-start cost,
so the test times out *before* the iter-359 three-phase
diagnostic line can fire. Three gold passes (358, 359, 360) have
happened without the diagnostic ever appearing in production,
which is the single most informative thing to break first —
without it every subsequent iteration on priority #1 lands blind.

Each spec already carries its own *inner* anti-flake budgets that
do the real regression detection (e.g.
`coldFromInactiveLiveEditableTest` asserts
`cmContentReadyMs ≤ 1000 ms` and `keystrokeAckMs ≤ 1000 ms`
*inside* the test — those still fail loud on a slow cold-start).
The outer `setTimeout` is a wall-clock safety net for the test
infrastructure; widening it does not relax any product invariant.

Concrete moves (parameter-only; no product code touched):

- `verifyLiveGt6LiveEditableState.spec.ts` (suspended):
  `testInfo.setTimeout(40_000)` → `120_000`.
- `verifyLiveGt6LiveEditableStateStopped.spec.ts` (stopped):
  `testInfo.setTimeout(60_000)` → `120_000`.
- `verifyLiveFullPipeline.spec.ts` (fresh full pipeline):
  `test.setTimeout(40_000)` → `150_000`. The inner
  `cmContent.waitFor({timeout: 120_000})` was already budgeted
  for 120 s of cold-start; the 40 s outer was cutting it off.
  150 s gives the 120 s locator headroom + ~30 s for type +
  pdf-segment poll + canvas-paint poll.

Update the inline rationale comments in each spec so the new
numbers carry their reason inline rather than just `git blame`-
ing back here. No shared-fixture changes — the
`coldFromInactiveLiveEditableTest.ts` inner phase timeouts
already cover their three phases.

`tests_normal/` is unaffected — these are `[live]`-only gold
specs.
