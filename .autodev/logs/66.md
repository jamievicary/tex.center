## Goal for this iteration

PLAN's "next ordinary iteration" (M6.3.1) is out-of-tree. File-tree
CRUD verbs (create iter 61, delete iter 62, rename iter 63) cover
the three create-or-edit-name affordances; client validation +
race-rejection feedback landed iter 64–65. GOAL.md item 3
explicitly lists **upload** alongside create / rename / delete.
Adding `upload-file` is a clean next slice that completes the
file-tree CRUD surface: drag-from-disk lands existing `.tex` /
`.bib` / `.cls` sources straight into the project.

Scope: text uploads only (UTF-8). Binary assets (images, PDFs)
fold into a future iteration once `Y.Text`'s text-only model gives
way to a binary-blob channel — that's a bigger design step.

## Approach

- Protocol:
  - New `ControlMessage` variant
    `{ type: "upload-file"; name: string; content: string }`.
  - Extend `file-op-error.op` union to include `"upload-file"` so
    the existing client banner surfaces server-side rejections
    (duplicate names, invalid characters) the same way it does
    for the other verbs.
- `ProjectPersistence.addFile(name, content?)`: optional second
  argument. When `content` is a non-empty string, the PUT carries
  encoded bytes and the in-memory `Y.Text` is populated within a
  single `doc.transact` so observers see one update. Empty /
  undefined content keeps the existing "create empty" path
  byte-for-byte identical. Mirrored in the no-blobStore branch
  so unit tests without a blob store still see the new text in
  the Y.Doc.
- Server: new `upload-file` handler, structurally identical to
  `create-file`, calling `addFile(name, content)` and broadcasting
  a refreshed `file-list` on success, unicasting a `file-op-error`
  with `op: "upload-file"` on rejection.
- `WsClient.uploadFile(name, content)`: thin send wrapper.
- `FileTree.svelte`:
  - Hidden `<input type="file" multiple>` triggered by a small
    "Upload…" button in the create-form row. `accept` is left
    permissive — the validator will reject names with disallowed
    characters anyway.
  - For each `File` whose name passes local validation, read
    `await f.text()` and call `onUploadFile(name, content)`. For
    names that fail validation, `window.alert(reason)` (same
    pattern as rename). The reset (`input.value = ""`) after the
    loop lets re-selecting the same file fire `change` again.
- Editor page: passes `onUploadFile={(n, c) => client?.uploadFile(n, c)}`.
- Tests:
  - Protocol codec round-trip for `upload-file` and for
    `file-op-error.op = "upload-file"`.
  - New `apps/sidecar/test/serverUploadFile.test.mjs`: upload
    succeeds → file-list contains it, Y.Text has content, blob
    has the bytes; duplicate upload rejected with a
    `file-op-error` carrying `op: "upload-file"`; invalid name
    rejected; cold restart sees the uploaded file with its
    content.
  - Register in `tests_normal/cases/test_node_suites.py`.

## Notes for future iterations

- Images and other binary assets need a parallel channel —
  `Y.Text` only handles UTF-8. A `binary-upload-file` verb
  carrying raw bytes + a Y.Map registry would be one shape;
  storing the bytes only in the blob store (with the registry as
  the in-memory truth) is probably simpler than mirroring them
  into Yjs.
- `addFile`'s content param means the future "upload to overwrite
  existing file" affordance can't reuse it as-is — it currently
  rejects duplicates. A separate `replace-file` verb or an explicit
  `overwrite: true` flag is the cleaner extension.
