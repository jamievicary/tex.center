## Goal for this iteration

Address the remaining live gold failure: GT-5
(`verifyLiveGt5EditUpdatesPreview`). Iter 207's harness pass shows
GT-A/B/C/D all green on live; only GT-5 fails with
"preview canvas hash did not diverge from the pre-edit snapshot
within timeout" after a 10s post-edit poll.

The 10s budget is the local issue: GT-D, which runs immediately
before GT-5 on the same shared project, takes ~35s and ends with
a 30s `waitForTimeout(30_000)` for the final coalesced compile.
GT-5 then types a substantial payload (~25 chars at 5ms delay) on
top of GT-D's typed body and only allows 10s for keystrokes →
coalescer → supertex round-trip → wire frame → PDF.js re-render
→ canvas re-paint. That's tight given observed live latencies
(GT-C single char: 2.6s; GT-D sustained: 35.7s).

The failure message is also under-informative. Strict
byte-exact "hash didn't diverge" conflates two distinct modes:

1. No post-edit pdf-segment frame arrived (compile/wire path
   broken).
2. Post-edit pdf-segment arrived but byte-identical to a prior
   segment (the iter-188 regression class the spec catches).

GT-5's `captureFrames` already tracks `pdfSegmentFrames`; reading
its count before/after the edit disambiguates.

## Approach

Two-part change to the GT-5 spec only — no production code
touched (GT-C/D demonstrate the live edit→PDF path is healthy):

- Bump the canvas-hash divergence timeout from 10s to 30s,
  matching the spec's own preamble (the snapshot poll already
  uses 30s) and the comment's stated "60s budget" intent.
- Before asserting hash diverged, snapshot
  `pdfSegmentFrames.length` immediately before typing and wait
  (bounded) for it to increase post-edit. A timeout here yields
  a specific "no post-edit pdf-segment frame" error; only after a
  new frame arrives do we enter the hash-divergence poll, whose
  timeout then unambiguously points at the iter-188
  byte-identical class.

## Actions

- Edited `tests_gold/playwright/verifyLiveGt5EditUpdatesPreview.spec.ts`:
  - Captured `pdfSegmentCountBeforeEdit = pdfSegmentFrames.length`
    after the pre-edit snapshot, before the typing block.
  - After `keyboard.type(...)`, added a 30s
    `expect.poll(() => pdfSegmentFrames.length).toBeGreaterThan(pdfSegmentCountBeforeEdit)`
    with a message naming the compile-path failure mode.
  - Bumped `expectPreviewCanvasChanged` `timeoutMs` from 10_000
    to 30_000.
