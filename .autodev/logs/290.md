## Approach

The algebra confirming the branch is dead (recapitulated for the
log so a future reviewer doesn't have to redo it):

After the entry-line clamps, `tree ≥ MIN_TREE_PX` and
`preview ≥ MIN_PREVIEW_PX`. Then `maxTree = total - dividers -
MIN_EDITOR_PX - preview`.

- **Floor path** (`maxTree < MIN_TREE_PX`): returns with both
  columns at their floors. The deleted branch is gated behind the
  `else` and never runs.
- **Tree-squeeze path** (`tree > maxTree`): after `tree = maxTree`,
  the recomputed `maxPreview = total - dividers - MIN_EDITOR_PX -
  tree = preview` exactly. So `preview > maxPreview` is
  `preview > preview` → false.
- **No-squeeze path** (`tree ≤ maxTree`): `tree ≤ total - dividers
  - MIN_EDITOR_PX - preview` rearranges to `maxPreview ≥ preview`.
  Again `preview > maxPreview` is false.

The branch is therefore dead. Deletion preserves observable
behaviour. The invariant the branch was nominally defending — that
the returned `{tree, preview}` plus dividers and editor min always
fits in `total` whenever `total` exceeds the absolute floor — is
worth pinning explicitly so future refactors can't silently break
it. Parameter sweep across plausible inputs is cheap (~210 calls).
