## Goal for this iteration

Iter 68's "notes for future iterations" flagged that the sidecar's
`/ws/project/:projectId` handler still falls back to `"default"`
when projectId is missing — a branch now unreachable from the
web app since `/editor/[projectId]` always supplies an id. Tighten
this: reject anything that doesn't match the workspace's existing
`^[A-Za-z0-9_-]+$` shape with a `1008 invalid projectId` close,
before `getProject` lazily seeds a Y.Doc / scratch dir for it. Small
defence-in-depth slice that doesn't need docker, postgres, or a
streaming compiler.

## Actions

- `apps/sidecar/src/server.ts`: `/ws/project/:projectId` handler
  now validates `params.projectId` against `/^[A-Za-z0-9_-]+$/`
  before calling `getProject`. Invalid (including empty/missing,
  defensively) → `socket.close(1008, "invalid projectId")` and
  return. The `?? "default"` fallback is removed.
- `apps/sidecar/test/serverProjectIdValidation.test.mjs`: new.
  Asserts `1008` close with `invalid projectId` reason for

  suite.
