## Goal for this iteration

Fix a real data-corruption hazard in `apps/sidecar/src/server.ts`'s
blob hydration path. Today, if `blobStore.get()` throws during
project hydration (transient outage, partial failure), the code
warn-and-continues with `persistedSource` left as `null`. The very
next compile then evaluates `source !== p.persistedSource`, which
is trivially true (empty Y.Text is `""`, vs. `null`), and the
hydration failure is "resolved" by overwriting the legitimate
remote blob with whatever's currently in the in-memory Y.Doc вЂ”
typically empty.

PLAN's current focus lists three larger candidates. M3.5 isn't
actionable here, multi-file is multi-iteration, M5 auth is
premature without the control plane. This is a small correctness
fix вЂ” a clear extension of the iter-28 work that decoupled
persistence from compile success.

## Approach

1. Track hydration outcome explicitly with a `canPersist: boolean`
   on `ProjectState`. Defaults `false`. Set to `true` only when
   hydration completed without throwing (whether or not the blob
   existed).
2. Gate the post-`writeMain` persistence block on `canPersist`.
3. Add a regression test in `serverBlobs.test.mjs`: wrap the real
   LocalFsBlobStore so the first `get()` throws; pre-populate the
   blob with sentinel content; connect; drive an edit; assert the
   blob is unchanged.

## Notes for future iterations

- `canPersist` is currently a one-shot session flag. A future
  hardening could re-attempt hydration on a periodic schedule or
  on `/healthz` blob-state recovery, flipping `canPersist` true
  once the blob is reachable again. Today, a permanent outage
  means edits in this process are never persisted; on restart
  the sidecar tries again. Acceptable for the per-project
  Machine model where Machines cycle frequently.
- When multi-file lands, `canPersist` becomes per-file (or per-
  project, with per-file `persistedSource` map). Keep the same
  invariant: never `put` a file whose pre-edit state is unknown.
