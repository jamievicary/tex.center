## Goal for this iteration

M7.4.x diagnose: confirm the iter-724 `--daemon` protocol works
locally on the same shape of fixture the live seeded project uses
(a 1-page hello-world document), and instrument the sidecar so the
next live gold run captures which of the three gates in
`vendor/supertex/tools/supertex_daemon.c:1100-1260` fires when
GT-3/GT-5 fail.

Per PLAN.md M7.4.x the iter-198 goal is two-part: (1) build
verification + capture one recompile-after-edit round, and (2)
diagnose which gate fires live. (1) is purely local. (2) needs
live observability — the sidecar already captures the daemon's
stderr into `stderrBuf` but only surfaces it on protocol violations
/ unexpected exit, so a path-2 (`WARN no usable rollback target`)
or path-3 (`WARN wait_for_resumed failed`) failure mode leaves no
trace in Fly logs today. Forwarding daemon stderr + parsed
protocol events to the host stderr stream is a small, focused
change that turns the next live run into a useful diagnostic.

## Approach

1. `make -C vendor/supertex -j all` to build the iter-724 binary.
2. Spawn `vendor/supertex/build/supertex --daemon DIR FIXTURE.tex`
   directly against (a) the 2-page fixture the existing
   `test_supertex_daemon_real` uses and (b) a 1-page fixture
   matching `MAIN_DOC_HELLO_WORLD` (the live seeded content).
   For each, capture stdout / stderr / chunks-dir state across an
   initial-compile + no-edit-recompile + edit-recompile sequence.
3. Forward daemon stderr (line-by-line) and parsed protocol
   events to `process.stderr` in `apps/sidecar/src/compiler/
   supertexDaemon.ts`, with `[supertex-daemon stderr]` /
   `[supertex-daemon event]` prefixes so the lines are
   recognisable in Fly's log stream.

## Notes for future iterations

- The single most informative next step is to run `flyctl logs
  --app tex-center-sidecar` (or `flyctl machine logs`) capturing
  output across a GT-3 failure window, and grep for
  `[supertex-daemon stderr] supertex: WARN`. The WARN message
  text tells you which gate fired:
    - `WARN no usable rollback target` → path 2 (process_event /
      resumed_pid). Upstream issue: re-open M7.4.x as an
      upstream supertex PR with a memory-pressure repro.
    - `WARN wait_for_resumed failed` → path 3. Same call:
      upstream.
    - `WARN resumed_pid=… did not exit within 120s` → path 3
      timeout. Same upstream call, but with a clearer remedy
      (increase the daemon's 120 000 ms cap in
      `supertex_daemon.c:1210`, or fix whatever is keeping the
      resumed sibling alive).
  If none of the three WARN strings appears but the
  `[supertex-daemon event]` log shows no `rollback` and no
  `shipout` between successive `round-done`s, the failure is
  even earlier than path 2 — likely an edit-detection miss in
  `run_baseline_diff`, which would also be upstream.
- The local probes captured here are the working baseline. Any
  deviation in the live event-log shape from
  `rollback K → I+1.out…K.out → round-done` is the bug.
- The iter-189 rollback-handler bug noted in `195_answer.md`
  (`if (ev.k < maxShipout)` initialised to -1 never fires) is
  still latent, but only matters under path 3. If path 3 is
  confirmed live, the sidecar-side fix is to treat
  `[rollback K]` followed by `[round-done]` with no chunks as
  "explicit no-op," distinct from "no rollback announced" — but
  shipping stale 1..K from disk would be byte-identical to the
  last segment, so the user-visible effect is the same. The
  real fix lives upstream.
