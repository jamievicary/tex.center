**Discussion mode.** Answer `284_question.md`: the user
challenges the iter-275/276/279 diagnosis of the M15 page-1-only

See `.autodev/discussion/284_answer.md` for the full answer.
Headline conclusions:

1. The prior diagnosis is unproven. It rests on a single daemon
   stderr line; we never logged the actual source bytes the
   sidecar wrote to disk before each compile, so we cannot
   distinguish "daemon mis-short-circuits a body insertion"
   from "daemon correctly ignores edits past `\end{document}`".
2. Iter 279 *claims* to have filed
   `vendor/supertex/discussion/764_question.md` but the diff
   (`git diff 6919b5e 93dd32b --stat`) shows that commit never
   touched the submodule. The file isn't there. PLAN.md has been
   waiting on an upstream question that was never asked.
3. Strong alternative hypothesis: the live test's keyboard
   sequence lands the cursor *after* `\end{document}` (SEED ends
   in `\n`, so `Ctrl+End` lands on a virtual blank line 5;
   `ArrowUp` → end of `\end{document}` line; `End` → no-op;
   `Enter` → newline after `\end{document}`; typed body is past
   the closing marker). Matches all live evidence: `sourceLen
   161`, daemon stderr "edit @85 past byte 70", `segments: 0`.
4. The local pin `supertexIncrementalMultipageEmit` constructs
   the file with body inserted before `\end{document}`, so its
   "green vs live red" comparison is not a meaningful local-vs-
   live signal — different shapes entirely.
