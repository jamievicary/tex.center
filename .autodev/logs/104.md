## Goal for this iteration

M7.1.2.1 — wire `createUpstreamResolver` into the production
control-plane entry. The pure resolver landed in iter 103; this
slice gates its construction on `FLY_API_TOKEN` +
`SIDECAR_APP_NAME` + `SIDECAR_IMAGE` (with `SIDECAR_PORT` /
`SIDECAR_REGION` defaults), constructs a real `MachinesClient` +
`dbMachineAssignmentStore`, and falls back to the existing static-
envvar upstream when any required env is missing. Factored into a
dependency-injectable helper so it can be unit-tested without
poking the db or the live Fly API.

## Approach

- New module `apps/web/src/lib/server/upstreamFromEnv.ts`:
  - `buildUpstreamFromEnv(env, deps)` returns
    `UpstreamResolver | null`.
  - `deps`: `makeMachinesClient({token, appName})` and
    `makeStore()`. Production wires real impls; tests pass stubs.
  - Returns `null` (preserving the static-envvar fallback) when
    any of `FLY_API_TOKEN`, `SIDECAR_APP_NAME`, `SIDECAR_IMAGE`
    is missing.
  - Otherwise constructs a `MachineConfig` from `SIDECAR_IMAGE`
    plus conservative defaults (`auto_destroy: false`,
    `restart: { policy: "on-failure" }`) and calls
    `createUpstreamResolver`. `sidecarPort` and `sidecarRegion`
    come from `SIDECAR_PORT` (parsed like the existing
    `resolveSidecarUpstream`) and `SIDECAR_REGION` (default
    `fra`, matching M7.0.2).
- `server.ts`: build deps with `new MachinesClient(...)` and
  `dbMachineAssignmentStore(getDb().db)`; lazy on first
  invocation so a missing `DATABASE_URL` doesn't crash boot of
  the non-resolver path. Spread `resolveUpstream` only when
  non-null.
- Test `apps/web/test/upstreamFromEnv.test.mjs`: missing env →
  null; full env → resolver invokes the stub `MachinesClient.
  createMachine` with `image` from `SIDECAR_IMAGE` and resolves
  to the expected 6PN address.

## Notes for future iterations

- M7.1.3 is the natural next step. Set `DATABASE_URL`,
  `FLY_API_TOKEN`, `SIDECAR_APP_NAME`, `SIDECAR_IMAGE` (and
  optionally `SIDECAR_PORT`/`SIDECAR_REGION`) as Fly secrets on
  `tex-center`; redeploy; extend `verifyLive.spec.ts` with a
  happy-path probe that mints a session cookie and verifies a
  `/ws/project/<id>` upgrade returns 101 from a real per-project
  Machine. That probe closes the M7.0.3.3 tail too.
- The `MachineConfig` baked into `upstreamFromEnv.ts` is minimal
  (just `image` + safety defaults). When M7.4 lands, the config
  will need to carry per-project env (checkpoint blob keys, db
  creds, etc.). Today the sidecar source has no `PROJECT_ID`
  reference, so a static config is sufficient.
- The eager `makeStore()` call inside `createUpstreamResolver`
  means the helper would crash boot if `FLY_API_TOKEN` were set
  without `DATABASE_URL`. That's the right failure mode for
  production — once the resolver path is on, the db must be
  configured — but if a future iteration wants to allow lazy
  store init, the change is local.
