## Design notes

- **Why live-only, not local-too.** Locally there is no sidecar
  and no per-project Machine — only the SvelteKit dev server. A
  `local` run would dial a `/ws/project/<id>` upstream that
  doesn't exist, the proxy's resolver would 502, and the test
  would fail for an entirely uninteresting reason. The whole
  point of M8.pw.4 is the integration with the deployed sidecar
  fleet, so it lives on the `live` project only.

- **Why `TEXCENTER_FULL_PIPELINE=1` is a separate gate from
  `TEXCENTER_LIVE_TESTS=1`.** Every other `live` spec is fast
  (sub-second probes). This one cold-starts a Machine + runs
  lualatex, easily 60–120 s for the first iteration after the
  Machine has destructed itself. Hiding it behind a second flag
  means deploy-touching iterations can opt in explicitly while
  routine live verifications keep their ~10 s budget.

- **Why `keyboard.type` survives `closeBrackets`.** Examined the
  basicSetup pipeline: `closeBrackets` registers handlers that
  (a) auto-insert the matching closer when an opener is typed,
  (b) skip insertion of a closer when the cursor already sits on
  the matching closer. Sequence `\begin{document}…\end{document}`:
  on `{` we get `{}` with the cursor between; the literal chars
  fill in; on `}` the extension detects the next char is already
  `}` and just advances the cursor. Net: the doc matches the
  typed source. Confirmed by reading
  `node_modules/@codemirror/autocomplete/dist/index.js`'s
  `closeBrackets` implementation; if a future CodeMirror upgrade
  changes the skip behaviour, swap to a paste event via
  `keyboard.insertText` (which bypasses keydown-driven extensions
  entirely).

- **Why the canvas pixel check is "any pixel < (240,240,240)"
  not a CRC.** A CRC-against-a-fixture canvas would flake on
  every minor PDF.js render-pipeline change (subpixel AA,
  device-pixel-ratio differences between WSL and the CI image,
  font fallback). The invariant we actually want is "PDF.js
  rendered *something* non-trivial" — one off-white pixel is
  enough to disprove both blank-canvas and security-tainted
  branches.

- **Why `createProject` directly instead of going through the
  `/projects` create form.** The create form is exercised by
  `projects.spec.ts` already; re-driving it here adds no
  coverage but spends 2–3 extra page navigations on the critical
  path. Cleanup via direct DB delete is symmetric.

## Notes for future iterations

- **Activation is operator work.** First live run needs
  `TEXCENTER_LIVE_TESTS=1` + `TEXCENTER_FULL_PIPELINE=1` +
  `TEXCENTER_LIVE_DB_PASSWORD` + `SESSION_SIGNING_KEY` +
  `TEXCENTER_LIVE_USER_ID` exported. The first cold-start will
  pull the sidecar image, dump `lualatex.fmt`, hydrate an empty
  Y.Doc, compile, emit the `pdf-segment`, and idle-stop ten
  minutes later. Subsequent runs within the idle window land on
  a warm Machine and complete in seconds.

- **The pixel sentinel is intentionally permissive.** If
  flakiness ever shows up here it almost certainly indicates
  PDF.js failing to render at all (canvas blank), not a near-
  white false positive — the `<240` threshold is far below any
  PDF text rendering.

- **`closeBrackets` interaction with backslash sequences.** Left
  a comment in the spec pointing at the fallback (paste-event via
  `keyboard.insertText`) so a future CodeMirror upgrade that
  changes skip behaviour has a recorded escape hatch.

- **Once both pw.3.3 and pw.4 activate live**, the priority block
  closes and the engine queue returns to M7.4.2 (upstream supertex
  daemon serialise/restore wire — gates checkpoint persistence
  doing anything observable in prod) and then M7.2
  (`/ws/project/<id>` routing-per-project plumbing). M7.4.2 is
  the natural next pick because M7.4.1 (the sidecar half) is
  already wired and waiting.
