## Goal for this iteration

PLAN's "next ordinary iteration" (M6.3.1) is out-of-tree.
File-tree CRUD verbs (create / delete / rename) all landed iter
61–63, but the client surfaces no feedback when the server
rejects a name — invalid characters, the reserved `main.tex`,
duplicates. The user sees nothing change and the rejection is
logged server-side only. Lift the existing
`validateProjectFileName` into the shared protocol package, mirror
the rule client-side in `FileTree.svelte`, and surface inline
error + alert for the create / rename affordances. The server
remains authoritative (defence-in-depth).

## Approach

- `@tex-center/protocol` gains `validateProjectFileName` (with the
  `FILE_NAME_RE` and length cap) — the shared rule lives where
  both halves of the wire already meet.
- `apps/sidecar/src/persistence.ts` drops its local copy and
  imports + re-exports the protocol version. No behavioural
  change on the server.
- `FileTree.svelte`:
  - `rejectionReason(candidate, ignore?)` combines pure validation
    with the duplicate-and-main-name rules from the server. The
    `ignore` parameter lets rename treat the old name as
    available (we never compare a rename target against itself).
  - Create input: `aria-invalid`, `disabled` button, and a small
    error paragraph below the form when the typed name would be
    rejected. Empty input is "no candidate yet" (button disabled,
    no error message).
  - Rename: `promptRename(f)` validates the prompt result and
    `window.alert(reason)` on failure rather than firing the
    request and silently failing.
- New unit tests in `packages/protocol/test/codec.test.mjs` cover
  accept and reject branches of the validator.

## Notes for future iterations

- The server-side rejection path is still log-only when the
  client *does* race a stale `files` list (e.g. another window
  just created the same name). Surfacing those late rejections
  would need a `file-op-error` control message keyed by a client
  request id. Out of scope for MVP; the in-tree validator covers
  the overwhelmingly common typo/main.tex cases.
- `window.prompt` / `window.alert` remain crude but adequate for
  single-user MVP. An inline rename input is a candidate UX
  polish iteration.
