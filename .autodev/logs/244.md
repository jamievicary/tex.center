## Goal for this iteration

Land step 1 of the delete-project sequencing in
`.autodev/discussion/241_answer.md`: a RED gold pin
(`verifyLiveDeleteProject.spec.ts`) that drives the user-visible
`/projects` delete flow on `live` and asserts the full reap —
projects row, machine_assignments row, and any Fly Machine tagged
`texcenter_project=<id>`. The implementation (endpoint + UI) lands
in a follow-up iteration; this iter's job is to pin the missing
behaviour as a gold-runner red, in the same style the iter-233 GT-6
and iter-224 GT-8 pins were landed.

The M9.live-hygiene metadata-tagging primitive landed iter 243, so
the (c) assertion has a stable key (`config.metadata.texcenter_project`)
to filter on without ambiguity against the shared `app`-tagged pool.

## Approach

- New spec at `tests_gold/playwright/verifyLiveDeleteProject.spec.ts`,
  live-only, gated on `TEXCENTER_FULL_PIPELINE=1` and
  `FLY_API_TOKEN`. Pattern mirrors GT-6 (`createProject` directly via
  `@tex-center/db`, dashboard click flow, `cleanupLiveProjectMachine`
  in `finally`).
- Flow: create fresh project → click into editor to guarantee
  per-project Machine creation (so the (c) assertion is non-vacuous)
  → confirm pre-conditions (assignment row + ≥1 tagged Machine) →
  back to `/projects` → locate row by `[data-project-id="<id>"]` →
  click `Delete` button → wait for link to vanish → assert DB row
  null, MA row null, no tagged Machine (with a brief poll because
  Fly destroy is async).
- Selector contract for the follow-up landing iter: each project
  row carries `data-project-id="<id>"`; inside it lives an
  accessibly-named `Delete` button. Any native `confirm()` dialog is
  pre-accepted via `authedPage.on("dialog", …)`.
- Teardown reaps via `cleanupLiveProjectMachine` + a defensive
  `delete from projects` so a RED run does not leak state across
  iterations.

## Notes for future iterations

- Next iter (delete-project landing): server-side form action on
  `/projects` (`?/delete`) that authorises owner, calls
  `cleanupProjectMachine` against the assignment row, deletes R2
  blobs for the project, deletes the projects row (cascades to
  machine_assignments), 303 redirect back. UI: a `Delete` button on
  each `/projects` row inside a `[data-project-id="<id>"]` wrapper,
  with a `confirm()` guard. Keep idempotency on Machine + blob reap
  so retries after partial failure are safe. The pin landing now is
  the acceptance criterion — when it flips green on live, the slice
  is done.
- R2-blob reap: requires the web side to talk to the same bucket
  the sidecar writes. Currently the `BLOB_STORE` binding lives only
  on per-project Machines; if that hasn't been hoisted to the web
  app by the time delete lands, the blob reap step is best-effort
  and the gold pin still passes because it does not assert R2
  state (intentional — the spec is verb-shaped, not storage-shaped).
- The `cleanupLiveProjectMachine` teardown means a RED run on this
  spec costs at most one extra Fly Machine create-then-destroy
  cycle per iteration; the count guardrail
  (`test_machine_count_under_threshold`) remains the safety net.
