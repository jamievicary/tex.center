`apps/web` from `@sveltejs/adapter-static` to `adapter-node`.
This is the smallest behaviour-preserving slice that unblocks the

## Architecture decision

`apps/web` becomes the control plane via `adapter-node`. Rejected
alternative: splitting a separate Fastify control-plane app
alongside `apps/sidecar`. Reasoning:

1. GOAL.md says "Control-plane web app … handles auth, the
   dashboard, the editor shell, and routes WebSocket connections"
   — one service with both pages and API.
2. "Fastify" in the GOAL tech stack is already realised by
   `apps/sidecar` (the per-project tier). A separate control-plane
   Fastify would mostly duplicate SvelteKit's Node server: route
   matching, static-asset serving, request lifecycle.
3. One origin = no CORS dance, one cookie domain, one deploy
   artefact. SvelteKit's `+server.ts` is exactly the right
   primitive for `/auth/google/start` and `/auth/google/callback`.
4. The white-page and editor still ship as **client-only** static
   artefacts (the existing `prerender = true; ssr = false` in
   `routes/+layout.ts` is preserved). adapter-node respects
   per-route prerendering, so only future `+server.ts` endpoints
   render dynamically — the M5.1 routes are server-only by
   construction.

If a future iteration needs a Fastify-specific feature (e.g. a
plugin that doesn't exist for SvelteKit), it can live in
`apps/sidecar` or a dedicated worker; nothing here forecloses
that.

## Notes for future iterations

- **M5.1.1 (next ordinary iteration).** Add
  `apps/web/src/routes/auth/google/+server.ts` for the start
  endpoint. Needs:
  - PKCE generation via `@tex-center/auth.generatePkce`.
  - A signed state cookie carrying `{state, verifier, exp}`.
    `signSessionToken` is shaped only for `{sid, exp}`; either
    extend its payload type (cheapest) or factor a small generic
    `signed.ts` underneath it. Pick at that iteration; either is
    one short module.
  - `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_REDIRECT_URI`, and
    `SESSION_SIGNING_KEY` from env. The route should refuse to
    boot if any are missing — silent fallback would generate a
    redirect Google can't honour.
  - 302 to `https://accounts.google.com/o/oauth2/v2/auth?…` with
    `scope=openid email`, `code_challenge_method=S256`,
    `prompt=select_account`.
- **`creds/google-oauth.json` per GOAL.md is the source of
  truth** for client_id/client_secret. The iteration that lands
  M5.1.1 should read it via a small loader (or refuse to start
  and tell the user which file to create).
- **adapter-node prod runner.** Future Dockerfile (M6) will
  `COPY apps/web/build /app && CMD ["node", "build/index.js"]`
  plus the production node_modules (or bundle). No need to
  precompute that now, just keep in mind that the dev workflow
  (`pnpm --filter @tex-center/web dev`) is unchanged by this
  swap.
- **`prerender` decision per future `+server.ts` route.** Server
  endpoints are not prerendered. The current global
  `prerender = true; ssr = false` in `+layout.ts` only applies to
  page routes; `+server.ts` is unaffected. If a page route later
  needs SSR (e.g. for an auth-redirect that can't be done
  client-side), it opts out per-page rather than flipping the
  global.
