## Goal for this iteration

Iter 67 landed the project-row storage primitives (`createProject`,
`getProjectById`, `listProjectsByOwnerId`) but noted that every
runtime codepath still uses the hardcoded literal `"default"`. The
natural next slice — and the one called out in iter 67's
"notes for future iterations" — is the projects dashboard: a
`/projects` list page with a "new project" form, per-project
editor routing `/editor/[projectId]`, and an ownership check on
the editor's server load. Closes GOAL.md item 5 (acceptance #6)
at the page-routing level; the sidecar already accepts the project
id as a path parameter, so this is mostly web-app work.

## Notes for future iterations

- The sidecar's `getProject(projectId)` path still falls back to
  `"default"` when the WS URL has no projectId. That branch is
  now unreachable from the web app, but the sidecar should
  probably reject empty/missing project ids outright; tightening
  is best done alongside a gold-test that exercises the new
  routing.
- A gold-test driving the full `/projects` → create → `/editor/<id>`
  → WS-connect chain would close the loop end-to-end. Needs
  Playwright (deferred to M8).
- `updateProjectName` / `deleteProject` primitives become useful
  once the dashboard exposes rename / delete affordances. Still
  deferred until those UI affordances are designed.
- The sidecar's per-project Y.Doc cache (`getProject`) is keyed
  on whatever string the WS URL provides, so two browsers
  hitting the same project id share state — the same
  collaborative behaviour the literal `"default"` route gave us.
  Per-user isolation isn't required for MVP (single-user
  allowlist), but a future iteration could narrow the cache key
  to `(ownerId, projectId)` for defence-in-depth.
