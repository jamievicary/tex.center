## Goal for this iteration

Land **M13.2(b).3 — cold-editable gold case**, the next sequenced
slice in `PLAN.md` after iter 254's optimistic delete and iter 255's
self-suspend resume fix. Adds a new aspirational gold spec that
exercises the suspended → resume → fully-editable path, asserting
tight 1000 ms budgets for both `.cm-content` ready and first Yjs
DOC_UPDATE wire frame after keystroke.

## Approach

The plan called for "Machine suspended ≥ 5 min" but in-test
wallclock is precious — Fly's idle timer is 600 s and we don't
need to wait for it to fire. The Fly Machines REST API exposes
the same `POST /machines/{id}/suspend` endpoint the sidecar's own
`createIdleHandler` calls. Driving suspend directly from the test
turns the multi-minute idle wait into a few-second API round-trip
plus state-transition polling.

Spec shape, mirroring GT-6/GT-8:
- Skip unless `live` + `TEXCENTER_FULL_PIPELINE=1` + `FLY_API_TOKEN`.
- Create fresh project, navigate `/editor/<id>`, wait for first
  pdf-segment (cold-start fully done).
- Navigate back to `/projects` so the page-side WS closes cleanly.
- Look up `machine_assignments.machine_id`, POST `/suspend`, poll
  `GET /machines/{id}` until `state === "suspended"` (60 s bound).
- From `/projects`, click the project link. Time from click →
  `.cm-content` containing the `documentclass` sentinel, and from
  keystroke → next `framesent` of tag 0x00 (TAG_DOC_UPDATE).
- Assert both ≤ 1000 ms.
- `afterEach`-style cleanup via `cleanupLiveProjectMachine`.

Per the M13.2 known-follow-ups note ("widen SSR seed for non-fresh
projects"), this spec is expected to go RED on the cm-content
budget until that follow-up lands (today's blank
`.editor-placeholder` ~11.5 s WS-open path). Gold failures don't
revert; they correctly gate `finished.md`.
