## Goal for this iteration

M7.0.1 ‚Äî provision the patched lualatex engine binary that the
sidecar Docker image's `/opt/engine/bin/` placeholder (iter 74)
expected. Without it, every `flyctl deploy` of the sidecar lands a
runtime image where `supertex`'s `find_engine` returns `None` and
the first project open immediately fails. PLAN offered two routes:
build from patched source in-Dockerfile (slow, reproducible) or
vendor a prebuilt ELF (fast, opaque). PLAN preferred (b) on
build-time grounds; this iteration takes (b).

## Approach

The maintainer already has a working build of the patched engine
on this host at `/tmp/luatex-append-build/build/texk/web2c/luatex`,
wired into `~/.luatex-append/binary` and exercised by their daily
`lualatex-append` workflow. Provenance check:

- Upstream repo: `github.com/jamievicary/luatex-incremental` (set
  as the `origin` remote of the build's source tree).
- Source HEAD at build time: `aa053dd-dirty` (`git describe` ‚Äî
  the trailing `-dirty` flags uncommitted local changes).
- Binary properties: 7.3MB stripped ELF, x86_64-linux,
  dynamically linked, `objdump -T | grep GLIBC_` tops out at
  `GLIBC_2.34` (Debian bookworm-slim ships glibc 2.36, so the
  binary loads on the sidecar image).
- Format file (`.fmt`, 11.9MB) is NOT vendored ‚Äî it's
  texlive-version-specific and the runtime image's `texlive-full`
  is the one we need to bind against.

## Notes for future iterations

- **Image is still dark code.** Like iter 74, the Dockerfile is
  not built anywhere in tests_normal ‚Äî the structural test only
  pins shape. The first real signal is M7.0.2's `flyctl deploy
  --remote-only`. Expect possible apt-package gaps for the fmt
  dump (e.g. missing `lualatex.ini` or `texsys.cfg` if
  `texlive-full` doesn't include them in the slim image ‚Äî though
  the metapackage should). If the `--ini` step fails, the next
  iteration can split it into a smaller diagnostic RUN.
- **Reproducibility gap.** `aa053dd-dirty` means a clean rebuild
  from the upstream commit produces a *different* binary. This is
  the FUTURE_IDEAS entry. Until that lands, `vendor/engine/` is
  an unhashed binary blob ‚Äî fine for now, must be revisited
  before any "you can rebuild tex.center end-to-end from this
  repo" claim.
- **Other arches.** `vendor/engine/x86_64-linux/` shape is
  deliberate; arm64 (Apple Silicon dev, Fly arm Machines) would
  add a sibling directory + a `--platform`-aware COPY. No
  current need.
- **fmt cache layer ordering.** The fmt-dump RUN sits after the
  `COPY --from=builder /repo /opt/repo` line, so any change in
  the workspace tree invalidates it. That's wrong for cache
  efficiency. A follow-up iteration can move the engine COPY +
  fmt dump above the workspace COPY (just below the apt-install
  layer), so a Node-code change doesn't re-dump the fmt.
- **M7.0.2 prerequisites.** Need to: pick a Fly app name
  (`tex-center-sidecar` per PLAN), write `apps/sidecar/fly.toml`
  (mirror `apps/web/fly.toml`'s shape ‚Äî `fra`, scale-to-zero,
  shared-cpu-1x but more RAM for texlive ‚Äî and disable public
  HTTP service since sidecar is reached only over 6PN), then run
  `flyctl apps create` + `flyctl deploy --remote-only`. First
  real image build is where we'll discover whether the dirty-
  tree binary actually runs and the fmt dumps cleanly.
