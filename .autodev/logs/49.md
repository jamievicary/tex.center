## Goal for this iteration

Small UX/correctness fix on the OAuth callback. Today, if Google
returns `?error=access_denied` — the canonical OAuth code for "user
clicked Cancel on the consent screen" — `resolveGoogleCallback`
emits a 400 with `OAuth error from Google: access_denied` as
plain-text body. That's wrong on two counts:

1. **It's not an error.** The user chose not to sign in. The
   correct response matches the allowlist-deny branch one step
   later: clear the state cookie and 302 back to `/` (the white
   sign-in page). Cancelling on Google and landing back on the
   sign-in page is the natural flow.
2. **The 400 body leaks the OAuth flow** to a casual user who hit
   the back button after consent. The white-page philosophy in
   GOAL #1 says unauth visitors see *only* the sign-in widget; a
   plain-text "OAuth error from Google" page contradicts that.

Other `error=` codes (`invalid_request`, `server_error`,
configuration mismatches, etc.) keep the 400 — those genuinely are
operator bugs we want surfaced loudly, not silently swallowed into
a redirect loop.

M6.3.1 (live cutover) still requires out-of-tree tokens; this is
the right shape of small slice in the meantime.

## Approach

- `apps/web/src/lib/server/oauthCallback.ts`: in the existing
  `queryError` branch, special-case `"access_denied"` → return the
  same `{kind: "redirect", location: signedOutPath, setCookies:
  [clearState]}` shape used by the allowlist-deny branch. Drop a
  short comment explaining *why* (cancel is normal, other codes
  are operator-visible). Everything else keeps the 400 path.
- Update `apps/web/test/oauthCallback.test.mjs`: existing
  "?error=access_denied → 400" case becomes "→ redirect to /";
  add a new case for a non-cancel error code (`invalid_request`)
  to lock in that the rest of the branch is unchanged. The
  "weird chars → unknown" case stays as-is (operator-visible
  branch).

No production wiring change needed — the route file already maps
`{kind: "redirect"}` to a 302, so this just falls through.

## Notes for future iterations

- There's a near-identical UX consideration on the 400 branches
  for malformed/expired state cookies. Those are mostly "user
  came back hours later from a stale tab" or "ran two parallel
  sign-in flows", not attack signals. Could redirect to `/` for
  those too — but it would mask genuine operator bugs (signing
  key rotation gone wrong, clock skew). Worth thinking about in
  M8 acceptance once we see real-user logs.
- `access_denied` from Google can also include `error_description`
  and `error_uri` query params; we ignore them. Fine — they're
  meant for the operator, not the user.
