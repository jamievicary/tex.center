## Goal for this iteration

PLAN priority 1: provision Tigris on Fly and wire the secrets onto
both `tex-center` and `tex-center-sidecar`. Iter 325 landed the
S3 adapter; iter 326 landed the AWS_*-name fallback so this slice
becomes a one-liner per app:
`flyctl storage create -a <app> -n texcenter-blobs -y` then
`flyctl secrets set BLOB_STORE=s3 -a <app>`. The auto-injected
AWS_* secrets (ACCESS_KEY_ID / SECRET_ACCESS_KEY / REGION /
ENDPOINT_URL_S3 / BUCKET_NAME) satisfy
`defaultBlobStoreFromEnv()`'s `s3` branch with no further
plumbing.

This unblocks `verifyLiveGt6LiveEditableStateStopped` from the
storage-architecture side. The cold-cycle gold spec (M20.3 second
half) is a follow-up; this iteration's success criterion is
"secrets in place on both apps, both deploys healthy, nothing
regressed in `tests_normal`".

## Approach

Two Fly apps need the AWS_* secrets that `defaultBlobStoreFromEnv`'s
S3 branch resolves through (per iter 326 fallback):
`tex-center` (web tier — composes the SSR seed-doc chain via
`coldSourceFor`) and `tex-center-sidecar` (per-project sidecar
Machines inherit the app's secret set on creation, so a single
`flyctl secrets set` covers every Machine spawned afterwards).

Wanted both apps reading/writing the same bucket so a stop on one
Machine and re-open on a fresh Machine recovers via shared
storage. `flyctl storage create -a tex-center` creates the bucket
and auto-injects the AWS_* set into that one app. The plan said
"repeat for sidecar mints app-scoped credentials against the same
bucket", but in practice `flyctl storage create -n texcenter-blobs`
on the second app errors with `Name has already been taken` — the
bucket-name namespace is global across Tigris projects and there's
no UI for "attach extra app to existing bucket". Pragmatic
resolution: read the AWS_* values out of the first app's
`storage create` stdout and `flyctl secrets set` them onto the
sidecar app verbatim. One bucket, one credential pair, both apps
read/write through the same identity. Saved a copy in
`creds/tigris-tex-center.txt` for re-deploy reference (the
`storage create` output is shown once and not recoverable).

## Notes for future iterations

- **Cold-start latency is the next M20.3 bottleneck.** A fresh
  cold start to first pdf-segment is ~89 s end-to-end as of this
  iteration. Some of that is Machine creation + image pull
  (typically ~10-15 s), some is supertex first-compile, and some
  may be the new S3 round-trip on the sidecar's
  `persistence.maybeRehydrate` path. The next slice toward M20.3
  closure should instrument this — log per-stage timings on a
  cold boot (Machine create / image pull / sidecar start /
  `maybeRehydrate` / first compile / first segment ship) and find
  the dominant term. The 60 s test budget in
  `verifyLiveGt6LiveEditableStateStopped` was set against a 12.5 s
  manual measurement (per the spec's own comment, 2026-05-15) and
  is the right north-star, but the gap is now ~7×.
- **Single-credential model** for the shared bucket is fine while
  the threat model is "both apps trusted equally". If a future
  decision splits trust (e.g. read-only on the web tier), use the
  Tigris dashboard / API to mint a second IAM-style credential and
  set it as a separate secret on the lower-trust app.
- **Cold-cycle preservation gold spec (M20.3 second half).**
  Should test "create project, edit non-trivial source, close
  editor, force-stop Machine, reopen → source matches what was
  typed". The existing GT-6-stopped spec only checks the seed
  placeholder (which the hello-world fallback also satisfies);
  preservation needs a unique edited string the test types and
  reads back. Defer until cold-start latency is reduced enough
  that the preservation phase has any budget at all.
