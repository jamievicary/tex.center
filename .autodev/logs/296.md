## Goal for this iteration

M21 max-visible-page wire signal (M21.1). Today
`apps/web/src/lib/pageTracker.ts` reports the *most-visible* page (the
one with the highest intersectionRatio); the editor pipes that into
`client.setViewingPage(...)` so the sidecar compiles up to that page.
GOAL item 4 needs *max-visible* — the highest-indexed page that has any
non-zero visibility — so a page whose top edge intrudes from below the
fold starts compiling proactively. Per PLAN M21.

Slice scope: pageTracker change + PdfViewer consumer + unit tests.
The gold spec ("3-page PDF, scroll so page 2 fully visible and page 3's
top edge intrudes → sidecar receives target=3") is a separable slice
(M21.2) — adding a real 3-page Playwright source plus sidecar
introspection is substantial test surface that doesn't share code with
the logic change.

## Approach

- Add `pickMaxVisible(items)` alongside `pickMostVisible` in
  `pageTracker.ts`. Same `ratio > 0` predicate, picks the highest page
  index (a complete page hidden at the top + a barely-visible page
  below both count; max-page wins).
- Widen `PageTracker.update()` return shape to
  `{ mostVisible: number | null; maxVisible: number | null }`. Each
  member non-null iff that value transitioned on this update. Tracker
  retains last-known semantics for both (preserves current behaviour
  on transition-to-empty).
- Add `tracker.maxVisible` getter symmetric with the existing
  `tracker.visible` (renamed conceptually but keeping `visible` for
  back-compat unit tests — it now reads `currentMost`).
- `PdfViewer.svelte`: switch the IO callback to consume
  `maxVisible` instead of the bare return value. Single-line change.
- Update `pageTracker.test.mjs` with the new return shape + cases
  exercising `pickMaxVisible` and the transition semantics.

## Notes for future iterations

- **`maxViewingPage(p)` reducer is now load-bearing in a new
  way.** The sidecar's `maxViewingPage` reduces across clients;
  each client now contributes its own max-visible. Multi-client
  edits (rare but possible — two browser windows on the same
  project) get the correct "compile up to whatever any client
  can see" behaviour for free. Worth recording so a future
  cleanup doesn't simplify this into `viewingPage = lastClient`.
- **M21.2 gold spec design.** Two sidecar introspection paths
  exist: (i) extend the existing `test_supertex_*` Python harness
  to receive viewing-page messages via the sidecar WS (heavy,
  duplicates Playwright); (ii) a Playwright spec that opens the
  editor, scrolls so page 3's top edge intrudes, and inspects
  the sidecar machine's `viewingPage` state via a debug HTTP
  endpoint that doesn't currently exist (would need adding).
  (ii) is lighter once the endpoint lands; bundle with a future
  debug-affordance slice.
- **M19 is the natural next pick.** Visible user-facing value
  (cog + settings + email-in-topbar from `293_answer.md` items
  2/3), well-scoped slices, sets up the runes-based settings
  store other features will reuse.
