## Goal for this iteration

M7.0.3.2 — auth gating on the control-plane WS proxy. Without it
`/ws/project/<id>` is the only unauth-accessible authenticated
surface on the control plane; an attacker can wake and talk to
the sidecar without ever signing in.

## Approach

Layer the auth check into the **pure** proxy module as an
optional injection point, then build the cookie-verifying wiring
one layer up so the proxy stays testable without DB / session
signing key. Decision matches the iter-94 follow-up note (option
(b): `authoriseUpgrade(req) → Promise<boolean>` on
`WsProxyOptions`).

Fail-closed everywhere: an authoriser that throws/rejects becomes
HTTP 401, never a leak. An entry that boots without a session
signing key configured installs an authoriser that returns false
unconditionally — same shape as `hooks.server.ts`'s "anonymous
everywhere", except on the WS side anonymous = blocked.

## Actions

1. `apps/web/src/lib/server/wsProxy.ts`:
   - New optional `WsProxyOptions.authoriseUpgrade`.
   - New `WsProxyEvent` kinds: `unauthorised`, `auth-error`.
   - Refactored: extracted `writeStatusAndClose(sock, status)` —
     both 404 (no-match) and 401 (rejected/threw) reuse it.
     Existing upstream-dial logic split into a `dialAndPipe(...)`
     helper so the upgrade handler stays linear (match path →
     authorise → proceed/reject). Promise.resolve covers both
     sync and async `authoriseUpgrade` return shapes.
2. `apps/web/src/lib/server/wsAuth.ts` (new):
   - `makeSessionAuthoriser({signingKey, sessionCookieName,
     lookupSession, now?})` → `(req) => Promise<boolean>`.
   - Reads `req.headers.cookie`, defers to `resolveSessionHook`
     (same path as `hooks.server.ts`), returns `session !==
     null`. `now` is injectable for tests.
3. `apps/web/src/lib/server/boot.ts`: accepts optional
   `authoriseUpgrade` and forwards. The conditional-spread
   pattern keeps `exactOptionalPropertyTypes` happy without
   widening the proxy's signature.
4. `apps/web/src/server.ts`: builds the production authoriser
   lazily — `loadSessionSigningKey()` once at module init,
   `lookupSession` closes over `getDb()` + `getSessionWithUser`.
   Missing/malformed key logs an error and installs an
   `async () => false` reject-everything authoriser.
5. Tests:
   - New `apps/web/test/wsAuth.test.mjs` (registered as
     `test_web_ws_auth`): valid cookie / unrelated cookies /
     missing cookie header / wrong cookie name / tampered sig /
     unknown sid / lookup-throws. Mints real signed tokens via
     `@tex-center/auth` so the b64u + HMAC paths are exercised
     end-to-end.
   - Extended `apps/web/test/wsProxy.test.mjs` with: reject path
     (401 from client, **0 upstream connections** asserted via
     server-side counter), accept path (bytes flow + dial
     happens), throwing authoriser (401 + `auth-error` event,
     still no upstream dial). The "upstream never dialled on
     reject" assertion is the load-bearing one — a future
     refactor that accidentally dialled-then-disconnected on
     reject would silently waste sidecar wake-ups but still pass
     a simpler test.

## Notes for future iterations

  next iteration: `flyctl deploy -a tex-center --config
  apps/web/fly.toml .`, then drive `wss://tex.center/ws/project/
  smoke` two ways:

       Upgrade response. Don't bother waiting for the sidecar.

       admin user → expect the upgrade to succeed and the
       sidecar machine (currently stopped) to wake. Allow ~5s

   slice; (2) doubles as the M7.0.3 closeout.

  gated behind `TEXCENTER_LIVE_TESTS` is the canonical place
  for the 401 probe — extend it in M8.pw.2 to assert the
  Upgrade response status without needing a real WS handshake.
- esbuild server bundle pulls in drizzle now. If image-size
  budget tightens, the easy lever is marking
  `@tex-center/db` (or specifically drizzle-orm/postgres) as
  external in `scripts/build-server-entry.mjs` and copying the
  resolved package out of adapter-node's `node_modules` —
  there's only one Postgres pool at runtime, the dupe is purely
  bundling.

  tests. Once the proxy is in production, plumbing it into
  `console.error` (or a structured logger) inside `server.ts`'s
  authoriser would surface DB outages earlier than a generic
  401 stream.
