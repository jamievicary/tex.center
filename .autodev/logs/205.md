## Goal for this iteration

Address the live GT-B/C/D/5 failures whose root cause was diagnosed
in iter 204's "Notes for future iterations": the gold harness's
`sharedLiveProject` warm-up consumes the project's initial compile,
so when a per-spec test later opens a fresh WS to the same project,
the supertex daemon short-circuits the unchanged-source `recompile`
to `{segments: []}` (`supertexDaemon.ts:131`). The new viewer never
receives a `pdf-segment` frame, the per-spec 5s assertion in GT-B
(and the equivalent pre-edit gate in GT-C/D/5) times out. Fix: cache
the last successfully-emitted PDF segment(s) in the sidecar project
state and replay them to a new WS subscriber on connect.

## Approach

Server-side replay, not a daemon-side fix. The daemon's short-circuit
is the *correct* response for a no-op recompile (iter 188's lesson:
synthesising a fake "fresh" PDF from stale disk chunks hid the
upstream rollback no-op and caused the edit→preview regression). The
right invariant for the sidecar is "a new subscriber sees the current
PDF state without needing an edit" — which is just an in-memory
replay of the most recent non-empty segment list. Source-of-truth
for "current PDF" is whatever the sidecar most recently broadcast.

## Actions

- `apps/sidecar/src/server.ts`:
  - `ProjectState.lastSegments: PdfSegment[]` (default `[]`).
  - `runCompile`: on a successful compile that produced ≥1 segment,
    replace `p.lastSegments` with `result.segments`. No-op compiles
    (`segments: []`, the daemon short-circuit) leave the cache
    untouched.
  - WS connect path: after the doc-update + file-list frames, replay
    every cached segment to the new client before calling
    `coalescer.kick()`. Socket-state guarded on each iteration.
- `apps/sidecar/test/serverSegmentReplay.test.mjs`: new unit test
  with a `OneShotCompiler` that emits a segment on its first call
  and returns `{segments: []}` on every subsequent call (mirroring
  the daemon's unchanged-source path). Two clients connect in turn;
  the second must receive the cached segment on connect.
- `tests_normal/cases/test_node_suites.py`: wired the new mjs into
  the unittest harness as `test_sidecar_server_segment_replay`.
- Removed three stale `.tmp.*` editor backup files left in
  `apps/sidecar/src/`, `apps/sidecar/src/compiler/`, and
  `apps/sidecar/test/` — purely orphan files from prior iter
  Write tool aborts, not part of any test path.

## Why this is not the iter-188 regression

Iter 188's bug was the *daemon* synthesising a segment from stale
on-disk chunks when its rollback path silently no-op'd a recompile —
the on-the-wire PDF bytes looked fresh but reflected an older
source state. Here the sidecar replays bytes it *itself* broadcast
on a prior successful compile. By construction those bytes are
consistent with the current source (the cache is only written on
`result.ok && segments.length > 0` from a real compile of the
current Y.Text). The daemon's no-op short-circuit remains the
correct signal that nothing has changed.

## Notes for future iterations

- If GT-C/D/5 are still RED after this lands but GT-B is GREEN, the
  remaining problem is on the post-edit compile path — i.e. the
  *second* segment isn't arriving, not the first. That would point
  back at the iter-202 daemon rollback codepath.
- If a viewer reconnects after a compile error (no segments cached
  yet), they get no replay and a kicked compile — same as today.
  Acceptable: a never-successfully-compiled project has no PDF state
  to replay.
- The cache holds whatever the last successful compile emitted —
  segment count and offsets. For the current single-PDF-segment
  daemon path that's one segment starting at offset 0; future
  multi-segment incremental updates can be handled by replaying the
  full list in order (which is what the code already does).
