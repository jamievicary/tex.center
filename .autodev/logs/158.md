## Goal for this iteration

Wire M8.pw.4 (`verifyLiveFullPipeline.spec.ts`) into the deploy
workflow so it runs against live on every push to `main`. This is
the gate PLAN's FREEZE explicitly names ("until M8.pw.4 runs
green automatically on every deploy, no other engineering work
proceeds"). Iter 157 just proved the underlying framed-payload
loop works in production via the standalone probe; M8.pw.4 is the
durable lock-in.

The PLAN's iter-158 entry conflated M8.pw.4 activation with
M8.pw.3.3's GCP-console operator step (`TEST_OAUTH_BYPASS_KEY`).
M8.pw.4 actually uses `authedPage` cookie injection, not the
real-OAuth round trip — so all that's needed is three GH Actions
secrets (already in `creds/`) and a post-deploy CI job.

## Approach

1. Push three secrets to `jamievicary/tex.center` repo via
   `gh secret set` (sourced from `creds/`):
   `TEXCENTER_LIVE_DB_PASSWORD`, `SESSION_SIGNING_KEY`,
   `TEXCENTER_LIVE_USER_ID`. These satisfy `authedPage`'s
   `resolveLiveDbConfig` contract.
2. Add a `live-pipeline` job in `.github/workflows/deploy.yml`,
   `needs: deploy`, that sets up Node 20 + pnpm via corepack +
   flyctl, installs Playwright chromium, and runs the live
   project with `TEXCENTER_FULL_PIPELINE=1` so the M8.pw.4 spec
   does not self-skip.
3. Lock the wiring with `test_deploy_workflow.py::test_live_pipeline_job_runs_full_pipeline_spec`
   — checks job exists, `needs: deploy`, has checkout/setup-node/
   setup-flyctl/playwright-install steps, and threads all four
   required env vars from repo secrets.

## Things not done (intentional)

- **Lifting the FREEZE.** The freeze must come down only after
  the new live-pipeline job actually runs green on a real push,
  not on the iteration that adds it. That's iter 159's call once
  the harness commit's CD has run.
- **M8.pw.3.3 activation.** Separate work, still operator-gated
  (GCP console + `scripts/google-refresh-token.mjs` browser
  flow). Not on the M8.pw.4 / FREEZE-lift path.
- **Manually triggering the new workflow.** The harness commit
  itself triggers deploy.yml, which will run live-pipeline; no
  need to `workflow_dispatch` and risk a leaked-subprocess class
  bug from `flyctl`-class commands.
