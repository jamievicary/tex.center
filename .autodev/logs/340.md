## Goal for this iteration

PLAN priority #1 (M20.3). Iter 339 enriched the GT-9 probe with a
per-iteration Fly Machine state fetch. The resulting transcript pins
the bug exactly:

```
phase=7-sentinel:poll                flyState=stopped
phase=7-sentinel:probe iter=1   ...  flyState=starting
phase=7-sentinel:probe iter=21  ...  flyState=suspended
phase=7-sentinel:probe iter=41…end   flyState=suspended (all the way to 200 s)
```

Read it: after the dashboard re-open click, the web tier's resolver
correctly drives the stopped Machine through `startMachine` →
`waitForStartedWithRetry` → `tcpProbe`. By probe iter=21 (~44 s wall),
the Machine has transitioned `stopped` → `starting` → **`suspended`**
without ever sustaining `started`. That's the sidecar's own
cold-boot suspend timer firing.

Root cause: `apps/sidecar/src/server.ts:338` calls `armIdleTimers()`
at startup, which arms **both** the 5 s `suspendStage` and the
5 min `stopStage`. The suspend timer was the right primitive for
"viewer disconnects" (~300 ms reconnect), but at cold-boot it races
against the upstream resolver's drive-to-started + tcpProbe + WS
upgrade forwarding chain — which on a cold-from-stopped path
routinely takes 20–60 s. The sidecar self-suspends mid-handshake
via `POST /machines/{self}/suspend`. After that, our web proxy's
direct 6PN TCP dial to port 3001 does **not** auto-resume a
suspended Machine (only HTTP-via-Fly-edge does), so the upgrade
stalls indefinitely. Both GT-9 (preserve-edits) and GT-6-stopped
fail with the same shape.

Fix: at startup, only arm the **stop** stage (orphan cleanup —
5 min is plenty for cold-handshake completion). The **suspend**
stage now arms only on a real `viewerCount` 1→0 transition, where
its 5 s timeout matches the "user closed tab" use case it was
designed for. iter-176's "billed indefinitely" rationale for
case-5 in `serverIdleStop.test.mjs` predates the stop stage; it's
now covered by stop-on-cold-boot, which still parks the orphan in
`stopped` (cold-load cost on next wake but zero RAM cost while
parked) without the 5 s race.

**Cron deferral.** Iter 340 is `N%10==0` (refactor iteration). I'm
deferring the refactor pick because closing M20.3 unblocks the
last MVP latency milestone and the diagnostic from iter 339 made
the bug actionable today. Next refactor iteration: 350 at the
latest (≤ 1 deferral, well inside the 3-iteration budget).
