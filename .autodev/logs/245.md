## Goal for this iteration

Land the delete-project endpoint + UI — the slice the iter-244 RED
pin (`tests_gold/playwright/verifyLiveDeleteProject.spec.ts`) was
written for, and the current live focus per PLAN. The pin's
selector contract is fixed: per-row wrapper carries
`data-project-id="<id>"` and contains an accessibly-named `Delete`
button; any native `confirm()` is pre-accepted by the spec.
Endpoint must reap the Fly Machine + cascade the
`machine_assignments` row + delete the `projects` row.

## Approach

1. Add `deleteProject(db, id)` db helper in
   `packages/db/src/projects.ts` (FK cascades the MA row).
2. Add `apps/web/src/lib/server/deleteProject.ts` server helper:
   look up the `machine_assignments` row → if present, construct
   a `MachinesClient` and call
   `destroyMachine(machineId, { force: true })` (404 swallowed as
   "already gone"; non-404 propagates so the action can fail and
   the row stays for retry) → delete the `projects` row. Fly
   destroy is skipped when `FLY_API_TOKEN`/`SIDECAR_APP_NAME`
   aren't set (so local dev still works).
3. Wire a `delete` form action on
   `apps/web/src/routes/projects/+page.server.ts`: owner check
   via `getProjectById` (403 on mismatch; 303 to `/projects` if
   already gone — idempotent), then call the helper, then 303.
4. UI on `apps/web/src/routes/projects/+page.svelte`: wrap each
   `<li>` with `data-project-id={p.id}`, add a per-row
   `<form method="POST" action="?/delete">` with a hidden
   `projectId` and a `Delete` button. Native `confirm()` on
   submit. Minor flex layout so button sits beside the link.
5. Locks (normal suite):
   - new `apps/web/test/deleteProject.test.mjs` against PGlite:
     no-assignment, with-assignment, 404-swallow, non-404
     propagate (+ row preserved), env-unset (Fly skipped).
   - extend `packages/db/test/projects-pglite.test.mjs` to cover
     the cascade through the FK and idempotence on missing rows.
6. PLAN updated to record the slice as LANDED; the iter-244 pin
   is now the live acceptance criterion.

R2-blob reap is intentionally out of scope this iter (iter-244
notes flagged it as best-effort pending the `BLOB_STORE` binding
moving to the web app; the gold pin is verb-shaped, not
storage-shaped, so doesn't gate on it).

## Notes for future iterations

- The `cleanupProjectMachine` helper in `tests_gold/lib/src/`
  remains the gold-side teardown verb. It and the new web-side
  `deleteProject` are near-duplicates by structure; consolidation
  is plausible later (move the shared core into a small package
  used by both, with the test-side helper re-exporting). Not
  worth doing this iter — the duplication is two short files and
  the gold helper has a slightly different surface
  (`AssignmentStore`/`MachineDestroyer` duck types parameterise
  unit-test stubs). Candidate for a future refactor iter.
- R2 blob reap: when the `BLOB_STORE` binding is hoisted to the
  web app, extend `deleteProject` to `list`+batched `delete`
  under the project's prefix before the DB row deletion.
  Idempotent on empty so retries after partial failure are safe.
- The non-404 destroy-error branch preserves the `projects` row
  intentionally so the user can retry. The form action surfaces
  this as a thrown 500 today; a follow-up could `return fail()`
  with a toast-friendly message, but the dashboard has no toast
  plumbing yet (M9.editor-ux.toast consumers cover the editor
  surface only).
- Local dev (no `FLY_API_TOKEN`/`SIDECAR_APP_NAME`) gets a
  silently-skipped Fly destroy — assignment rows can accumulate
  if local development somehow creates them. PGlite local dev
  doesn't currently spawn per-project Machines, so this is a
  non-issue in practice.
