## Approach

- New module `apps/web/src/lib/server/upstreamResolver.ts`:
  - `createUpstreamResolver({machines, store, sidecarPort,
    sidecarRegion, machineConfig})` returns the resolver.
  - State machine over `MachineState`: `started` â†’ use; `created`
    / `starting` â†’ wait; `stopped` / `suspended` â†’ start + wait;
    `stopping` / `suspending` â†’ wait quiescent, then start + wait;
    `destroying` / `destroyed` / `replacing` â†’ drop the cached row
    and recreate. Anything else throws.
  - `cachedStateOf` normalises Fly's wider union into the narrow
    `'starting' | 'running' | 'stopped'` that the
    `machine_assignments.state` column declares.
  - In-process dedup: `Map<projectId, Promise<â€¦>>` shares one
    create/start round-trip across concurrent upgrades for the
    same project. Entry cleared on settle.
  - `MachineAssignmentStore` is a narrow capability interface, so
    tests can stub it with a plain `Map`; `dbMachineAssignmentStore`
    in the same file adapts the iter-102 Drizzle primitives.
- `wsProxy.ts`: widen `WsProxyOptions.upstream` to `SidecarUpstream
  | (projectId)=>SidecarUpstream | Promise<â€¦>`. Resolution runs
  inside `proceed()` (i.e. after `authoriseUpgrade`), so an
  unauthorised caller never triggers the Fly API. Resolver
  throw/reject â†’ 502 + new `resolve-error` `WsProxyEvent`;
  upstream never dialled.
- `boot.ts`: `BootOptions.resolveUpstream?: UpstreamSource` lets
  callers supply a per-project resolver; fallback path is the
  unchanged static envvar.

## Notes for future iterations

- `server.ts` (M7.1.2.1) is the next step: read `FLY_API_TOKEN`,
  `SIDECAR_APP_NAME`, `SIDECAR_IMAGE`, `SIDECAR_REGION` (default
  `fra`) from env; if all present, construct a `MachinesClient`
  + `dbMachineAssignmentStore(getDb().db)` + resolver and pass it
  as `resolveUpstream` to `boot()`. Fall back to the static envvar
  upstream if anything is missing. A small integration test can
  cover the gating logic with a stub `MachinesClient`.
- `MachineConfig` is currently passed in as an opaque blob. The
  production config will need (at least) `image`, `env` (sidecar
  needs `PROJECT_ID`, `DATABASE_URL`, blob creds), `guest`
  (cpu/memory), and possibly `services` if any port has to be
  exposed publicly. The 6PN-only sidecar deployed in M7.0.2 has
  `no public IPs`; per-project Machines should match.
- The terminal-state recreate path drops the row and creates a
  new Machine â€” fine for the MVP, but worth revisiting when M7.4
  lands (a destroyed Machine may still own a checkpoint blob
  that needs migrating to the new Machine, otherwise the user
  loses the in-flight compile state).
- The in-process dedup is intentionally per-resolver-instance,
  not per-host. Two control-plane Machines can race; the upsert
  in `ensureMachineId` will lose one row but each will end up
  routing through a different Machine until the cache reconciles.
  Acceptable for MVP (one control-plane Machine); revisit if we
  scale the control plane horizontally.
