## Goal for this iteration

N=200 → refactor iteration. The single highest-leverage refactor
is extracting the compile coalescer out of
`apps/sidecar/src/server.ts` (684 lines, the largest file in the
tree). Today four fields on `ProjectState`
(`compileInFlight`, `pendingCompile`, `debounceTimer`,
`highestEmittedShipoutPage`) plus three closure functions
(`kickCompile`, `maybeFireCompile`, `maybeKickForView`) inside
`buildServer` implement a self-contained edge-triggered state
machine. They are referenced only within `server.ts` and already
have a dedicated unit test
(`apps/sidecar/test/serverCompileCoalescer.test.mjs`).

Extracting into a `CompileCoalescer` class in a new module
`apps/sidecar/src/compileCoalescer.ts`:
- shrinks `ProjectState` to its actual concerns (Yjs doc, viewers,
  compiler, workspace, persistence, restorePromise) plus a single
  `coalescer` field;
- removes 3 closure-over-`buildServer` functions;
- makes the state machine independently readable and easier to
  reason about in isolation;
- keeps the existing test as-is, since the behaviour is unchanged
  and the test interacts through the WS surface.

Pure refactor, no behaviour change.

## Approach

1. New `apps/sidecar/src/compileCoalescer.ts` exporting
   `CompileCoalescer` with `kick()`, `kickForView(maxPage)`,
   `cancel()`, and a public `highestEmittedShipoutPage` field.
   Constructor takes `{ debounceMs, run: () => Promise<void> }`.
2. `server.ts`: drop the four `ProjectState` fields and the three
   functions; instantiate `CompileCoalescer` per project with
   `runCompile` (curried with `p`) as the `run` callback; update
   call sites and the `onClose` teardown.
3. Run unit tests; iterate.


## Actions

- Added `apps/sidecar/src/compileCoalescer.ts` — new
  `CompileCoalescer` class with `kick()`, `kickForView(maxPage)`,
  `cancel()` and public `highestEmittedShipoutPage`. Constructor
  takes `{ debounceMs, run }`. Behaviour is byte-identical to the
  inline state machine that used to live in `server.ts`.
- `apps/sidecar/src/server.ts`:
  - Dropped four `ProjectState` fields (`compileInFlight`,
    `pendingCompile`, `debounceTimer`,
    `highestEmittedShipoutPage`); replaced with a single
    `coalescer: CompileCoalescer` field.
  - Removed three closure functions (`kickCompile`,
    `maybeFireCompile`, `maybeKickForView`, ~50 lines).
  - `getProject` instantiates the coalescer with
    `() => runCompile(state)` as its `run` callback (late-bound
    via a `null as unknown as CompileCoalescer` placeholder before
    `state.coalescer` is assigned — `state` needs to be in scope
    inside the closure).
  - Call-site updates: hydrate path + `onTextChange` use
    `project.coalescer.kick()`; `view` control msg uses
    `project.coalescer.kickForView(maxViewingPage(project))`;
    socket-close and `onClose` use `coalescer.cancel()`.
  - `runCompile` bumps
    `p.coalescer.highestEmittedShipoutPage` instead of the
    removed direct field.

## Notes for future iterations

- `server.ts` is now 627 lines (was 684); the buildServer closure
  is one state machine simpler to read top-to-bottom.
- The late-bound `state.coalescer = …` after the object literal
  is a small wart driven by needing `state` in `run`'s closure.
  An alternative is passing `state` into a `run(p)` shape but
  that re-exposes the parameter that motivated the closure; the
  current factoring keeps the call shape clean. If a future
  iteration wants to remove this wart, a `bindRun` setter on
  `CompileCoalescer` would suffice.
- Two orphaned `apps/sidecar/test/*.test.mjs.tmp.*` files exist
  in the test dir (visible in `ls`); they predate this iteration
  and are out of scope for a coalescer refactor.
