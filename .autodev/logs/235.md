## Goal for this iteration

M13.1 instrumentation slice — wire the four remaining
`performance.mark` points on the editor hydrate path. Iter 206
scaffolded `apps/web/src/lib/editorMarks.ts` (mark-name constants
+ idempotent `markOnce` helper) and wired `EDITOR_ROUTE_MOUNTED` in
`onMount`. Remaining: `EDITOR_WS_OPEN`,
`EDITOR_YJS_HYDRATED`, `EDITOR_FIRST_TEXT_PAINT`,
`EDITOR_FIRST_PDF_SEGMENT`. These five marks are the probe needed
to diagnose GT-6's ~5 s `.cm-content` appearance lag (PLAN
M9.editor-ux.regress.gt6 / M13.1 / `231_answer.md` hypothesis):
once all marks fire, a live run can read the timeline and identify
which sub-step dominates the gap.

This iteration only wires the marks — the live-run timeline
extraction + GT-6 fix follows in a later iteration. Keeping it
this small because (a) the wiring is straightforward but touches a
load-bearing route file, (b) live timeline collection wants its
own iteration with debug-toast bridge or a dedicated harness, and
(c) the scaffolding pattern PLAN explicitly endorses (§3
"Scaffolding / dark code") fits — the marks are inert observers
until something reads them.

## Approach

Add a single `$effect` in `apps/web/src/routes/editor/[projectId]/+page.svelte`
that reacts to `snapshot`/`text` runes and calls `markOnce` for
each of the four predicates:

- `snapshot.status === "open"` → `EDITOR_WS_OPEN`
- `snapshot.hydrated` → `EDITOR_YJS_HYDRATED`
- `text !== null` → `EDITOR_FIRST_TEXT_PAINT`
- `snapshot.pdfBytes !== null` → `EDITOR_FIRST_PDF_SEGMENT`

`markOnce` is already idempotent (guards on
`performance.getEntriesByName(name).length > 0`), so the effect
re-running on every snapshot delta is harmless — the first true
transition records the mark and subsequent runs are no-ops.

For ws status the existing enum value is `"open"` (see
`apps/web/src/lib/wsClient.ts:22`), not `"connected"`. The mark
name stays `editor:ws-open`, matching that enum.
