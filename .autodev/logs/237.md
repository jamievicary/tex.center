## Goal for this iteration

Two coupled changes driven by the iter-236 GT-6 timeline:
`route-mounted=+0ms ws-open=+11546ms yjs-hydrated=+11564ms
first-text-paint=+1ms first-pdf-segment=(absent)`, source actually
in `.cm-content` at +11572ms.

1. Fix `EDITOR_FIRST_TEXT_PAINT` so it measures the real first text
   paint instead of firing at +1ms. The current predicate
   `text !== null` fires immediately because
   `Y.Doc#getText(name)` returns a non-null Y.Text right away
   (length=0). Retarget at `text.length > 0` via a Y.Text observer.
2. Record the M13.1 diagnostic finding in PLAN, and commit to an
   M13.2 fix direction: SSR-side seed content so the editor paints
   while the per-project Machine cold-starts in parallel. The WS
   upgrade in `wsProxy.ts:200` blocks until the sidecar TCP-binds;
   for a fresh project the seed is just the static
   `MAIN_DOC_HELLO_WORLD` template, so the control plane already
   has everything needed to paint immediately.

## Actions

- `apps/web/src/routes/editor/[projectId]/+page.svelte`: split the
  M13.1 mark `$effect`. The first effect still fires WS_OPEN,
  YJS_HYDRATED, FIRST_PDF_SEGMENT off the snapshot. A new effect
  watches `text` and registers a `text.observe()` observer that
  fires `EDITOR_FIRST_TEXT_PAINT` the first time `text.length > 0`
  (or immediately if the bound text is already non-empty after a
  file switch). Returns an unobserve cleanup so file switches
  re-bind cleanly. No production behaviour changes; this only
  shifts when the mark fires.
- `.autodev/PLAN.md`: updated the M13 section with the iter-236
  diagnostic timeline, the instrumentation-artefact note for
  `first-text-paint=+1ms`, and the M13.2 fix direction (SSR-side
  seed body via the editor route's `+page.server.ts` `load`, with
  the cold-fresh-project shape as the smallest first slice).
  Updated the "Current state" pointer to reference iter 237.

## Notes for future iterations

- **Iter 238**: implement the smallest version of M13.2 (a) from
  PLAN. In `apps/web/src/routes/editor/[projectId]/+page.server.ts`
  (create if absent), include seed bodies for the project's files
  in the `load` return. For the cold-project shape this is just
  `{ "main.tex": MAIN_DOC_HELLO_WORLD }`. On the client, before
  WsClient is constructed, write that seed into the local Y.Doc so
  CodeMirror paints immediately. Gate this on a flag in the load
  return so we only seed when the server believes the project is
  cold/fresh (e.g. R2 listing returned zero blobs for this project,
  or — simpler — only on the first SSR ever for this project per
  the DB row). On reconnect, do not seed locally; the sidecar's
  authoritative state wins.
- **Followup**: once first-text-paint is honest, also widen the
  GT-6 diagnostic to surface `.cm-content` first non-empty
  `textContent` time alongside the marks, so the gap between
  Y.Text content arrival and DOM materialisation is visible.
  Currently noted in iter-236 "possible follow-up to widen the
  probe" — deferred again to keep this slice narrow.
- **Risk for M13.2**: if a project has been edited and the client
  somehow loads stale SSR seed bytes ahead of the live state, the
  Yjs CRDT merge should still resolve to the sidecar's view, but
  there is a brief flash window of the seed body. Gating the seed
  injection on a "cold/fresh" server flag avoids this entirely;
  iter 238 must be careful that the flag is correct.
