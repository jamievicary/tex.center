## Goal for this iteration

Iter 336 raised the outer Python wall to 1200 s and added a
`TimeoutExpired` handler so the iter-335-style outer-wall fire now
flushes Playwright stdout. The iter-336 gold pass exercised that —
but the more interesting signal it surfaced was: GT-9 hit its own
8-min per-test wall stuck in the post-reopen sentinel poll loop with
zero per-iteration diagnostic output, despite the iter-335 per-phase
markers wrapping the spec.

The transcript ends at
`[verifyLiveGt9StoppedPreservesEdits] elapsedMs=19774
phase=7-sentinel:poll` and the test wall fires 460 s later with
"Target page, context or browser has been closed" on
`waitForTimeout(150)` inside the loop. Phase `7-sentinel:done` never
logs. That's structurally inconsistent with the loop's
`while (Date.now() < sentinelDeadline)` exit on a 90 s budget — the
only explanation is that one loop iteration blocked for hundreds of
seconds. The body has two awaits:
`locator(".cm-content").textContent().catch(() => "")` and
`waitForTimeout(150)`. The former has no per-call timeout; per
Playwright docs locator methods retry until `actionTimeout` fires,
and `actionTimeout` is unset in `tests_gold/playwright.config.ts`,
so the default is 0 (no limit). On a cold-restart, the editor page
may have arrived (`waitForURL` succeeded at phase 6-reopen) before
client-side hydration mounted `.cm-content`, leaving textContent
parked indefinitely on a missing element.

This iteration fixes the diagnostic blackout in the poll loop:

1. Add `timeout: 1000` to the `textContent({...})` call so a
   missing `.cm-content` returns "" within 1 s and the loop
   advances. Budget arithmetic (90 s ÷ ~1.2 s/iter ≈ 75 iterations)
   sits well within the test wall.
2. Add a `7-sentinel:probe` `phase()` log on iter 1 and every 20th
   subsequent iteration (when sentinel still not found), recording
   `iter`, `cmLen`, and the first 80 chars of `cmContent`. A
   timeout-fired run now tells us: did the editor render at all? Is
   the cm-content the canonical hello-world seed (cold-source path
   broken — fell back to `MAIN_DOC_HELLO_WORLD`)? Is it persisted
   content minus the sentinel (sentinel-specific persistence bug)?
3. Augment `7-sentinel:done` with the same `cmLen` field so we
   stop having to read the success-path diagnostic line.

No production code change. The existing `pdf-segment` proof-of-
persistence still holds; this just unblocks investigation if the
spec is still RED on the next gold pass.

Out of scope: I noticed a separate fly-log signal worth recording.
On every fresh-project cold-start the `compiler warmup` (iter 331
hook) spawns the supertex daemon before `runCompile.writeMain`
materialises `main.tex` on disk; the daemon errors out with
`supertex: /tmp/.../main.tex: no such file` and the warmup `.catch`
swallows it. The first `compile()` then detects the dead child,
respawns, and proceeds normally — but the warmup-overlap design
gains nothing on fresh-project bookings because the spawn is
guaranteed to fail. On stopped→start of a Machine with a prior
main.tex on disk this doesn't bite. Filing as a separate slice
under M20.3(a)3 in `.autodev/FUTURE_IDEAS.md` rather than bundling
it with the diagnostic.
