## Goal for this iteration

`N=326` ordinary iteration (no `%10` obligation). PLAN priority 1 is
M20.3 — Tigris deploy wiring + cold-cycle gold spec. M20.2(d) landed
iter 325: `S3BlobStore` + `BLOB_STORE=s3` branch of
`defaultBlobStoreFromEnv` requiring five `BLOB_STORE_S3_*` env vars.

The natural next step is to provision Tigris on Fly and wire the
secrets onto both `tex-center` and `tex-center-sidecar`. There is
one piece of plumbing that should land *first*, recorded explicitly
as a follow-up in the iter 325 notes ("AWS_* env-var compat
(deferred)"): the `flyctl storage create` flow auto-injects the
Tigris credentials into the target app under the standard AWS-SDK
env-var names (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`,
`AWS_REGION`, `AWS_ENDPOINT_URL_S3`, `BUCKET_NAME`). If
`defaultBlobStoreFromEnv` accepts those names as a per-field fallback
under `BLOB_STORE=s3`, the provisioning iteration that follows
becomes a one-liner per app — `flyctl storage create -a <app>` plus
a single `flyctl secrets set BLOB_STORE=s3 -a <app>` to flip the
mode. Without this, the provisioning step has to mirror five
secrets per app into our own naming scheme.

This iteration lands the AWS_* fallback as a pure-code, fully
unit-testable slice. Tigris provisioning itself is a follow-up
iteration: it touches live infrastructure and deserves its own
clean window so the bucket-create output / secrets-push diff are
easy to reason about (and roll back if needed).

## Approach

`packages/blobs/src/envSelect.ts` per-field resolution:

- For each of the five required fields, prefer the explicit
  `BLOB_STORE_S3_*` name; fall back to its AWS-SDK equivalent if
  unset. Mapping:
  - `BLOB_STORE_S3_ENDPOINT`           ← `AWS_ENDPOINT_URL_S3`
  - `BLOB_STORE_S3_REGION`             ← `AWS_REGION`
  - `BLOB_STORE_S3_BUCKET`             ← `BUCKET_NAME`
  - `BLOB_STORE_S3_ACCESS_KEY_ID`      ← `AWS_ACCESS_KEY_ID`
  - `BLOB_STORE_S3_SECRET_ACCESS_KEY`  ← `AWS_SECRET_ACCESS_KEY`
- Missing-field errors name both prefixes
  (`"BLOB_STORE=s3 requires BLOB_STORE_S3_ENDPOINT or
  AWS_ENDPOINT_URL_S3"`), so a misconfigured deploy sees both
  candidate names in the failure.
- `BLOB_STORE=s3` remains the explicit cutover signal — AWS_* on
  their own do not trigger s3 mode. That keeps the cutover
  deliberate; a future iteration that wants implicit-cutover-on-
  detected-credentials can flip it locally without affecting the
  name-resolution shape.

Test surface (`packages/blobs/test/s3.test.mjs` envSelect block):

- All AWS_*, no BLOB_STORE_S3_* → wires up an `S3BlobStore`
  (the Fly-auto-injected shape).
- Mixed (some of each) → wires up.
- BLOB_STORE_S3_* takes precedence when both are set (asserted via
  a test that points the two prefixes at different bucket names and
  checks which one wins by exercising the round-trip against the
  stub).
- Each field missing from *both* prefixes is named in the error
  with both prefixes in the message.

## Notes for future iterations

- **Tigris provisioning (next slice).** Per-app:
  `flyctl storage create -a tex-center -n texcenter-blobs -y` then
  `flyctl secrets set BLOB_STORE=s3 -a tex-center` (the AWS_*
  secrets are auto-injected by `storage create`). Repeat the same
  two commands with `-a tex-center-sidecar` — a second
  `storage create` against the same bucket name mints an
  independent set of access credentials scoped to the new app.
  Both apps will now read/write the same Tigris bucket through
  `defaultBlobStoreFromEnv()` with no further code change.
- **Cold-cycle gold spec (M20.3).** With Tigris wired, write a
  spec that:
  1. Creates a project, edits, lets the sidecar settle (persist).
  2. Stops the Machine via Fly Machines API (or waits past
     SIDECAR_STOP_MS via env override).
  3. Re-opens the editor; asserts the original content reappears
     in `.cm-content` (via the SSR seed chain or the Yjs sync
     after the cold-boot) and that further keystrokes round-trip.
- **`flyctl storage create` output handling.** The command prints
  the access key + secret on stdout once and never again — capture
  to a one-shot file in `creds/` if a manual re-deploy ever needs
  to reproduce them. For automated re-bind, the simplest path is
  to re-run `storage create -a <app>` which mints a new key.
