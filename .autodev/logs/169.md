  spec project (e.g. `7128ec25-…` at 14:35:38Z, `dd0480a3-…` at

  worked** — the ECONNREFUSED-after-`started` race is closed.
- sidecar: every project's Machine logged `Server listening at
  http://[::]:3001` followed by exactly one Fastify `incoming
  request` for `/ws/project/<id>`. And then **nothing**: no
  `client doc-update`, no `compile start`, no `compile ok` — the
  precise log lines iter 163 added to apps/sidecar/src/server.ts to
  distinguish the next four failure branches.

Cross-checked `flyctl image show -a tex-center-sidecar` and
`flyctl ssh console -a tex-center -C 'printenv SIDECAR_IMAGE'`:
every per-project Machine and the `SIDECAR_IMAGE` secret are pinned
to `sha256:5513f7f38b…` — the digest iter 153 hand-pinned. Sidecar
CD has been pushing new images on iters 152, 156, 163, 167 but the
control-plane resolver creates every new per-project Machine from
the stale pinned digest, so **none** of iter-154-onwards' sidecar
code (including the diagnostic logs we were relying on to debug
this very class of failure) has been in prod.

Latest sidecar digest:
`sha256:56ee6e143a7e18e3c55d0a08fd75acc4030eb7fe435c8efedee75a0e3a36be00`
(iter-167 deploy, registry push line in run 25740652789 logs).

## Approach

Two fixes, both required:

1. **Operational:** update the `SIDECAR_IMAGE` Fly secret on
   `tex-center` to the iter-167 digest *now* so the next M8.pw.4 run
   (incl. the harness's own at the end of this iteration) actually
   exercises iter-163's observability.
2. **Code:** wire sidecar CD to auto-sync the pin going forward, so
   the same drift can't recur next time someone pushes a sidecar
   fix and assumes "green CD = code in prod".
