## Goal for this iteration

Round out the M5 leaf module so M5.1 (OAuth callback) doesn't
have to also invent its primitives. Add PKCE (RFC 7636, S256) to
`packages/auth`: a fresh `{verifier, challenge}` generator and a
deterministic `computeChallenge` for the server-side compare /
tests. Pure logic, no I/O, mirrors the shape M5.0 set.

PLAN listed M3.5 (not actionable in-repo), a multi-file project
slice (multi-iteration, frontend-heavy), and M5.1 (needs apps/web
server tier — a real architecture decision). PKCE is the single
missing pure-logic primitive between M5.0 and M5.1; landing it in
isolation keeps M5.1 focused on the server-tier choice + JWKS +
cookie wiring rather than RFC plumbing.

## Approach

- New `packages/auth/src/pkce.ts`:
  - `generatePkce()` returns `base64url(randomBytes(32))` (43
    chars, 256 bits) plus `base64url(SHA256(verifier))`. The
    base64url alphabet `[A-Za-z0-9_-]` is a strict subset of
    RFC 7636 §4.1's verifier alphabet, so the generated string is
    always spec-valid by construction.
  - `computeChallenge(verifier)` is the deterministic S256
    transform — useful both for callback-side compares and tests.
    Validates input against the spec alphabet; throws on invalid
    so callers can't quietly hash garbage.
  - `isValidVerifier(s)` exposes the alphabet+length check (43–
    128, RFC §4.1) for callers that want a guard without throwing.
  - S256 only. "plain" is banned by spec for confidential clients
    and we have no reason to ever emit it.
- Extracted shared `src/b64u.ts` from `session.ts` (one
  encode/decode pair) since two callers now need it. No behaviour
  change to `session.ts`; the helpers were already private and
  byte-identical to the new shared ones.
- Test suite `packages/auth/test/pkce.test.mjs`:
  - Length and alphabet of generated verifier+challenge.
  - Two calls produce distinct pairs (entropy smoke test).
  - `computeChallenge(verifier) === pair.challenge` (cross-check
    the bundled hash against an independent `crypto.createHash`).
  - **RFC 7636 Appendix B vector** (`dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk`
    → `E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM`) — the
    canonical interop-checked pair, so a future b64u typo or
    SHA-256 mis-encoding fails loudly.
  - `isValidVerifier` boundaries: 42 (reject), 43 (accept), 128
    (accept), 129 (reject); bad chars; non-string inputs; the
    `+` from raw base64 (rejected — it's not in the RFC alphabet
    even though it's a base64 char).
  - `computeChallenge` rejects under-length and bad-alphabet
    input.
- Wired `test_auth_pkce` into `tests_normal/cases/test_node_suites.py`.

## Notes for future iterations

- M5.1 now has all the pure primitives it needs in
  `@tex-center/auth`: allowlist, signed session tokens, PKCE.
  Remaining work for that iteration: the apps/web server-tier
  choice (adapter-node swap vs. split Fastify control-plane —
  see iter-32 notes), the actual `/auth/google/start` +
  `/auth/google/callback` routes, JWKS-based ID-token verify, and
  the cookie wiring. No more pure-logic leaves before that.
- The PKCE verifier is intentionally sized at the 43-char floor
  (32 random bytes → base64url) rather than the 128-char ceiling.
  Keeps the redirect URL short while still hitting 256 bits of
  entropy; if a future audit wants more, change the `randomBytes`
  argument and nothing else needs touching.
- An OAuth `state` parameter is the next obvious primitive but is
  trivially expressible as `signSessionToken({sid: stateUuid, exp})`
  with a short TTL — no new module needed unless we want to bind
  the verifier into the state payload itself, which is an M5.1
  call.
