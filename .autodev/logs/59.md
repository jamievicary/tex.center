## Goal for this iteration

PLAN's "next ordinary iteration" (M6.3.1) is out-of-tree (live Fly
+ Cloudflare). Picking the queued in-tree slice: multi-file
persistence. After iter 58, every project file hydrates into its
own `Y.Text`, but only `main.tex` is persisted back ‚Äî non-main
files are gated read-only in `editor/+page.svelte` to prevent the
silent-edit-loss footgun. This iteration lifts that gate by
making `maybePersist` track per-file last-persisted source and
PUT each changed file to its own blob key on every compile.

## Approach

- `persistence.ts`: replace the singleton `persistedSource: string
  | null` + `canPersist` with `persistedByName: Map<string,string>`
  and `knownFiles: Set<string>`. After hydration, `knownFiles` is
  the union of `MAIN_DOC_NAME` and every listed key. For each
  listed entry whose bytes loaded (including empty bytes),
  `persistedByName[name]` is set to its decoded text. Files that
  weren't listed (only `main.tex` if the blob is brand-new) keep
  no entry, so the first compile still establishes the blob ‚Äî
  preserving today's "first-compile creates main.tex" semantics.
- `maybePersist()` drops its `source: string` arg. Iterates
  `knownFiles`; for each, reads `doc.getText(name).toString()`,
  PUTs to `projects/<id>/files/<name>` if it differs from the
  recorded entry (or no entry exists), and updates the map on
  success. Individual PUT failures are logged and don't abort the
  rest ‚Äî one transient outage on `refs.bib` shouldn't lose a
  `main.tex` edit. The flaky-hydration data-loss guard (Run 1c in
  `serverBlobs.test.mjs`) remains intact: when hydration throws,
  `canPersist` stays false and `maybePersist` is a no-op for
  every file.
- `server.ts`: drop the `source` arg at the call site. Replace the
  per-client `project.text.observe(onTextChange)` with
  `project.doc.on("update", onTextChange)` so edits to any file's
  Y.Text trigger compile-and-persist, not only `main.tex`.
- `+page.svelte`: drop `editorReadOnly`. Every file in the tree
  is now editable.
- `serverBlobs.test.mjs` Run 1: also drive an edit to `refs.bib`
  through the client doc and assert the sibling blob is updated.
  Run 2 (cold-start hydration) gains an assertion that the
  rehydrated `refs.bib` reflects the edit.
