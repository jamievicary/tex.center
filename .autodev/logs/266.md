## Goal for this iteration

M13.2(b).4 — stopped-state cold-editable pin. Add a sibling spec
to `verifyLiveGt6LiveEditableState.spec.ts` that drives the
per-project Machine into `stopped` (via Fly's `POST
/machines/{id}/stop`) rather than `suspended`, then asserts the
same 1000 ms `.cm-content`-ready / 1000 ms keystroke-ack budget
on a dashboard click. Per `PLAN.md` §M13.2(b).4 the slice is
expected to land RED — it pins the path the user reported as
"20 s+" cold-load (`260_answer.md`) so M13.2(b).5 has a concrete
target.

Iter 265 noted the existing `.3` spec may itself be flaking
between green (260: 857 ms) and red (265: 5884 ms); both data
points are observations of the suspended-resume path. Capturing
the stopped-resume path as a separate pin lets the diagnosis
distinguish "/suspend-resume regressed" from "/stop-resume was
always slow" — the two paths share Fly Machine cold-start cost
but the suspend path keeps memory mapped, so they should bottom
out at different floors.

## Approach

1. New file `tests_gold/playwright/verifyLiveGt6LiveEditableStateStopped.spec.ts`.
   Copy the structure of `verifyLiveGt6LiveEditableState.spec.ts`
   but:
   - Replace `POST /machines/{id}/suspend` with `POST
     /machines/{id}/stop`.
   - Poll until `state === "stopped"` rather than `"suspended"`.
   - Bump the settle deadline to ~120 s — Fly `stop` is normally
     fast but can take longer than `suspend` because the runtime
     is fully torn down.
   - Tag every diagnostic line with the new spec name so the gold
     transcript distinguishes the two paths.
2. Keep the 1000 ms budgets (M13.2(b) target) so the spec is
   honest about the goal; RED is the expected initial state, per
   `PLAN.md`.
3. Pure additive — no changes to product code, no changes to the
   existing `.3` spec.
4. Run `bash tests_normal/run_tests.sh`.

## Notes for future iterations

- **M13.2(b).5 selection.** PLAN names two candidate fixes:
  - **R1 — widen SSR seed for non-fresh projects.** Fetch persisted
    source from a shared blob store so `.cm-content` paints from
    SSR HTML during the Machine cold-start. Prerequisite: shared
    `BLOB_STORE` binding (Cloudflare R2 or D1, not yet wired). This
    is the more general fix and would also flip M13.2(b).3 from its
    current ~6 s ceiling to within 1000 ms.
  - **R2 — eliminate `stopped` as a reachable per-project state.**
    Either (a) keep one warm Machine (shared-pool M7.0.2), or
    (b) convert the sidecar idle-handler `process.exit(0)`
    fallback to a hard-error retry so Machines only ever land in
    `suspended`. R2(b) is the lower-risk, smaller change; M13.2(b).4
    pin then narrows to the case "suspended is more reliable than
    we measured" rather than depending on R1's blob-store work.
  Next iteration should probably do R2(b) as the cheapest narrowing
  before committing to R1.
- **Suspended-resume variability.** Iter 260 saw cmContentReadyMs
  = 857 (GREEN); iter 265 saw 5884 (RED). Single observations on
  either side; the path is variable enough that any sub-1000 ms
  observation should be treated as luck rather than evidence the
  budget is achievable without R1.
- **Resource hygiene.** The new spec uses the same
  `cleanupLiveProjectMachine` finalizer as `.3`, so per-project
  Machines created by this case are reaped on test exit. No new
  long-lived helpers introduced.
