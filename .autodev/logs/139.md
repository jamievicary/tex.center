## Goal for this iteration

M7.2.1 â€” close the explicit follow-up from iter 138: "401 vs 403"
on the WS upgrade gate. The boolean authoriser shape lumped
"no session" and "wrong owner" both to 401, which is misleading
for an authenticated caller (they would be invited to re-auth
for a project they already can't access). Widen the authoriser
to a discriminated union and let the proxy map deny reasons to
status codes.

## Approach

- `UpgradeAuthDecision = { kind: "allow" } | { kind: "deny-anon" }
  | { kind: "deny-acl" }`. Sentinel constants `ALLOW`, `DENY_ANON`,
  `DENY_ACL` exported from `wsAuth.ts` so call sites don't sprinkle
  object literals.
- `makeSessionAuthoriser`: session present â†’ `ALLOW`, else
  `DENY_ANON` (every miss is "no session by the proxy's reckoning"
  â€” the sub-reason whether the token was missing/tampered/expired
  doesn't matter at this layer).
- `makeProjectAccessAuthoriser`:
  - no session (incl. `resolveSessionHook` outer throw) â†’ `DENY_ANON`
  - authenticated + owner-lookup throws â†’ `DENY_ACL`. Caller is
    already authed; downgrading to 401 would invite a pointless
    re-auth. Treat the outage as an ACL deny (defence-in-depth: the
    proxy still doesn't dial upstream).
  - authenticated + project missing â†’ `DENY_ACL`. We deliberately
    hide existence; if we returned 404 here the proxy would become
    a project-id existence oracle.
  - authenticated + owner â‰  session user â†’ `DENY_ACL`.
  - authenticated + owner === session user â†’ `ALLOW`.
- `wsProxy.ts`: route `allow` â†’ proceed, `deny-acl` â†’ 403 + new
  `{ kind: "forbidden", projectId }` event, `deny-anon` â†’ 401 +
  existing `unauthorised` event. Authoriser throws stay mapped
  to 401 (preserves the "DB-momentarily-down must not admit"
  guarantee even on synchronous throws before we can read the
  decision shape).
