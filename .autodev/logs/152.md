## Goal for this iteration

PLAN designated this iteration as the live cutover (after iter 151's
sidecar-CD fix unblocked the deploy). On orient I checked the iter-151
CD run and found it had **failed**, but at a new step — the submodule
clone now works (iter 151's PAT fix landed), and the build progressed
to the runtime stage for the first time, where it crashed with
`/bin/sh: 1: /opt/engine/binary: Permission denied` (exit 126) during
the `--ini` fmt dump. The cutover cannot happen because there is still
no fresh sidecar image. This iteration becomes "fix the next CD bug",
cutover slides to iter 153.

## Root cause

`vendor/engine/x86_64-linux/lualatex-incremental` is the prebuilt
patched-luatex ELF vendored in iter 75. It is stored in git as mode
`100644` (`git ls-files -s` confirmed). The runtime stage `COPY`s it
to `/opt/engine/binary` and immediately tries to run it. With mode
644, exec is refused → 126.

Two reasons this stayed invisible since iter 75:

- The Dockerfile build never reached the runtime stage on CI prior
  to iter 151 — `actions/checkout@v5` died at the private-submodule
  clone before `flyctl deploy` ever ran. So this is the *first* CD
  attempt to fail downstream of the submodule, which is exactly when
  this latent bug becomes reachable.
- Local docker builds on the maintainer's WSL filesystem always
  report `vendor/engine/x86_64-linux/lualatex-incremental` as
  `-rwxrwxrwx` regardless of the git index mode, so any local
  `docker build` of `apps/sidecar/Dockerfile` *did* succeed. (The
  current production sidecar image must therefore have been built
  via a local push or an earlier path that bypassed CD; the iter-149
  bind-host fix is consequently *not* in the live runtime, which is
  why the live WS still 502s today.)
