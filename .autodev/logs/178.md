## Goal for this iteration

Land the M9.editor-ux compile-coalescer slice. The
pre-coalescer sidecar gated only on a pending debounce timer
(`scheduleCompile` at `apps/sidecar/src/server.ts:255`), so a
doc-update during an in-flight compile reached the underlying
compiler and tripped its "another compile already in flight"
guard — the root cause of GT-B/C/D failing live. Bundles the
fake-compiler harness called out as missing in `.autodev/PLAN.md`
§3 with the state-machine change itself, since the harness has
no consumer apart from this test.

## Approach

1. Add an optional `shipoutPage` to `CompileSuccess` (server-side
   only; no protocol wire change). `SupertexDaemonCompiler`
   surfaces `events.maxShipout` when ≥0; other compilers leave
   it absent.
2. Add `compileInFlight`, `pendingCompile`, `debounceTimer`,
   `highestEmittedShipoutPage` to `ProjectState`. Replace
   `scheduleCompile` with three small functions:
   - `kickCompile(p)` — set `pendingCompile=true`; if no
     debounce timer running, start one.
   - `maybeFireCompile(p)` — edge-triggered: when called and
     not in flight and pending, clear pending, set in-flight,
     `runCompile().finally(...)`. In the finally clear
     in-flight; if pending again, re-arm debounce.
   - `maybeKickForView(p)` — idle-only kick when the highest
     viewer page exceeds `highestEmittedShipoutPage`. Wired into
     the `view` control message branch.
3. `runCompile` bumps `highestEmittedShipoutPage` after emitting
   segments, using the new `shipoutPage` field on `CompileSuccess`.
4. Hydrate-path and doc-update path both call `kickCompile`
   instead of `scheduleCompile`. Cleanup paths rename
   `compileTimer` → `debounceTimer`.
5. Unit test via a `ManualCompiler` whose `compile()` returns a
   promise the test explicitly resolves (success-with-shipoutPage
   or synthetic error). Four cases: burst-collapse, error-clears-
   in-flight, view-fire-through (positive + negative), quiescent.
