## Goal for this iteration

M21.3b — sidecar post-round observability. The pre-round
`daemon-stdin` debug log (iter 282) records each `recompile,…`
write with `{ round, target, sourceLen }`, but the emit decision
itself (whether the round shipped any chunks, hit an error, or
no-op'd) is invisible to anyone reading the log. M21.3a/b/c are
queued behind M20.2 in the PLAN priority order, but M21.3b is the
pre-requisite for diagnosing the user-reported "edit on hidden
page still ships" pattern: without the post-round emission, the
sidecar log cannot answer "did the daemon return maxShipout=-1
(genuine no-op) or did it ship a segment we then suppressed
elsewhere?".

This is a small, self-contained observability slice — not picked
over M20.2 because M20.2 is taking on infra weight (S3-compatible
blob store + creds + deploy) that won't fit one iteration. M21.3b
fits cleanly here and unblocks the diagnostic path the PLAN's
M21.3c step depends on. (Picking M20.2 instead would require
either scoping it down to pure scaffolding code with no live
exercise, or splitting it across 3+ iterations starting next time.
This iteration takes M21.3b; M20.2 starts with the next ordinary
non-maintenance iteration.)

Bonus: a single shape — `daemon-round-done` with
`{ round, maxShipout, errorReason, violation? }` — covers the
happy, error, no-op, and protocol-violation paths uniformly. The
existing `collectRound` already returns exactly these fields, so
the wire is a literal pass-through after the `await` site.

## Approach

- `apps/sidecar/src/compiler/supertexDaemon.ts:compile()`: after
  `collectRound` returns and before any short-circuits, emit one
  `daemon-round-done` log line with `{ projectId, round: roundSeq,
  maxShipout, errorReason }` plus `violation` when set. Emission
  is unconditional (gated only on `this.log` being wired) — the
  post-round event is exactly when the emit decision was made, so
  the diagnostic is most useful here even when the result is an
  error / no-op / violation.

    `maxShipout=-1`.
  - Violation (case 5) — `violation` set, plus `maxShipout` and
    `errorReason` shown.
  - No-op (case 12) — `maxShipout=-1, errorReason=null`.
