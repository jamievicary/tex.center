## Goal for this iteration

PLAN's "next ordinary iteration" (M6.3.1) is out-of-tree. Picking
the next file-tree CRUD slice from `FUTURE_IDEAS.md`: `delete-file`.
After iter 61 the user can create files but not remove them.
Rename is more involved (Y.Text has no rename primitive);
deferring.

## Approach

- Protocol: extend `ControlMessage` with
  `{ type: "delete-file"; name: string }`.
- Persistence: `deleteFile(name)` — rejects `MAIN_DOC_NAME` and any
  name not in `knownFiles`. On accept: deletes the blob (when
  `canPersist`), clears the file's `Y.Text` via
  `t.delete(0, t.length)` (Y.Doc has no remove-type primitive, so
  the type still exists but its contents are empty), removes from
  `knownFiles` so `maybePersist` no longer touches the key, and
  drops the `persistedByName` entry. The no-blob-store variant
  also tracks deletion in-memory so the session view stays
  consistent.
- Server: `delete-file` control handler calls `deleteFile` and
  broadcasts a refreshed `file-list` on success; logs the
  rejection reason on failure (defence-in-depth — the client UI
  hides the affordance for main.tex).
- `WsClient.deleteFile(name)` sends the control message.
- `FileTree`: small "×" button per row (suppressed for
  `MAIN_DOC_NAME` and when `onDeleteFile` is unwired). Row layout
  becomes a flex row so the delete sits next to the select
  button.
- Editor page: on delete, if the currently-selected file is the
  one being removed, switch selection back to `MAIN_DOC_NAME`
  before sending the message (the editor's `$effect` then rebinds
  to main's `Y.Text`; the soon-to-be-cleared text is no longer
  attached).

## Notes for future iterations

- Rename is the last file-tree CRUD verb. Y.Text has no rename;
  the simplest path is a transactional copy: create the new file
  (PUT empty blob + add `knownFiles` entry), copy
  `oldText.toString()` into the new `Y.Text`, then delete the old
  in the same `doc.transact` block so observers see one update.
  Care with `persistedByName`: the new entry should start unset
  so the next `maybePersist` writes the actual contents.
- The deleted Y.Text type still exists on the doc (Yjs limitation)
  but is empty and no longer in `knownFiles`. If a client still
  has it selected at the moment of delete, the editor page swaps
  selection back to `main.tex` before sending the request so the
  user never sees a phantom empty buffer with the deleted file's
  name in the file-tree highlight.
