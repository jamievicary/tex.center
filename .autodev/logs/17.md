## Goal for this iteration

Land M4.1 — Drizzle integration. Add `drizzle-orm` to
`packages/db`, declare a typed `pgTable` for each entity, and
add a cross-check test that asserts the Drizzle schema agrees
with the existing `TableSpec` view (which drives the SQL
migration test). This is the next slice in M4: the spec stays
the source of truth for SQL DDL, the Drizzle tables are the
source of truth for query typing, and the test makes drift a
build failure rather than a runtime surprise.

The actual `drizzle(client, { schema })` instantiation needs a
postgres driver, which lands with M4.2 alongside docker-compose
bring-up; this iteration is just the schema seam.

## Approach

1. Add `drizzle-orm@^0.45.2` to `packages/db/package.json` and
   run `pnpm install` (lockfile churn unavoidable, package is
   small).
2. New file `packages/db/src/drizzle.ts`: explicit `pgTable`
   declarations for `users`, `sessions`, `projects`,
   `projectFiles`, `machineAssignments`. Mirror the migration's
   FKs (cascade), indexes (`sessions_user_id_idx`,
   `projects_owner_id_idx`), and the `(project_id, path)` unique
   constraint on `project_files`. `state` on
   `machine_assignments` keeps its TS literal-union typing via
   `.$type<...>()`. Defaults (`now()`) match the SQL.
3. Re-export from `packages/db/src/index.ts`.
4. Cross-check test `packages/db/test/drizzle.test.mjs`: walks
   `allTables`, looks up the matching Drizzle table via
   `getTableConfig`, asserts each spec column has the same SQL
   type (`getSQLType()`), `notNull`, `primary`, and FK target.
5. Register the new suite in
   `tests_normal/cases/test_node_suites.py` as
   `test_db_drizzle`.
6. PLAN.md: mark M4.1 done, refresh the current-focus block.

Explicit hand-written `pgTable` (vs spec-driven generation) is
the right call for the typed-query goal — Drizzle's column types
are the whole point, and a generator would have to reproduce
them as a giant conditional. The cross-check test handles the
drift risk.

## Notes for future iterations

- `drizzle-orm` ships its own helpful column kinds we didn't
  need yet (`boolean`, `integer`, `jsonb`, custom `bytea` via
  `customType`). The spec's `ColumnType` union still lists them
  for forward compatibility, but no migration uses them today.
  When one does, extend `SQL_TYPE_BY_SPEC_TYPE` in the test and
  add the matching builder in `drizzle.ts`.
- M4.2 will pick a postgres driver. `postgres-js` is the modern
  default for `drizzle-orm/postgres-js`; `pg` (`node-postgres`)
  is the alternative. Either works behind the same `schema`
  re-export.
- The `(project_id, path)` constraint is a `UNIQUE` constraint
  in the SQL but is expressed as a `uniqueIndex` in
  `drizzle.ts` — Drizzle treats a unique index as the FK
  partner, and Postgres backs the unique constraint with an
  identical index, so this is equivalent at the DB level. If a
  later test needs to assert the constraint name verbatim it
  may want to switch to `unique('project_files_project_id_path_uq').on(...)`.
