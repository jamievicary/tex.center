## Goal for this iteration

M22.4b — wire-shipoutPage batch. The PLAN priority queue points to
this as the next coherent slice after M23 closed in iter 316. Spec
was already fully nailed down in `306_answer.md` items 7–8 and the
PLAN's M22.4b bullet: extend the binary `pdf-segment` header from
13 → 17 bytes incl. tag with a new `uint32 shipoutPage` (0 =
unknown sentinel), stamp it from `events.maxShipout` in the
daemon, surface it on `WsDebugEvent.pdf-segment`, change the toast
text format to `[${n}.out] ${bytes} bytes`, and extend
`compileCycleTracker` to prefix segment toasts with `${elapsedMs}s
— ` when a cycle is in progress.

## Approach

Wire-format change is atomic across web ↔ sidecar because both
import `@tex-center/protocol`. Single coherent commit that:

- Adds the field to both the protocol `PdfSegment` and the
  compiler-internal `PdfSegment`. The compiler-internal type is a
  separate interface in `apps/sidecar/src/compiler/types.ts` (the
  sidecar's seam between web/wire concerns and compiler output);
  structural-typed flow into `encodePdfSegment(seg)` at
  `server.ts:552` and the segment-replay path at `server.ts:658`
  carries the field across the seam.
- Widens encoder (`Uint8Array(13) → Uint8Array(17)`) and decoder
  (`body.length < 12 → < 16`, plus a fourth `getUint32(12)`).
  Decoder treats `0` as the absent sentinel and omits the field;
  every positive integer becomes a real `shipoutPage`.
- `supertexDaemon.compile()` stamps `segment.shipoutPage =
  events.maxShipout` right after `assembleSegment` and before
  returning. The result-level `shipoutPage` field stays — the
  server uses it at `server.ts:557–562` to update
  `coalescer.highestEmittedShipoutPage`, and removing it would
  force the coalescer to read through `result.segments[0]`. Both
  encode the same info; minimal change to keep both.
- `WsClient.handleMessage` for `case "pdf-segment"` emits the
  debug event with `shipoutPage` only when present (so consumers
  can distinguish "stamped 0" from "stamped N", though only the
  daemon stamps today and the FixtureCompiler doesn't).
- `debugEventToToast` chooses between two text formats; aggregate
  key unchanged (`debug:pdf-segment`) so coalescing within a
  burst still collapses to one `×N` badge.
- `compileCycleTracker` factored: `compile-status` branch
  unchanged from M22.4a; new `pdf-segment` branch prefixes when
  `cycleStart !== null` and explicitly does NOT clear the cycle
  (a segment is mid-cycle, idle/error is end-of-cycle). Other
  event kinds pass through unmodified.

Locks (per PLAN bullet):
- `packages/protocol/test/codec.test.mjs` — added header-width
  assertion (`frame.length === 17 + bytes.length`), the sentinel
  read (bytes 13..16), a round-trip with stamped page, a
  truncation-rejection case for the old 13-byte shape, and
  shipoutPage-validation throws.
- `apps/sidecar/test/supertexDaemonCompiler.test.mjs` happy-path
  case 1 now asserts `seg.shipoutPage === 3` (matching
  `FAKE_TOTAL=3` chunks).
- `apps/web/test/wsClientDebugEvents.test.mjs` got a new case 1b
  (stamped frame → event carries `shipoutPage`), an extra entry
  in the case-7 matrix, and explicit text-format assertions for
  the unstamped and stamped formats.
- `apps/web/test/compileCycleTracker.test.mjs` cases 5 and 6
  rewritten for the new format; new case 5b pins the
  "segment-without-running" pass-through.

## Notes for future iterations

- The result-level `CompileSuccess.shipoutPage` is now redundant
  with `segments[0].shipoutPage` for the daemon compiler (both
  carry `events.maxShipout`). The coalescer's
  `highestEmittedShipoutPage` could read either. Today the
  result-level field still drives the coalescer at
  `server.ts:557–562`; a follow-up could remove it and switch to
  `result.segments[0].shipoutPage`, but the saving is minor and
  the dual-write is harmless. Leaving for a maintenance cycle.
- M22.2 (GT-F local Playwright cases for the debug-toast wire
  coverage) is the natural next pickup per PLAN priority queue.
  It exercises the `?debug=1` flow end-to-end, which now has the
  segment-toast format change visible to assert on.
- The 0 sentinel handling is at decode time, so an old client
  talking to a new server would see a header-truncated error
  (body 12-byte vs. required 16-byte) — but the entire fleet is
  one-version because the protocol package is published only
  via the workspace, so a fan-out skew can't happen in practice.
