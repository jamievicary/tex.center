## Goal for this iteration

M18 PDF preview quality: DPR-aware canvas sizing in
`PdfViewer.svelte`. Today the preview canvas is allocated at viewport
pixels of a hard-coded `scale: 1.5`, then CSS-scaled to fit the
wrapper via `width:100%`. On HiDPI screens (typical 2× retina) every
backing pixel is stretched 2:1 on display → the visible pixelation
the user reported (`293_question.md` item 1). Fix: multiply the
render scale by `window.devicePixelRatio` so the backing store
matches the device, and pass CSS-px dimensions (not device-px) to the
fade controller's `setWrapperGeometry` so layout is unchanged.

Slice scope: smallest visible improvement only. Extract a tiny
`pdfRenderScale(baseScale, dpr)` helper and unit-test it. Wire into
`PdfViewer.svelte`. Defer the ResizeObserver + gold visual-snapshot
slices to a follow-up — they depend on this landing first and add
test surface a single iteration shouldn't bundle.

## Approach

- New module `apps/web/src/lib/pdfRenderScale.ts`: pure function
  `pdfRenderScale(baseScale, dpr) → { cssScale, pixelScale }`.
  Tolerates non-finite / non-positive `dpr` (falls back to 1) so a
  freak browser report can't crash the renderer.
- `apps/web/test/pdfRenderScale.test.mjs`: covers DPR=1 / 2 /
  fractional / 0 / NaN / negative.
- `PdfViewer.svelte`: read DPR once per render (cached at top of the
  page loop, *not* recomputed per page — guarantees all pages of a
  single commit share one DPR even if the screen changes mid-render),
  build a `pixelScale` viewport for `page.render`/canvas.width, and a
  separate `cssScale` viewport for the descriptor handed to the
  controller. Add `tests_normal/cases/test_node_suites.py` entry for
  the new test.

## Notes for future iterations

- **Verifying the fix in production.** The next time the user opens
  a project on a HiDPI screen, the canvas backing store will be
  `2 × 1.5 = 3` device pixels per PDF user-unit (vs. 1.5 before).
  Visible sharpness improvement should be immediate; if not, the
  failure is in `page.render({ viewport: pixelViewport })` not
  scaling the underlying glyphs, which would be a pdfjs config bug
  to investigate.
- **DPR vs. divider drag.** A user who drags the preview divider
  wider than the original CSS width will see the canvas up-scaled
  until the next commit. That's M18.2 — needs a ResizeObserver +
  100ms-trailing coalesced re-render. Don't paper over it by always
  rendering at maximum possible width; cost would be wasted GPU on
  the typical case. Wait until the user reports the issue.
- **fade-controller test surface.** `pdfFadeController.test.mjs` is
  untouched: descriptor shape is unchanged (still `{canvas, width,
  height}`), only the semantics of width/height shifted from "device
  px" to "CSS px". The controller's setWrapperGeometry call already
  treats them as CSS px (uses them in `style.width` / `aspect-
  ratio`), so the change is a clarification of existing contract.
- **M21 (max-visible-page).** Still tagged as small in PLAN; could
  be paired with M18.2 in a single iteration if the diff stays
  clean. The two share zero code paths but both touch PdfViewer.
