## Decisions

- **303 not 302 on logout.** 303 forces method-rewrite to GET on
  the redirect; 302 leaves it ambiguous (some clients re-POST).
  The callback route uses 302 because it's already redirecting a
  GET; here we're redirecting a POST, so 303 is the right verb.
- **POST-only, no GET branch.** A GET logout link is a known XSS
  / pre-fetch / image-tag CSRF vector (an `<img
  src="/auth/logout">` on any page would sign the user out). The
  same-origin form gets us CSRF safety for free under
  SameSite=Lax. Future link-driven flows go through a CSRF-
  protected shim.
- **Pure orchestrator for a 30-line route.** Slightly
  ceremonious for the size, but matches the surrounding shape
  (`oauthStart`, `oauthCallback`, `sessionHook` all separate
  policy from wiring) and makes the cookie-attribute assertions
  testable without a SvelteKit harness.
- **`+page.server.ts` not `+layout.server.ts`.** Putting the load
  on the layout would force every route through SSR — root
  layout still has `prerender = true; ssr = false` so the white
  sign-in page stays a static artefact. Editor is the only page
  with a user header for now; when a dashboard lands, factor up.
- **Delete error propagates (no swallow).** Logout failing on a
  DB blip is the right user-visible outcome — they get a 500 and
  retry; the alternative is a clear-cookie that leaves a live
  server-side session, which on the next reauth would re-attach
  to the same DB row anyway, but it's confusing UX. Cheap to
  revisit if it shows up in prod.
