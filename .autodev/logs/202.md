2. Smoking gun in the post-deploy machine

   `ea04b3b1-c0e5-4d70-b501-275cf9acc7bd`) at 07:54:42 UTC:

   ```
   [supertex-daemon stderr] supertex: edit detected at .../main.tex:73
   [supertex-daemon stderr] supertex: WARN no usable rollback
     target for .../main.tex@73 (checkpoint=1 resumed_pid=-1)
   [supertex-daemon event] round-done
   compile ok elapsedMs=5184 segments=0 bytesShipped=0
   ```

   The 5s elapsed window matches the daemon's deadline; the
   round emits `[round-done]` with **no** intermediate
   `[shipout n=…]`, so the sidecar reaps an empty segment list
   and `assembleSegment` short-circuits — no `pdf-segment` frame.
   This is **Path 2** in PLAN M7.4.x (no usable rollback target;
   `eff_n == 0 || resumed_pid <= 0`).

3. Located the WARN site in
   `vendor/supertex/tools/supertex_daemon.c`. **Two** copies exist:

   - **Lines 1230-1238** (handshake-mode `recompile,N` driver).
     Inside the `else` of `if (per == 0 && resumed_pid > 0)`
     at line 1131. Emits the WARN and **falls straight through
     to `[round-done]` at line 1257 with no recovery**. The
     sidecar's `recompile,N` round therefore returns
     `segments=0`, which the sidecar treats as a no-op compile.
   - **Lines 1695-1702** (post-handshake `run_watch_loop_c`
     inotify-driven path). The iter-711 (NEX slice 6) fix sets
     `rc = WATCH_LOOP_RECOMPILE_RC` here; main() catches it at
     line 2197 and `goto recompile;`s to re-fork the engine
     in-process, emitting fresh chunks for the new PDF state.

   The asymmetry is the bug. The post-handshake path recovers
   from `eff_n == 0 || resumed_pid <= 0` by doing a full
   in-process recompile; the handshake path emits a bare
   `[round-done]` and lets the round end. Iter-198 local probes
   never tripped this branch because the test fixtures never
   exhausted the frozen-sibling chain (`resumed_pid=-1` arises
   when the previously-frozen sibling for the selected
   checkpoint has died or been reaped, which only manifests
   after a substantial edit sequence — exactly the live-typing
   case).

4. Live trace also shows the precursor: edits at byte offsets
   ~54-66 all succeed (`[rollback k=0]` → `[shipout n=1]` →
   `[round-done]`, `segments=1`). The first empty round is at
   edit byte 69 (07:54:36) — `[rollback k=0]` then `[round-done]`
   with `segments=0` and no shipout. Edit byte 73 produces the
   explicit WARN. Hypothesis: the chain-exhaustion is gradual —
   the frozen sibling lingers via fork from the last successful
   resume; eventually `process_event`'s `select_for(eff_n=1)` lands
   on a checkpoint whose frozen sibling is gone (`resumed_pid=-1`).

## Approach for the fix (next iteration)

**Upstream PR against `github.com/jamievicary/supertex`**: mirror
the iter-711 in-process recompile recovery into the handshake
`recompile,N` path. Concretely, in the
`supertex_daemon.c:1230-1238` else-branch:

1. Refork the engine in-process (same approach as
   `WATCH_LOOP_RECOMPILE_RC` → `goto recompile;` at line 2207).
2. Discard the existing daemon-dir chunks and re-emit chunks
   from page 1 for the fresh PDF state.
3. Then emit `[round-done]` so the handshake protocol round
   still closes cleanly.

This makes the handshake protocol byte-identical to the watch
loop's recovery: every `recompile,N` round either produces
shipouts or, on chain exhaustion, transparently does a full
recompile and produces the chunks anyway. The sidecar layer
needs no change — the existing `assembleSegment` path consumes
the chunks just as it does today on a normal rollback.

Alternative considered and rejected: detect `segments=0`
round-done in the sidecar and force-restart the daemon. Slower
(daemon respawn ≈ 4-5s vs in-process refork ~hundreds of ms),
duplicates the recovery logic across two layers, and the
upstream fix is the correct architectural home.
