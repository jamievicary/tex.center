## Goal for this iteration

The iter-357 gold run gave us the first reproducible Bug B / Bug B-
adjacent signal in CI: `verifyLiveFullPipelineReused` FAILED with
the reused-fixture project `00000000-0000-4000-8000-000000000001`
showing `compile-cycles=2 zero-segment-cycles=2 pdf-segment-bytes=0`,
i.e. zero pdf-segments after the typed SRC. But the timeline line
`+14.118s in control:compile-status state=error` shows the cycles
ended in `error`, not `idle` ‚ÄĒ the actual signature is **two
back-to-back compile errors** on the reused-project cold-open path,
not the silent zero-segment shape Bug B was originally about.

The PLAN's Bug B decision tree assumes we have sidecar logs to look
up `replay-segments`/`compile ok`/`round-done`. flyctl logs only
retain a few minutes' window; by the time iter-358 starts the
iter-357 gold window has rolled off. We need the same evidence to
travel back to the iteration log in the SAME run, so the next gold
failure pins root cause without a separate observability sortie.

Coherent slice: extend the wire-timeline diagnostic
(`wireTimelineFormat.ts` + `wireFrames.ts`) so a `compile-status
state=error` entry carries the `detail` field (the error message
the sidecar already broadcasts on the wire ‚ÄĒ see `server.ts:587-621`
which puts `result.error` into `detail`). Pure-data change, no
sidecar/web changes, locked by the existing
`tests_normal/cases/wireTimelineFormat.test.mjs` formatter spec
with a new case for the detail-on-error rendering. Net effect on
the next gold pass: the failing timeline line becomes e.g.
`state=error detail=supertex daemon error: <reason>`, which
collapses the Bug B decision tree to a one-liner and lets the
following iteration (or the human, if interesting upstream) act
without log-fetching.

   new cases ‚ÄĒ error with detail rendered (and running line

   trailing `detail=`).

every error (server.ts:587, 621). Nothing on the
runtime-data-flow path is touched.
