## Goal for this iteration

PLAN priority #1 (M20.3 closeout). Iter 340 landed a fix that
arms only `stopStage` at cold boot, on the hypothesis that the
cold-handshake-vs-5 s-suspend race was the GT-9 + GT-6-stopped
RED cause. Iter 341 was the scheduled plan-review and incidentally
recorded a fresh gold pass: **both specs are still RED**, with the
same `flyState=starting → suspended` shape iter 339 first pinned.

Iter 342 reconciles that: confirm the fix is live in prod, frame
the residual failure mode, and add the minimal sidecar
instrumentation the next iteration needs to root-cause it.

## Investigation

**Deploy is live.** `gh run view 25954977375` shows the iter-340
sidecar workflow set `SIDECAR_IMAGE=…@sha256:e9c4a5d…0fa` on
`tex-center` at 06:28:32 Z; the parallel `deploy.yml` run finished
the web rolling restart at 06:29:11 Z. Iter 341's GT-9 ran from
~06:30 Z onwards — the spawned Machine should have used the new
digest. `flyctl image show -a tex-center-sidecar` confirms the
shared-pool app's two release-managed machines (`683437eb1e3378`,
`d895e7ea479958`) now carry that digest; per-project Machines I
sampled below also point at the new digest *if* they were created
post-secret-rotation.

**Server code is correct.** `apps/sidecar/src/server.ts:356` is
`stopStage.arm();` only; `armIdleTimers()` is *only* called from
`noteViewerRemoved` (line 367) on a viewerCount 1→0 transition.
There is no other site that arms `suspendStage`. Cold-boot suspend
arm is genuinely gone.

**Failure shape unchanged.** Iter 341's gold transcript:

```
phase=5-stop:settled       lastState=stopped       (30 s)
phase=6-reopen:click                               (30 s)
phase=7-sentinel:probe iter=1   flyState=starting   wsOpens=2 wsFrames=11 wsCloses=1
phase=7-sentinel:probe iter=21  flyState=suspended  wsOpens=2 wsFrames=11 wsCloses=1
phase=7-sentinel:probe iter=121 flyState=suspended  wsOpens=2 wsFrames=11 wsCloses=2
                                                     ↑ first new-WS close, page-error logged
```

Read it: the test client *does* observe a WS open on the new page
(`wsOpens` 1→2) within ~1 s of the reopen click — meaning the
upstream sidecar accepted the upgrade and sent 101. But the new
WS never delivers any frames (`wsFrames` stays at 11, all from the
phase-1 connection). Between iter=1 and iter=21 (~23 s window),
the Fly state walks `starting → suspended`. There is no clean
`started` observation; the probe sampling could have missed a
brief `started` interval.

**Hypothesis.** The new-WS upgrade reaches the sidecar long enough
to call `noteViewerAdded`, but then the connection closes before
any application frame is delivered to the client. `socket.on("close")`
then drives `noteViewerRemoved` → `armIdleTimers()` → the 5 s
suspend stage fires → Machine state flips to `suspended`. The
client thinks its `OPEN` socket is still alive (proxy-side
buffering) and only sees the close ~140 s later as the proxy
pipeline finally gives up.

This is consistent with everything observed: the iter-340 fix
removed the *startup* suspend arm but did not touch the
disconnect-arm path, and that path is the actual primitive being
exploited by a transient cold-boot WS open-then-close cycle.

## Approach

Add focused diagnostic logs to the sidecar so the next gold pass
records WHEN viewer-add/remove fires, WHEN each idle stage arms
and fires, and the close-event metadata. The logs are info-level
so they appear in `flyctl logs` by default. No behaviour change.

Once iter 343 captures a fresh GT-9 transcript paired with the
sidecar's `viewer-added` / `viewer-removed` / `idle-arm` /
`idle-fire` records, the hypothesis becomes a one-line
verification: did `viewerCount` go 0→1→0 between the WS upgrade
and `Suspend` firing? If yes, the disconnect-arm path is the
remaining bug; the fix is small (defer the suspend-arm on the
*first* 1→0 transition until the viewer has held the connection
for some minimum, or coalesce arm-then-clear cycles that don't
imply real idleness). If no, the bug is elsewhere and the logs
will point at the actual culprit.

## Actions

`apps/sidecar/src/server.ts`:

- `createIdleStage.arm`: log `{stage, ms} idle-arm` on every arm.
- `createIdleStage` timer callback: log `{stage} idle-fire` before
  `persistAllCheckpoints` runs.
- `noteViewerAdded`: log `{viewerCount} viewer-added` after
  clearing timers.
- `noteViewerRemoved`: log `{viewerCount} viewer-removed` after
  decrementing, *before* the conditional arm.
- WS upgrade handler (post `noteViewerAdded`): log
  `{projectId, viewerCount} ws-upgrade-open`.
- WS upgrade `socket.on("close", (code, reasonBuf) => …)`: capture
  the close `code` and `reason`, log
  `{projectId, hadViewer, code, reason} ws-upgrade-close` before
  the per-client cleanup branch.

No production behaviour changes. Sidecar tests
(`serverIdleStop.test.mjs`, `idleSuspend.test.mjs`,
`serverCheckpointWiring.test.mjs`) and the sidecar's typecheck all
stay green.
