## Approach

Reviewed `aaa625a` (supertex commit) — confirmed contract:
chunk_writer tail-scans for `%SUPERTEX-LAST-PAGE` and emits
`[pdf-end]\n` after the matching `[N.out]`, before
`[round-done]`. Reviewed current sidecar segment-assembly path:
`assembleSegment(maxShipout)` concatenates chunks 1..maxShipout
from disk into ONE `PdfSegment` with `offset:0`. So the user-
reported "(c) only target-page chunk arrives today" is **not**
explained by sidecar single-segment-per-round shipping (which
carries all chunks); more likely an FE rendering / cache-
invalidation bug or an upstream supertex shipout-elision bug.
Answer routes (c) to "capture daemon-round-done transcript first,
then route by `maxShipout` value" rather than speculatively
changing segment-assembly.

Concur with iter A / iter B split (small, forward-compatible
sidecar + protocol slice first; FE + targetPage swap +
(c) routing next). PLAN priority queue updated: new priority #1
is this work; M13.2(b).4 demoted to #2; subsequent items
renumbered.
