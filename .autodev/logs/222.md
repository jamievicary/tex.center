## Findings (load-bearing — read carefully)

**The bug does not reproduce locally even under stress.** Tried:

1. Native daemon cold-start (~370 ms) + 80 doc-updates across two
   phases (50 zero-spaced, 30 at 100 ms). Result: 2 compile calls
   total (initial + one queued follow-up), no overlap-error frames.
   Test PASSES.
2. Artificially slow first compile (5 000 ms forced delay via
   `SlowFirstCompileCompiler`) + same 80-update burst. Total compile
   span ~10 s. Result: still 2 compile calls, still no overlap-error
   frames. Test PASSES.

So the coalescer's `inFlight` gate is **not** simply failing under
"slow first compile + concurrent kicks" — that scenario was what
220_answer.md predicted as the necessary trigger, and it does hold.
This rules out three of the iter-221-listed candidates:

- Pure cold-start latency × debounce arithmetic is sufficient on its
  own to overdrive the gate. **Wrong** — it does not.
- The bug is a missing kick-side guard. **Wrong** — `kick()` only
  arms a timer; the gate is in `maybeFire`, and that gate is
  observably effective under the patterns we can construct locally.
- The unit test (`serverCompileCoalescer.test.mjs` case 1) was
  somehow not representative of real-daemon behaviour. **Less
  wrong** — the real-daemon path with the wrapper is now also
  exercised and also holds the gate.

What remains plausible as the production trigger:

- A second `CompileCoalescer` or `ProjectState` for the same project
  id arising from a production-only code path I have not yet
  identified. `getProject(id)` is sync and idempotent, but maybe the
  WS upgrade or the blob-hydration path indirectly creates state I
  haven't traced.
- Something in real-blob-store hydration that fires Y.Doc updates
  *before* the WS handler attaches `onTextChange`. Locally
  hydration is in-memory; on Fly it's R2-backed and may interleave
  with viewer-add differently.
- A subtle interaction between `noteViewerAdded`/`noteViewerRemoved`
  and `coalescer.cancel()` (line 586) — if the last viewer
  disconnects mid-cold-start and another reconnects, the cancelled
  timer scenario may put the state machine in a place the unit
  test never visits.

I did not eliminate any of these in this iteration; the local repro
work showed only that the *naive* "slow compile + rapid kicks" model
is insufficient.

## Notes for future iterations

- The next probe must focus on what's different between local
  buildServer and the deployed sidecar at cold-start. Two concrete
  candidates:
  1. Add structured `coalescer.run` / `coalescer.maybeFire` /
     `.finally` trace logs (gated on an env var, e.g.
     `SIDECAR_TRACE_COALESCER=1`) that emit
     `{event: "fire" | "skip-inFlight" | "skip-pending" | "finally"}`
     plus a sequence number. Deploy briefly, repro GT-7, scrape
     `flyctl logs`. The trace will show whether `inFlight` is ever
     observed false during a slow first compile — settling whether
     this is a logic bug or a state-pollution bug.
  2. Stand up a local sidecar against a real `LocalFsBlobStore`
     populated with a 70-byte main.tex blob, and a WS sequence that
     disconnects mid-cold-start then reconnects (mimicking a Fly
     proxy hiccup). The coalescer's `cancel()` on last-viewer
     disconnect + re-arm on reconnect is the closest thing to an
     untested edge in the current state machine.
- Augmented GT-7 will go RED on the next live gold pass if the
  production bug recurs without an underlying fix. If it goes
  GREEN, that itself is a data point — perhaps a redeploy with the
  iter-220 file-key refactor or some other recent change calmed the
  failure mode. Either outcome is a useful next signal.
- The new sidecar-level gold case is a regression lock on the
  invariant even though it does not currently trigger the bug. A
  future change that lets `compiler.compile` overlap on the
  real-daemon path will fail this test.
