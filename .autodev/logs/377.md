## Goal for this iteration

Fix the intermittent 412 race in `upstreamResolver`'s
stopping→start path that surfaced on iter 376 as the
`verifyLiveGt6LiveEditableStateStopped` gold failure. Fly's
`POST /machines/{id}/stop` is asynchronous: the API reports
`state=stopped` to the polling client *before* the runtime has
finished tearing down, and a subsequent `startMachine` call
inside the same race window returns
`412 failed_precondition: machine still active, refusing to
start`. The resolver propagates the 412 to the WS proxy, which
writes 502 to the client; the editor never opens, the test
times out at 120 s. Other recent gold passes happened to find
the runtime fully torn down (8 consecutive greens, iters
369-375) — classic intermittent race.

## Diagnosis

Fly Machines API event timeline for machine `7849752ad12268`
(project `e93d1e63`, iter-376 GT-6 stopped failure), extracted
from `/v1/apps/tex-center-sidecar/machines/<id>/events`:

| ts (UTC) | event |
|---|---|
| 22:53:23.786 | launch created (user) |
| 22:53:26.273 | start started (flyd) |
| 22:53:35.301 | stop stopping (user; test's POST /stop) |
| 22:53:42.103 | exit stopped (flyd; runtime SIGTERM exit_code 143) |

Web app log at the same instant:

```
22:53:42Z resolve-error projectId=e93d1e63 message:
  Fly Machines API 412 .../machines/7849752ad12268/start:
  {"error":"failed_precondition: machine still active,
  refusing to start"}
```

So at 22:53:42 the resolver's `driveToStarted` saw `stopping`
on `getMachine`, called `waitForState("stopped")`, and the
moment that returned called `startMachine` — exactly when Fly
flipped to `stopped` per `flyd`'s `exit` event but before the
runtime was fully reaped. 412 is the API's response to "start
on a still-active machine".

The 408-retry loop in `waitForStartedWithRetry` only catches
`/wait` timeouts. The `startMachine` call sits outside that
loop in `driveToStarted` and has no retry.

Verified the three OTHER iter-376 failures (GT-8, bootstrap
cascade, seeded multi-page) are unrelated to this 412 race:
each test's `cmContent.waitFor` exceeded its inner budget
(30 s / 60 s / 60 s respectively) on a cold-start image-pull
tail, then the `finally` block called `cleanupLiveProjectMachine`
which `DELETE`d the just-launched machine, and the in-flight
`/wait?state=started` returned 404 ~7 s later as a downstream
effect of the self-destroy. Confirmed via the Fly events
endpoint showing each failing machine's destroy timestamp
matching the test's `finally`-block wallclock to within a
second. Those are cold-start budget tunes (separate iteration);
the 412 race above is a real code bug.

## Approach

Wrap `startMachine` in a bounded 412-retry loop. The 412 race
is structurally identical to the 408 race already handled in
`waitForStartedWithRetry`: Fly's async state-machine reports a
state moments before the underlying runtime catches up, and a
follow-up call within the gap is rejected with a deterministic
precondition error. A short budget (10 s with 250 ms backoff,
~40 attempts) is enough to absorb the post-stop runtime-reap
tail (≤8 s observed; the `exit` event in the trace above is
7 s after the stop command).

Implementation: add `startMachineWithRetry(machineId)` helper
alongside the existing `waitForStartedWithRetry(machineId)`,
use it in both `driveToStarted` branches that call
`startMachine` (stopped/suspended + stopping/suspending).
Budget configurable via a new
`startMachineRetryTimeoutSec` option (default 10).

Catching only status 412 (not all errors) keeps real
"machine vanished" / auth-token-revoked errors propagating
immediately. Non-412 errors throw on the first call.

A clean alternative — drive by event subscription rather than
poll-then-start — would require Fly to expose a "stopped and
reaped" terminal signal, which it doesn't. Retry is the right
primitive.

## Actions

- `apps/web/src/lib/server/upstreamResolver.ts` — added
  `startMachineRetryTimeoutSec` to `UpstreamResolverOptions`
  (default 10), `startMachineWithRetry(machineId)` helper that
  retries on `err.status === 412` until the budget elapses
  (250 ms sleep between attempts), replaced both direct
  `opts.machines.startMachine(machineId)` calls in
  `driveToStarted` with the wrapper. Non-412 errors propagate
  on the first attempt; persistent 412 past the budget
  rethrows the last 412 error.

- `apps/web/test/upstreamResolver.test.mjs` — added two cases:
  - Case 11: 412 on `startMachine` for a stopped machine →
    retries → second attempt succeeds → resolver returns the
    upstream. Asserts the start call was retried.
  - Case 12: persistent 412 past the retry budget →
    `resolve()` rejects with `/412/`, the start call was
    invoked more than once.

  Both cases use a configurable `startMachineRetryTimeoutSec`
  override to keep test wallclock at sub-second.

## Notes for future iterations

- The 412 race is symmetrical for the
  `suspending → suspended → start` path on
  `verifyLiveGt6LiveEditableState.spec.ts` (the suspended
  variant). Iter-376 didn't surface it because the suspend
  path is much faster (~78 ms resume per PLAN), shrinking the
  race window. The same retry now covers it for free.
- The cold-start tail that bursted iter-376's GT-8 / bootstrap
  cascade / seeded budgets is a separate concern. Two
  candidate fixes: (a) bump `cmContent.waitFor` to 120 s in
  the three specs (papers over Fly image-pull variance,
  acceptable for gold), (b) investigate whether the sidecar
  image-pull / boot can be sped up further. Defer to next
  iteration; not in scope here.
- If a future iter-N gold pass shows a 412 in web logs *with*
  a retry that doesn't succeed within 10 s, bump
  `startMachineRetryTimeoutSec` — the tail-of-tail for
  `exit stopped` may be harder to characterise than 8 s on
  rare slow runs.
