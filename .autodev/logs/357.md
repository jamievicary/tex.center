## Bootstrap-timeout findings (not actioned)

Detailed timeline from sidecar logs of `08003d7c053648`
(extracted via `flyctl logs -a tex-center-sidecar -i ... --
no-tail`):
- `11:49:46Z` page.goto starts on the bootstrap browser →
  Fly Machine create.
- `11:51:09Z` Fly emits `started` event (83s elapsed).
- `11:51:12Z` sidecar bootElapsedMs=271ms, listening, then
  `ws-upgrade-open` viewerCount=1 reaches it.
- `11:51:13Z` `ws-upgrade-close` code=1001 hadViewer=true (1s
  after open). Code 1001 = "going away" — client-initiated.
- `11:51:16Z` bootstrap deadline (started + 90s).
- `11:51:17Z` supertex `daemon-round-done` maxShipout=1,
  `compile ok elapsedMs=4694 segments=1` — but the WS was
  already closed, the segment never reached the browser.
- `11:51:17Z` follow-on `daemon-stdin` round 2, no-op
  (already-rendered source), `daemon-round-done` maxShipout=-1
  (no shipout).

Two open puzzles, neither addressed this iteration:

- **83s Fly create→started.** `196_answer.md` documents 60-90s
  as the realistic worst case, so this is at the upper edge but
  not anomalous. Cause: probably image-pull cold cache on a Fly
  host that hadn't run a sidecar instance recently. Mitigation
  options: bump the warm-up budget to absorb (anti-flake-rule
  adjacent — but `WARMUP_TIMEOUT_MS` is a "wait long enough for
  external infra" deadline, not a sync primitive); or
  accept-and-retry with a different strategy.
- **WS close 1s after open.** This is the load-bearing item —
  even if Fly cold-start were 30s, the WS close races segment
  delivery. Hypotheses: (a) editor JS errors out before the
  compile completes (the page closes / reloads), (b) Playwright's
  default 30s navigation timeout fires AFTER the page-load event
  but somehow tears down the page mid-WS, (c) Fly proxy
  artifacts on the multi-minute hold of a pending WS upgrade.
  No conclusive answer; deferred.

The right next step on this front is either to capture
browser-console logs from the bootstrap's transient browser
(it currently doesn't log anything from the page) OR to add an
explicit page.on("close") / page.on("framenavigated") logger so
we know precisely why the WS closes. Both are 1-iteration
investigations IF the failure recurs in iter 357's gold. If iter
357's gold passes cleanly, this stays as a recorded one-off.
