## Goal for this iteration

Advance M7.4.x by enriching GT-5's failure diagnostics. Iter 210
established that GT-A/B/C/D are GREEN on live and only GT-5 is RED
with "no post-edit pdf-segment arrived". The plan's diagnostic
probes (cross-spec state pollution; Yjs ops not reaching the WS;
swallowed compile error) all live in the failure path of a single
GT-5 run, but the current error message ("no post-edit pdf-segment
frame arrived within timeout — keystroke → … wire path broken …")
tells the next iteration nothing about which probe to follow.

This iteration adds inline capture in GT-5 — pre-edit document
text + cursor position, outgoing-WS frame counter across the type,
and ALL control-frame payloads — and replaces the `expect.poll`
with a manual bounded loop so the timeout message can incorporate
state that was captured *after* the poll was constructed (Playwright
freezes `expect.poll`'s `message` at call-site time). The next live
gold pass will, in failure, produce a diagnostic that names the
probe.

No fix attempt this iteration; pure observability. Coherent because
both probes (#1 and #2) require the same instrumentation surface
and would otherwise be wasted as two iterations.

## Approach

Edit `tests_gold/playwright/verifyLiveGt5EditUpdatesPreview.spec.ts`:

1. Import `TAG_CONTROL` (was already exported by
   `fixtures/wireFrames.ts` for the existing inline filter).
2. Add a second `authedPage.on("websocket")` listener registered
   *before* `goto`. The existing `captureFrames` listener already
   collects `pdfSegmentFrames` and overlap-error control frames; the
   new listener increments a `framesSent` counter on every
   non-empty outgoing payload and accumulates *every* received
   `TAG_CONTROL` frame body as a string (not just the `already in
   flight` subset). Multiple `websocket` listeners coexist —
   Playwright fans them out.
3. After the `Control+End → ArrowUp×2 → End` cursor navigation but
   before `keyboard.type`, snapshot `cm-content`'s full text and
   probe `window.getSelection()` for the focused-node text +
   startOffset. These name the document state and cursor landing
   line at the moment of the edit.
4. Capture `framesSentBeforeEdit` and `framesSentAfterType` either
   side of `keyboard.type`. A healthy GT-C run sends ≥1 frame per
   keystroke; a near-zero delta localises the bug to the page → WS
   edge.
5. Replace the `expect.poll(() => pdfSegmentFrames.length, …)` with
   a manual bounded poll over the same 30 s timeout. On timeout,
   throw a multi-line `Error` whose message includes pre-edit doc
   tail, cursor info, framesSent before/after/now, overlap-error
   count, and the first 10 control-frame payloads.

Behaviour on the success path is unchanged (the poll still resolves
the moment a new pdf-segment arrives).

## Notes for future iterations

When the next live gold pass produces the GT-5 failure, read the
multi-line error message in the gold-suite output (search
`.autodev/state/last_gold_output.txt` for "no post-edit pdf-segment
frame arrived"). Interpretation guide:

- **`framesSent delta during type` is 0 or 1** (vs. ~24 chars in
  `EDIT_PAYLOAD` with 5 ms delay): the page → WS edge is dropping
  ops. Probe #2 is the cause. Next iteration looks at the
  CodeMirror → Yjs binding for the `\section{...}` payload shape —
  possibly the `\` or `{`/`}` characters routing through an
  unbound key handler.
- **`framesSent delta` is healthy but `pdfSegmentFrames` count
  unchanged**: ops reached the WS but the daemon never emitted a
  segment. Inspect `controlFrames[]` for a `compile-status
  state:error` payload; if present, probe #3 is the cause. If
  absent, the daemon silently dropped the round — that is an
  upstream blocker.
- **`focused line` contains `\end{document}` or `pre-edit doc tail`
  shows the body has been pushed past `\end{document}`**: probe #1
  is the cause. Next iteration gives GT-5 its own freshly-seeded
  project in `globalSetup.ts` (a second `bootstrapLiveProject`
  call with a different env-var prefix, and a new fixture).

If GT-5 unexpectedly turns GREEN, the diagnostic capture stays in
place — its overhead is two evaluate calls and one extra WS
listener on the success path, both negligible. The richer error
message remains useful as protection against regression.
