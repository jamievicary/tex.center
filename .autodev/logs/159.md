## Diagnosis

Run `25735693190` (commit `7a0263a`):

- ✓ `runtime image smoke` (38s)
- ✓ `flyctl deploy` (49s)
- ✗ `live full pipeline (M8.pw.4)` — failed at step
  `pnpm install --frozen-lockfile` after only 6 s.

Failure log:

```
ERR_PNPM_ENOENT  ENOENT: no such file or directory, mkdir
'/home/runner/work/tex.center/tex.center/node_modules'
```

`actions/checkout@v5` ran clean before pnpm. Yet pnpm can't
mkdir its own target. The reason: **`node_modules` is tracked
in git as a symlink** (mode 120000), committed all the way back
in iter 4's commit `23513e5`. On the maintainer's WSL2
checkout, `tests_normal/setup_node.sh` repoints that symlink at
an ext4-backed cache under `$HOME/.cache/tex-center-nm/<hash>/
node_modules` to dodge DrvFs handle races. On a clean GH
runner, the symlink target doesn't exist, so what `checkout`
lays down is a *dangling* symlink. `pnpm install`'s `mkdir
node_modules` follows the link and ENOENTs on the absent
target.

This bug has been latent since iter 4 because no GH Actions
workflow in this repo did `pnpm install` directly on a runner
until iter 158. The two existing jobs are:

- `smoke` → runs `bash scripts/smoke-runtime-image.sh` which
  builds the runtime Docker image; pnpm install happens INSIDE
  the image where no host symlink exists.
- `deploy` → runs `flyctl deploy --remote-only` which ships the
  source to Fly's remote builder and builds there.

The `live-pipeline` job is the first runner-side pnpm install,
hence the first to surface this.

## Fix

- `git rm --cached node_modules` — drops the dangling-symlink
  blob from the index. .gitignore's `node_modules/` rule keeps
  it from being re-added; `tests_normal/setup_node.sh` continues
  to create the symlink in the working tree on local clones.
- `tests_normal/cases/test_no_tracked_node_modules.py` — new
  regression lock.
  - `test_no_node_modules_tracked`: no path segment equal to
    `node_modules` appears in `git ls-files`.
  - `test_no_tracked_symlinks_outside_vendor`: more general
    guard — any tracked symlink (mode `120000`) outside
    `vendor/` is rejected, since they almost always encode a
    maintainer-local absolute path that breaks on CI.
    `vendor/` excluded so submodule internal layout is free to
    use symlinks where it needs to.
