- `apps/sidecar/src/index.ts` rewritten so `main()` is a thin
  composition of two new exported building blocks:

  - `buildSuspendSelfFromEnv(env)` â€” returns `null` unless
    `FLY_APP_NAME`, `FLY_MACHINE_ID`, and `FLY_API_TOKEN` are all
    set. When wired, returns a function that POSTs to
    `https://api.machines.dev/v1/apps/{app}/machines/{id}/suspend`
    with a `Bearer` header and throws on non-2xx.
  - `createIdleHandler({ getApp, suspendSelf, log, exit })` â€”
    fires once (re-entrancy guarded), `await app.close()`,
    `await suspendSelf?.()`, then `exit(0)`. Exit is the
    backstop on every code path: missing creds, fetch error,
    non-2xx response, and the in-theory case where suspend
    returns instead of freezing.

  Production: in the steady state the suspend call freezes the
  VM mid-fetch, so the `exit(0)` after it is dead code (and a
  log line marks the "returned without freezing" anomaly if it
  ever fires). Local dev: `FLY_API_TOKEN` is unset â†’
  `suspendSelf` is `null` â†’ behaves exactly like the previous
  `process.exit(0)` path.

- `apps/sidecar/test/idleSuspend.test.mjs` â€” new unit suite:
  - `buildSuspendSelfFromEnv` returns `null` on each of the
    three single-env-missing cases.
  - Happy path: composes the right URL + `Authorization` header
    and uses `POST`.
  - Non-2xx response (e.g. 422 from a hypothetical "suspend not
    supported"): the returned function rejects with the body
    folded into the message.
  - `createIdleHandler` with `suspendSelf: null` closes the
    app and exits cleanly with code 0; no log lines.
  - With `suspendSelf` that resolves: still closes, calls
    suspend, and exits 0.
  - With `suspendSelf` that throws: logs the error and still
    exits 0.
  - Double-firing the handler only triggers a single
    close/exit cycle.
