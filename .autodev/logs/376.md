## Goal for this iteration

Fix the iter-375 bootstrap-cascade gold case that landed RED in iter
375's first gold pass with shape
`shapes=[{"bytes":57878,"shipoutPage":2,"lastPage":false}]
compileStatusEvents=running×3,idle×3`. The cascade chain itself is
working (page 2 was shipped) — the test's terminal expectation
(`lastPage=true AND shipoutPage≥2 after one cascade`) is structurally
incompatible with the iter-372 `target = maxViewingPage(p)` design,
not a bug in the runtime.

## Diagnosis

Trace with `STATIC_TWO_PAGE` (2-page document) under `target =
maxViewing`:

1. Hello-world initial compile: `maxViewing=1`, `target=1`, 1-page
   document → supertex hits `\enddocument` → `lastPage=true`, no
   placeholder.
2. `Ctrl+A → STATIC_TWO_PAGE`: `maxViewing` still 1, `target=1`,
   2-page doc → supertex stops at page 1 (target reached) →
   `lastPage=false` → PdfViewer mounts `.pdf-page-placeholder`
   for page 2.
3. Scroll into placeholder-2: IO fires → `maxViewing=2` →
   `kickForView(2)` → `target=2`, supertex ships pages 1+2 then
   stops (target reached) → `lastPage=false` → PdfViewer renders
   two real pages **and remounts a new placeholder for page 3**
   (because lastPage===false still signals "more might exist").
4. Scroll into placeholder-3: `maxViewing=3` → `kickForView(3)` →
   `target=3`, supertex hits `\enddocument` before shipping page
   3 → `lastPage=true`, no further placeholder.

So the design intentionally requires **N cascades to terminate an
N-page document** with `lastPage=true` — the user must scroll past
the last real page into the "probe" placeholder. The iter-375 test
asserted termination after one cascade, which only works for a
1-page document.

The placeholder is fundamentally ambiguous under `target =
maxViewing`: it means "more might exist; scroll to find out". The
explicit "the doc has exactly K pages" signal (`lastPage=true`)
fires only when target *exceeds* the doc length. This is correct
per design (no wasted compile work probing arbitrary numbers of
pages ahead) but means the test must walk one more cascade hop.

## Approach

Extend the bootstrap-cascade case to exercise **both** cascade
hops:

- **Hop 1 (already in the spec).** Wait for placeholder-2, scroll
  it into view, await a post-cascade segment with `shipoutPage≥2`
  (any `lastPage`). Verify DOM has 2 real pages. This validates
  the demand-fetch chain (IO → setViewingPage → view frame →
  kickForView → recompile → bigger segment → render).
- **Hop 2 (new).** Wait for a new placeholder to mount (page-3
  slot, because cascade-1's segment had `lastPage=false`). Scroll
  it into view, await a segment with `lastPage=true`. Verify DOM
  ends at 2 real pages + 0 placeholders. This validates the
  termination signal — supertex emits `[pdf-end]`, sidecar stamps
  `lastPage=true`, PdfViewer removes the placeholder.

Failure-mode messages stay structured so each hop's break shape
is identifiable. The outer envelope (240 s) already covers one
extra cascade compile (each ~10-20 s on a warm Machine).

A simpler alternative — accept `lastPage=false` after one cascade
and just check 2 real pages — was considered but rejected: it
would silently regress the iter-A `[pdf-end]` plumbing, since the
final termination signal would never be observed in any gold
spec. The two-hop variant pays one extra compile to lock both
signals.

GT-8 also red in the same gold pass (`.cm-content` not visible
within 30 s on a cold Machine, similar shape to the cold-start
tunes already in PLAN priority #2). Not bundling: GT-8 is a
separate cold-start budget tune that warrants its own iteration
with its own diagnostic shape.

## Notes for future iterations

- **Probe-ahead (target = maxViewing + 1) was considered and
  rejected.** It would let a 2-page doc terminate in *one*
  cascade, but for an N-page doc with the user viewing page K
  it would still need K-many cascades to reach lastPage=true
  (each cascade ships K pages with target=K+1, lastPage=false
  unless K+1 > N). So it only saves one cascade per session, at
  the cost of one extra compile per scroll. Net negative; the
  two-hop test variant matches the chosen design directly.
- **If the iter-376 reshape goes RED on hop 2 with `lastPage=
  false` segments only**, the most likely cause is the WsClient
  → +page.svelte → PdfViewer prop wiring losing the tri-state
  byte on the second `.lastPage=true` segment. Check
  `apps/web/src/lib/WsClient.ts` `pdfLastPage` snapshot field
  and the lastPage `$effect` in `PdfViewer.svelte:131-135`
  (newly-added `$effect` separate from the `src` effect, runs
  syncPlaceholder() independently of re-render).
- **GT-8 cold-start budget** (iter-376 gold-pass red) is the
  next obvious slice if priority #1 closes. Test gives
  `.cm-content` only 30 s on a cold per-project Machine; iter
  362's tune for similar shapes was to bump to 60-90 s on the
  inner `waitFor`. The Slow-test-file output showed the GT-8
  test took 31.2 s total, so the test gave up just past 30 s —
  a small tune (60 s) should be enough. Defer to next iter
  since this iteration's coherence is "fix iter-375's cascade
  case".
