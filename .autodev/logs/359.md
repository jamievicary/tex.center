## Goal for this iteration

PLAN priority #1 is `verifyLiveGt6LiveEditableStateStopped` (M13.2(b).4)
— RED in iter 347, 354, 355, 357, 358. Iter 358's gold pass gave us
the first hard numbers: `cmContentReadyMs=5372 keystrokeAckMs=8`. The
stopped-Machine cold-resume path takes 5.3 s end-to-end vs a 1 s
budget. Reading the wire timeline (lines 660-670 of
`last_gold_output.txt`): hello frame for the second WS lands at
clickAt+≈5.3 s, file-list / doc-update follow in 100 ms, and
`.cm-content` populates immediately after. So practically all of the
5.3 s is "before any wire frame arrives" — Fly start + WS handshake
+ upstream resolver `driveToStarted`. The cm/yjs/hydrate part is
fast.

Iter 358's gold pass also surfaced a second failure I want pinned
before it gets lost: `verifyLiveGt6LiveEditableState` (M13.2(b).3, the
SUSPENDED variant) regressed from GREEN (iter 347, 355, 357) to a
40 s testTimeout. Browser console: `WebSocket connection ... failed:
Error during WebSocket handshake: Unexpected response code: 502`. The
spec's diagnostic `console.log` line never fired — the test never
reached step 4 measurement. Without intermediate timing we can't tell
whether the regression is `driveToStarted` taking >5 s connect timeout,
a Fly proxy hiccup, or a real bug.

Both specs share a diagnostic gap: the failure modes are different
(stopped: budget overshoot; suspended: 502) but the
`console.log` line at end emits only `cmContentReadyMs` /
`keystrokeAckMs` / `cmTextPrefix`. We have richer signals already
plumbed: `editor:route-mounted`, `editor:ws-open`, `editor:yjs-hydrated`,
`editor:first-text-paint`, `editor:first-pdf-segment` performance
marks — GT-9 already reads them in its probe loop. WS-side counters
(open/close/error counts) are also already captured by
`captureFramesAuto` via the `authedPage` fixture.

Coherent slice: extend both specs' end-of-test diagnostic to emit
the post-click WS-open / first-frame timestamps (relative to
clickAt) plus open/close counters. The change is purely
observational — no product code, no new asserts, no new wire frames.
Locks: none needed (the diagnostic format isn't asserted; it's
stdout for the next iteration's reader). Failures still fail in the
same place; they now carry actionable detail. Next gold pass either
gives us the timing breakdown of the 5.3 s on the stopped path
(directly informing the M13.2(b).5 architectural choice) or pins the
suspended-502 to a specific phase.

scope here. The wire-side timestamps below are independent of the
mark plumbing and don't require any product change to land.

## Notes for future iterations

- **Reading the next gold pass for these specs.** Three-phase
  decomposition: `clickToWsOpenMs` covers Fly start +
  `driveToStarted` (`startMachine` → `waitForState(started)`);
  `clickToFirstFrameMs - clickToWsOpenMs` covers handshake + sidecar
  boot + hello/file-list; `cmContentReadyMs - clickToFirstFrameMs`
  covers Yjs hydrate + CodeMirror render. For iter 358's
  `cmContentReadyMs=5372` (stopped variant), expectation is
  `clickToWsOpenMs` ≈ 4-5 s (Fly cold-start cost), the rest small.
  If `clickToWsOpenMs` is small and the gap before frame is large,
  that's a sidecar-boot issue. If `clickToFirstFrameMs` is `(none)`,
  the WS opened but no hello arrived in 1 s — investigate sidecar
  start-up code path.
- **GT-6-suspended 502.** Iter 358's specific failure (`opens:0`
  expected if WS upgrade itself returns 502) points at the upstream
  resolver's `driveToStarted` for `suspended` state at
  `apps/web/src/lib/server/upstreamResolver.ts:293`. The resolver
  calls `startMachine` (which Fly accepts on `suspended` to
  un-suspend). `waitForStartedWithRetry` retries on 408. If the
  Machine takes >60 s × N to come back, the `wsProxy`
  `connectTimeoutMs=5000` already expired and 502 was written. The
  next-gold-pass numbers will tell us which.
- **GT-6-suspended budget tightness.** `testInfo.setTimeout(40_000)`
  for a flow with up to 20 s step-1 + 15 s suspend-settle + step-4
  measurement is too tight on a colder Fly host. If the next gold
  recurs the 502 + timeout shape, bumping to 60 s aligns with the
  stopped variant and surfaces the underlying issue rather than
  testInfo-timing it.
- **M13.2(b).5 architecture.** Once we know the breakdown, the fix
  is one of: (a) widen SSR seed to serve persisted blob even when
  `machine_assignments` row exists; only addresses the visible-seed
  latency, not `.cm-content` per current code (seed renders in
  `<pre class="editor-seed">`, not the CM mount); (b) eliminate
  `stopped` state — keep Machines running OR move to suspend on
  idle (requires solving the iter-340/343 auto-resume race); (c)
  add a per-cycle mark set so step-4 cold-resume of an SPA
  re-mount actually fires marks (would also unlock a future
  `verifyLiveGt6Reopen`-style spec).

  Plan-review N=361.
