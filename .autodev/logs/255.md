## Goal for this iteration

Fix the M13.2(b).1 self-suspend bug surfaced by iter 254's lone gold
failure (`verifyLiveFullPipelineReused`). Diagnosis: iter 249's
`createIdleHandler` assumed Fly would freeze the VM mid-`fetch` on
`POST /machines/{self}/suspend`, but Fly actually sends the HTTP
response *then* freezes. When the Machine is later resumed via
`/start`, the awaited fetch resolves, the handler falls through to
`exit(0)`, and the sidecar exits ~1 s after every wake. Live
evidence: Machine `1859d5dc2d6158` (the reused-project Machine)
events showed `started` → `exit(stopped)` 912 ms apart.

Effect: every project's first access after an idle-suspend exited
the sidecar before serving any traffic, so `.cm-content` never
appeared. M13.2(b).1's intent (cheap resume from suspended state)
was inverted — resumes became single-use deaths.

## Approach

Rewrite `createIdleHandler` so that:

- When `suspendSelf` is wired (production), the handler does NOT
  close the Fastify app and does NOT exit. After the suspend fetch
  resolves it treats itself as post-resume, calls `ctx.rearm()` to
  re-arm the server's idle gate, clears its in-flight flag, and
  returns. The listener stayed bound across the freeze, so the
  incoming WS handshake that triggered the resume is served
  normally.
- When `suspendSelf` is `null` (local dev / missing creds) or the
  suspend call throws, fall back to the historical close + `exit(0)`
  path. That keeps the local idle-stop story unchanged.
- A second concurrent firing while a suspend is in flight is a
  no-op (the re-entrancy guard is preserved), but after a successful
  suspend/resume cycle the handler can re-fire for the next idle
  window.

Server side (`apps/sidecar/src/server.ts`): the idle timer callback
now invokes `onIdle({ rearm })` where `rearm` re-arms the idle
timer iff `viewerCount === 0`. If a viewer connected during the
suspend window (extremely narrow but possible), they've already
cleared the timer via `noteViewerAdded`, so we don't double-arm.

  with a fake fetch — only the live behaviour exposes it.
