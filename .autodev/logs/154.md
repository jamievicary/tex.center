## Goal for this iteration

PLAN iter-154 entry: bump per-project Machine memory to 1024MB. Iter
153's live cutover succeeded on the WS upgrade handshake, but
per-project Machines are still being created at the Fly Machines API
default (~256MB). The earlier OOM-reboot loop on the old image
happened on first WS request at that size (anon-rss small but
total-vm 1.26GB → cgroup kill); the new image will hit the same wall
once real Yjs traffic arrives. Surgical fix: add a `guest:` block
with `memory_mb: 1024` to `machineConfig` in `upstreamFromEnv.ts`,
lock the floor in `upstreamFromEnv.test.mjs`.

## Actions

- `apps/web/src/lib/server/upstreamFromEnv.ts`: added
  `guest: { memory_mb: 1024, cpu_kind: "shared", cpus: 1 }` to
  `machineConfig`. Comment explains the cgroup-total-vm-kills-rss
  reasoning so a future trim doesn't try to shrink it back.
- `apps/web/test/upstreamFromEnv.test.mjs` (happy-path block):
  asserts `createCall.req.config.guest` exists and
  `guest.memory_mb >= 1024`. Picked `>= 1024` rather than `== 1024`
  to leave headroom for a future bump (full TeX Live's `lualatex
  --ini` peak is real and could go higher).

The `MachineConfig` type already declared `guest.memory_mb?: number`
(see `apps/web/src/lib/server/flyMachines.ts:46`), so no type
changes were needed; the field was always shaped, just unset.
