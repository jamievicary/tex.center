## Goal for this iteration

Refactor iteration (N % 10 == 0). Survey identified a clear
highest-leverage cleanup: four near-identical hand-rolled cookie
formatters across `apps/web/src/lib/server/`. Every cookie this
app emits (state-cookie set/clear, session-cookie set/clear) and
every `Cookie:` header read shares the same attribute shape —
`Path; HttpOnly; SameSite=Lax; Max-Age; [Secure]` — but the M5
work landed each implementation as a private helper at the bottom
of its consumer:

- `oauthStart.ts` — set state cookie (attrs array, manual join).
- `oauthCallback.ts` — `clearCookie()` + `formatSessionCookie()`
  (two near-identical attrs arrays).
- `logout.ts` — `clearSessionCookie()` (third copy).
- `sessionHook.ts` — `clearSessionCookie()` + `readCookie()`
  (fourth copy + first parser copy).
- `routes/auth/google/callback/+server.ts` — `readCookie()`
  again (second parser copy).

Drift between them is a real security concern: a missing
`HttpOnly` or `SameSite=Lax` in one branch and not another would
ship undetected. Extract to a single `cookies.ts` module.

## Approach

`apps/web/src/lib/server/cookies.ts` exports three pure helpers:

- `formatSetCookie({name, value, path, maxAgeSeconds, secure})` —
  the one canonical builder.
- `formatClearCookie({name, path, secure})` — sugar for
  `formatSetCookie` with empty value + Max-Age=0.
- `readCookie(header, name)` — single parser.

All five consumers rewritten to import these. No public API
changes; the existing unit tests assert wire format (`Max-Age=0`,
`tc_session=;` prefix, `Secure` presence) and continue to pass
unchanged.

## Actions

- `routes/auth/google/callback/+server.ts`: dropped private
  `readCookie`; imports the shared one.
