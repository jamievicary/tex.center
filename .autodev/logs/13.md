## Goal for this iteration

Land M3.3 — `SupertexWatchCompiler`. Replace the per-compile
fork-exec of M3.2 with one persistent supertex watch process per
project, driven by writes to `main.tex` (server already mirrors
the doc there before each compile) and synchronised on a
"compile round done" stdout marker line. Selection via
`SIDECAR_COMPILER=supertex-watch`; default stays `fixture`. The
plan calls for a paired `pgrep`-empty test to prove the child is
reaped by `Compiler.close()`.

## Approach

The watch process model raises one design question the
once-compiler didn't: **how does the sidecar know a compile round
has finished?** `--live-shipouts` only marks per-shipout
boundaries (page/offset), not end-of-round. Quiet-period polling
violates the anti-flake rule. The cleanest synchronisation is a
stdout marker line emitted by supertex after each watch round —
this is an upstream PR (filed under M3.5 in PLAN.md updates
below) but is straightforward to specify now and the sidecar
half can be tested today against a fake binary that already
honours the contract. Documenting the marker as part of the
sidecar↔supertex protocol is the load-bearing decision here.

Lifecycle:

- Lazy spawn on first `compile()` (the server has already mirrored
  the source by then, so the watcher's initial compile uses it).
- `compile()` awaits the next READY marker from stdout, with a
  configurable wallclock cap.
- A pending-marker counter prevents loss of a READY that lands
  between writeMain returning and compile() registering its
  resolver — without it a fast inotify path could race the
  resolver setup. Single compile in flight per project, so the
  counter is bounded.
- `close()` SIGTERMs the child, awaits exit, escalates to
  SIGKILL after 2 s. Tested by recording the pid before close
  and asserting `process.kill(pid, 0)` throws ESRCH after.

## Notes for future iterations

- Real supertex is unusable as a watch backend until the
  `--ready-marker` upstream PR (M3.5) lands. A streak of
  M3.x sidecar-only iterations is fine, but M3.4 (per-shipout
  deltas) doesn't strictly need the marker — it can prototype
  against the same fake watcher, since the shipouts file format
  is real. M3.5 then unlocks production use of both M3.3 and
  M3.4 simultaneously.
- The pending-marker counter in `SupertexWatchCompiler` is
  essentially defensive against a hypothetical race where READY
  arrives between writeMain returning and compile() registering
  its resolver. With single-compile-in-flight per project (server
  debounce + serial runCompile) it's hard to provoke; I kept it
  rather than removing it because the cost is one int and the
  failure mode it guards is silent.
- The fake binary uses 20 ms mtime polling rather than inotify
  to keep the test self-contained (no Linux-specific deps in the
  test infra). Real supertex uses inotify; the contract on the
  marker is the same either way.
