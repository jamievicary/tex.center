## Goal for this iteration

Re-land iter 148's CRITICAL PATH code-side fix. Iter 148 was
reverted for wallclock timeout — most of the 45m went to the
remote-only sidecar Docker build during the live cutover. This
iteration deliberately lands only the in-tree code changes and
stops; live deploy + secret-roll + Machine-destroy + re-probe
is split into a follow-up iteration so wallclock budget is not
at risk.

Specifically:
- `apps/sidecar/src/index.ts`: default `HOST` → `"::"`, factor
  out `resolveBindHost` + `DEFAULT_BIND_HOST` so a regression
  test can lock it.
- `apps/sidecar/Dockerfile`: runtime `HOST=0.0.0.0` → `HOST=::`.
- Regression tests under `apps/sidecar/test/bindHost.test.mjs`,
  `tests_normal/cases/test_sidecar_dockerfile.py` flipped, and
  wired into `tests_normal/cases/test_node_suites.py`.
- `apps/web/src/lib/server/wsProxy.ts`: write `HTTP/1.1 502 Bad
  Gateway` to the client before destroying the socket on
  pre-connect `upstream-error`. Post-connect errors mid-pipe
  still destroy silently.
- `apps/web/test/wsProxy.test.mjs` Case 4 strengthened to assert
  the 502.

## Actions

1. `apps/sidecar/src/index.ts`: exported `DEFAULT_BIND_HOST =
   "::"` and a small `resolveBindHost(env)` helper (treats
   undefined and empty-string as unset; both fall back to the
   default). `main()` now goes through the helper. Compose /
   shell setups that propagate empty env values would have
   silently regressed the fix without the empty-string clamp.
2. `apps/sidecar/Dockerfile`: runtime stage `HOST=0.0.0.0` →
   `HOST=::`. The vendored engine + texlive logic above this
   line is untouched.
3. `tests_normal/cases/test_sidecar_dockerfile.py`:
   `test_runtime_listens_on_all_interfaces` now requires
   `HOST=::` on its own line in the runtime ENV block and
   asserts the file contains no `HOST=0.0.0.0` anywhere — so a
   future "make IPv4 explicit" edit can't accidentally land in
   the runtime stage.
4. `apps/sidecar/test/bindHost.test.mjs`: locks
   `DEFAULT_BIND_HOST === "::"`, exercises `resolveBindHost`
   (default, empty-string, IPv4 override, IPv6 loopback
   override), and re-reads the Dockerfile to ban a regression.
5. `tests_normal/cases/test_node_suites.py`: new
   `test_sidecar_bind_host` runs the mjs test under tsx.
6. `apps/web/src/lib/server/wsProxy.ts`: `dialAndPipe` now
   tracks `upstreamConnected` (flipped to `true` in the
   `connect` handler). `cleanup` writes
   `HTTP/1.1 502 Bad Gateway` + `Connection: close` +
   `Content-Length: 0` to `clientSocket` before destroying it,
   gated on `reason.kind === "upstream-error" &&
   !upstreamConnected`. Post-connect errors (`client-error`,
   mid-pipe `upstream-error`, `closed`) still destroy silently
   — by then the WS handshake has been forwarded and the
   socket is no longer in HTTP land.
7. `apps/web/test/wsProxy.test.mjs` Case 4: now passes an
   `onEvent` collector, reads the response with `readUntil`
   and asserts a leading `HTTP/1.1 502 Bad Gateway`, then
   asserts a recorded `upstream-error` event. The
   "socket-not-closed" guard now early-returns when the socket
   is already destroyed (the 502 + `cleanup` may race; either
   close ordering is acceptable).

## Notes for future iterations

- **Iter 150 cutover.** This commit modifies `apps/sidecar/**`
  and `apps/sidecar/Dockerfile`, so the deploy-sidecar
  workflow's path filter will fire on push to main and build
  a new image. Grab the resulting digest from the workflow
  log (or `flyctl image show -a tex-center-sidecar`), roll
  `SIDECAR_IMAGE` on `tex-center` via `flyctl secrets set`,
  destroy the existing per-project Machines so the resolver
  recreates them at the new sha, then run
  `scripts/probe-live-ws.mjs` and expect `kind: "upgrade",
  status: 101`. Bogus-id control: 403. The first WS upgrade
  after Machine destroy takes ~80 s (image pull + Firecracker
  + tsx warm-up); if the probe's connect-timeout window is
  shorter you'll see the new wsProxy 502 (two `via` hops) — re-
  probe after the new Machine is in `started` state.
- **Why the cutover is not in this iteration.** Iter 148 packed
  the rebuild + roll + destroy + probe into one iteration and
  the remote-only build alone consumed most of the 45 min
  wallclock. Splitting them costs one extra commit but the
  code-side fix is now durable in tree and the cutover
  iteration only spends wallclock on `flyctl` calls + the
  probe, not on a Docker build.
- **The wsProxy 502 write is intentionally pre-connect only.**
  Do not extend it to `client-error` / mid-pipe `upstream-
  error` / `closed` — those events fire after the upgrade
  handshake has been forwarded, and an HTTP-looking byte
  string mid-frame would corrupt the WebSocket stream.

  records the code-side relanding, iter 150 = live cutover,
  iter 151 = M8.pw.4 activation. FREEZE-lift gate moved from
  iter 149 to iter 151 accordingly.
