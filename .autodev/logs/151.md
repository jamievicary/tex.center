## Goal for this iteration

PLAN had this as the live-cutover iteration (set `SIDECAR_IMAGE` to
the latest sidecar sha and destroy stale per-project Machines). On
orient I checked the actual deploy state and found a much bigger
problem: **sidecar CD has been silently failing since at least iter
124.**

`gh run list --workflow=deploy-sidecar.yml` shows every recent run
exited non-zero in ~10–20 s. The failed step is
`actions/checkout@v5` with `submodules: recursive`: the default
`GITHUB_TOKEN` only has access to the current repo, and
`vendor/supertex` is a private repo on the same account, so the
submodule clone fails with `Repository not found`. This means the
iter-149 bind-host fix never reached the production sidecar image —
the "cutover to a fresh sha" in PLAN's iter-151 entry was based on a
wrong assumption.

So this iteration becomes "fix sidecar CD" instead. Cutover slides
to iter 152, M8.pw.4 activation slides to iter 153.

  `SIDECAR_IMAGE` on `tex-center`, destroys stale per-project

  `wss://tex.center/ws/project/<id>` for `kind: "upgrade",
  status: 101`.
- If the iter-151 CD run *also* fails, that's surprising and worth
  reading the new log before doing anything else — possible
  causes: PAT lacks `repo` scope on private repos owned by the
  user (try `read:org`?), or the supertex repo is in an org and
  needs different scope. Both are diagnosable from the
  `gh run view --log-failed` output.
