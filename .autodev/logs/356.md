## Goal for this iteration

Root-cause and resolve `verifyLiveFullPipelineReused` RED. Iter 355
landed FE+fixture instrumentation expecting to capture a PDF.js
parse error on the next gold pass, but the iter-355 gold transcript
shows **no** `browser diagnostics:` block for the failing reused
project — i.e. no `console.error` and no `pageerror` fired. That
rules out the iter-355 "silent PdfViewer render failure" hypothesis
and forces a different look at the iter-354/355 evidence.

The smoking gun was already in the iter-354 transcript and was
mis-explained by `354_answer.md`: the pdf-segment frames carry
`shipoutPage=626017350`. That's not a stamped page number — it's
the 32-bit big-endian decode of the bytes `0x25 0x50 0x44 0x46`
= ASCII `%PDF`, i.e. the PDF magic. The decoder is reading the
first four bytes of the PDF payload as the `shipoutPage` field,
which means the **sidecar is writing the pre-M22.4b 13-byte
pdf-segment header (no shipoutPage)** while the FE has long since
moved to the 17-byte version.

This is precisely the staleness the PLAN flagged in iter 342 ("if
`verifyLiveFullPipelineReused`'s intermittent failures persist,
recreating its seeded project against the latest image would
isolate iter-340 from any older bug"). Iter 355's answer
explicitly dismissed staleness as the axis — that was the wrong
call.

## Approach

Three coherent steps, all addressing the same root cause:

1. **Ops fix.** Destroy the stale per-project Machine for the
   fixed-UUID reused project (`00000000-0000-4000-8000-
   000000000001`) and clear its `machine_assignments` row so the
   next gold pass provisions a fresh Machine on the current image.
2. **Observability.** Surface `decodeFrame` failures in
   `WsClient.handleMessage` via `console.error` so future wire-
   protocol drift fails loudly in the iter-355 fixture
   (`tests_gold/playwright/fixtures/authedPage.ts`) instead of
   being swallowed into `_lastError`. The previous code only set
   `_lastError` and emitted; the iter-355 fixture watched
   `console.error` (and `pageerror`), so the iter-355 gold pass
   showed no diagnostics block for the failing test.
3. **Pin.** Normal test
   `apps/web/test/wsClientDecodeError.test.mjs` asserts that a
   truncated pdf-segment frame both sets `lastError` and emits
   exactly one `[WsClient] decodeFrame failed: <msg>` console
   error.

  → throws `"pdf-segment payload truncated"`.

  caught the throw, set `_lastError`, emitted, returned. No
  `console.error`, so the iter-355 fixture captured nothing.
