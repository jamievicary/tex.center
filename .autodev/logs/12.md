## Goal for this iteration

Land M3.2 — `SupertexOnceCompiler`. The M3.1 mirror (iter 9)
already lays `main.tex` down on disk before each compile; this
iteration adds the compiler that actually spawns supertex against
that mirror, reads the resulting PDF, and returns it through the
existing `Compiler` seam. Default selection stays `fixture` so the
dev loop doesn't suddenly require a working supertex build; the
new path is opt-in via `SIDECAR_COMPILER=supertex-once`.

## Approach

Three concerns, kept in one coherent slice:

1. **The compiler itself.**
   `apps/sidecar/src/compiler/supertexOnce.ts` is a thin spawn-and-
   read implementation. It does not assume the binary location:
   the caller passes `supertexBin` and `workDir` in. Args match
   what the iter-8 survey block in PLAN.md identified as the M3.2
   shape: `<source> --once --output-directory <ws>/out
   --live-shipouts <ws>/out/shipouts`. It enforces a 60 s timeout
   (SIGKILL) so a hung engine can't pin a sidecar — important
   even at this stage, since the sidecar is the per-project
   Machine entry point. Failure modes (non-zero exit, missing PDF,
   ENOENT on the binary) all surface as `{ ok: false, error }`
   so the existing compile-status-error wire path keeps working.

2. **Wiring.** `buildServer` previously took a `compilerFactory:
   () => Compiler`. To make a real-supertex factory possible
   without re-doing the workspace-rooting plumbing every time,
   I changed the factory shape to `(ctx: { projectId, workspace })
   => Compiler`. The default factory now reads
   `$SIDECAR_COMPILER` (default `"fixture"`) and produces either
   a `FixtureCompiler` or a `SupertexOnceCompiler` rooted at
   `workspace.dir`; an unknown value throws on boot rather than
   silently falling back. `SUPERTEX_BIN` is required only when
   `supertex-once` is selected — fail loudly at boot, not on
   first compile.

3. **Tests.** `tests_normal/` can't depend on a built supertex,
   so I drove the compiler against a fake Node script that
   mimics the CLI's flag parsing and writes a stub PDF embedding
   the source bytes — that lets the happy-path test prove we
   really did spawn the binary against the on-disk source from
   `ProjectWorkspace.writeMain`. Failure-path coverage uses two
   tiny env-injecting wrapper scripts (`FAKE_FAIL=1` →
   non-zero exit, `FAKE_NO_PDF=1` → zero exit but no PDF) plus a
   nonexistent binary path for the ENOENT case.

A discovery worth recording in PLAN.md: the supertex executable
lives at `vendor/supertex/src/supertex` (Python entry point), not
`bin/supertex` as earlier plan revisions assumed. The compiler
takes the path as a constructor argument so this only matters at
deploy time; updated PLAN.md to set the right expectation.

`targetPage` from `CompileRequest` is plumbed in but currently
ignored — supertex's `--target-page` is documented but not
implemented (CLI errors today). M3.5 picks up that thread; the
sidecar already threads `targetPage` through the `Compiler`
boundary, so wiring it through is one flag-add when supertex
gains the feature.

## Notes for future iterations

- M3.3 (`SupertexWatchCompiler`) needs a way to detect
  per-shipout completion — tail `--live-shipouts` and compare
  byte counts. The M3.2 implementation doesn't read that file
  yet but it's already being written by the spawn, so the
  layout is ready.
- For the first real end-to-end test (gold suite) we'll need
  `vendor/supertex` actually built, which means `make` in the
  submodule plus a working `lualatex-incremental`. That's a
  gold-only dependency — no need to touch normal-suite setup.
- The compiler-factory signature change is a breaking API touch
  for any future caller, but the only caller today is the
  sidecar's own `buildServer` and the test suite, both updated.
