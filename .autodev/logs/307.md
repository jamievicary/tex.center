## Goal for this iteration

Open active-queue item #7 — **M11.2** file-tree CRUD via keyboard.
Today every file row exposes inline `✎` (rename) and `×` (delete)
buttons, but a user who has selected a row by clicking on it expects
F2 / Delete to also work (matching every OS file-manager). This
slice adds those two keybindings. The right-click context menu is
the other half of M11.2 and lands separately (M11.2b) — it has its
own UX surface (positioning, click-outside, Esc handling) that
deserves its own focused iteration rather than being bundled here.

GT-F (M22.2), max-visible (M21.2), and shared blob-store (M20.2)
sit higher on the priority queue but each needs a non-trivial
scaffold (test-fixture accessor / 3-page Playwright + sidecar
introspection / shared bucket binding). M11.2a is a clean, local,
unit-testable slice that moves a real product surface forward
without those prereqs.

## Approach

- New pure helper `apps/web/src/lib/fileTreeKeyboard.ts` exporting
  `decideFileRowAction(key, path, mainDocName) → "rename" | "delete" | null`.
  Mirrors today's visibility rules for the `✎` / `×` buttons:
  - `MAIN_DOC_NAME` is never editable (rename or delete).
  - F2 → "rename"; Delete (or Backspace on macOS) → "delete".
  - Modifier keys (Ctrl/Alt/Meta/Shift) suppress so OS shortcuts
    aren't hijacked.
  - Any other key → null (caller falls through to default handling).
- `apps/web/src/lib/FileTree.svelte`: add `onkeydown` on the
  `<button class="file-row">` element. On a non-null helper result:
  - "rename" → existing `promptRename(path)` flow.
  - "delete" → `window.confirm("Delete X?")` then `onDeleteFile(path)`.
    Confirm gate is deliberate — Delete is one key away from being
    pressed accidentally on a focused row. The explicit `×` button
    stays one-click (visible affordance, deliberate action).
  - `e.preventDefault()` only when the helper resolved to an action,
    so unrelated keystrokes (arrow keys for focus traversal, Enter
    for activate, etc.) keep their default semantics.
- New unit test `apps/web/test/fileTreeKeyboard.test.mjs` and wire
  through `tests_normal/cases/test_node_suites.py`.

## Notes for future iterations

- **`decideFileRowAction` and the inline button-visibility guards
  share a contract: `MAIN_DOC_NAME` is non-renamable and
  non-deletable.** If a future iteration relaxes that (e.g. allow
  renaming `main.tex` once the seed-doc plumbing supports it),
  update both `FileTree.svelte`'s `{#if … row.data.path !==
  MAIN_DOC_NAME}` guards on the `✎` / `×` buttons *and* the
  `path === mainDocName` early-out in `fileTreeKeyboard.ts`.
- **M11.2b context menu.** The right-click affordance is a separate
  UX surface — anchored menu near the pointer, click-outside +
  Escape dismissal, focus capture on open. Reuses
  `decideFileRowAction`'s permission rules (e.g. greys out
  Rename/Delete on `MAIN_DOC_NAME`) so the two entry points stay
  policy-consistent. Likely shape: a small `ContextMenu.svelte`
  positioned absolutely under the cursor, with three buttons that
  call the existing imperative flows (`promptRename`, `onDeleteFile`
  with confirm, and a focus-the-create-input shortcut for `Create`).
- **Keyboard delete confirms; button delete does not.** Deliberate
  asymmetry — the `×` glyph is itself a deliberate affordance, but
  Delete on a focused row is one keystroke away from accident.
  Equal-treatment alternatives (both confirm, neither confirms)
  were considered and rejected: a confirm on the `×` click adds
  friction to the dominant deletion path; removing the keyboard
  confirm risks losing the user's work to a stray key.
