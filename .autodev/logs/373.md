## Goal for this iteration

Unblock the only red gold case — `verifyLiveFullPipelineReused` —
which is failing for the **same reason iter 356-357 diagnosed**:
wire-format drift between the FE decoder (now 18-byte
`PdfSegment` header after iter 370's `lastPage` tri-state byte)
and a stale per-project sidecar Machine still running the
pre-iter-370 image (17-byte header). Browser console:
`[WsClient] decodeFrame failed: decodeFrame: pdf-segment payload
truncated`. The fixture's shape-tolerant decoder picked up the
segment fine (`bytes=56557 shipoutPage=1`) so the timeline looks
healthy until you cross-check the browser-diagnostics line.

Mass-destroy was tagged "now a known-good pattern" in
357_log#151; this is its second invocation. The reused project's
UUID is stable (`00000000-0000-4000-8000-000000000001`) — it
won't be reaped by `test_cleanup_old_playwright_projects` (only
test-spawned UUIDs are), so until manually destroyed its Machine
keeps coming back stopped → started with the same pinned image.

## Approach

1. **Identify staleness.** `flyctl machines list -a
   tex-center-sidecar --json` joined with the metadata
   `texcenter_project` field. The Machine for the reused project
   (`00000000-0000-4000-8000-000000000001`) is
   `d8d3e90a001ee8`, image digest tail `28f33e05132ac9bff19d4e94ffb2f4`,
   created `2026-05-16T12:09:34Z` — long before iter 370's
   deploy at `2026-05-16T21:08:25Z` (deploy-sidecar.yml run
   `25972987926`, success). All five non-app-pool Machines were
   created pre-iter-370; this iteration targets only the reused
   project's Machine because the other four are time-bounded
   Playwright transients that the existing cleanup gold case
   reaps. (The cleanup case ran green in this gold pass; those
   transients were under its retention threshold or its scan was
   the prior gold's responsibility.)
2. **Destroy the one stale Machine.** `flyctl machine destroy
   --force d8d3e90a001ee8 -a tex-center-sidecar`. The resolver
   (`apps/web/src/lib/server/upstreamResolver.ts`) creates a fresh
   Machine on next request, pinned to whatever `SIDECAR_IMAGE`
   currently is — that secret was bumped to the iter-372 deploy
   digest by `deploy-sidecar.yml` step "Pin SIDECAR_IMAGE on
   tex-center to just-deployed digest", so the next reused-spec
   gold run handshakes against an 18-byte-header sidecar.
3. **Clean up `machine_assignments`.** Per iter-357 hygiene,
   leaving a `machine_assignments` row pointing at a destroyed
   Machine ID means the next resolver call tries to start a
   non-existent Machine and synthesises a fresh assignment. The
   resolver already handles "Machine not found" by recreating
   (verified via spec
   `test_machine_count_under_threshold` continuing to pass after
   357 with the same code path), so this step is no-op so long
   as the resolver's "machine vanished" branch still works.
   I'll verify by reading the resolver code before deciding
   whether to manually delete the assignment row.
4. **FUTURE_IDEAS entry.** This is the second occurrence of the
   same wire-protocol-drift symptom. Add the "wire-version smoke
   gold test" idea the 357 notes flagged but didn't ticket, so
   if it recurs a third time there's an existing primitive to
   reach for instead of another targeted destroy.
5. **PLAN update.** Priority #4 is currently "defensive only" /
   "no proactive work needed until then." It now needs a
   sentence noting this is a recurrence with a different root
   cause (wire-shape drift, not the old back-to-back-compile-
   errors shape), the targeted-destroy remediation already
   applied, and the wire-version-smoke FUTURE_IDEAS pointer.
