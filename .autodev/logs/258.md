## Goal for this iteration

Wire a boot-time session sweeper (FUTURE_IDEAS bullet). The
storage primitive `deleteExpiredSessions(db, now)` landed iter 54
and has had no periodic caller — every expired row in
`tc_sessions` lingers until something deletes it. The two larger
PLAN options (M13.2 seed-widening; M11.1 file tree) are both
gated on shared R2 / multi-segment filenames respectively, which
are multi-iteration substrate changes. This iteration takes a
small, deterministic, end-to-end slice instead.

## Approach

Mirror the `bootMigrations` shape exactly: a pure
`runBootSessionSweep(env, deps, now)` driver gated by
`DATABASE_URL` + `SWEEP_SESSIONS_ON_BOOT=1`, with a default deps
adapter that opens a short-lived postgres-js connection. Wired
from `apps/web/src/server.ts` right after migrations and before
`boot()`. A sweep failure is logged and swallowed (the schema is
still usable; a future deploy retries).

Boot-time one-shot is the simplest scheduler that's strictly
better than the status quo. A periodic in-process timer is a
fine future upgrade; recorded in FUTURE_IDEAS.

## Notes for future iterations

- Deploy-time enable: set `SWEEP_SESSIONS_ON_BOOT=1` on
  `tex-center` (Fly secret or env). Not flipped this iteration —
  the change should land in a separate deploy with the binary
  already carrying the driver, so a sweep failure under load
  doesn't co-mingle with the rollout's first-traffic surface.
- The success branch on a real DB is exercised indirectly through
  `packages/db/test/users-sessions-pglite.test.mjs` (which tests
  `deleteExpiredSessions` end-to-end against PGlite). The boot
  driver only adds env-gating + connection lifecycle, which the
  stubbed-deps unit tests cover.
- Periodic in-process timer is the obvious next step
  (FUTURE_IDEAS); only worth doing once deploy cadence drops to
  the point where boot-only is insufficient.
