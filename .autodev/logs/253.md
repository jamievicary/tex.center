## Goal for this iteration

Fix the one gold failure from iter 252:
`verifyLiveFullPipelineReused.spec.ts` timed out waiting 30 s for
`.cm-content` to become visible. The sibling fresh-project spec
(`verifyLiveFullPipeline.spec.ts:96`) waits 120 s for the same
locator. The reused-project path can need to cold-start an
idle-stopped per-project Machine before iter 177's no-flash gate
releases `.cm-content`, so 30 s is too tight — the rest of the 300 s
test budget is wasted on what should be a slow-path-tolerant wait.

## Approach

Bump the `cmContent.waitFor` timeout from 30 000 ms to 120 000 ms to
match the sibling spec, and pin the reasoning in a comment so a
future iteration tightening editor-load latency doesn't silently
re-introduce the flake. No other change.

## Actions

`tests_gold/playwright/verifyLiveFullPipelineReused.spec.ts` lines
113–116 → 114–122. Timeout 30_000 → 120_000, plus a 6-line comment
explaining the reused vs. fresh-path difference and citing iter 252
as the failure evidence.

## Tests

- `bash tests_normal/run_tests.sh` → expected PASS (no
  tests_normal/ change).
- `tests_gold/` not run inside this iteration; harness runs it
  post-edit. Expected effect: `verifyLiveFullPipelineReused` either
  passes within 120 s (whichever Machine state Fly is in) or — if
  the failure is something other than a slow cold start —
  surfaces a clearer error than "30 000 ms waiting for .cm-content".

## Notes for future iterations

- Iter 252 was the first failure of this spec since at least iter
  239, so this is most likely a one-off cold-start window the
  shorter timeout couldn't absorb. If the failure persists after
  the timeout bump, the bug is elsewhere (Machine stuck stopped,
  upstreamResolver wedged, sidecar dead) and warrants a real
  investigation rather than another timeout nudge.
- Tightening editor-load latency in a future milestone should
  *not* be paired with shrinking this timeout: the 120 s ceiling
  is about Fly cold-start variance, not editor code performance.
