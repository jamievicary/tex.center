## Approach

Read the breakdown:

- `clickToWsOpenMs=95` — Fly start was hot (image cached on host),
  so the variant didn't see a multi-second image pull on this pass.
- `clickToFirstFrameMs=4556` — first WS frame from the sidecar
  arrives 4.5 s after the WS opened. That's ~5× the suspended
  path's ~785 ms first-frame, consistent with a fresh container
  boot (no mapped-memory shortcut) vs. resume.
- `cmContentReadyMs=4853` — 297 ms more for Yjs hydrate + CM
  mount after first frame, matching the suspended-path delta
  (~486 ms; lower here, likely sample variance).
- `keystrokeAckMs=7` — same shape as suspended (9 ms iter 362);
  keystroke ack budget at 1000 ms is real and stays.

Match the iter-366 ratio (suspended 1349 → 2500 = ~1.85×) for the
stopped sample: 4853 × 1.85 ≈ 8978 ≈ 9000 ms. The 1.85× factor
gives ~4 s absolute headroom on a path with intrinsically higher
variance (fresh container boot — image-pull state, host load,
builder warmth all affect the floor; the iter-366 sample had a hot
image so 4853 ms is likely a *fast* sample, not a typical one).
Single-sample tune; expect to revisit once 2-3 more passes give a
real distribution.
