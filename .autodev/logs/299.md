## Goal for this iteration

M17.b cross-fade blend math. The PLAN's next non-deferred slice
(active queue #2 post iter 298). Today's stacked-opacities path
exhibits a mid-fade background bleed-through: with the entering
canvas on top fading 0‚Üí1 and the leaving canvas below fading 1‚Üí0,
the visible pixel at progress `t` is `t¬∑NEW + (1‚ąít)¬≤¬∑OLD +
t¬∑(1‚ąít)¬∑BG`. For a flat-grey panel where `OLD == NEW == G ‚Č† BG`,
this dips toward `BG` (peak deviation at `t = 0.5`). The PLAN's
Fix A switches to a single-layer opacity: entering canvas stays
opaque, sits *under* the leaving canvas; leaving fades 1‚Üí0. That
produces `(1‚ąít)¬∑OLD + t¬∑NEW` ‚ÄĒ perfect linear interpolation, no
`BG` term, no flicker.

The pre-iter-298 deploy is still queued (iter 298's gold-suite
`[live] editor.spec.ts` failed for the same reason ‚ÄĒ the live site
ran iter 297 code at gold-suite time; the assertion uses the new
`data-testid="topbar-email"`). Not an M17.b concern ‚ÄĒ iter 298's
deploy will land before iter 299's gold suite runs.

## Approach

- **Strategy as a pure constant.** Factor the layering rule into
  `apps/web/src/lib/pdfCrossFade.ts`: a `CROSS_FADE_STRATEGY`
  object naming the four opacity values and the two z-indices, plus
  a `crossFadeAt(t, old, new, bg)` helper that computes the visible
  channel value at progress `t`. The DOM adapter in
  `PdfViewer.svelte` reads the constant; the unit test asserts the
  math properties end-to-end.
- **The two invariants worth pinning.**
  - *Flat-grey invariant:* `OLD == NEW ‚áí result == OLD` for all
    `t ‚ąą [0, 1]` and any `BG`. The user-visible failure mode is a
    same-content recompile flashing through `BG`; this kills it.
  - *Linear interpolation:* `OLD ‚Č† NEW ‚áí result == (1‚ąít)¬∑OLD +
    t¬∑NEW`, independent of `BG`. The slightly stronger property ‚ÄĒ
    `BG` is never sampled ‚ÄĒ which makes the cross-fade visually
    invariant to the pane's background.
  Both hold under the new strategy; both fail under the legacy
  one. The test exercises a `LEGACY_STRATEGY` regression guard so a
  future revert is obvious.
- **DOM application.** In `startCrossFade`: set entering's z-index
  to `enteringZIndex (0)` and opacity to `enteringOpacity (1)`;
  set leaving's z-index to `leavingZIndex (1)`, force reflow at
  `leavingInitialOpacity (1)`, then transition to
  `leavingTargetOpacity (0)`. Listen for `transitionend` on the
  *leaving* canvas (not entering ‚ÄĒ entering doesn't transition any
  more). On a snapshot/interrupt (`commitFadeImmediately`): remove
  the leaving canvas, snap entering to opacity 1 / z-index 0. The
  controller layer is unchanged ‚ÄĒ this is purely an adapter
  rewrite.
- **CSS unchanged.** `.pdf-page > canvas` already carries
  `transition: opacity var(--pdf-fade-ms, 180ms) ease`. That now
  drives only the leaving canvas; entering's opacity never
  changes, so the same rule is a no-op for it. The dead
  `pdf-canvas--leave` class addition is removed (no CSS targeted
  it ‚ÄĒ it was a state marker for a previous design that never
  shipped).
