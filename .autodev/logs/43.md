## Approach

- Single TOML file at repo root (Fly's default search location;
  `flyctl deploy` finds it without a `--config` flag).
- Spec straight from PLAN's M6.1 entry: `app = "tex-center"`,
  Frankfurt region, build context pointing at
  `apps/web/Dockerfile`, scale-to-zero http_service on internal
  port 3000.
- No `[checks]` block. The iter 42 notes flag that `apps/web`
  exposes no `/healthz` yet (only the sidecar does). Adding a
  check against a route that always 404s would gate every deploy
  on a never-healthy Machine ‚Äî strictly worse than no check, and
  trivially fixable in a later iteration once the route lands.
- Pick `shared-cpu-1x` with 512MB. The default Fly VM is 256MB,
  but Node + adapter-node + Yjs server workload will graze that
  on a cold session-bootstrap; 512MB has the same hourly price
  on `shared-cpu-1x` and removes a likely first-deploy OOM
  footgun.
- Static structural test rather than a `flyctl validate` shell-
  out ‚Äî `flyctl` isn't on the runner, and the value we're paying
  for is "drift between Dockerfile port and fly.toml port doesn't
  reach prod", not "fly.toml is parseable" (Python's `tomllib`
  gives us that for free).
