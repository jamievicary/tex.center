## Goal for this iteration

M13.2(b).5 R2 — eliminate `stopped` as a sidecar-reachable state.
Currently `createIdleHandler` falls back to `app.close() +
exit(0)` when `suspendSelf` is wired but throws (e.g. Fly 5xx,
bad token). That fallback parks the Machine in `stopped`, the
exact state the user reported as 20 s+ cold-load
(`260_answer.md`) and the state
`verifyLiveGt6LiveEditableStateStopped` pins RED.

Change the failure contract so a thrown `suspendSelf` logs and
re-arms the idle gate instead of exiting. Next idle window
retries. The `suspendSelf === null` branch (local dev / missing
creds) keeps the close-and-exit behaviour — that's the
documented local-dev path and exiting is correct there.

Net effect: in production, the only way for a Machine to land in
`stopped` is an external force (Fly host eviction, manual
`flyctl machine stop`, or the gold spec calling `/stop`). R1
(SSR seed widening, requires shared blob store) remains the open
piece for the externally-stopped path; R2 closes the sidecar's
own contribution.

## Actions

1. `apps/sidecar/src/index.ts` — `createIdleHandler`: when
   `suspendSelf` is set, both the success and the throw paths
   now keep the listener bound and call `ctx.rearm()`. The throw
   path logs but never reaches `app.close()` / `exit(0)`. The
   `suspendSelf === null` (local-dev) path is unchanged.
2. `apps/sidecar/test/idleSuspend.test.mjs` — replaced the
   prior "throw → close + exit" expectation with the new
   stay-alive contract (no close, no exit, rearm fires). Added a
   second-idle-window retry case to pin that `inFlight` is
   correctly cleared after a failed suspend so subsequent idle
   timers actually re-fire.
3. `tests_normal/cases/test_node_suites.py` — wired
   `idleSuspend.test.mjs` into the normal runner. It was
   previously not registered, so the lock had never executed in
   CI. Now it does.

## Notes for future iterations

- **Deploy required.** The R2 fix only matters live once a
  `flyctl deploy` ships the new sidecar image. Schedule
  alongside R1 work (one deploy covers both) or take a
  dedicated deploy iteration.
- **R1 architecture sketch.** Shared `BLOB_STORE` binding on the
  web app, sidecar writes persisted source keyed by
  `project_id`, `+page.server.ts` reads when a
  `machine_assignments` row exists. Requires shared blob bucket
  on Fly + a sidecar-writes-blob-on-edit signal. Until then,
  externally-stopped Machines still hit the 20 s+ cold-load path.
- **Failure-loop cost.** If `suspendSelf` is permanently broken
  (e.g. revoked token), the Machine now stays running with the
  idle handler re-firing every `SIDECAR_IDLE_TIMEOUT_MS` (10 min
  default). By design — the alternative (drop into `stopped`
  and force a 20 s+ cold-load on next access) is worse UX.
  Operationally surfaced as repeated "suspend failed, staying
  alive to retry next idle window" log lines.
