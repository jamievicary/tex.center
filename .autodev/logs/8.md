## Goal for this iteration

Open M3 with two coherent pieces: (a) a survey of
`vendor/supertex` to know what daemon support already exists and
what M3 actually has to build, (b) the M3 scaffolding seam — a
`Compiler` interface in the sidecar so future iterations can drop
in a real supertex-driven implementation without churning
`server.ts`. The default compile path stays the fixture PDF; no
visible behaviour change. Plan the M3 sub-milestones explicitly
so the next iteration's slice is obvious.

## Approach

Survey: read `vendor/supertex/GOAL.md`, `PLAN.md`, the CLI entry
in `vendor/supertex/src/supertex`, and the `--help` flag list.
Findings (full version in `.autodev/PLAN.md`): supertex already
has a watch mode (default invocation), `--once`, `--live-shipouts`
giving a per-shipout `page<TAB>offset` map, and `--output-directory`.
`--target-page=N` is declared but not implemented — that is the
sole hard upstream-PR dependency for the M3 endgame. Supertex
watches the source files itself via inotify, so the sidecar's
job is to mirror the Yjs document to disk and read the resulting
PDF + shipout map back; no custom IPC needed.

Scaffolding seam: add `apps/sidecar/src/compiler/{types,fixture}.ts`.
`Compiler` is a tiny interface — `compile({source, targetPage})
→ {ok, segments[]} | {ok:false, error}` and `close()`.
`FixtureCompiler` is the existing fixture-PDF behaviour moved out
of `server.ts`. `server.ts` calls the compiler from `runCompile`
and threads the maximum `viewingPage` of the connected viewers
through as `targetPage` (currently ignored by `FixtureCompiler`,
but the value flows). `buildServer` accepts an optional
`compilerFactory` for tests / future env-var-driven selection.
The `viewingPage` plumbing closes the iter-7 note about
forwarding to supertex's `target_page=N`: the sidecar half is now
done; the upstream half is M3.5.

## Actions

- `apps/sidecar/src/compiler/types.ts` — `PdfSegment`,
  `CompileRequest`, `CompileResult` (`ok: true | false`
  discriminated union), `Compiler` interface.
- `apps/sidecar/src/compiler/fixture.ts` — `FixtureCompiler`,
  caches the on-disk fixture; returns `{ok: false, error}` on
  read failure rather than throwing, so `server.ts`'s compile
  loop has a single shape to handle.
- `apps/sidecar/src/server.ts` — refactored: `ProjectState` holds
  a `Compiler`; `runCompile` calls `compiler.compile({source,
  targetPage})`, broadcasts each segment, then a final
  `compile-status idle` (or `error`); `onClose` awaits
  `compiler.close()`. New `compilerFactory` option on
  `SidecarOptions`. Imports use `.js` (NodeNext-required)
  rather than `.ts` (typecheck rejects the latter without
  `allowImportingTsExtensions`).
- `apps/sidecar/test/fixtureCompiler.test.mjs` — covers the
  happy path (segment shape, %PDF magic, byte-cache identity
  across calls) and the missing-file failure path.
- `tests_normal/cases/test_node_suites.py` — new
  `test_sidecar_fixture_compiler` case.
- `.autodev/PLAN.md` — new "Survey of `vendor/supertex` (iter 8)"
  block + an explicit M3.0–M3.5 sub-milestone breakdown. M3.0
  marked done. M3.5 records the upstream PR dependency.

## Problems encountered

- First attempt used `./compiler/types.ts` import — typecheck
  rejected with TS5097 because `apps/sidecar/tsconfig.json`
  inherits `moduleResolution: NodeNext` (overridden in the
  sidecar) and we don't set `allowImportingTsExtensions`.
  Switched to `.js` extensions to match `index.ts`'s existing
  pattern. tsx still resolves the `.ts` source at runtime via
  the standard NodeNext suffix-stripping behaviour.

## Notes for future iterations

- M3.1 next: build a `ProjectWorkspace` that mirrors the
  `Y.Text` to a per-project scratch dir on every text change
  (debounced with the same 100ms beat as the compile loop, or
  collapsed into one timer). This remains under
  `FixtureCompiler` for the iteration that lands it; M3.2
  flips the compiler.
- The `targetPage` parameter on `Compiler.compile` is wired
  through but ignored by `FixtureCompiler`. Avoids a churn
  iteration when the supertex compiler arrives.
- `Compiler.close()` is awaited in `onClose`. The fixture
  implementation just nulls its cache; the supertex impl will
  need to reap the watch process and assert the absence of
  leftover children.
