## Goal for this iteration

Re-land the small focused half of iter 198 that the wallclock
revert removed: forward `supertex --daemon` stderr lines and parsed
protocol events to the sidecar's `process.stderr` with stable
prefixes, so the next live gold run's Fly logs capture which gate
(path 2 / path 3 / earlier) fires when GT-3 / GT-5 RED. The local-
probe half of iter 198 is recorded in `.autodev/logs/198.md`
(probes captured local-only paths 1+2+3 succeed against the
iter-724 binary); the wallclock burn came from doing both halves
plus building supertex in one iteration. This iteration ships
only the production-code instrumentation.

## Approach

`apps/sidecar/src/compiler/supertexDaemon.ts`:

1. In the stderr handler, in addition to appending to `stderrBuf`,
   buffer + line-split and write each complete line to
   `process.stderr` with `[supertex-daemon stderr] ` prefix.
2. In `enqueueEvent`, also write each `DaemonEvent` to
   `process.stderr` with `[supertex-daemon event] ` prefix, via a
   small `describeEvent` helper.

Both are pure side-effects on stderr — no behaviour change for the
unit suite. Existing tests should remain green; the daemon test
asserts on returned segments and protocol violations, not on
stderr text.

## Actions

- Edited `apps/sidecar/src/compiler/supertexDaemon.ts`:
  - New `stderrLineBuf` field + `forwardStderrLines(chunk)` helper:
    line-splits the daemon's stderr stream and writes each complete
    line to `process.stderr` with `[supertex-daemon stderr] ` prefix
    (partial trailing lines held back until the newline arrives).
  - `enqueueEvent` now writes `[supertex-daemon event] <describe>`
    to `process.stderr` before queueing/dispatching the event.
  - New `describeEvent` helper formats each `DaemonEvent` variant
    (`shipout n=N`, `rollback k=K`, `error reason=…`, `round-done`,
    `violation raw=…` truncated to 200 chars).
- `stderrBuf` is unchanged in role (still feeds the synthetic-
  violation `child exited` and ready-marker timeout messages); the
  new line buffer is parallel state purely for forwarding.

## Notes for future iterations

- Iter 200 is a `N%10==0` refactor cron. The next live gold run
  (whenever the harness next runs gold) is the data source for
  picking M7.4.x's actual fix; collect logs first, then on iter
  201+ tighten `collectRound` or open an upstream PR per
  PLAN.md M7.4.x. Refactor iter should not consume that data.
- The forwarding is unconditional (no env flag). For a
  long-running production sidecar this is roughly one short stderr
  line per protocol event — far below any sane log budget for a
  per-project Machine, and matches the iter-187 `?debug=1`
  philosophy that supertex-protocol detail belongs on the wire
  whenever a compile is in flight.
