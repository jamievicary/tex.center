## Goal for this iteration

`N=10` — refactor iteration per `engineer.md` §3 maintenance rule.
Survey of `src/` and tests after iter 9 turned up one clear
duplication worth fixing: the magic string `"main.tex"` was hardcoded
in four places as the *Yjs `Y.Text` identifier* — `server.ts:87`,
`wsClient.ts:52`, `editor/+page.svelte:19`, and the sidecar server
test. Both ends of the WebSocket must agree on this string for sync
to work; a typo on either side would silently break the doc round-trip
with no type-system signal. M4 (multi-file) will introduce more
filename-as-doc-key strings, so promoting this one to a named export
of the `@tex-center/protocol` package now establishes the convention.

Other surveyed candidates were rejected as not worth the churn:

- `runCompile` has 4 `compile-status` broadcasts; an `emitStatus`
  helper would save little and obscure the call sites.
- `Buffer→Uint8Array` and `ArrayBuffer→Uint8Array` shapes in
  `server.ts` and `wsClient.ts` look similar but the input types
  differ across the Node/browser boundary, so they aren't real
  duplication.
- The procedural `*.test.mjs` files are small and idiomatic; a test
  framework would be over-engineering for the current scale.

The on-disk filename `"main.tex"` (`workspace.ts:39` and tests) is
deliberately *not* unified — it represents the sidecar's local
filesystem layout, a separate concern from the wire-protocol doc
identifier. They happen to share a value today; conflating them
would couple two contracts that are free to diverge.

## Approach

- Export `MAIN_DOC_NAME = "main.tex"` from `packages/protocol`, with
  a comment naming it as the agreed Yjs key and pointing at M4 as
  the place this generalises.
- Replace the literal at the four call sites listed above.
- Add a one-line assertion in the codec test that `MAIN_DOC_NAME`
  has the expected value, so a future rename is reflected in the
  test record.
