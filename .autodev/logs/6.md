## Goal for this iteration

M2 **browser half**: wire `apps/web` to the sidecar over WebSocket so
the CodeMirror buffer is the canonical Yjs state, edits propagate to
the server, the static `/fixture.pdf` URL is replaced by an
incrementally-patched `Uint8Array` fed from server `pdf-segment`
frames, and `/ws/*` is proxied to `127.0.0.1:3001` in dev. After
this iteration the M1 static fixture-PDF reference becomes
deletable; we delete it as part of the cutover.

## Approach

- Add `yjs` and `y-codemirror.next` to `apps/web`. The latter brings
  `y-protocols` (used only client-side; wire format unchanged).
- New `apps/web/src/lib/pdfBuffer.ts` — pure logic that maintains a
  growable `Uint8Array` and applies `{totalLength, offset, bytes}`
  segments. Unit-tested under tsx (no DOM). This is the only piece
  that's worth a unit test; the rest is glue.
- New `apps/web/src/lib/wsClient.ts` — opens the project's
  WebSocket, decodes frames via `@tex-center/protocol`, applies
  `doc-update` to a `Y.Doc`, accumulates `pdf-segment` frames in a
  `PdfBuffer`, and exposes a small reactive surface (status,
  pdfBytes, error). Outbound: it forwards local `Y.Doc` updates not
  originating from itself, plus `view` control messages on
  `setViewingPage(n)`.
- `Editor.svelte` switches from a `string`-bound prop to a `Y.Text`
  prop, using `y-codemirror.next`'s `yCollab` extension. Syntax
  mode for LaTeX deferred (M3 cleanup); basicSetup keeps the
  experience plausible.
- `PdfViewer.svelte` accepts `Uint8Array | string`, re-runs
  `getDocument` whenever the buffer reference changes, cancels
  in-flight renders cleanly.
- `+page.svelte` constructs one `WsClient`, threads its `Y.Text`
  into `Editor` and its `pdfBytes` into `PdfViewer`, sends
  `view` on a stub page-1 default (real page tracking deferred).
- `vite.config.ts` adds `server.proxy['/ws']` to
  `ws://127.0.0.1:3001` with `ws: true`.
- Delete `apps/web/static/fixture.pdf` and the structural-check
  rule that pinned it (if any).

- `apps/web/src/lib/pdfBuffer.ts` — `PdfBuffer` class. Grows the

## Problems encountered

- First svelte-check pass flagged `private readonly onChange?:
  (snap: …) => void` under `exactOptionalPropertyTypes: true` —
  the optional-with-`?` is rejected for callback fields.
  Switched to `((…) => void) | undefined`. Same shape, accepted.

## Notes for future iterations

- `Editor.svelte` is keyed by `text` so swapping the placeholder
  Y.Text for the WS-owned one in `onMount` reliably remounts the
  editor. A future iteration can drop the placeholder by gating
  the editor on `client !== null`. Low priority.
- `PdfViewer.svelte` currently re-renders by tearing down all
  canvases and rendering all pages from scratch on every update.
  That's fine for a 1-page fixture but the M3 "patch only the
  changed pages" optimisation is the whole point of the
  incremental wire format — keep page canvases keyed by page
  number and only re-render pages whose bytes changed. M3 work.
- Real "viewing page N" tracking: `IntersectionObserver` on each
  page canvas, debounce, send `view` when the most-visible page
  changes. Today the page is hardcoded to 1.
- `WsClient` reconnect: not implemented. On `close`/`error` the
  client just sets state and stops. Adding exponential-backoff
  reconnect is a good late-M2/early-M3 task.
- The sidecar dev port is hardcoded (3001) in `vite.config.ts`.
  Once the control plane spawns per-project Machines, the proxy
  target becomes "look up assignment in Postgres" — i.e. M7
  rewires this entirely.
