## Goal for this iteration

Step 1 of the iter-224 next-iteration plan
(`.autodev/PLAN.md::M9.editor-ux.regress.gt7`): build a fast local
repro inside `tests_gold/lib/test/` that spawns `supertex --daemon`
directly and drives the user's literal `\newpage XX` stdin sequence
from `.autodev/discussion/220_question.md`. Goal: trigger the
upstream SIGABRT (code 134) outside the live Playwright loop so the
upstream bug can be debugged with iteration latency measured in
seconds rather than minutes.

## Approach

New gold case: `tests_gold/lib/test/supertexColdNewpageCrash.test.mjs`
+ Python wrapper `tests_gold/cases/test_supertex_cold_newpage_crash.py`,
following the `supertexOversizeTarget` pattern (drives a real
`SupertexDaemonCompiler` against a temp workdir, self-skips when the
supertex binary or system `lualatex` is missing).

Two probes in a single mjs script:

1. **Steady ramp** — baseline compile of the 4-line `MAIN_DOC_HELLO_WORLD`
   seed (identical to what `persistence.ts:255` inserts for a brand-new
   live project), then 20 rounds at 500 ms inter-round delay. Each
   round appends one more `\newpage NN` line just before
   `\end{document}` and drives `recompile,T` with `T = page_count`.
2. **Coalesced big-paste** — baseline, then ONE round with +15
   `\newpage` lines (modelling the sidecar coalescer collapsing
   every edit that arrives while the slow first lualatex round is
   in flight), then 5 single-newpage follow-up rounds at 500 ms.

Both probes assert the daemon does not emit any of
`protocol violation` / `child exited` / `stdin not writable` (the
exact substrings GT-8 keys on).

## Findings — load-bearing

**Both probes PASS locally on every run.** The upstream `supertex
--daemon` binary survives the user's literal stdin sequence,
including the coalesced big-paste variant that more faithfully
models what the cold-start window actually presents to the daemon.

This contradicts the iter-224 "next iteration plan" assumption that
the fast local repro would reliably trigger code 134. **The bug is
not purely stdin-pattern-driven.** Something the live environment
adds beyond the in-process daemon's stdin protocol is required.

Observed stderr per round (probe 1, steady ramp): every round after
the first emits `supertex: WARN no usable rollback target for
.../main.tex@NN (checkpoint=6 resumed_pid=-1)` and ships zero
segments — the daemon falls back to a no-op because the only
available checkpoint (page 6 from baseline) is too far back to
resume against the growing source. Only the final liveness round
(which renames the visible text) triggers a real rollback + shipout
burst. Probe 2 has a nicer profile: the big-paste round rolls back
to k=0 and ships pages 1..16; subsequent rounds roll back to
k=15, 16 successfully; only the last 3 rounds fall into the
"no usable rollback target" no-op shape. **Neither shape crashes.**

## Hypotheses for what the live environment adds

In rough order of how easy each is to probe next:

1. **R2 chunk hydration.** The live sidecar restores `chunks/`
   from R2 on hydrate (see `apps/sidecar/src/persistence.ts:274..`).
   A brand-new project has nothing in R2 (correct hello-world
   seed only), but the *Machine* still ran the restore code path,
   which may have left the chunks dir in a different state than
   the local `mkdir -p` does. Cheap test: pre-create chunks dir
   with the artefacts a real restore would leave (touch nothing,
   touch random subdirs, etc.) and rerun.
2. **Concurrent Yjs hydration writes.** The sidecar can call
   `writeMain(source)` while the Yjs doc is still hydrating from
   R2 (chunk-by-chunk). On cold-start the first `runCompile` may
   race with a doc-update applying a chunk to the source, so the
   bytes the daemon sees mid-`recompile,T` aren't stable. Worth
   instrumenting `runCompile` with a "did the source change
   during compile?" assertion.
3. **CPU/memory pressure on shared-cpu-1x Fly Machine.** Cold
   lualatex runs on a 1GB Fly Machine; locally we have 8+ GB and
   a real CPU. An assert in supertex's incremental engine that
   fires only under timing-dependent memory pressure is harder
   to rule out without instrumenting the daemon.
4. **Subtle `\newpage NN` content difference.** The user said
   `\newpage XX` literally; live Playwright types `\\newpage 00`
   etc. (matching what we do here). Unlikely to be the
   distinguishing factor since the live spec DOES reproduce.

## Notes for future iterations

- The new local probe is **green**, by design — it pins the
  "stdin-only sequence doesn't crash" invariant. If a future
  refactor of the upstream daemon breaks that invariant, this
  test catches it. But it is NOT the GT-8 pin; GT-8 (live
  Playwright) remains the only red pin for the user-visible bug.
- Next-step priority is hypothesis 1 (R2 chunk hydration). It's
  the cheapest delta to try and matches the only structural
  difference between local and live for a brand-new project.
- Once a local repro DOES trigger code 134, debug supertex with
  `gdb --args supertex --daemon DIR main.tex` to get the abort
  backtrace — `vendor/supertex` builds with frame pointers per
  `vendor/supertex/build/Makefile`. Until then, upstream debugging
  is bottlenecked on reproducibility.
