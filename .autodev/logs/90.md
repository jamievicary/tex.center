## Problems encountered

### Default-goal of `vendor/supertex/Makefile` is `build/baseline_snapshot`, not `all`

`make -C vendor/supertex` (no explicit target) compiles only
`build/baseline_snapshot`. Reason: the Makefile declares
`$(UTIL_TOOL_BINS): tools/util.h src/supertex_records.h
tools/generated_exts.h` (a prerequisite-only rule, no recipe)
*before* `all: ...`, so the first listed target â€” i.e.
`build/baseline_snapshot`, the first element of `UTIL_TOOLS` â€”
becomes make's default goal. `make all` builds everything as
intended.

This bug was present before the bump too. The sidecar
Dockerfile (line 61) currently runs `make -C vendor/supertex
-j` and would therefore build only `baseline_snapshot` in the
runtime image. Not visible during iter-89's deploy because:

- the sidecar starts without spawning the compiler;
- `SUPERTEX_BIN=/opt/supertex/src/supertex` already pointed at
  a non-existent path (the Python entry that used to live
  there is gone post-bump, replaced by `build/supertex`); and
- the Fly trial cap stops machines after 5 min, before any
  real compile request hits.

Did **not** fix the Dockerfile in this iteration â€” that's
M7.5.2 territory (`SupertexDaemonCompiler` wires the new
daemon ELF and the bin-path fix is naturally bundled). Recorded
the carry-over note inside the M7.5.0 PLAN entry so future
iterations don't have to rediscover.
