## Goal for this iteration

M6.3.1 (live cutover to `tex.center` DNS + cert) is out-of-tree —
it needs live tokens against Fly + Cloudflare and can't run inside
autodev. Smaller alternatives in the plan are: multi-file project
slice on the sidecar (large), `awaitPdfStable` wiring (waits on a
streaming compile path that doesn't exist yet), `S3BlobStore`
(waits on docker-compose). That leaves the M5 tail items in
FUTURE_IDEAS as the highest-leverage small slice. Pick JWKS
clock-skew tolerance — iter 36 explicitly flagged it as a known
gap, and it's a defensive hardening of the auth path that's now
live in production-ready form.

## Approach

- `verifyGoogleIdToken` is currently a const that captures a
  module-level lazy JWKS resolver — fine for prod, but untestable
  without spinning up a real JWKS HTTP server.
- Replace it with `makeVerifyGoogleIdToken(config)` builder that
  accepts (a) `clockToleranceSeconds` (default 60s, matching
  google-auth-library), and (b) an injectable `keyInput` so unit
  tests can sign locally with `jose.SignJWT` + a generated
  keypair and pass the public key directly.
- Keep a `verifyGoogleIdToken` constant — `makeVerifyGoogleIdToken()`
  with defaults — so `+server.ts` doesn't need to know.
- Test against real `jwtVerify`: sign a token, verify it; sign one
  with `exp` 30s in the past, verify it succeeds with the 60s
  default tolerance and fails when tolerance is set to 0 or to
  10s with a 300s-expired token.

## Problems encountered

- First pass of the test asserted that `iat` slightly in the future
  would be rejected without `clockTolerance` and accepted with it.
  jose v5 doesn't actually check `iat` at all by default — `iat`
  only matters when `maxTokenAge` is set. The real defensive gain
  from tolerance is on `exp` (and `nbf`, which Google ID tokens
  don't carry). Rewrote the cases to drive on `exp` past the
  configured tolerance and updated the doc comment in
  `googleTokens.ts` to match reality.
- Initial typecheck failed: the union `KeyLike | Uint8Array |
  JWTVerifyGetKey` matches neither of `jwtVerify`'s two overloads
  alone. Branching on `typeof keyInput === "function"` narrows it
  to a single overload per arm.

## Notes for future iterations

- The new test pattern (local `generateKeyPair` + `SignJWT` +
  injected `keyInput`) is reusable if more ID-token validation
  logic lands (e.g. requiring `email_verified`, parsing
  `hd` for hosted-domain checks). The `keyInput` plumbing keeps
  unit tests offline.
- `verifyGoogleIdToken` const still creates the `createRemoteJWKSet`
  resolver at module-import time. It doesn't fetch eagerly — the
  fetch only happens on first `jwtVerify` call — but it does
  allocate the resolver. Acceptable; lazy-creating it would just
  add a one-time guard for no real benefit.
