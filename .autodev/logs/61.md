## Goal for this iteration

Add a `create-file` protocol verb so users can create new project
files at runtime. Today `knownFiles` is frozen at hydration time —
the FileTree shows whatever was already in the blob store, but
there is no way to add a new file without restarting the sidecar.
This is the smallest coherent slice from the "file-tree CRUD" item
in `FUTURE_IDEAS.md` and unblocks the natural "create a new .tex,
edit it, see it persist" flow.

PLAN's "next ordinary iteration" (M6.3.1) is out-of-tree.

## Approach

- Protocol: extend `ControlMessage` with
  `{ type: "create-file"; name: string }`.
- Persistence: new `addFile(name)` method. Validates the name as a
  single-segment blob key (no `/`, no `.`/`..`, matches the
  existing `BlobStore` segment regex), rejects duplicates, and —
  when `canPersist` is true — PUTs an empty blob immediately so
  the file survives a session restart even if the user never
  edits it. Returns `{ added: boolean; reason?: string }`.
- Server: when a client sends `create-file`, call
  `addFile(name)`; on success, broadcast a refreshed `file-list`
  to every viewer of the project. Reject silently (log only) on
  invalid/duplicate names — the client UI also validates, this is
  defence-in-depth.
- `WsClient.createFile(name)` sends the control message.
- `FileTree`: add a small "+ new file" input row at the bottom of
  the list with a submit handler that invokes a `onCreateFile`
  prop. The editor page wires `client.createFile`.
- Tests: extend `serverBlobs.test.mjs` with a third run that
  creates a file via the protocol, asserts the broadcast, asserts
  the blob exists, and asserts a fresh sidecar rehydrates it.

## Notes for future iterations

- Rename / delete are the remaining file-tree verbs. Delete must
  also `blobStore.delete` the per-file key and clear the entry in
  `persistedByName`. Rename = move blob + reassign `Y.Text`
  content under the new name (Y.Text doesn't support renaming;
  the simplest path is `create new + copy contents + delete old`,
  carefully sequenced to avoid losing in-flight edits).
- Creating a file with non-empty content is not currently
  expressible — the empty-blob create + Yjs edit flow works but
  a future "upload file" affordance would need bytes-on-create.
