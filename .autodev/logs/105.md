## Goal for this iteration

M7.1.3 prep — add migration-on-boot to the control plane so that
when M7.1.3.1 lands `DATABASE_URL` (via `flyctl postgres attach`),
the app applies pending DDL before serving traffic. Without this,
attaching a freshly-provisioned Postgres would either crash auth
queries against missing tables or require an out-of-band
`pnpm db:migrate` step via `flyctl proxy`. Split M7.1.3 into
0/1/2 so this prep is captured separately from the live
provisioning. M7.1.3.1 was the originally-planned slice but
provisioning postgres has cost and irreversibility characteristics
worth landing alongside the boot-time wiring rather than ahead of
it — doing the helper first lets the next iteration attach +
verify in one shot.

## Approach

- `apps/web/src/lib/server/bootMigrations.ts`:
  - `runBootMigrations(env, deps)` returns a discriminated
    `BootMigrationsStatus`: `skipped-no-database-url`,
    `skipped-disabled`, `no-migrations-found`, `applied`.
  - Pure logic; the default deps adapter wraps `loadMigrations`
    and `createDb + applyMigrations(postgresJsDriver(...))` from
    `@tex-center/db`, using a dedicated `max:1` handle that is
    closed before boot proceeds.
  - Gating: `DATABASE_URL` present **and** `RUN_MIGRATIONS_ON_BOOT
    === "1"` (strict equality so future docs can't accidentally
    pass `RUN_MIGRATIONS_ON_BOOT=true` and silently enable in
    prod via a different flag value).
  - `MIGRATIONS_DIR` overrides the default `/app/migrations`.
- `apps/web/src/server.ts`: top-level await on
  `runBootMigrations(process.env)` before `boot()`; log the
  human-readable status. esbuild's `node20` target supports
  top-level await in ESM bundles, verified by running
  `apps/web/scripts/build-server-entry.mjs` locally
  (`build/server.js 320.1kb`).
- `apps/web/Dockerfile`: copy
  `packages/db/src/migrations/` from the builder stage to
  `/app/migrations` so the runtime image contains the SQL.
- Test `apps/web/test/bootMigrations.test.mjs`: stub
  `BootMigrationsDeps`; asserts every status branch, that
  non-"1" flag values stay disabled, that
  `MIGRATIONS_DIR` overrides the default, that `applyWith`
  errors propagate, and that `describeBootMigrationsStatus`
  returns sensible strings for each variant.
- Wired into `tests_normal/cases/test_node_suites.py` as
  `test_web_boot_migrations`.

## Notes for future iterations

- M7.1.3.1 should provision Fly Postgres (unmanaged,
  single-node `shared-cpu-1x` in `fra`, smallest volume), run
  `flyctl postgres attach tex-center-db --app tex-center` to
  set `DATABASE_URL` automatically, then `flyctl secrets set
  -a tex-center RUN_MIGRATIONS_ON_BOOT=1 FLY_API_TOKEN=…
  SIDECAR_APP_NAME=tex-center-sidecar SIDECAR_IMAGE=registry.
  fly.io/tex-center-sidecar:<digest>`. Deploy will re-roll the
  image with the migrations copied in and `runBootMigrations`
  will apply `0001_initial.sql` on first boot. Verify with
  `curl https://tex.center/healthz` reporting `db.state: ok`.
- `FLY_API_TOKEN` for the control plane must be a deploy token
  scoped to the org (not the personal token in `creds/fly.token`,
  which carries too much authority for a long-lived secret).
  `flyctl tokens create deploy --app tex-center-sidecar -x
  720h` is the right shape.
- If MIGRATIONS_DIR or `RUN_MIGRATIONS_ON_BOOT` ever need a
  per-Machine override (e.g. a multi-Machine future where only
  one runs migrations), the boot helper already factors into a
  pure function — wrap the env check with a leader-election
  primitive at that point.
