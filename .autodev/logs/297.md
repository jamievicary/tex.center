## Goal for this iteration

M19.1 + M19.2 (slider half). PLAN's next pick (per iter 296
"Notes for future iterations"). Add a runes-driven editor
settings store, a cog button in the editor topbar that opens
a popover, a single fade-duration slider (0–3s, 0.05s step,
default 0.18s — the values frozen in `293_answer.md`), and
wire the chosen value to `PdfViewer.svelte` through a
`--pdf-fade-ms` CSS custom property. Persist via
`localStorage["editor-settings"]` as a single JSON object so
future settings can extend the same key. Defer the
displayName→email topbar swap (M19.2 remainder) and the
keyboard/a11y polish (M19.3) — both small and separable, and
the slider is the shape that establishes the store contract.

## Approach

- Pure-helpers + Svelte-side reactive state, not a `$state`-
  carrying `.svelte.ts` module. Keeps the store SSR-safe and
  unit-testable without DOM globals: `parseSettings(raw)` /
  `serializeSettings(s)` / `clampFadeMs(n)` plus the storage
  key and slider bounds. The Svelte component owns the
  `$state<EditorSettings>` and the `localStorage` I/O. This
  pattern matches the existing `editorPanelLayout.ts` /
  `pageTracker.ts` shape (logic + state separated).
- CSS-var driven application. `.shell` carries
  `--pdf-fade-ms: {n}ms`. `PdfViewer.svelte` transitions read
  `var(--pdf-fade-ms, 180ms)` so a value change is purely a
  style recalc — no controller wiring, no per-render re-arm.
  The previously-unused `const FADE_MS = 180` is removed
  (it was a comment-anchor only; never load-bearing).
- Outside-click dismisses the popover via a window-level
  `pointerdown` listener that's installed in `onMount` and
  removed in the cleanup return. The cog itself is excluded
  so clicking it toggles rather than dismiss-then-toggle.

## Actions

- `apps/web/src/lib/settingsStore.ts` (NEW). Pure helpers:
  `EditorSettings { fadeMs }`, `DEFAULT_SETTINGS`,
  `clampFadeMs`, `parseSettings`, `serializeSettings`,
  `SETTINGS_STORAGE_KEY = "editor-settings"`, plus
  `FADE_MS_MIN/MAX/STEP/DEFAULT`. `parseSettings` tolerates
  null / malformed JSON / non-object payloads / wrong-typed
  fields → defaults; clamps to range on read; drops unknown
  extra keys. `serializeSettings` clamps on write too so a
  buggy caller can't poison storage.

## Notes for future iterations

- **M19.2 email-swap.** Trivial one-line change in
  `+page.svelte` (`displayName ?? email` → `email`), but the
  local editor.spec.ts seeded user probably has a `displayName`
  set by the fixture — confirm before changing and update the
  spec if it asserts the visible text. Check
  `tests_gold/playwright/fixtures/authedPage.ts` first.
- **M19.3 a11y.** Pattern: trap `keydown` Escape on the
  popover, return focus to the cog on close, move focus into
  the slider on open. Don't add a full focus-trap library —
  one cog + one slider doesn't justify it. Roving-focus
  becomes worth it when the popover grows past 3 controls.
- **Settings extension point.** Adding a setting later: extend
  `EditorSettings`, give it a default in `DEFAULT_SETTINGS`,
  add a clamp in `parseSettings` mirroring `clampFadeMs`, and
  add another `.settings-row` to the popover. The store's
  `unknown-extra-keys-dropped` rule means partial rollback is
  safe.
- **Cross-fade flatness pin (M17.b).** With the slider in
  place, that future spec can drive a known fade duration via
  the public setting (set `localStorage["editor-settings"]`
  before navigation) rather than monkey-patching CSS, which
  was the cleaner test design discussed in `293_answer.md`.
