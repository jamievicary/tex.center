## Goal for this iteration

Fix the iter-206 GT-A regression. Root cause: `verifyLiveGt1NoFlashLoad`
(and the four sibling GT specs) call `test.setTimeout(60_000)` /
`90_000` / `120_000` inside the test body, but the worker-scoped
`sharedLiveProject` fixture's warm-up has a 180s budget for the
cold Fly Machine boot + sidecar start + first lualatex round.
Fixture setup runs *before* the test body, so an inner
`test.setTimeout()` cannot raise the budget that already covered
setup — the default 30s applies. When warm-up exceeds 30s (which
it routinely will on the FIRST spec to trigger the fixture,
especially during a deploy as in iter 206), GT-A fails with
exactly the observed signature: "Test timeout of 30000ms exceeded
while setting up 'liveProject'", trace pointing at
`sharedLiveProject.ts:168 page.waitForTimeout(250)`. Subsequent
specs pass because Playwright re-runs fixture setup and the
second attempt finds a partially-warm sidecar.

The inner `test.setTimeout()` calls aren't just ineffective —
they're actively harmful: per Playwright semantics, calling
`setTimeout(N)` mid-test resets the *remaining* budget and throws
if elapsed > N. So if fixture setup took, say, 90s, an inner
`setTimeout(60_000)` would throw the moment the body starts.

## Approach

Move the timeout into the config, where it covers fixture setup
on the first test, and remove the per-test overrides that
shadowed it. The config-level `timeout` is global; local-target
specs are unaffected because their internal `waitFor` timeouts
still bound them.

## Actions

- `tests_gold/playwright.config.ts`: added top-level
  `timeout: 240_000` with a comment explaining the warm-up budget
  rationale.
- Removed `test.setTimeout(...)` calls inside the test body of all
  five `verifyLiveGt[1-5]_*.spec.ts` files (60s / 60s / 60s / 120s /
  90s). The config-level 240s budget supersedes them.
