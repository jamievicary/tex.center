## Approach

`startFlyProxy({app, localPort, remotePort, command?, extraArgs?,
readyTimeoutMs?, probeIntervalMs?, closeTimeoutMs?})` spawns
`flyctl proxy LOCAL:REMOTE -a APP`, polls `127.0.0.1:LOCAL`
with `net.connect` until a connection succeeds, and returns
`{localPort, close()}`. Failure modes (per PLAN's "distinct-
failure-mode health check"):

- happy → resolves with the handle.
- child exits before port opens → reject with code/signal +
  captured stderr in the message.
- port never opens within `readyTimeoutMs` → reject with the
  port + timeout in the message.
- spawn ENOENT (`/nonexistent/flyctl-binary`) → reject with the
  underlying error message.

`close()` is idempotent and races `exitPromise` against
`spawnErrorPromise` so that ENOENT failures don't hang. SIGTERM
first; SIGKILL after `closeTimeoutMs`.

`command` is parameterised (default `"flyctl"`) so tests can
substitute a stand-in binary — the test writes a tiny bash
wrapper that execs `node SCRIPT` with one of three behaviours
(`listen`, `exit-fast`, `sleep-no-listen`).

## Problems encountered

1. First-pass `sleep-no-listen` stub used
   `process.stdin.resume()` to keep the process alive, but with
   `stdio: ["ignore", ...]` stdin is `/dev/null` and emits `end`
   immediately, so the process exited cleanly. Fixed by switching
   the stub to `setInterval(() => {}, 1000)`. Caught it because
   the timeout test reported "child exited before ready" instead
   of "did not open within 800ms" — i.e. the helper's
   distinct-failure-mode wiring was correct, the *stub* was wrong.

2. Spawn-ENOENT test hung in `close()`. Root cause: my close()
   awaited `exitPromise` unconditionally, but with ENOENT node
   fires `error` and does not reliably emit `exit`, so
   `exitPromise` never resolves. Fixed by tracking `spawnError`
   alongside `exited` and racing both in close() / waitForReady.
   Better than papering over with a timeout, because the helper
   now has one consistent "child is gone" signal.
