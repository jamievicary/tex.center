## Diagnosis

Wrote a local probe (`/tmp/probe_multipage*.mjs`) driving
`SupertexDaemonCompiler` directly against `vendor/supertex` on a
5-page `\newpage`-chain source. Result:

- `targetPage=5` → 5 shipouts, 5 page refs in the assembled PDF
  (126944 bytes).
- `targetPage=1` → 1 shipout, 1 page ref (120664 bytes).
- `targetPage=0` (= `recompile,end`) → 5 shipouts, 5 page refs.

Root cause in `apps/sidecar/src/server.ts`: `runCompile` passed
`targetPage: maxViewingPage(p)`. `ProjectClient.viewingPage`
defaults to 1; no viewer can set `viewingPage > 1` until a page-2
canvas exists; no page-2 canvas exists until page 2 is shipped.
Chicken-and-egg: every compile shipped page 1 only. The user-
visible symptom was "only page 1 of my multipage doc renders".

The 241_answer.md diagnosis tree (a/b/c) didn't include this
fourth hypothesis — a sidecar-side per-target clamp orthogonal to
the wire format, viewer logic, or CSS overflow. The new hypothesis
is the load-bearing one; the original (a) is ruled out (segment
totalLength matches `bytes.length` in both probe arms), (b) is
ruled out (viewer iterates `pdf.numPages` faithfully — but
`numPages` was being kept at 1 by the sidecar), (c) is ruled out
(`.preview` has `overflow: auto` and `.host` is `flex-direction:
column`).

## Fix

`apps/sidecar/src/server.ts` `runCompile`: pass `targetPage: 0`
unconditionally. Comment explains the M15 chicken-and-egg. The
per-viewer `kickForView` path is still wired but now redundant —
`highestEmittedShipoutPage` always grows to the document's full
page count after the next compile. The optimisation hatch is left
in place for re-introduction if long-document perf becomes
load-bearing.

`maxViewingPage` retained because `kickForView`'s API takes a
`maxViewingPage` argument; refactoring it out is a follow-up if
ever the optimisation hatch is formally removed.

## Notes for future iterations

- **Live failure shape was misleading.** The iter-268 gold output
  reported "no post-edit pdf-segment arrived" (i.e.
  `pdfSegmentFrames.length` stayed at 1). My local probe showed a
  `targetPage=1` compile DOES ship a (page-1-only) segment, so the
  segment count should have grown to 2 on live. The most likely
  explanation: the `\newpage` keystroke flurry produced multiple
  intermediate compiles, some with `[error]` (mid-typing source
  like `\newp`) and some no-op (page 1 byte-identical to prior
  compile). Sidecar's iter-188 no-op fallback returns
  `segments=[]` for a no-shipout round; no segment ships. The
  final stable compile then short-circuits on byte-identical
  page-1 output and again ships nothing. After the targetPage=0
  switch, the same edit produces a 5-page PDF whose byte sequence
  differs strongly from the seed, so a segment will always ship.
- **`maxViewingPage` is now near-dead.** If `kickForView` is
  retired, drop `maxViewingPage`, `ProjectClient.viewingPage`,
  the `view` control-frame type, and the corresponding sidecar
  tests in one sweep — a small refactor iteration.
- **Performance.** On a 5-page doc the cost difference between
  `recompile,1` and `recompile,end` was negligible (a single
  cold lualatex run dominates). For very long documents (50+
  pages) the per-viewer clamp would matter; reintroduce as a
  user-driven scroll-down trigger if user reports surface it.
- **R1 still owed** for the M13.2(b).5 cold-stopped UX (20s+
  blank editor). Blocked on shared blob-store binding.
