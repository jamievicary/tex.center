## Goal for this iteration

N=240 — refactor iteration (`N % 10 == 0`). Survey for the
single highest-leverage cleanup. The clearest target is the
top entry in `.autodev/FUTURE_IDEAS.md`: remove the iter-228 GT-5
diagnostic seam (`CompileSuccess.noopReason` field +
"compile no-op (no pdf-segment shipped)" warn-log branch in
`apps/sidecar/src/server.ts`). GT-5 has now been GREEN across
iters 231→239 — far past the "2–3 consecutive full gold passes"
threshold the FUTURE_IDEAS entry named as the gate. The seam was
explicit dark-diagnostic code, useful only as a probe; with the
upstream fix landed iters 759–764 of `vendor/supertex` and locked
both at the daemon-CLI level (`supertexWarmDocBodyEditNoop.test.mjs`)
and live (GT-5), the seam can come out cleanly.

While in tree, also reaped two orphan
`tests_gold/playwright/*.spec.ts.tmp.<pid>.<ts>` files left over
from a previous crashed `Edit` operation — playwright's default
`testMatch` doesn't pick them up, so they were harmless, but
they're pure clutter.

## Approach

1. Drop `noopReason?: string` from `CompileSuccess` in
   `apps/sidecar/src/compiler/types.ts`.
2. Drop the field population in `supertexDaemon.ts` — the no-op
   round still returns `{ ok: true, segments: [] }`, just without
   the free-form reason.
3. Collapse the `if (segments.length === 0 && result.noopReason)`
   warn-log branch in `server.ts` back to a single `app.log.info`
   call; the `segments` field already conveys whether a delta was
   shipped, and the no-op shape is no longer a load-bearing
   diagnostic.
4. Rewrite `apps/sidecar/test/serverCompileNoop.test.mjs` to
   continue pinning the wire shape (no `pdf-segment`,
   `compile-status:idle` still emitted, no
   `compile-status:error`) but drop the warn-log/noopReason
   propagation assertions. Removing the test wholesale would
   weaken the iter-188 regression lock — the wire-shape half of
   that pin is still valuable.
5. Drop the corresponding noopReason assertions in
   `apps/sidecar/test/supertexDaemonCompiler.test.mjs` no-op
   case; `r.ok && r.segments.length === 0 && r.shipoutPage ===
   undefined` is the underlying invariant and is preserved.
6. Update `.autodev/PLAN.md` and `.autodev/FUTURE_IDEAS.md` to
   reflect the removal.

  iteration. The shared-pool machines (`fly_process_group=app`)
  are intentional and should NOT be destroyed by any cleanup
  step. Possible follow-up shapes: (a) audit
  `tests_gold/playwright/*.spec.ts` for `bootstrapLiveProject` /
  fresh-project flows that miss `cleanupLiveProjectMachine` in
  `afterEach`; (b) add a final-teardown sweeper that destroys any
  metadata-less machine older than the suite start time; (c) tag
  every machine created by the control plane with a
  `texcenter_project=<id>` metadata key so leaks are
  programmatically distinguishable from the shared pool. (c) is
  the cleanest because it makes the guardrail test capable of
  triaging itself.
- The iter-228 GT-5 diagnostic seam is gone; if a future
  regression of the same class appears, the place to add the next
  probe is `apps/sidecar/src/server.ts`'s `compile ok` log line —
  add structured fields for whatever was missing. Avoid resurrecting
  the `noopReason` shape unless an upstream API actually carries
  a reason; the previous one was a free-form sidecar-side string
  that duplicated information already implicit in `segments.length`.
